# Sprint v0.2 Scope — CLI + LLM

> **本文是 Sprint v0.2 的范围定义**。回答"v0.2 做什么、不做什么、怎么算做完"。
>
> 详细设计见 [v0.2-design.md](./v0.2-design.md)。本文件不写实现细节，只写边界。

---

## 1. Sprint 目标

**用一句话**：在终端里跟"会说 persona 的话的 LLM"聊至少 5 轮，全程在 CI 中可重放（Replay 模式），需要时切 Live 模式能调真 API。

**为什么是 v0.2 而不是 v0.3 先做存储**：

- 状态机、记忆、工具在 v0.2 都是噪音
- 验证"LLM 接到提示词能正确按 persona 说话"是后续所有 sprint 的基础
- 没有 LLM 接通，后面所有"自动化测试"都是空话

---

## 2. 范围（In）

### 2.1 必须交付

| 交付物 | 验证方式 |
|---|---|
| 终端能跑 REPL，键入消息 → 看到 LLM 回复 | 手动 `pnpm dev` |
| `.env` 加载、校验、缺 key 立即报错 | L1 测试 |
| `prompts/SOUL.md` / `AGENTS.md` / `USER.md` 读入并拼成 system prompt | L1 测试 |
| MiniMax-M3 provider 接通（Anthropic-SDK 协议） | 手动 Live 跑通 |
| Replay provider 跑通，CI 不打外网 | L1 + L3 测试 |
| `thinking` 内容块能被识别（不打到 stdout） | L1 测试 |
| L1 测试 ≥ 5，L3 测试 ≥ 1 | `pnpm test` 通过 |
| `pnpm typecheck` / `pnpm lint` / `pnpm build` 全绿 | CI 模拟 |

### 2.2 Demo 场景（手动 Live 验证用）

```bash
$ pnpm dev
English Oral Teacher Agent — CLI mode
Persona: 13 岁中国中学生陪练
Type "exit" to quit.

> Hi
[Teacher]: Hi there! How are you doing today? ...
> I am fine, thanks.
[Teacher]: Great to hear! What did you do today? ...
> ...
```

至少 5 轮，LLM 回复用英文 + 引导式提问（与 SOUL.md 里写的人设一致）。

## 4. LLM Provider 决策（已拍板）

> 拍板时间：2026-06-03，源自 ARCHITECTURE §9 + 此次 sprint 启动会。

| 项 | 决定 | 理由 |
|---|---|---|
| v0.2 provider | **仅 MiniMax M3** | 用户已选定 MiniMax M3；多 provider 复杂度不必要 |
| API 协议 | **Anthropic Messages API 兼容** | MiniMax 官方示范代码基于 `anthropic` SDK |
| SDK | **`@anthropic-ai/sdk`**（TS 版） | Anthropic 官方 TS SDK；MiniMax 协议兼容 |
| 模型 ID | **`MiniMax-M3`** | MiniMax 提供 |
| Base URL | **`https://api.minimaxi.com/anthropic`** | MiniMax 提供，通过 `ANTHROPIC_BASE_URL` 注入 |
| API key 变量 | **`MINIMAX_API_KEY`** | 独立变量名，未来加 provider 不混淆 |
| 内容块 | 响应可能含 `thinking` + `text` 两种 block | v0.2 需识别但不打印 thinking |
| 切 Live 开关 | `RUN_LIVE_LLM=1`（默认 Replay） | CI 不打外网；Live 模式留给手动验证和发版门 |

**默认 `.env` 配置**（v0.2 落地后）：

```env
LLM_PROVIDER=minimax
MINIMAX_API_KEY=<你的 key>
ANTHROPIC_BASE_URL=https://api.minimaxi.com/anthropic
LLM_MODEL_MAIN=MiniMax-M3
LLM_MODEL_SUMMARIZER=MiniMax-M3
```

## 5. 测试策略

> 完整测试架构见 [DEVELOPMENT_PLAN §4](../../DEVELOPMENT_PLAN.md)。v0.2 只需要 L1 + L3。

### 5.1 L1（纯单元，4 个文件 ≥ 5 个 case）

| 文件 | 测试 |
|---|---|
| `tests/config/env.test.ts` | (1) 缺 `MINIMAX_API_KEY` 抛错；(2) 缺 `LLM_MODEL_MAIN` 用默认值；(3) 非法 `LLM_TEMPERATURE` 抛错；(4) `.env` 不存在用 process.env |
| `tests/prompts/loader.test.ts` | (1) 读 SOUL.md 返回非空字符串 |
| `tests/llm/anthropic.test.ts` | (1) user/assistant/system 消息格式正确转换 |
| `tests/llm/testing.test.ts` | (1) ReplayProvider 从 fixture 读响应 |

### 5.2 L3（集成，1 个）

| 文件 | 测试 |
|---|---|
| `tests/cli/repl.test.ts` | spawn `tsx src/cli.ts`，注入 stdin，断言 stdout 含 greeting 关键词 |

L3 用 ReplayProvider 跑，**不打外网**。

### 5.3 L4 烟雾测试（手动 / 发版前，不进 CI）

```bash
RUN_LIVE_LLM=1 pnpm dev
# 输入 "hi"，验证回复非空、引用 persona 风格
# 用脚本录制 fixture：RUN_LIVE_LLM=1 pnpm test:record
```

## 6. 风险与回退

| 风险 | 影响 | 应对 |
|---|---|---|
| MiniMax API 不稳定 / 限流 | Live 验证失败 | Live 仅手动跑；CI 永远 Replay |
| `thinking` block 格式变化 | AnthropicProvider 输出乱 | L1 测试守住格式转换；Live 失败时降级只取 `text` |
| MiniMax-M3 模型名变更 | provider 调不通 | `.env` 可改，代码不动；文档提醒用户同步 |
| `tsx` 跨平台 spawn 问题（Windows） | L3 测试在 Windows CI 挂 | 用 `node --import tsx` 而非 `tsx` 命令行；如仍挂，标记为已知跳过 |

6. Replay 模式 CI 跑通
