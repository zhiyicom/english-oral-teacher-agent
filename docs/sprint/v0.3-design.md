## 2. 模块设计

### 2.1 `src/storage/db.ts` —— 连接 + Migration Runner

**职责**：

- 创建 / 打开 SQLite 文件（`APP_DATA_DIR/oral-teacher.db`）
- 跑待应用的 migration（事务式，幂等）
- 跑 `PRAGMA integrity_check`，返回结果
- 暴露一个 `getDb()` 工厂给 DAO 用

**接口**：

```ts
import type Database from 'better-sqlite3';

export interface DbOptions {
  dataDir: string;                      // APP_DATA_DIR
  dbFilename?: string;                  // 默认 'oral-teacher.db'
}

export interface DbHandle {
  raw: Database.Database;               // 暴露给 DAO 用
  close(): void;
  integrityCheck(): 'ok' | string;      // 'ok' 或错误消息
}

export function openDb(opts: DbOptions): DbHandle;
export function applyMigrations(handle: DbHandle): void;   // 同步跑所有未跑 migration
```

**实现要点**：

```ts
import Database from 'better-sqlite3';
import { existsSync, mkdirSync, readdirSync, readFileSync } from 'node:fs';
import { join, resolve } from 'node:path';
import { fileURLToPath } from 'node:url';

const __dirname = fileURLToPath(new URL('.', import.meta.url));
const MIGRATIONS_DIR = resolve(__dirname, 'migrations');

export function openDb(opts: DbOptions): DbHandle {
  if (!existsSync(opts.dataDir)) {
    mkdirSync(opts.dataDir, { recursive: true });
  }
  const dbPath = join(opts.dataDir, opts.dbFilename ?? 'oral-teacher.db');
  const raw = new Database(dbPath);
  raw.pragma('journal_mode = WAL');
  raw.pragma('foreign_keys = ON');
  return {
    raw,
    close: () => raw.close(),
    integrityCheck: () => {
      const row = raw.prepare('PRAGMA integrity_check').get() as { integrity_check: string };
      return row.integrity_check;       // 'ok' 表示通过
    },
  };
}

export function applyMigrations(handle: DbHandle): void {
  const { raw } = handle;
  raw.exec(`
    CREATE TABLE IF NOT EXISTS schema_migrations (
      version TEXT PRIMARY KEY,
      applied_at TEXT NOT NULL
    );
  `);

  const applied = new Set(
    (raw.prepare('SELECT version FROM schema_migrations').all() as { version: string }[])
      .map(r => r.version)
  );

  const files = readdirSync(MIGRATIONS_DIR)
    .filter(f => f.endsWith('.sql'))
    .sort();                              // '001_init.sql' < '002_*.sql'

  for (const file of files) {
    const version = file.replace(/\.sql$/, '');
    if (applied.has(version)) continue;
    const sql = readFileSync(join(MIGRATIONS_DIR, file), 'utf8');
    const tx = raw.transaction(() => {
      raw.exec(sql);
      raw.prepare('INSERT INTO schema_migrations (version, applied_at) VALUES (?, ?)')
        .run(version, new Date().toISOString());
    });
    tx();
  }
}
```

**关键决策**：

- **同步 API** —— better-sqlite3 本身就是同步的；CLI 单线程用同步最简单、避免 `await` 满天飞
- **WAL 模式** —— 减少 `database is locked` 概率，未来 UI 多连接也兼容
- **`PRAGMA foreign_keys = ON`** —— 每个连接都要设（SQLite 默认关闭 FK）；不开的话 `ON DELETE CASCADE` 形同虚设
- **Migrations 目录硬编码** —— `__dirname/migrations` 在 `dist/` 下可定位；测试可通过环境变量覆盖（见 §6.1）
- **不写 `BEGIN` / `COMMIT` 显式** —— 用 better-sqlite3 的 `db.transaction()` 包装，自动管理

### 2.2 `src/storage/migrations/001_init.sql`

```sql
-- 001_init.sql: 初始 schema — sessions + messages
-- 完整字段预留；v0.3 只读写核心列（id / started_at / ended_at / duration_min / role / content / ts / session_id）
-- 后续 sprint 通过 002_*.sql / 003_*.sql 增量补全（含错例 / 词汇 / 作业 / topic_stats 表）

CREATE TABLE sessions (
  id              TEXT PRIMARY KEY,
  started_at      TEXT NOT NULL,
  ended_at        TEXT,
  duration_min    INTEGER,
  phase_history   TEXT,             -- JSON (v0.4)
  summary         TEXT,             -- v0.5
  keywords        TEXT,             -- v0.5 (JSON 字符串)
  topics_used     TEXT,             -- v0.6 (JSON 字符串)
  homework        TEXT,             -- v0.7
  transcript_path TEXT              -- v0.6
);

CREATE TABLE messages (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id  TEXT NOT NULL,
  role        TEXT NOT NULL,        -- 'user' | 'assistant' | 'system'
  content     TEXT NOT NULL,
  ts          TEXT NOT NULL,
  voice_used  INTEGER DEFAULT 0,    -- v0.9
  FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);

CREATE INDEX idx_messages_session ON messages(session_id, ts);
CREATE INDEX idx_sessions_started ON sessions(started_at);
```

**为什么不像 PRD 那样一次写 6 张表**：

- v0.3 sprint 目标只是"消息能查回"，加 4 张表是噪音
- 增量 migration 风险小、review 容易、字段不混淆
- 真要 ALL-in-one 写完 6 张表，未来补字段得改 001_init.sql + 兼容老库——不优雅

### 2.3 `src/storage/sessions.ts` —— Sessions DAO

**职责**：session 行的 CRUD 子集（v0.3 用到的部分）。

**接口**：

```ts
import type { DbHandle } from './db.js';

export interface Session {
  id: string;                          // UUID v4
  started_at: string;                  // ISO 8601
  ended_at: string | null;
  duration_min: number | null;
  // 其他字段暂不暴露给 v0.3 调用方
}

export interface CreateSessionInput {
  id?: string;                         // 可选：测试时传固定 id；CLI 不传，自动 generate
  startedAt?: string;                  // 可选：测试 mock 时间；默认 new Date().toISOString()
}

export interface SessionsDao {
  create(input?: CreateSessionInput): Session;
  get(id: string): Session | null;
  list(): Session[];
  markEnded(id: string, opts: { endedAt?: string; durationMin?: number }): Session;
}

export function createSessionsDao(handle: DbHandle): SessionsDao;
```

**实现要点**：

```ts
import { randomUUID } from 'node:crypto';

const SELECT_COLS = 'id, started_at, ended_at, duration_min';

export function createSessionsDao(handle: DbHandle): SessionsDao {
  const { raw } = handle;
  const insert = raw.prepare(`
    INSERT INTO sessions (id, started_at) VALUES (?, ?)
  `);
  const selectOne = raw.prepare(`SELECT ${SELECT_COLS} FROM sessions WHERE id = ?`);
  const selectAll = raw.prepare(`SELECT ${SELECT_COLS} FROM sessions ORDER BY started_at DESC`);
  const updateEnd = raw.prepare(`
    UPDATE sessions SET ended_at = ?, duration_min = ? WHERE id = ?
  `);

  return {
    create(input = {}) {
      const id = input.id ?? randomUUID();
      const startedAt = input.startedAt ?? new Date().toISOString();
      insert.run(id, startedAt);
      return { id, started_at: startedAt, ended_at: null, duration_min: null };
    },
    get(id) {
      return (selectOne.get(id) as Session) ?? null;
    },
    list() {
      return selectAll.all() as Session[];
    },
    markEnded(id, opts = {}) {
      const endedAt = opts.endedAt ?? new Date().toISOString();
      const durationMin = opts.durationMin ?? computeDurationMin(id);
      updateEnd.run(endedAt, durationMin, id);
      return { id, started_at: '...', ended_at: endedAt, duration_min: durationMin };
    },
  };
}
```

**`computeDurationMin` 内部实现**：取 `started_at` 算 `(endedAt - startedAt) / 60_000` 向下取整。

### 2.4 `src/storage/messages.ts` —— Messages DAO

**职责**：单条 message 的 append + 顺序读。

**接口**：

```ts
export interface MessageRow {
  id: number;
  session_id: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  ts: string;
  voice_used: 0 | 1;
}

export interface AppendMessageInput {
  sessionId: string;
  role: 'user' | 'assistant' | 'system';
  content: string;
  ts?: string;                         // 默认 now；测试可 mock
  voiceUsed?: 0 | 1;                   // 默认 0
}

export interface MessagesDao {
  append(input: AppendMessageInput): MessageRow;
  getBySession(sessionId: string): MessageRow[];
  countBySession(sessionId: string): number;
}

export function createMessagesDao(handle: DbHandle): MessagesDao;
```

**实现要点**：

```ts
export function createMessagesDao(handle: DbHandle): MessagesDao {
  const { raw } = handle;
  const insert = raw.prepare(`
    INSERT INTO messages (session_id, role, content, ts, voice_used)
    VALUES (?, ?, ?, ?, ?)
  `);
  const select = raw.prepare(`
    SELECT * FROM messages WHERE session_id = ? ORDER BY ts ASC, id ASC
  `);
  const count = raw.prepare(`
    SELECT COUNT(*) AS n FROM messages WHERE session_id = ?
  `);

  return {
    append(input) {
      const ts = input.ts ?? new Date().toISOString();
      const voiceUsed = input.voiceUsed ?? 0;
      const info = insert.run(input.sessionId, input.role, input.content, ts, voiceUsed);
      return {
        id: info.lastInsertRowid as number,
        session_id: input.sessionId,
        role: input.role,
        content: input.content,
        ts,
        voice_used: voiceUsed,
      };
    },
    getBySession(sessionId) {
      return select.all(sessionId) as MessageRow[];
    },
    countBySession(sessionId) {
      return (count.get(sessionId) as { n: number }).n;
    },
  };
}
```

**关键决策**：

- `ORDER BY ts ASC, id ASC` —— `ts` 可能同毫秒（CLI 极快连续 append），加 `id` 兜底保稳定顺序
- `voice_used` v0.3 总是 0，v0.9 才会用到——但字段先建好
- 不暴露 `update` / `delete` —— v0.3 append-only，错了就重新建 session

### 2.6 `src/cli.ts` —— 改造

在 v0.2 基础上：

```ts
import { createMessagesDao, createSessionsDao, applyMigrations, openDb } from './storage/index.js';

export async function main(): Promise<void> {
  const env = loadEnv();

  // v0.3 新增：打开 DB + 跑 migration + 拿 DAO
  const db = openDb({ dataDir: env.APP_DATA_DIR });
  applyMigrations(db);
  const sessions = createSessionsDao(db);
  const messages = createMessagesDao(db);

  // v0.3 新增：开 session 行
  const session = sessions.create();
  // ... 现有 system prompt / client / readline 初始化

  // v0.2 既有循环
  try {
    while (true) {
      // ... 读 input
      history.push({ role: 'user', content: input });
      messages.append({ sessionId: session.id, role: 'user', content: input });  // 新增

      process.stdout.write('[Teacher]: ');
      let response = '';
      for await (const chunk of client.chatStream({ system: systemString, messages: history })) {
        if (chunk.type === 'text') {
          process.stdout.write(chunk.delta);
          response += chunk.delta;
        }
      }
      process.stdout.write('\n\n');
      history.push({ role: 'assistant', content: response });
      messages.append({ sessionId: session.id, role: 'assistant', content: response });  // 新增
    }
  } finally {
    // v0.3 新增：mark ended + 关 db
    sessions.markEnded(session.id);
    rl.close();
    db.close();
  }
}
```

**关键决策**：

- **session 行在 `main` 开始就建**——`started_at` 准确；万一启动后立刻崩溃，未结束 session 留作 v0.4 的 "resume" 提示用
- **每 turn 写 2 条**（user + assistant）——append-only，崩溃时已写的部分可查
- **`finally` 块保证 markEnded** —— 正常 `exit` / Ctrl+D / 异常都能跑
- **history 内存数组 + DB 双写**——v0.2 内存数组仍保留（喂给 LLM 用），DB 是持久化镜像
- **v0.3 不实现 `/history` 命令**——L3 测试直接调 DAO 查询（比 CLI 内置一个命令简单）；CLI 内置 `/history` 是 v0.8 UI 范围

```ts
import { spawn } from 'node:child_process';
import { mkdtempSync, rmSync } from 'node:fs';
import { tmpdir } from 'node:os';

import { afterEach, beforeEach, describe, expect, it } from 'vitest';

describe('CLI persistence (5 turns → DB readable)', () => {

  });
  afterEach(() => {
    rmSync(dataDir, { recursive: true, force: true });
  });

  it('persists 5 turns of conversation to SQLite', async () => {
    const cli = resolve('src/cli.ts');
    const child = spawn(process.execPath, ['--import', 'tsx', cli], {
      env: {
        ...process.env,
        RUN_LIVE_LLM: '0',
        MINIMAX_API_KEY: 'sk-test',
        APP_DATA_DIR: dataDir,
      },
      stdio: ['pipe', 'pipe', 'pipe'],
    });

    let stdout = '';
    child.stdout.on('data', d => { stdout += d.toString(); });
    child.stderr.on('data', d => { stdout += d.toString(); });   // 调试用

    // 5 轮对话（间隔 700ms 给流式响应时间）
    const inputs = ['hi', 'i am fine thanks', 'i played minecraft', 'i built castles', 'creepers blew them up'];
    for (const input of inputs) {
      await new Promise(r => setTimeout(r, 700));
      child.stdin.write(`${input}\n`);

    expect(msgs).toHaveLength(10);                              // 5 user + 5 assistant
    expect(msgs.filter(m => m.role === 'user')).toHaveLength(5);

    const userContents = msgs.filter(m => m.role === 'user').map(m => m.content);
    expect(userContents).toEqual(inputs);

    // assistant 至少有一条非空（证明 Replay 响应真的落了）
    const assistantTexts = msgs.filter(m => m.role === 'assistant').map(m => m.content);
    expect(assistantTexts.every(t => t.length > 0)).toBe(true);

    db.close();
  }, 15000);
});
```

> **为什么不开第二个子进程验证重启**：subprocess 启动 / signal / port 冲突在 Windows CI 上很 noisy。`openDb({ dataDir })` 在父进程开新物理连接 = "用全新 DB 句柄读已落盘文件"——本质等同"重启进程后查"。

## 5. 错误处理

| 场景 | 行为 |
|---|---|
| `APP_DATA_DIR` 不可写 | `openDb` 抛 `SQLITE_CANTOPEN`，CLI 退出 1 + 打印路径 |
| Migration SQL 语法错 | `applyMigrations` 抛错，CLI 退出 1 + 打印出错文件名 + 行号 |
| 重复 migration（v0.4 写 002_*.sql 后再跑 v0.3 DB） | `schema_migrations` 跟踪版本，001 跳过、002 跑 |
| ON DELETE CASCADE 失效 | 跑前 `PRAGMA foreign_keys = ON` 强制开启；v0.3 不删 session，但开 FK 保险 |
| DB 文件被外部锁（同时两个 CLI 跑） | `database is locked` 错误传播，CLI 提示"另一个实例在跑？"并退出 1 |
| 进程崩溃中途 | `db.transaction()` 包住的写是 atomic；未 markEnded 的 session 留作 v0.4 resume 提示 |
| L1 测试间 DB 状态污染 | `beforeEach` 重新建临时目录，`afterEach` `rm -rf` |
| L3 测试在 Windows 卡死 | 设置 `testTimeout: 15000`；如还 flaky 改"等待 stdout 含 `[Teacher]: ` 后再写下一行" |
| `node:fs` 不存在（rare） | 测试中 `import { existsSync } from 'node:fs'`，无 fallback |

## 8. 风险与对策（设计层）

| 风险 | 对策 |
|---|---|
| `@types/better-sqlite3` 版本与 better-sqlite3 不匹配 | 装时挑 stable v11.x 对应的 types（v0.3 实装时 npm 验证） |
| `better-sqlite3` 编译失败（Windows 缺 MSVC） | 安装步骤写到 README "Prerequisites"；如真不行 v0.3 退回 `node:sqlite`（Node 22+ 内置，sync API 略弱但够用） |
| Migration 写错导致 schema 进退两难 | `schema_migrations` 跟踪；v0.3 是初始 migration，错了改文件重跑即可（库未发布） |
| L3 测试 Windows 平台 spawn 不稳 | 用 `node --import tsx`（v0.2 验证过）；testTimeout 15s；输入间隔 700ms（v0.2 600ms OK，加点 buffer） |
| 测试间 DB 状态污染 | 每个测试用 `os.tmpdir()` 子目录 + `mkdtempSync`（UUID 命名） + `afterEach` 删 |
| ON DELETE CASCADE 默默失效 | 每个连接 `PRAGMA foreign_keys = ON`；L1 测试可加一条 delete cascade 验证（v0.3 不写，留作 v0.4） |
| 同 session 同 ts 的两条 message 顺序不稳定 | `ORDER BY ts ASC, id ASC`，id 自增兜底 |
| CLI 启动崩溃（DB 打开失败）但 session 已建 | v0.3 接受；v0.4 resume 提示会用——目前的"未结束 session"列表是 v0.4 的入口 |
| `tsx` watch + better-sqlite3 文件锁 | `pnpm dev` 改 watch 模式会 hot-reload；不重启时 DB 句柄复用 OK |
| Live 模式 + `tsx watch` 调真 API | `tsx watch` 在文件变更时重启；用户改 `.env` 会触发重启，符合预期 |
