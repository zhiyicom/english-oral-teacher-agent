## 1. Sprint 目标

**用一句话**：CLI 跑一次对话 → 退出 → 重启 → 历史消息能查回来（落 SQLite）。

**为什么是 v0.3 而不是 v0.4 先做状态机**：

- v0.2 已经在内存里维护 `history: Message[]`，但**关掉就丢**——零持久化
- 后续所有 sprint（v0.4 状态机 / v0.5 记忆 / v0.6 摘要）都要查历史，没 SQLite 一切空谈
- v0.3 把"消息能落盘"这条最短链路打通，证明 schema 形态对、性能 OK、能容灾

**Demo 形态**：

```bash
$ pnpm dev
> hi
[Teacher]: Hi there! ...
> exit
$ ls data/
oral-teacher.db
$ pnpm dev
> /history
[DB query, 回到上次的对话]
```

（`/history` 命令是 v0.3 L3 测试的需求接口；UI 上的"主界面列表"是 v0.8）

### 2.3 Schema 范围（v0.3 创建的表）

**只创建** `sessions` 和 `messages` 两张表 + `schema_migrations` 元表。完整 PRD schema 中的 `mistakes` / `vocabulary` / `homework` / `topic_stats` 留到对应 sprint（v0.5 / v0.6 / v0.7）通过**增量 migration**（002_*.sql）补齐。

> **决策依据**：一次性 001_init.sql 写全 6 张表 vs 增量 migrations。v0.3 选增量——单次 migration 职责单一、风险小、好 review；后续 sprint 改 schema 不需要动 001_init.sql。

`001_init.sql` 大致内容（详细字段见 [PRD §3.2](../../PRD.md)）：

```sql
CREATE TABLE sessions (
  id              TEXT PRIMARY KEY,
  started_at      TEXT NOT NULL,
  ended_at        TEXT,
  duration_min    INTEGER,
  phase_history   TEXT,             -- JSON (v0.4 填)
  summary         TEXT,             -- v0.5 填
  keywords        TEXT,             -- v0.5 填
  topics_used     TEXT,             -- v0.6 填
  homework        TEXT,             -- v0.7 填
  transcript_path TEXT              -- v0.6 填
);

CREATE TABLE messages (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id  TEXT NOT NULL,
  role        TEXT NOT NULL,        -- 'user' | 'assistant' | 'system'
  content     TEXT NOT NULL,
  ts          TEXT NOT NULL,
  voice_used  INTEGER DEFAULT 0,    -- v0.9 填
  FOREIGN KEY (session_id) REFERENCES sessions(id) ON DELETE CASCADE
);

CREATE INDEX idx_messages_session ON messages(session_id, ts);
CREATE INDEX idx_sessions_started ON sessions(started_at);
```

> v0.3 不写这些可选列（`summary` / `keywords` / `phase_history` / `topics_used` / `transcript_path` / `voice_used`），但**字段先建出来**——以后补数据不动 schema。代码层 DAO 暂时只暴露 v0.3 实际用到的字段。

## 4. LLM Provider 决策（沿用 v0.2）

无变化。v0.3 复用 v0.2 的 MiniMax-M3 / Replay 双模式：

| 项 | 决定 |
|---|---|
| Provider | 仅 MiniMax M3 |
| API 协议 | Anthropic Messages API 兼容（`https://api.minimaxi.com/anthropic`） |
| Replay / Live 开关 | `RUN_LIVE_LLM=1`（默认 Replay） |

## 5. 测试策略

> 完整测试架构见 [DEVELOPMENT_PLAN §4](../../DEVELOPMENT_PLAN.md)。

### 5.1 L1（纯单元，≥ 5 个 case）

| 文件 | 测试 |
|---|---|
| `tests/storage/db.test.ts` | (1) 临时 DB 创建后 `PRAGMA integrity_check` 返回 `ok`；(2) 跑 migration 后 `sessions` 和 `messages` 表存在；(3) 重复跑 migration 不会重复创建（`schema_migrations` 跟踪） |
| `tests/storage/sessions.test.ts` | (1) create → get 返回同一条；(2) list 按 started_at DESC 排序；(3) markEnded 后 `ended_at` 与 `duration_min` 已填 |
| `tests/storage/messages.test.ts` | (1) append + getBySession 返回顺序正确（按 ts 升序）；(2) 不同 session 的 message 互不串 |

> 共 8 个 case（3 + 3 + 2），超出最低 5。

### 5.2 L2（单元 + I/O，≥ 2 个 case）

| 文件 | 测试 |
|---|---|
| `tests/storage/db.test.ts`（同一文件） | (1) 真实文件 DB 在 `os.tmpdir()` 子目录创建；(2) 删文件后下次 init 重建（验证 cleanup） |
| `tests/storage/integrity.test.ts` | (1) 创建 DB → 写几条数据 → 关掉 → 重开 → 数据还在（验证持久化）；(2) 模拟损坏文件（截断 header）→ 启动报错（而不是默默建空库） |

### 5.3 L3（集成，1 个）

`tests/cli/persistence.test.ts`：spawn CLI 跑 5 轮对话 + exit，**不重启进程**，用 DAO 直接查 DB：

1. 临时 DB 路径（覆盖 `APP_DATA_DIR`）
2. spawn CLI，注入 5 轮 Replay-friendly 输入（`hi` / `fine` / `played minecraft` / `castles` / `creepers`）
3. `exit` 后
4. 用 `sessions.list()` 断言：1 条 session，`ended_at IS NOT NULL`，`duration_min >= 0`
5. 用 `messages.getBySession(sessionId)` 断言：10 条 message（5 user + 5 assistant），内容顺序匹配输入
6. **关键断言**：content 字段就是当时 LLM 的回复（用 Replay fixture 内容校验）—— 证明 SQLite 落盘 + 重启可查**这条路通**

> **为什么不重启子进程**：v0.3 L3 的本质是"验证数据落盘"——在子进程退出后，从父进程**开新 DB 连接**读数据，物理上等同于"重启进程再开 DB 读"。这避免了 spawn / signal / port 冲突等 CI 噪音，又守住了"持久化生效"这条断言。

### 5.4 L4 烟雾测试（手动 / 发版前，不进 CI）

```bash
RUN_LIVE_LLM=1 pnpm dev
# 跑 5 轮对话 + exit
# 切回 Replay：pnpm dev
# 输入 /history，断言能看到上一轮内容
# sqlite3 data/oral-teacher.db "SELECT role, substr(content, 1, 30) FROM messages;"
```

## 6. 风险与回退

| 风险 | 影响 | 应对 |
|---|---|---|
| `better-sqlite3` 编译失败（缺 C++ 工具链） | v0.3 整 sprint 走不通 | 安装 Python + MSVC build tools（Windows）；优先跑 `pnpm install` 验证；不行就退回 `node:sqlite`（Node 22+ 内置，sync API 略弱） |
| `better-sqlite3` 跨平台 spawn 出问题 | L3 测试 Windows 挂 | L3 不 spawn 新进程（见 §5.3 解释），用同进程 DAO 查询 |
| Replay fixture 不匹配 | L3 测试 "沉默通过" 或抛 fixture not found | v0.3 复用 v0.2 现有 3 个 fixture + 加 2 个新 fixture（`castles` / `creepers`），覆盖 L3 的 5 轮输入 |
| Migration 重入产生 duplicate column error | 重复跑 migration 挂 | `schema_migrations` 表跟踪已跑版本，runner 跳过已应用 |
| ON DELETE CASCADE 在 SQLite 默认未启用 | 删 session 时 FK 失效 | 跑 migration 前 `PRAGMA foreign_keys = ON;` |
| 并发写（CLI 退出时 LLM 流式仍在写） | `database is locked` | CLI 单线程、用 sync API、退出时先 await 最后一条 message 写入再 close db |
| 测试间 DB 状态污染 | 测试 flaky | 每个 L1/L2 用 `os.tmpdir()/storage-test-${randomUUID()}/` 临时目录，`afterEach` 删目录 |
| 进程崩溃中途 | DB 半写 | better-sqlite3 transaction 默认 atomic；schema 用 `IF NOT EXISTS` 让 migration 可重入 |

## 7. Definition of Done

按 [DEVELOPMENT_PLAN §6](../../DEVELOPMENT_PLAN.md) 7 条硬性条件，v0.3 必须全部满足：

1. `pnpm typecheck` 绿
2. `pnpm lint` 绿
3. `pnpm test` 全绿（L1 ≥ 5 + L2 ≥ 2 + L3 ≥ 1 落地）
4. `pnpm build` 成功
5. Live 模式手动跑通 ≥ 5 轮对话 + DB 落盘 + 重启可查
6. Replay 模式 CI 跑通（L3 用 Replay）
7. 本 scope 文档中"不做"清单没有偷偷做（git diff 与 scope 一致）
