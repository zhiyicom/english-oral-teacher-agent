## 2. Schema 设计

### 2.1 `sessions` 表 —— **零迁移**

`001_init.sql` 已经在 `sessions` 表预留了：

```sql
summary         TEXT,             -- v0.5
keywords        TEXT,             -- v0.5 (JSON 字符串)
```

所以 **v0.5 不需要 ALTER TABLE**。Migration 002 是一个**空文件 + 注释**，只起"v0.5 启用"标记作用：

```sql
-- 002_session_memory.sql
-- v0.5 启用 summary + keywords 列
-- 字段定义在 001_init.sql 已存在；本迁移为空操作，仅作为版本标记
SELECT 1;
```

> **为什么不开新表 / 不加列？** 旧 DB（v0.3 跑过 001 的）已经有这 2 列；新 DB（v0.4 之后）也跑过 001。**v0.5 是 v0.3 schema 预留的"激活"**，不是新 schema。`schema_migrations` 跑完 002 后版本号从 1 → 2，下一次再跑会跳过。

### 2.2 `LastReview` interface

```ts
// src/agent/retrieval.ts
export interface LastReview {
  sessionId: string
  startedAt: string             // ISO 8601
  endedAt: string | null
  durationMin: number | null
  summary: string               // 50-150 tokens
  keywords: string[]            // 3-8 个
  daysAgo: number               // 整数；now 距 startedAt 的整天数
}
```

`daysAgo` **不存 DB**，由 `loadLastReview(handle, now = new Date())` 现算。这样 `daysAgo` 永远反映"调用时刻"（CLI 启动时 = 此刻）。

### 2.3 `Summary` interface + zod schema

```ts
// src/agent/summarizer.ts
import { z } from 'zod'

export const SummarySchema = z.object({
  summary: z.string()
    .min(20, 'summary 太短 (< 20 chars)')
    .max(800, 'summary 太长 (> 800 chars)'),
  keywords: z.array(z.string().min(1).max(40))
    .min(3, 'keywords 至少 3 个')
    .max(8, 'keywords 最多 8 个'),
})

export type Summary = z.infer<typeof SummarySchema>
```

**边界值**：
- summary 长度：20-800 chars（≈ 5-200 tokens）—— 比 PRD §6.4 的 "50-150 tokens" 略宽，给 LLM 一点弹性
- keywords 数量：3-8（与 PRD §F4 一致）
- keyword 单词长度：≤ 40 chars（防 LLM 输出整句）

## 3. 算法 / 伪代码

### 3.1 `summarize(messages, client)`

```ts
export async function summarize(
  messages: Message[],
  client: LLMClient,
): Promise<Summary> {
  // 1. 截断：> 25 条 → 前 20 + 后 5
  const truncated = messages.length > 25
    ? [...messages.slice(0, 20), ...messages.slice(-5)]
    : messages

  // 2. 拼成 transcript
  const transcript = truncated
    .map((m) => `${m.role.toUpperCase()}: ${m.content}`)
    .join('\n')

  // 3. 构造单条 user message（"summarize" 这个词是关键 ——
  //    Replay fixture 匹配 userSaysContains: "summarize"）
  const summaryRequest: Message[] = [{
    role: 'user',
    content:
      `Please summarize the following English conversation session between ` +
      `a teacher (Alex) and a student. Return a JSON object with two fields:\n` +
      `- "summary": 1-3 sentences (50-150 tokens) describing what the student ` +
      `practiced and what the teacher focused on.\n` +
      `- "keywords": 3-8 lowercase English words or short phrases capturing ` +
      `the key topics, vocabulary, and themes.\n\n` +
      `Output ONLY the JSON, no other text.\n\n` +
      `<conversation>\n${transcript}\n</conversation>`,
  }]

  // 4. 调 LLM
  const system = await loadSummarizerSystemPrompt()
  const response = await client.chat({ system, messages: summaryRequest })

  // 5. 解析 + 校验
  const parsed = JSON.parse(response.content)  // Replay fixture 是合法 JSON
  return SummarySchema.parse(parsed)
}
```

**为什么 LLM 调用要构造单 user message**（而不是把 transcript 直接塞 messages）：

- Replay provider 匹配 `userSaysContains`（`src/llm/testing.ts:24-35`），需要最后一条 user 消息里有 "summarize"
- Live 模式（Anthropic）这样构造也更清晰：单 user message 包住 transcript 是常见 summarization prompt pattern
- 第一行带 "Please summarize" 也让 Live LLM 知道这是 summarization task

### 3.2 `loadLastReview(handle, now?)`

```ts
export function loadLastReview(
  handle: DbHandle,
  now: Date = new Date(),
): LastReview | null {
  const row = handle.raw.prepare(`
    SELECT id, started_at, ended_at, duration_min, summary, keywords
    FROM sessions
    WHERE summary IS NOT NULL
    ORDER BY started_at DESC
    LIMIT 1
  `).get() as SessionRow | undefined

  if (!row || !row.summary) return null

  return {
    sessionId: row.id,
    startedAt: row.started_at,
    endedAt: row.ended_at,
    durationMin: row.duration_min,
    summary: row.summary,
    keywords: row.keywords ? JSON.parse(row.keywords) : [],
    daysAgo: Math.floor(
      (now.getTime() - Date.parse(row.started_at)) / 86_400_000
    ),
  }
}
```

**关键点**：
- `WHERE summary IS NOT NULL` —— 跳过 v0.3 / 早期 v0.5 session（没 summary 的）
- `ORDER BY started_at DESC LIMIT 1` —— PRD §3.4 "最近 1 条"（不是 top 2 相关，那是 v0.6+）
- `daysAgo = Math.floor(...)` —— 边界值测试：startedAt = 23:59:59、now = 00:00:01 → daysAgo = 0（不是 1）。这是 [System Context] 文案期望："a few hours ago" 比 "1 day ago" 准

### 3.3 `buildSystemContext` 扩展

```ts
// src/agent/context-injector.ts
import type { LastReview } from './retrieval.js'
import type { SessionState } from './state-machine.js'

export function buildSystemContext(
  state: SessionState,
  lastReview: LastReview | null = null,
): string {
  const lastTransitionAgo = Math.max(0, state.elapsedMin - state.lastTransitionAt)
  const lines = [
    '[System Context]',
    `- Phase: ${state.phase}`,
    `- Elapsed: ${state.elapsedMin.toFixed(1)} min`,
    `- Silence: ${state.silenceMin.toFixed(1)} min`,
    `- Last transition: ${lastTransitionAgo.toFixed(1)} min ago (entered ${state.phase})`,
  ]
  if (lastReview) {
    const kwStr = lastReview.keywords.join(', ')
    const durStr = lastReview.durationMin != null ? `${lastReview.durationMin} min` : 'unknown'
    lines.push(
      `- Last session (${lastReview.daysAgo} day${lastReview.daysAgo === 1 ? '' : 's'} ago, ${durStr}): ${lastReview.summary}`,
    )
    lines.push(`- Last session keywords: ${kwStr}`)
  }
  return lines.join('\n')
}
```

**示例输出**（lastReview 非空）：

```
[System Context]
- Phase: WARM_UP
- Elapsed: 0.0 min
- Silence: 0.0 min
- Last transition: 0.0 min ago (entered WARM_UP)
- Last session (1 day ago, 3 min): Student practiced talking about Minecraft castles and creepers.
- Last session keywords: minecraft, castle, creeper, wall, build
```

### 3.4 CLI 集成

#### 3.4.1 启动时（一次）

```ts
// src/cli.ts (在 sessions.create() 之后、loadSystemPrompt 之后)
const lastReview = loadLastReview(db)
let isFirstTurn = true
```

#### 3.4.2 主循环

```ts
// 每个 turn
state = applyEvent(state, { type: 'TICK' }, clock)
// ... (phase transition logging, stop detection, USER_MSG) ...

const system = buildFinalSystem(systemPrompt, state, isFirstTurn ? lastReview : null)
isFirstTurn = false  // ← 关键：之后所有 turn 不再注入 last review

process.stderr.write(`[cli] ctx: ${state.phase} elapsed=${state.elapsedMin.toFixed(1)} silence=${state.silenceMin.toFixed(1)}\n`)
reader.writePrompt('[Teacher]: ')
let response = ''
for await (const chunk of client.chatStream({ system, messages: history })) {
  // ...
}
```

> **first-turn-only 实现**：`isFirstTurn` 在第一次 `buildFinalSystem` 后变 false。`buildFinalSystem` 第二次及以后收到的 `lastReview` 是 null（caller 控），所以 injector 不需要知道"是否第一次"。这是 v0.5 scope §2.5 决策 B。

#### 3.4.3 退出时

```ts
} finally {
  try {
    // 1. NEW: 收集所有 messages 调 summarizer
    const allMessages = messages.getBySession(session.id).map(m => ({
      role: m.role,
      content: m.content,
    }))

    let summary: string
    let keywords: string[]
    try {
      const review = await summarize(allMessages, client)
      summary = review.summary
      keywords = review.keywords
    } catch (err) {
      // 降级：fallback summary，不阻塞 END
      process.stderr.write(`[cli] summarizer failed: ${(err as Error).message}\n`)
      summary = '(summarization failed)'
      keywords = []
    }

    // 2. 写 DB
    process.stderr.write(`[cli] markEnded ${session.id} reason=${exitReason}\n`)
    sessions.markEnded(session.id, {
      phaseHistory,
      summary,
      keywords,
      reason: exitReason,
    })
    process.stderr.write('[cli] markEnded done\n')
  } finally {
    rl.close()
    db.close()
  }
}
```

**关键点**：
- `summarize()` 包在 `try/catch` 里 —— LLM 失败 / JSON 解析失败 / zod 失败都用 fallback（不抛出去）
- `markEnded` 仍走 —— phase_history 不丢
- messages 用 DAO 的 `getBySession` 取（v0.3 已有）

### 3.5 `markEnded` SQL 扩展

```sql
-- src/storage/sessions.ts
UPDATE sessions
SET
  ended_at = ?,
  duration_min = ?,
  phase_history = COALESCE(?, phase_history),
  summary = COALESCE(?, summary),         -- v0.5
  keywords = COALESCE(?, keywords)         -- v0.5
WHERE id = ?
```

`COALESCE(?, summary)` 保证不传时不覆盖已有值（与 v0.4 phase_history 一致）。

`Session` interface 扩 2 个字段：

```ts
export interface Session {
  id: string
  started_at: string
  ended_at: string | null
  duration_min: number | null
  phase_history: string | null
  summary: string | null        // v0.5
  keywords: string | null       // v0.5 (JSON)
}
```

`SELECT_COLS` 同步加 2 个列：

```ts
const SELECT_COLS = 'id, started_at, ended_at, duration_min, phase_history, summary, keywords'
```

`serializePhaseHistory` 类似地加 `serializeKeywords`：

```ts
function serializeKeywords(kw: string[] | undefined): string | null {
  if (!kw) return null
  return JSON.stringify(kw)
}
```

## 4. Fixture 设计

### 4.1 `tests/fixtures/replay/summarize.json`

```json
{
  "name": "summarize",
  "match": {
    "userSaysContains": "summarize"
  },
  "chunks": [
    {
      "type": "text",
      "delta": "{\"summary\":\"Student practiced talking about Minecraft castles and creepers. Teacher suggested building walls for defense.\",\"keywords\":[\"minecraft\",\"castle\",\"creeper\",\"wall\",\"build\"]}"
    }
  ],
  "result": {
    "content": "{\"summary\":\"Student practiced talking about Minecraft castles and creepers. Teacher suggested building walls for defense.\",\"keywords\":[\"minecraft\",\"castle\",\"creeper\",\"wall\",\"build\"]}"
  }
}
```

**为什么 match 是 "summarize"**：summarizer 构造的单 user message（§3.1 step 3）以 "Please summarize the following..." 开头，Replay provider 的 `pickFixture` 用 `includes` 匹配 `userSaysContains`，所以这个 fixture 在 summarizer 调用时一定命中。

**为什么 `chunks` + `result.content` 都是同一段 JSON**：
- `chunks` 给流式调用用（v0.5 不调 chatStream，但保持 fixture 完整）
- `result.content` 给 `chat()` 用（summarizer 用的是 `chat`）
- 两边必须一致 —— Replay provider 期望如此

### 4.2 `prompts/summarizer-system.md`

```markdown
# Role

You are a session summarizer for an English oral teacher agent.

# Task

Given a conversation transcript, produce a structured summary capturing:
- What the student practiced (topics, grammar, vocabulary, themes)
- What the teacher focused on (techniques, corrections, scaffolding)
- Any notable observations (engagement level, struggles, breakthroughs)

# Output format

Output ONLY a JSON object with these two fields:

```json
{
  "summary": "1-3 sentences, 50-150 tokens",
  "keywords": ["3-8 lowercase English words or short phrases"]
}
```

Do not include any prose, explanation, or markdown formatting outside the JSON.
```

**Live 模式行为**：MiniMax M3 看到 system prompt 后会按 JSON 格式输出。zod schema 校验失败时 fallback（scope §6 风险 #1 已定）。

## 7. 风险与处置

| 风险 | 概率 | 影响 | 处置 |
|---|---|---|---|
| LLM 输出 JSON 不规范（缺字段 / 越界） | 中 | summarizer 抛 → session 写不进 | zod 校验；失败 → fallback `summary = "(summarization failed)"` + `keywords = []`；不阻塞 markEnded |
| LLM 输出格式完全错（不是 JSON） | 低 | JSON.parse 抛 → 同上 fallback | 同上 |
| Replay fixture 不命中（match 词不在 user 消息里） | 低 | session END exit 1 | fixture match 用 "summarize"；summarizer 永远以 "Please summarize..." 起的 user 消息 调 LLM（§3.1 step 3）|
| summarizer LLM 慢 | 中（Live）| session 退出延迟 5-10s | 可接受——END 阶段不在主路径；用户已按 stop 预期等 |
