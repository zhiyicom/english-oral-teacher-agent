# Sprint v0.5 Scope — Memory: Last-Review Recall

> **本文是 Sprint v0.5 的范围定义**。回答"v0.5 做什么、不做什么、怎么算做完"。
>
> 详细设计见 [v0.5-design.md](./v0.5-design.md)。本文件不写实现细节，只写边界。

---

## 1. Sprint 目标

**用一句话**：新 session 启动时自动把"上次 session 的摘要"塞进 [System Context]，session 结束时 LLM 自动生成摘要 + 关键词写回 DB——agent 能"记住"学生上次聊过什么。

**为什么是 v0.5**：

- v0.4 把 30 分钟 session 切成 4 phase，但 **session 之间完全失忆**：新 session 起来，agent 不认识学生，聊过的话题、上次学了什么、错在哪——全清零
- 这是 v0.7 工具调用（`mark_mistake` / `mark_vocabulary`）的前置：错例 / 词汇表落盘要等 session END 有 summary 字段才好关联
- 也是 v0.6 `topic_stats` 选题的前置：需要 `session.keywords` 列先存在
- **不是** v0.5：向量化、LanceDB、跨 session 语义检索、错例 / 词汇表（属于 v0.6 / v0.7）

**Demo 形态**：

```bash
# 第一次：跑完一个 session
$ MOCK_TIME=1 pnpm dev
> hi
[Teacher]: Hi there! How are you today?
> castle
[Teacher]: Castles are cool! ...
> stop
$ sqlite3 data/oral-teacher.db "SELECT summary, keywords FROM sessions ORDER BY started_at DESC LIMIT 1;"
"Student talked about Minecraft castles and creepers. Asked what she builds. Suggested she try walls." | ["minecraft","castle","creeper","wall"]

# 第二次：再开一个 session —— 启动时 agent 看到上次回顾
$ MOCK_TIME=1 pnpm dev
[System Context]
- Last session (1 day ago, 3 min): Student talked about Minecraft castles and creepers. ...
- Last session keywords: minecraft, castle, creeper, wall
> hi
[Teacher]: Hey! Last time we talked about Minecraft castles. Did you try building walls around your base?
```

### 2.2 `LastReview` 数据形状

```ts
interface LastReview {
  sessionId: string
  startedAt: string             // ISO 8601
  endedAt: string | null
  durationMin: number | null
  summary: string               // 1-3 句，50-150 tokens
  keywords: string[]            // 3-8 个英文词或短语
  daysAgo: number               // 整数，距今多少天（用于 [System Context] 显示）
}
```

> `daysAgo` 由 `loadLastReview` 算好后返回（不存 DB）。其他字段直接从 `sessions` 表读。

### 2.3 [System Context] 块扩展

v0.4 的块长这样：

```
[System Context]
- Phase: WARM_UP
- Elapsed: 0.0 min
- Silence: 0.0 min
- Last transition: 0.0 min ago (entered WARM_UP)
```

v0.5 末尾追加 "Last Session" 段（仅当 `lastReview` 非空）：

```
- Last session (1 day ago, 3 min): <summary>
- Last session keywords: kw1, kw2, kw3
```

> **只在 WARM_UP 阶段注入**——这是 v0.5 的决策（见 §2.5）。MAIN_ACTIVITY / WRAP_UP 阶段不重复塞历史，避免长 session 把"上次"信息稀释到噪声里。Phase 检查由 caller（CLI）控制，不是 injector 内部逻辑。

### 2.4 Session END 流程

CLI 退出时（`markEnded` 之前）新增一步：

```ts
// 1. 已有：调 LLM
// 2. 已有：写 messages
// 3. NEW: 调 summarizer（Replayed / Live）
const review = await summarize(allMessages, client)
// 4. NEW: 把 summary + keywords 写进 sessions 行
sessions.markEnded(session.id, { phaseHistory, summary: review.summary, keywords: review.keywords, reason })
// 5. 已有：close DB
```

`markEnded` 入参扩展：

```ts
interface MarkEndedInput {
  endedAt?: string
  durationMin?: number
  phaseHistory?: PhaseTransition[]   // v0.4 已有
  summary?: string                    // ← v0.5 新增
  keywords?: string[]                 // ← v0.5 新增（存时 JSON.stringify）
  reason?: string                     // v0.4 已有
}
```

SQLite 写入用 `UPDATE sessions SET summary = COALESCE(?, summary), keywords = COALESCE(?, keywords) WHERE id = ?`——和 v0.4 phase_history 一样，**不传时不覆盖**已有值，向后兼容。

### 2.5 WARM_UP-only 注入决策

| 选项 | 描述 | 选 |
|---|---|---|
| **A**: 每个 turn 都注入 | 全程带 last review，但 30 分钟 long session 后期会显得 stale | |
| **B**: 只在 session 启动时（首次 LLM call）注入 | 注入 1 次，写到 system prompt 后整个 session 不变 | ✅ |

B 更省 token、更贴近 PRD §5.3 的"启动时检索注入一次"。CLI 实际做法：构建 system string 时**第一次**带 last review，存到本地变量；后续 turn 复用同一个 system string，不再追加。状态机换 phase 时，buildSystemContext 仍重算（因为 elapsed / silence 在变），但 last review 部分从 system string 里"冻结"。

> 实现细节在 design doc 写。Scope 只定决策：**first-turn-only**。

### 2.6 Summarizer LLM 调用

- **输入**：session 全部 messages（user + assistant 交替），可以压缩：截前 20 条 + 后 5 条 + 中间密度抽样
- **输出 schema**（zod 校验）：
  ```ts
  {
    summary: string,   // 50-150 tokens
    keywords: string[]  // 3-8 个，全部小写英文 / 短语
  }
  ```
- **调用方式**：复用现有 `LLMClient.chat(opts)`（不是 chatStream——summarizer 不需要流式）
- **system prompt**：写一个 `prompts/summarizer-system.md`（commit 进 git）—— 告诉 LLM "你是会话摘要器，输出 JSON"
- **Replay fixture**：`tests/fixtures/replay/summarize.json`，提供一份标准响应
- **Live 模式**：真打 MiniMax M3

### 2.7 Schema 影响

| 改动 | 文件 |
|---|---|
| 新增列 `summary TEXT` | `sessions` 表 |
| 新增列 `keywords TEXT`（JSON 数组） | `sessions` 表 |
| `Session` interface 加字段 | `src/storage/sessions.ts` |
| `markEnded` 入参加 `summary?` / `keywords?` | 同上 |
| `markEnded` SQL 扩展 | 同上 |
| 迁移文件 | `src/storage/migrations/002_session_memory.sql` |

> **不开新表**。`mistakes` / `homework` / `vocabulary` / `topic_stats` 都留给 v0.6 / v0.7。

| 不做 | 留给 | 备注 |

| 跨天恢复 / resume 中断 session | v1.0 前 | PRD §4.4 提了，没排进 sprint |
| 启动时引用相关历史 top 2（向量）| v0.6+ | PRD §3.4 的 "可选" |
| process 不退出（多次开 session）| v0.8 UI | v0.5 仍是 CLI REPL，启一次跑一个 session |
| TTS / 语音 | v0.9 | |
| 进程内缓存 / 性能优化 | 量起来再说 | v0.5 数据量小（一人 1-2 session/day） |

## 6. 风险与回退

| 风险 | 影响 | 应对 |
|---|---|---|
| **LLM 生成的 JSON 不规范**（少字段 / keywords 数量超界） | summarizer 抛错，session 写不进 END | zod 严格校验；校验失败 → 写一个 fallback `summary: "(summarization failed)"` + 空 keywords，不阻塞 session 结束 |
| **Summarizer LLM 慢**（Live 模式）| session 退出要等 5-10s | 可接受——END 阶段不阻塞其他逻辑；用户已按 stop，预期要等 |
| **Last review 注入造成 token 膨胀** | 单轮 prompt 超过 6K | summary 限 150 token，keywords 限 8 个，预算 250 token |
| **Replay fixture 不匹配 summarizer 调用** | L3 测 exit 1 | fixture 匹配 key 用 `userSaysContains` 检查消息里有无 "summarize" 触发字 |
| **Migration 002 与 001 不兼容** | 旧 DB 升级失败 | migration 全部用 `ALTER TABLE ... ADD COLUMN`（v0.3 schema 已支持 `IF NOT EXISTS`） |
| **CLI 第一次 LLM call 失败导致 END 流程断** | summary 写不进去 | 已有 `finally` 块保护 markEnded；summarizer 失败时 fallback summary 仍写 |
| **Days-ago 边界（恰好跨天）** | 显示 "0 day ago" 但其实是昨天 | 用 `Math.floor((now - startedAt) / 86400000)`；边界值测试 |
| **多 session 并发写 last review** | A 写完 B 启动读到的还是旧 review | v0.5 CLI 是单 session 串行，无并发；风险不存 |

## 7. Definition of Done

按 [DEVELOPMENT_PLAN §6](../../DEVELOPMENT_PLAN.md) 7 条硬性条件，v0.5 必须全部满足：

1. `pnpm typecheck` 绿
2. `pnpm lint` 绿
3. `pnpm test` 全绿（L1 ≥ 3 + L2 ≥ 2 + L3 ≥ 2 落地，v0.5 实际远超）
4. `pnpm build` 成功
5. Live / Replay 手动跑通：连跑 2 session，第二次启动时 [System Context] 可见 "Last session" 段 + DB 落 `summary` / `keywords`
6. Replay 模式 CI 跑通（L3 用 Replay + summarizer fixture）
7. 本 scope 文档中"不做"清单没有偷偷做（git diff 与 scope 一致）
