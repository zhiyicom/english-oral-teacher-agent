| 文件 | 匹配 `userSaysContains` | 用途 | v0.5 新增 |
|---|---|---|---|
| `tests/fixtures/replay/greeting.json` | `hi` | 首轮打招呼 | |

| `tests/fixtures/replay/hobby-played.json` | `played` | 学生说玩了什么之后的追问 | |
| `tests/fixtures/replay/castle.json` | `castle` | castle 后的追问 | |
| `tests/fixtures/replay/creeper.json` | `creeper` | creeper 后的追问 | |
| `tests/fixtures/replay/stop.json` | `stop` | 用户说停后的道别 | |
| `tests/fixtures/replay/summarize.json` | `SUMMARIZER_INSTRUCTION` | 退出时 summarizer 调用的回复 | ✅ |

6+1 = 7 个 fixture。`summarize.json` 是 v0.5 唯一新增：提供 `summary: "Student practiced talking about Minecraft castles and creepers. Teacher suggested building walls for defense."` + `keywords: ["minecraft","castle","creeper","wall","build"]`。

**为什么需要 unique marker**：`pickFixture` 是 first-match-wins（按文件系统顺序），summarizer 把 transcript 当上下文送给 LLM，但 fixture 匹配只看**最后一条 user message**。如果最后一条是 transcript 内容（如 "castle"），就会错配 `castle.json`。用 `[SUMMARIZER_INSTRUCTION]` 这种独特 marker 作最后一条 user message，transcript 里绝不会出现，自然只匹配 `summarize.json`。**`buildSummaryInstruction` 故意只输出 marker + JSON 格式要求，不复制 transcript**——见 `src/agent/summarizer.ts:buildSummaryInstruction` 注释。

## 6. 架构与代码改动

### 6.1 `src/agent/retrieval.ts`（新增）

| 项 | 内容 |
|---|---|
| 导出 | `LastReview` interface + `loadLastReview(handle, now?)` 函数 |
| SQL | `SELECT id, started_at, ended_at, duration_min, summary, keywords FROM sessions WHERE summary IS NOT NULL ORDER BY started_at DESC LIMIT 1` |
| `daysAgo` | `Math.max(0, Math.floor((now.getTime() - Date.parse(started_at)) / 86_400_000))` |
| 返回 null 条件 | 库空 / 所有 session `summary IS NULL` |
| `now` 注入 | 可选参数，测试用 `new Date('2026-06-09T00:00:00Z')` 之类的固定时间求 `daysAgo` |

### 6.2 `src/agent/summarizer.ts`（新增）

| 元素 | 职责 |
|---|---|
| `SummarySchema` (zod) | `summary: 20..800 chars` + `keywords: 3..8 strings` |
| `truncateMessages(msgs)` | ≤ 25 条不变；> 25 条取 first 20 + last 5（设计 §3.2） |
| `buildSummaryInstruction()` | 返回**仅含** `[SUMMARIZER_INSTRUCTION]` marker + JSON schema 说明的 user message；**不复制 transcript**——让 fixture 唯一匹配 `summarize.json` |
| `summarize(messages, client)` | 1) 截断 2) 拼 `messages: [...truncated, buildSummaryInstruction()]` 3) `client.chat({...})` 4) JSON.parse 5) `SummarySchema.parse` 6) 失败任一步抛错 |

### 6.3 `src/agent/context-injector.ts`（扩）

`buildSystemContext(state, lastReview?)` 新增 `lastReview` 可选参数。当非 null 时在 v0.4 4 字段后追加两行：

```
- Last session (X day(s) ago, Y min): <summary>
- Last session keywords: k1, k2, k3
```

- `day` / `days` 单复数
- `durationMin === null` → "unknown"
- keywords 数组空时也显示 keywords 行（v0.5 设计：保留行让 LLM 知道"上次有回顾但无关键词"）

### 6.4 `src/agent/prompt-builder.ts`（扩）

`buildFinalSystem(prompt, state, lastReview?)` 把 `lastReview` 透传给 `buildSystemContext`。**注意**：`LastReview` 类型从 `./retrieval.js` 单独 import，**不从 `./index.js`**——后者会拉入 summarizer 形成循环依赖（retrieval → sessions → 无 summarizer；summarizer → client → 无 retrieval；OK）。`prompt-builder` 在 summarizer 之后被引用，不在循环里。

### 6.5 `src/cli.ts` 改造

| 时机 | 行为 |
|---|---|
| 启动 | `loadLastReview(db)` → 缓存 `lastReview: LastReview \| null` + `isFirstTurn = true` |
| 每 turn | `buildFinalSystem(sp, state, isFirstTurn ? lastReview : null)` → 用完即 `isFirstTurn = false` |
| finally 块 | `messages.getBySession(id)` → `summarize(messages, client)` → 拿 `{summary, keywords}` → catch 兜底为 `"(summarization failed)"` + `[]` → `sessions.markEnded({phaseHistory, summary, keywords, reason})` |
| stderr | 新增 `[cli] summarize ok summary=Xc keywords=Y` / `[cli] summarize failed: ...` / `[cli] markEnded <id> reason=<r>` / `[cli] markEnded done` |

**wiring sanity 为什么用 `summarize ok` 当 proxy**：L3 不能直接看到 LLM 收到的 system 字符串（CLI 没把 system 写 stderr）。但要证明 "Last session 真的进了 LLM 输入"，需要观察 LLM 的反应。简化办法：检查 `summarize` 调到 LLM（→ `summarize ok`）且其 fixture 匹配正确（→ fixture summary 含 "minecraft" 关键字）——能匹配上，证明 `client.chat` 收到的最后一条 user message 含 `[SUMMARIZER_INSTRUCTION]`（而非 transcript 里的 "castle" 之类）。这间接覆盖了"transcript + lastReview 串到 LLM"。

### 6.6 `src/storage/sessions.ts` 扩展

| 字段 | 变化 |
|---|---|
| `Session.summary: string \| null` | ✅（v0.5 落地字段） |
| `Session.keywords: string \| null` | ✅（JSON-encoded string[]） |
| `MarkEndedInput.summary?: string` | ✅ |
| `MarkEndedInput.keywords?: string[]` | ✅ |
| `markEnded` SQL | 增 `summary = COALESCE(?, summary)` + `keywords = COALESCE(?, keywords)` |
| `serializeKeywords(arr)` 内部 helper | keywords 数组 → JSON 字符串（兜底空数组 → `"[]"`） |
| `create()` 行为 | 不变（返回 `summary: null, keywords: null`） |

### 6.7 迁移 002_session_memory.sql

空操作（`SELECT 1;`）。**字段 `summary` / `keywords` 在 001_init.sql 已存在**（v0.3 schema 设计时预留）。v0.5 002 仅作为版本标记，让 `applyMigrations` 知道 v0.5 跑过——便于以后真有 schema 变化时挂 ALTER TABLE。

## 8. 风险与遗留问题

| 风险 / 遗留 | 影响 | 处置 |
|---|---|---|
| **Summarizer fixture 误匹配** | transcript 里有 "castle" → castle.json 先匹配 → "summarize failed" | `[SUMMARIZER_INSTRUCTION]` unique marker + `buildSummaryInstruction` 不复制 transcript；L1 测了 17 个分支 |
| **Migration 002 是空操作** | 用户怀疑"我跑了迁移为啥 DB 没变？" | 注释里说明（`src/storage/migrations/002_session_memory.sql` + 设计 §2.1）；字段早就在 001 里 |

| **Last review 只在 first turn 注入** | 第 2+ 轮 LLM 看不到上次回顾 | 设计 §2.5：first-turn-only 防 stale drift；用户能感知第一句"上次我们聊了 Minecraft" |
| **Summarizer LLM 失败兜底太薄** | 失败时只写 "(summarization failed)" + `[]` | 留作 v0.5+ 增强：失败时存 raw LLM 输出到 `raw_summary_json` 列（不动 v0.5 范围） |
| **biome auto-fix 改 7 个文件** | 提交前 lint 失败 | `pnpm exec biome check --write --unsafe` 自动修模板字面量；无功能改动 |
| **`loadLastReview` 缺列不报错** | 旧 DB 没有 summary/keywords 列时返回 undefined 行 | L1 `retrieval.test.ts` 测了缺列 → 抛错而非崩 |
| **多 session 同秒结束** | `ORDER BY started_at DESC LIMIT 1` 选 started_at 最大的（不是 ended_at），可预期 | OK，当前用法合理 |
| **Live 模式没真打过 LLM** | LLM 真实响应未在本机验证 | L1 `summarize.test.ts` Replay 集成测了 fixture 路径；Live 模式跑 LLM 需 `RUN_LIVE_LLM=1` + 真实 key，跨网延迟 + token 成本不在本 sprint 范围 |
| **关键词大小写 / 中英混排** | LLM 可能输出 "Minecraft" / "我的狗" | 现状接受（"minecraft" 在 fixture 里就是小写）；中英混排是下个 sprint 的多语种 LLM 测试覆盖 |
