# Sprint v0.6 Scope — 话题匹配 + topic_stats

<!-- STATUS: PLANNING -->

> **本文件是 Sprint v0.6 的计划**。详细设计见 [v0.6-design.md](./v0.6-design.md)。
>
> v0.6 核心：**session END 时把本次 keywords 跟一个预定义 topic 库做 Jaccard 匹配，匹配上则更新 topic_stats**（讨论次数 + 最后讨论时间）。同时把这些 topic_stats 注入 `[System Context]`，让 LLM 知道"哪些话题学生最近反复聊过"，自然地延续或切换话题。

---

## 1. 背景

v0.5 实现了"上次 session 摘要 → 下次首轮注入"。v0.6 沿同一条价值链往前走一步：

- 学生围绕某个话题（minecraft / school / sports ...）连聊了多轮
- 每次结束都生成 keywords，但**没有聚合视角**
- 老师（L4M）不知道"这孩子最近 3 次都在聊 minecraft"——只看得到本次
- topic_stats 提供**横向统计**：某个话题被讨论过几次、最近一次何时

这是后续 v0.7 工具调用（`mark_vocabulary` / `mark_homework`）的基础设施：工具能基于"这个话题被讨论 N 次"做决策（重复练习 vs 推进）。

### 1.1 与 DEVELOPMENT_PLAN §3 路线图的关系

路线图原 v0.6 = "摘要 + 话题"。**摘要部分已在 v0.5 完成**（summary + keywords + LLM summarizer + lastReview 注入）。v0.6 实际范围 = 路线图 v0.6 的"话题"部分 + topic_stats。代码量更小（~2 天 vs 路线图估的 2 天，持平）。

路线图原 v0.6 §5 提到的 Jaccard 算法（"已在 v0.5 写过，验证复用"）——v0.5 实际没用 Jaccard，v0.6 引入。

## 3. 关键决策

### 3.1 Topic 库用 DB 表 vs 硬编码

**选 DB 表**。理由：topic 是"领域知识"，v0.6+ 可能加新 topic（v0.7 工具发现时），DB 表好维护、好查、好备份。硬编码要改代码 + 重新部署。

### 3.2 Jaccard 阈值

**选 0.3**。理由：13 岁孩子口语关键词噪声大（"I think"、"also"、"really"），阈值太低会误匹配；太高又匹配不上。Minecraft fixture 关键词 5 个全在 topic 关键词集里（10 个），Jaccard=0.5 远超 0.3；空交集 keywords 不会匹配。0.3 是平衡点（先 ship，看实际误匹配率再调）。

### 3.3 一次 session 匹配几个 topic

**最多 1 个**（best match）。理由：v0.6 简化逻辑；多 topic 真实场景少（13 岁口语 5-10 轮一般聚焦 1 主题）；v0.7 工具调用时可以发现"这次聊了 2 个主题"再做。Jaccard tie（2 个 topic 同分）按 topic name 字典序取第一个，结果确定性。

### 3.4 Active topics 注入 [System Context] 吗

**注入**。理由：v0.5 已把 lastReview 注入；v0.6 沿同一条价值链。LLM 看到 "Minecraft (3 times, 2 days ago)" 会自然延续或切换；不注入则 topic_stats 是"死数据"。注入位置：[System Context] 末尾，Last session 段之后。

### 3.5 Active topics 显示几个

**最多 5 个**。理由：5-10 分钟 session 范围内 5 个 topic 足够代表"最近聊过的话题"；再多会让 [System Context] 臃肿（v0.5 已 6 字段 + 2 段）。无 topic_stats 时整段不显示（与 Last session 段同处理）。

### 3.6 注入格式

```
- Active topics: minecraft (3 times, 2 days ago), school (1 time, today)
```

- "times" 永远复数（"1 time" 简化为 "1 time" 而非 "1 times"——美式英文容忍）
- 时间差复用 v0.5 的 `daysAgo` 计算（Math.floor days）
- 0 days ago → "today"
- 0 times 不出现（topic_stats 不会出现 count=0 的行，count 只增）

## 4. 数据模型

### 4.1 `topics` 表

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | TEXT PRIMARY KEY | topic 名（e.g., "minecraft"） |
| `keywords_json` | TEXT NOT NULL | JSON 数组，topic 的种子关键词 |
| `description` | TEXT | 一句话描述（v0.6 暂不展示，预留给 v0.7 工具） |
| `created_at` | TEXT NOT NULL | ISO timestamp |

### 4.2 `topic_stats` 表

| 字段 | 类型 | 说明 |
|---|---|---|
| `topic` | TEXT PRIMARY KEY | 外键 → topics.name |
| `discussion_count` | INTEGER NOT NULL DEFAULT 0 | 累计匹配次数 |
| `last_discussed_at` | TEXT | 最近一次匹配时间（ISO）；NULL 表示从未匹配 |
| `first_discussed_at` | TEXT | 首次匹配时间 |

**注**：外键约束不开（SQLite 缺省 PRAGMA foreign_keys=OFF，且 topic 被删时 cascade 复杂）。DAO 层保证 `matchAndUpdateTopic` 只 insert 已知 topic。

### 4.3 Seed 数据（迁移 003）

| name | keywords | description |
|---|---|---|

| school | school, class, teacher, homework, exam, friend, lunch, recess | School life |
| sports | soccer, basketball, swim, run, ball, team, match, win | Sports |
| food | food, eat, dinner, lunch, breakfast, restaurant, delicious, cook, recipe | Food & meals |
| family | family, mom, dad, brother, sister, pet, dog, cat, home | Family & pets |
| movies | movie, film, watch, cartoon, character, story, episode | Movies & TV |
| music | music, song, sing, dance, band, instrument, piano, guitar | Music |

7 个 starter topics。后续 v0.7 工具可发现新 topic 再加。

## 8. 风险

| 风险 | 影响 | 处置 |
|---|---|---|
| **Jaccard 阈值 0.3 误匹配** | 无关 keywords 匹到 sports 之类 | L1 fixture 测正反例；live demo 验证；v0.6.1 可调阈值 |
| **Jaccard tie** | 2 topic 同分，结果不确定 | 字典序取第一个；L1 测稳定 |
| **Active topics 段让 [System Context] 过长** | 5 个 topic × 20c ≈ 100c；与 Last session 合计 ~200c | 限制 5 个；后续 v0.7 视情况砍 |
| **Topic seed 7 个不够** | 真实学生聊的话题超出 seed | DAO 开放 `addTopic` 给 v0.7 工具自动发现 |
| **Topic stats 在 lastReview 之上还是之下** | 影响 LLM 优先级 | 放在 lastReview 之后（topic 是聚合视角，lastReview 是单 session 视角，前者更宏观） |
| **Pre-built migration 003 跟 v0.5 的 002_session_memory.sql 命名风格不一致** | 看代码的人困惑 | 002 写"002_session_memory"已经定，003 写 "003_topic_stats" 对齐 |
