# v0.7 Tool-Call Protocol — Side-by-Side Comparison

> 用于决定 v0.7 走哪条路。两个方案都"对"，本表只列 trade-off。

---

## TL;DR

| 维度 | A: 自定义文本协议 | B: Anthropic 原生 function calling |
|---|---|---|
| LLMClient 改 | 0 改（只走 text chunk） | 改 ChatOpts/ChatChunk/Result；加 tool 反馈循环 |
| Replay fixture 改 | 0 改（content 是文本里夹 `<tool>`） | 改 fixture schema，加 `tool_use` 字段 |
| 系统提示 | 加一段"调用语法说明"（~10 行） | 0 改（Anthropic 自动注入工具 schema） |
| 单元测试 | 测 parser 正反例 | 测 parser + schema validation + 反序列化 |
| L3 集成 | LLM 输出 → parser → 调 tool → text 续流 | LLM 输出 tool_use → CLI 调 tool → 喂 tool_result → 续 LLM |
| 真实 LLM 验证 | prompt 写不好 LLM 漏写或写错 | 概率高；Anthropic 训练目标之一 |
| 工时 | **~1 天** | **~2-3 天** |
| 长期灵活性 | 想换 provider 不用改协议 | 绑死 Anthropic 的 tool_use 形状 |

**推荐 A**——理由见 §4。

---

## 1. 代码形状对比

### 1.1 A: 自定义文本协议

**System prompt 新增段**（写到 `prompts/AGENTS.md` 或新 prompt）：

```markdown
## Tool Calling

To call a tool, output EXACTLY this block on its own line:

<tool>mark_mistake({"original": "...", "corrected": "...", "category": "grammar|vocabulary|spelling"})</tool>

Do NOT add explanation before/after. After the tool block, you may continue
speaking to the student. Only one tool per turn.
```

**LLM 实际输出**：

```
<tool>mark_mistake({"original":"I go to school yesterday","corrected":"I went to school yesterday","category":"grammar"})</tool>
Good try! I noticed you used "go" but it should be "went" because...
```

**CLI 解析**（cli.ts 内，~30 行）：

```ts
const TOOL_REGEX = /<tool>(\w+)\((\{.*?\})\)<\/tool>/s
const match = TOOL_REGEX.exec(response)
if (match) {
  const name = match[1]
  const args = JSON.parse(match[2])
  const tool = toolRegistry.get(name)
  const result = tool.execute(args)  // 直接 DB 落库
  // LLM 看不到 tool result（同步 tool，不需反馈循环）
  // 把 tool_use 痕迹从 response 里去掉，再 stdout 给学生
  response = response.replace(match[0], '').trim()
}
```

**Replay fixture**（无改动）：

```json
{
  "name": "tool-mistake.json",
  "match": {"userSaysContains": "yesterday"},
  "chunks": [{"type": "text", "delta": "<tool>mark_mistake(...)"},
             {"type": "text", "delta": "</tool> Good try!"}],
  "result": {"content": "<tool>mark_mistake(...)</tool> Good try!"}
}
```

**tool registry**（src/agent/tool-registry.ts，~30 行）：

```ts
export interface Tool {
  name: string
  schema: z.ZodSchema  // 参数校验
  execute(args: unknown): unknown  // 同步执行，DB 落库
}

export class ToolRegistry {
  register(tool: Tool): void
  get(name: string): Tool | undefined
  list(): Tool[]
}
```

**mark_mistake 实现**（~50 行）：

```ts
export function createMarkMistakeTool(db: DbHandle): Tool {
  const mistakes = createMistakesDao(db)
  const schema = z.object({
    original: z.string().min(1),
    corrected: z.string().min(1),
    category: z.enum(['grammar', 'vocabulary', 'spelling']),
  })
  return {
    name: 'mark_mistake',
    schema,
    execute(args) {
      const parsed = schema.parse(args)
      return mistakes.append({ ...parsed, ts: new Date().toISOString() })
    },
  }
}
```

### 1.2 B: Anthropic 原生 function calling

**LLMClient 改**（types.ts，~20 行 + 改 anthropic.ts ~30 行 + testing.ts ~40 行）：

```ts
export interface ChatOpts {
  system: string
  messages: Message[]

}

  | { type: 'thinking'; delta: string }
  | { type: 'tool_use'; id: string; name: string; input: Record<string, unknown> }

  content: string
  toolCalls?: { id: string; name: string; input: Record<string, unknown> }[]
  // ...
}
```

**CLI 多轮循环**（cli.ts，~50 行）：

```ts
async function runTurn(system, history, client) {
  // 1st call
  let result = await client.chat({ system, messages: history })
  while (result.toolCalls && result.toolCalls.length > 0) {
    // 把 assistant tool_use 塞回 history
    history.push({ role: 'assistant', toolCalls: result.toolCalls })
    // 调 tool, 拿结果
    for (const call of result.toolCalls) {
      const tool = registry.get(call.name)
      const output = tool.execute(call.input)
      history.push({ role: 'user', content: '', toolResult: { id: call.id, output } })
    }
    // 2nd call (LLM 看 tool_result 后继续)
    result = await client.chat({ system, messages: history })
  }
  return result.content
}
```

## 2. 测试形状对比

### 2.1 A: 自定义文本协议

**L1 parser 测**（~15 case）：

```ts
// tool-parser.test.ts
- parser('text <tool>f({})</tool> more text') → ('f', {}, 'text  more text')
- parser('text') → null
- parser('<tool>f(invalid json)</tool>') → throws
- parser('<tool>f({})</tool> extra <tool>g({})</tool>') → first only
- parser('no <tool>only open') → null
// etc
```

**L1 tool schema 测**（~6 case）：zod happy/sad path

**L3 cli-integration**（~3 case）：

```ts
- 'yesterday' input → mark_mistake tool 调 → DB 出现 1 行 mistake
- 多 turn 'mistake + follow up' → DB 累计 2 行
- 错例注入 [System Context] 下轮可见（needs Replay fixture 变体）
```

**Replay fixture 总数**：**0 新增**（复用现有 `yesterday` 关键词匹配 + summary 那条 fixture 的形式）

### 2.2 B: 原生 function calling

**L1 parser 测**（~20 case）：同 A + tool_result 序列化 + 多 tool 调用的 history 拼装

**L1 tool schema 测**：同 A

**L1 LLMClient 测**（~10 case）：streaming tool_use chunks 合并；testing.ts fixture schema 反序列化

**L3 cli-integration**（~3 case）：同 A 但验"tool_result feedback loop 跑通"（断言 2 次 LLM call、history 含 tool_use + tool_result）

**Replay fixture 总数**：**+1 新增**（tool_use 变体）+ 可能要 +1 改旧 fixture（Re-exam 流式 chunk 形状）

## 3. 风险对比

### 3.1 A: 自定义文本协议

| 风险 | 触发 | 处置 |
|---|---|---|
| LLM 不按格式输出 | real LLM 漏 `<tool>` 标签 | 强 prompt 约束；fallback：检测到错例 text 但没调 tool，CLI 日志告警 |
| LLM 输出多个 `<tool>` | 一次 LLM 调多 tool | 取第一个；prompt 写"Only one tool per turn" |
| JSON.parse 失败 | LLM 写错格式 | try/catch，stderr 告警，跳过该 turn 的 tool（不阻塞对话） |
| LLM 在 `<tool>` 外说"我要调 mark_mistake..." | 噪声 | 接受；不影响行为（只解析 tag） |

### 3.2 B: 原生 function calling

| 风险 | 触发 | 处置 |
|---|---|---|
| Tool schema 跟 LLM 期望对不上 | 必填字段缺 | zod 提前校验；CLI 喂 LLM 错误信息 |
| LLM 不调 tool | LLM 选不出该用哪个 | prompt 工程；不会崩（fallback 到纯对话） |
| Tool result 超长 | 填入 messages 占用 token | 截断；或 v0.7+ 改成"tool 改 DB，不反馈" |
| 多 tool 链式调用 | 5 turn 后 round-trip 累加 | 设上限（如 max 2 round-trip） |
| Replay fixture 升级 | fixture 写错结构 | L1 fixture parse 测试覆盖 |

---

## 4. 推荐 A 的理由

1. **Reuse first**——`[System Context]` + LLMClient 现有架构完全不动；CLI 现有"每 turn 一次 LLM call"循环不动
2. **单 tool 简洁**——v0.7 范围小（1 tool），A 的实现量 < 1 天，B 要 2-3 天；性价比差异大
3. **fixture 维护成本低**——A 不动 Replay；B 要给 fixture 加 schema 字段 + 写新 fixture
4. **未来可升级**——v0.8+ 如果想换 B，tool-registry 的 Tool interface 不变，只要把"LLM 怎么表达 tool call"那一层切了就行
5. **跟 v0.5/v0.6 的"先 ship 最小可用"风格一致**——v0.5 没上 LanceDB、v0.6 没上 embedding，都是先 ship 简单版

## 5. 不推荐 B 的理由

1. v0.7 范围只有 1 tool，B 的工程量大头（多轮 feedback loop）在 1 tool 场景**用不上**（mark_mistake 是同步 side-effect tool，不需要 LLM 看完 result 后继续生成）
2. B 绑定 Anthropic tool_use 形状，未来想试 Mistral / Qwen 等 provider 的 tool call 要再写一层适配
3. Replay fixture 升级是 L1 测试都要跟着改的"高风险改动"

## 6. 但如果……

- **如果未来要上 `memory_search` 这种"需要 LLM 看完 result 后继续生成"的 tool**——B 更合适
- **如果 v0.7 要上 ≥ 3 个 tool**——B 的工程量摊薄，单 tool 性价比才上来
- **如果用户硬要 LLM 真实调过**——A 的 prompt 依赖训练偏弱，B 在 live 模式概率高

但这些都不在 v0.7 范围。

---

## 7. 我的建议

**v0.7 选 A**。理由 4+5。如果 v0.8+ 上 memory_search，再考虑 B 或 A+B 混合（CLI 解析 + 喂 LLM 文本反馈，绕过 tool_use schema）。
