# Sprint v0.7 Scope — Tool Calling (mark_mistake)

<!-- STATUS: PLANNING -->

> **本文件是 Sprint v0.7 的计划**。详细设计见 [v0.7-design.md](./v0.7-design.md)；协议选型见 [v0.7-protocol-compare.md](./v0.7-protocol-compare.md)。
>
> v0.7 核心：**agent 在对话中能调 mark_mistake 工具记录学生错例**。LLM 在响应里输出 `<tool>mark_mistake({...})</tool>` 文本块，CLI 解析后落 SQLite。同步 side-effect tool，不需要 LLM 看完 result 再续生成。

---

## 1. 背景

v0.5 scope §2.5 把"错例（mistake）记录 + correction 注入"推迟过来。v0.6 拿了 topic 部分。v0.7 拿 mistake 部分（v0.5 deferred 的"记录"部分先做，correction 注入留 v0.7.1）。

口语陪练最值钱的差异化：**让学生感觉"我犯的错被记住了，下次别再犯"**。这要求：
1. 错例有地方存（DB）
2. agent 能在对话中触发存储（tool call）
3. 后续能拿出来用（v0.7.1+ 再做）

v0.7 做 1+2。3 推迟。

### 1.1 协议选型

走 **A: 自定义文本协议**（见 [v0.7-protocol-compare.md](./v0.7-protocol-compare.md)）：

- LLM 在 system prompt 学到 `<tool>name({json})</tool>` 语法
- LLM 响应里夹带这个文本块
- CLI regex 解析 → tool registry → execute → DB 落库
- 同步 side-effect，**不需要** LLM 看完 result 再续生成
- LLMClient 0 改，Replay fixture 0 改

理由：1 tool + sync side-effect 场景下，B（Anthropic function calling）的多轮 feedback 优势用不上；A 的工程量是 B 的 1/3，fixture 0 改动。如果未来上 `memory_search` 这种"需要 LLM 看完 result 续生成"的 tool，再切 B 或 A+B 混合。

## 3. 关键决策

### 3.1 Tool 调用的解析时机

**在 CLI 主循环的"LLM 响应之后、stdout 之前"解析**。一次 LLM call = 一次 tool call（取第一个匹配块），不在 stream 阶段解析（更简单）。

### 3.2 一次 LLM 响应里允许多少个 tool

**最多 1 个**（取第一个匹配）。理由：13 岁口语上下文，1 turn 1 个错例足够；多个 tool 增加 parser 复杂度和 zod 校验工作量。LLM prompt 明确"Only one tool per turn"。

### 3.3 Tool 块是否对学生可见

**不可见**。Parser 解析成功后，把 `<tool>...</tool>` 块从 response 里去掉再 stdout。学生只看到 LLM 解释的话（"Good try! You said 'go' but it should be 'went'..."）。tool 块是 LLM 给 CLI 的"内部信"，不是给学生的话。

### 3.4 Tool 失败时怎么办

**告警 + 跳过，不阻塞对话**。3 个失败点：
1. JSON.parse 失败 → stderr `[cli] tool parse error: <e>`，跳过该 turn tool
2. zod schema 校验失败 → stderr `[cli] tool schema error: <e>`，跳过
3. tool.execute() 抛错（如 DB 故障） → stderr `[cli] tool execute error: <e>`，跳过

LLM 仍然看到完整 response 文本（tool 块已经在 stdout 前从 response 里去掉的话），但 LLM 不知道 tool 没真调——这是 v0.7 的简化。如果 tool 真的必须调成功才有用，下次 v0.7.1 加"tool result 反馈到 LLM"。**当前不重要**：mark_mistake 是 best-effort 记录，调失败只是少了一条错例，不影响对话继续。

### 3.5 Mistake 字段

```
{ id, session_id, original, corrected, category, ts }
```

- `category`: enum `grammar | vocabulary | spelling`（zod 强制）
- `ts`: SQLite 默认 CURRENT_TIMESTAMP
- 不存 explanation（自由文本，模型给的纠错话术由 LLM 在 response 里讲；DB 只存"错在哪"和"对的是什么"）

### 3.6 Tool prompt 写在哪

新文件 `prompts/tools.md`。理由：
- 不污染 `AGENTS.md`（那是 agent 行为守则）
- 独立文件，未来加 tool 只改这一个文件
- prompt-builder 可以选择性加载（v0.7 阶段无条件加载）

## 5. Tool 系统

### 5.1 Tool interface

```ts
// src/agent/tool-registry.ts
export interface Tool {
  name: string
  description: string  // 给 system prompt 用的简短描述
  schema: z.ZodSchema
  execute(args: unknown): unknown  // 同步 side-effect，返任意结果
}

export interface ToolRegistry {
  register(tool: Tool): void
  get(name: string): Tool | undefined
  list(): Tool[]
}
```

### 5.2 Parser

```ts
// src/agent/tool-parser.ts
export interface ParsedToolCall {
  name: string
  args: Record<string, unknown>
  rawMatch: string  // 整段 <tool>...</tool>，用来从 response 删掉
}

const TOOL_REGEX = /<tool>(\w+)\((\{[\s\S]*?\})\)<\/tool>/

export function parseToolCall(response: string): ParsedToolCall | null
export function stripToolCall(response: string, parsed: ParsedToolCall): string
```

`parseToolCall` 行为：
- 没匹配到 → null
- 匹配到但 JSON.parse 失败 → throws（caller 兜底）
- 匹配到但 name 不在 registry → 返 ParsedToolCall（caller 调 registry.get 决定）

### 5.3 mark_mistake

```ts
// src/agent/tools/mark-mistake.ts
export function createMarkMistakeTool(db: DbHandle): Tool {
  const mistakes = createMistakesDao(db)
  const schema = z.object({
    original: z.string().min(1).max(500),
    corrected: z.string().min(1).max(500),
    category: z.enum(['grammar', 'vocabulary', 'spelling']),
  })
  return {
    name: 'mark_mistake',
    description: 'Record a mistake the student just made. Use when the student says something ungrammatical, uses wrong word, or misspells.',
    schema,
    execute(args) {
      const parsed = schema.parse(args)
      return mistakes.append({ sessionId: currentSessionId, ...parsed })  // sessionId 由 factory 注入
    },
  }
}
```

注意：`sessionId` 不是 LLM 传的参数，而是 tool factory 注入的（CLI 启动时 createMarkMistakeTool(db, sessionId)）。这样 LLM 不用关心 sessionId，schema 干净。

## 6. System Prompt 段

写到 `prompts/tools.md`：

```markdown
# Tool Calling

You have one tool: `mark_mistake`. Call it when the student makes a grammar
mistake, uses the wrong word, or misspells. The tool records the mistake
to a database for future reference.

To call a tool, output EXACTLY this block on its own line, BEFORE your
spoken response to the student:

<tool>mark_mistake({"original": "<the student's wrong phrase>", "corrected": "<the corrected version>", "category": "grammar|vocabulary|spelling"})</tool>

After the tool block, you may continue speaking to the student (correcting
them gently, continuing the conversation, etc.). The tool block itself is
NOT shown to the student.

Only call this tool when you are CONFIDENT the student made a mistake
worth recording. Don't call it for stylistic preferences or things
that are debatable.
```

CLI 在 `buildFinalSystem` 之后、send 给 LLM 之前，把这段拼到 system 末尾（与 `prompts/loader.ts` 已有的 loader 模式一致——读 SOUL/AGENTS/USER，v0.7 加一个 readTools()）。

### 7.1 启动

### 7.2 主循环

```ts
// 在 chatStream 拿到 response 之后
let response = ''
for await (const chunk of client.chatStream({...})) {
  if (chunk.type === 'text') {
    process.stdout.write(chunk.delta)  // 还是 stream 给学生
    response += chunk.delta
  }
}

// stream 完了之后，解析 + 调 tool
const parsed = parseToolCall(response)
if (parsed) {
  // 在 stdout 上把 tool 块从 [Teacher]: 行里"擦掉"——
  // 但 stream 已经写了，需要"重写"这一行？
  // v0.7 简化：tool 块通常出现在 response 开头，stream 时会先输出
  // 我们的处理：tool 块不在 stdout 显示，但 stream 已发出去了。
  // 方案 1: stream 阶段不输出，buffer 全部 response 后再处理
  // 方案 2: 接受 tool 块在 stdout 出现一次然后被擦掉（终端会闪一下）
  // 选方案 1，更干净
  // ...
}
```

**等等——v0.7 改 streaming 行为？** 让我看现有 cli.ts 怎么 stream。

看 cli.ts:206-214，stream 阶段直接 stdout.write(chunk.delta)。如果 LLM 流式输出 `<tool>...</tool>`，学生终端会看到这块文本。

**两个方案**：
- **A. 改 stream 行为**：buffer 全部 response，stream 完了再决定哪些输出
- **B. 接受 tool 块闪烁**：让 stream 输出 tool 块，然后 ANSI escape code 删行（脆性）
- **C. 改 LLMClient 返回值**：让 client.chatStream 在末尾标一个 "complete"，CLI 在 complete 之后做 tool 解析 + 重写 stdout

**推荐 A**。理由：
- 干净（学生终端看不到 tool 块）
- 简单（buffer 即可，不增加 LLMClient 类型）
- LLM 响应本来就不长（一次 50-200 token），buffer 没性能问题
- 跟 v0.5/v0.6 "每 turn 一次 LLM call" 模式一致

代码形状：

```ts
let response = ''
for await (const chunk of client.chatStream({...})) {
  if (chunk.type === 'text') {
    response += chunk.delta
  }
}
// stream 完了，处理 tool + 输出
const parsed = parseToolCall(response)
let display = response
if (parsed) {
  display = stripToolCall(response, parsed)
  // 调 tool
  try {
    const tool = registry.get(parsed.name)
    if (tool) {
      tool.execute(parsed.args)
      process.stderr.write(`[cli] tool call: ${parsed.name}(${JSON.stringify(parsed.args)})\n`)
    } else {
      process.stderr.write(`[cli] tool unknown: ${parsed.name}\n`)
    }
  } catch (err) {
    process.stderr.write(`[cli] tool error: ${(err as Error).message}\n`)
  }
}
process.stdout.write(display + '\n\n')
```

**注意**：v0.4-v0.6 现有 cli.ts 是 stream 阶段就 `process.stdout.write(chunk.delta)`。v0.7 改成"buffer 后再输出"是 breaking change for 现有 L3 测试？需要看 L3 测试是否依赖 streaming 的中间状态。

让我 check tests/agent/cli-integration.test.ts —— 之前看的时候它只断言 `result.stdout` 整个字符串，所以 buffer 模式应该 OK。tests/cli/repl.test.ts 同样也是看整体 stdout。Live demo 体验也好（学生感觉不到 tool 块出现）。改 buffer 模式。

## 11. 风险

| 风险 | 影响 | 处置 |
|---|---|---|
| LLM 不按格式输出 | real LLM 漏 `<tool>` | prompt 强约束；L3 走 Replay fixture 验证；live demo 手验 |
| LLM 输出多个 `<tool>` | 多 tool 解析 | 只取第一个；prompt "Only one tool per turn" |
| stream → buffer 改 breaking change | 现有 L3 测试 | 现有测试只看最终 stdout，buffer 模式 OK |
| Tool execute 抛错阻塞对话 | DB 故障 | try/catch + stderr 告警 + 跳过 |
| zod schema 跟 LLM 期望对不上 | 必填字段缺 | zod 严格校验 + 错误信息 stderr |
| Fixture 误匹配（"yesterday" 命中其他 fixture） | Replay 顺序 | 单独文件 mistake-yesterday.json，pickFixture first-match-wins，需要"yesterday" 字段唯一 |
| sessionId 注入到 tool 内部 | tool 不纯，跨 session 不可用 | factory 模式 + sessionId 在 CLI 启动时绑定 |
