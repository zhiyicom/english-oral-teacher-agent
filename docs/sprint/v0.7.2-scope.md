# Sprint v0.7.2 Scope — Vector Store + Relevant-Past-Sessions Injection

<!-- STATUS: PLANNING -->

> **本文件是 Sprint v0.7.2 的计划**。详细设计见 [v0.7.2-design.md](./v0.7.2-design.md)。
>
> v0.7.2 核心：**补 v0.5 砍掉的向量层**——session END 时把 summary 嵌入向量存进 SQLite；新 session 启动时用上一次 session 的关键词做语义检索，把"相关历史 session"注入 `[System Context]`。同时给 v0.7.3 的 `memory_search` tool 铺好底座（embedder + vector-store + retrieve API 三件套）。

---

## 1. 背景

v0.5 原计划落 LanceDB + embedding，实际只做了"上次回顾"（`loadLastReview` 读最近一条 session 摘要，纯 SQL）。结果：

- **跨 session 语义检索缺失** —— PRD §3.4 写明"向量检索 `session_summaries`，query = 选题关键词，top 2"完全没落地
- **v0.7.3 `memory_search` tool 没底座** —— LLM 想搜"上次讲宠物聊到啥"没接口
- **DEVELOPMENT_PLAN §3 v0.5 行的 L2 / L4 测试要求只完成了一半**（向量相关的两条 fail）

v0.7.2 把这条线接上：

1. **写 embedding**：session END summarizer 完成后，把 summary 嵌入 → 写 `sessions.embedding`
2. **读 retrieval**：新 session 启动时，用 `lastReview.keywords.join(' ')` 当 query embed → 跟 DB 所有已嵌入 session 算 cosine → top-2（排除 lastReview 自身）
3. **注入 [System Context]**：新增 "Relevant past sessions" 段（first-turn-only，跟 Last session 段并列）

不做 `memory_search` tool（留 v0.7.3）；不做错例向量化（留 v0.8+）。

## 3. 关键决策

### 3.1 Vector storage backend：SQLite BLOB（**偏离 ARCH §3.4**）

**选 SQLite BLOB + 内存 brute-force cosine**，不用 LanceDB。理由：

| 维度 | LanceDB（ARCH 原方案）| SQLite BLOB（本 sprint 选）|
|---|---|---|
| 新 native binding | 有（重演 better-sqlite3 Windows prebuild 坑）| 0（复用现有 better-sqlite3）|
| 新依赖 | `@lancedb/lancedb` ~30MB | 0 |
| 存储格式 | LanceDB 文件目录 | SQLite 列（BLOB）|
| 数据一致性 | sessions 表 + LanceDB 两套，删 session 要双写 | 单表 FK 自动级联 |
| Scale 上限 | 百万级 | ~1 万 session（你这场景永远撞不到） |
| 检索性能 @ 1000 session | <1ms（索引）| <5ms（brute force, 384 dim × 1000 × cosine）|
| 检索性能 @ 1 万 session | 同 | ~50ms（仍可接受）|
| 跟 v0.7.2 embedding 决策一致性 | 矛盾（你刚为了避免 native binding 选了 transformers.js）| **一致** |

**ARCH §3.4 需要更新**：把 "LanceDB（嵌入式，文件式，零外部依赖）" 改成 "SQLite BLOB + 内存 cosine（嵌入式，零额外依赖；< 1 万 session 性能充足）"。本 sprint 内不改 ARCH，**v0.7.2 commit 时一并加 ARCH 修订**（v0.7.2 收尾时一次提交）。

### 3.2 Embedding 模型

**`Xenova/all-MiniLM-L6-v2`，int8 量化版**：
- 维度 384（PRD §3.3 写的 768 同步更新为 384）
- 量化后 RAM ~120MB，磁盘 ~25MB
- 单次 embed 80-150ms（CPU）
- 不联网（首次启动从 HF CDN 下载，缓存到 `~/.cache/huggingface/`）

### 3.3 Embedder 懒加载

`getExtractor()` 第一次调用时才走 `pipeline('feature-extraction', ...)`。CLI 启动本身不变；只有 session END（embed summary）或新 session 启动（embed query）才真正加载模型。模型加载只发生一次/进程。

### 3.4 Embed 时机

只在 **session END 后** 嵌入（紧跟 summarizer 之后）。**不**在每 turn 嵌入。理由：
- 每 turn 嵌入毫无意义（嵌的是当前 turn 文本，没有用例消费）
- session END 是天然的"一次性写入"点
- 失败可降级：embed 抛错时 log 警告 + 跳过（session 仍正常保存 summary，下次启动 `listWithEmbeddings` 自动排除它）

### 3.5 Backfill 策略：**不做**

v0.5 / v0.6 / v0.7 / v0.7.1 已存在的 session（有 summary 但无 embedding）**不回填**。理由：
- 这些 session 都是 dev 自己跑的 demo，不是真实学生数据
- 加 backfill 脚本会扩大 sprint 范围
- 真实情况下你只关心"从 v0.7.2 开始的新 session 之间能相互检索"
- 如未来需要，单独写 `scripts/backfill-embeddings.ts`（v1.0 打磨时考虑）

### 3.6 Retrieval query：用 `lastReview.keywords.join(' ')`

新 session 启动时的检索查询从哪儿来？**用上一 session 的关键词拼成的字符串当 query**：

```ts
const queryText = lastReview.keywords.join(' ')   // "minecraft castle creeper"
const queryVec = await embedder.embed(queryText)
const relevant = retrieveRelevant({
  queryVec,
  topK: 2,
  excludeSessionId: lastReview.sessionId,   // 上一 session 已经在 "Last session" 段里
})
```

理由：
- 上一 session 关键词是"今天最可能继续聊"的最强信号（你昨天聊了 X，今天大概率还聊 X 相关）
- 排除自身避免 "Last session" 和 "Relevant past sessions" 段重复
- 没有 lastReview（第一个 session）→ 跳过检索，整段不出现
- v0.7.3 加 `memory_search` 后，LLM 可以主动用对话中的关键词检索（query 来源更丰富）

### 3.7 Top-K = 2

跟 PRD §3.4 "top 2 相关历史摘要" 一致。理由：
- token 成本可控（每条 summary ~100 token + 元数据 ~30 token，2 条 ~260 token）
- 多了会稀释——LLM 一次看 5 条相关历史会"无所适从"
- 跟 active topics top 5 / recent mistakes N=5 区分：那两个段是 whole-session 反复看，这个段是 first-turn-only

### 3.8 段格式

```
- Relevant past sessions (N=2):
  - 3 days ago: "We talked about Minecraft castles and creepers..." (keywords: minecraft, castle, creeper)
  - 7 days ago: "Discussed favorite hobbies and weekend activities..." (keywords: hobby, weekend, basketball)
```

理由：
- `N days ago` 给 LLM 时间感（不是 "session_id=abc123"）
- summary 取前 80 字 + `...` 截断（防一条太长占满 context）
- keywords 列出（让 LLM 一眼判断是否相关）
- **不**含 cosine score（LLM 用不上）

### 3.9 [System Context] 段顺序

```
[System Context]
- Phase: ...
- Elapsed: ...
- Silence: ...
- Last transition: ...
- Last session: ...                  (v0.5, first-turn-only)
- Relevant past sessions (N=2):      (v0.7.2 NEW, first-turn-only)
  - ...
- Active topics: ...                 (v0.6, whole-session)
- Recent mistakes (N=5):             (v0.7.1, whole-session)
  - ...
```

Relevant past 跟 Last session 物理相邻，逻辑上"历史块"聚在一起；后跟 Active topics（话题统计）→ Recent mistakes（错例）。这个顺序回归测试稳定。

## 4. 数据模型

### 4.1 Migration 005

```sql
-- 005_session_embeddings.sql
-- v0.7.2: store session.summary embeddings as BLOB for in-memory cosine retrieval
ALTER TABLE sessions ADD COLUMN embedding BLOB;
```

- BLOB 列存 Float32Array（384 floats × 4 bytes = **1536 bytes per session**）
- 100 session × 1.5KB = 150KB（DB 增量微不足道）
- NULL 表示尚未嵌入（lookup 时 WHERE embedding IS NOT NULL 排除）
- 删 session 时 BLOB 跟着行走，无需额外清理

### 4.2 DAO 扩展

`src/storage/sessions.ts`（已有）加 2 方法：

```ts
setEmbedding(sessionId: string, vec: Float32Array): void
listWithEmbeddings(): Array<{
  id: string
  startedAt: string
  summary: string
  keywords: string[]
  embedding: Float32Array
}>
```

## 9. 风险

| 风险 | 影响 | 处置 |
|---|---|---|
| **transformers.js Windows 首次跑模型下载失败** | embedder 抛错，整个 sprint 卡住 | T1 优先做 smoke；失败时 fallback 到手动下模型放 `~/.cache/huggingface/`；最坏切 onnxruntime-node 重新评估（但要重谈 native binding 决策） |
| **transformers.js RAM 占用超预期** | CLI 进程吃 300MB+ | int8 量化 + 懒加载已是最优；如仍超，关掉量化精度损失换 RAM 不值；切更小模型 `Xenova/all-MiniLM-L12-v2` 反更大；保留风险记录到 test-report |
| **PRD §3.4 写的 cross-session "向量补充 top 2 相关"被 LLM 当成"必须引用"** | LLM 每次开场都引用历史，体验差 | prompt 加 "reference only if topically relevant"；live demo 观察 |
| **cosine 在小 session 数下意义不大**（N=2 时几乎啥都返回） | 早期会出现"相关性弱仍被注入"的尴尬 | top-K=2 + 截断 80 字 summary，token 浪费有限；LLM 会自行判断是否引用 |
| **embed 模型加载延迟（2-4s）** | 第一次 session END 慢 | 用户感知不强（session END 本就是写盘 + summarizer LLM call，多 2s 在阴影里）；如真碍事，可在 CLI 启动后 nextTick 异步预热 |
| **Float32Array ↔ Buffer 序列化在 Windows / Linux 上字节序差异** | 跨平台数据库不兼容 | 用 `Buffer.from(f32.buffer)` + `new Float32Array(buf.buffer, buf.byteOffset, 384)`，依赖 V8 native byte order（x64 全是 little-endian，本项目无 ARM）；test-report 加说明 |
| **L3 mock embedder 跟真 embedder 行为脱节** | L3 绿了但 live 跑挂 | L2 跑真 embed 守住 dim + 一致性；live demo (DoD #5) 端到端验证；mock 只作 retrieval 排序断言 |
| **migration 005 在已有 dataDir 上跑（已有 sessions 行）** | ALTER TABLE 在 SQLite 是 atomic，新列默认 NULL | OK；DAO `listWithEmbeddings` 自带 `WHERE embedding IS NOT NULL` 过滤 |
| **embedding 写入失败但 summary 写入成功** | sessions 行有 summary 没 embedding，listWithEmbeddings 自动跳过 | 已是 graceful 行为；log warning + 继续，不阻塞 session END |
| **`@xenova/transformers` license** | Apache-2.0，OK | 无 |
