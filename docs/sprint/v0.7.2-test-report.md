## 6. 架构与代码改动

### 6.1 `src/memory/embedder.ts` 新增

| 元素 | 内容 |
|---|---|
| 接口 | `Embedder { embed(text): Promise<Float32Array>, dim: number }` |
| Real 实现 | `createTransformersEmbedder()`：动态 `import('@huggingface/transformers')` 懒加载 pipeline + 模块级 `extractorPromise` 单例；模型 `Xenova/all-MiniLM-L6-v2` + `dtype: 'q8'`（int8 量化，~25MB） |
| Mock 实现 | `createMockEmbedder({ table?, dim? })`：从 table 取，否则按 text codepoint seed + `Math.sin(seed + i)` 生成确定性 vec（dim 默认 384） |

### 6.2 `src/memory/vector-store.ts` 新增

| 元素 | 内容 |
|---|---|
| `f32ToBuffer(vec: Float32Array): Buffer` | `Buffer.from(vec.buffer, vec.byteOffset, vec.byteLength)`（V8 native byte order，x64 little-endian，与 SQLite BLOB 直接字节兼容） |
| `bufferToF32(buf: Buffer): Float32Array` | `new Float32Array(buf.buffer, buf.byteOffset, buf.length / 4)` |
| `cosineSimilarity(a, b): number` | dot(a,b) / (norm(a) * norm(b))；dim mismatch throw `dim mismatch` |

### 6.3 `src/memory/retrieve-relevant.ts` 新增

| 元素 | 内容 |
|---|---|
| 签名 | `retrieveRelevant({ candidates, queryVec, topK, excludeSessionId?, now? }): RelevantSession[]` |
| 行为 | 1) filter `excludeSessionId`；2) 算 cosine；3) DESC 排序；4) slice topK；5) 加 `daysAgo = (now - startedAt) / 86400_000` |
| 形态 | **纯函数**：不碰 DB，candidates 由调用方从 DAO 拿，test 容易（构造 candidates 数组即可） |

### 6.4 `src/storage/migrations/005_session_embeddings.sql` 新增

```sql
ALTER TABLE sessions ADD COLUMN embedding BLOB;  -- raw Float32Array bytes, 384×4=1536 bytes, NULL=not embedded
```

- 无 INDEX（brute-force cosine 在 1000 × 384 维 < 1ms，索引是负优化）
- NULL = not embedded（listWithEmbeddings 自动 filter）
- 不影响已有列；migration 用 `schema_migrations` 表跟踪（PRAGMA user_version 升到 5）

## 8. 风险与遗留问题

### 8.1 Live demo N=1 vs 设计预期 N=2

设计 §10.1 期望 session C 启动时 `retrieved 2 relevant sessions`。实际跑出 N=1，原因：

- Session C 启动时 `lastReview = B`（最近结束的 session）
- candidates = `listWithEmbeddings()` = `[A, B]`（都已被 embed）
- `retrieveRelevant({ ..., excludeSessionId: B })` → 排除 B → `[A]`
- topK=2 但只剩 1 个候选 → N=1

**这是正确行为**：lastReview 已经在 "Last session" 段注入到 [System Context]，再把它当 Relevant past session 注入会重复（同样的 summary + keywords）。excludeSessionId 是在 design §6.1 明确规定的。

**怎么办**：
- (a) 把 design §10.1 期望改成 N=1，与实际一致
- (b) demo 用 4 个 session（A, B, C, D），D 启动时 lastReview = C，candidates = [A, B, C] → 排除 C → [A, B] → N=2

选 (a)：1 个 live demo 已能验证代码路径（`retrieved 1 relevant sessions` + [System Context] 含 "Relevant past sessions (N=1):"），且与 L3 测试一致。改 design 文档期望即可（已记到 §10 变更记录）。

### 8.2 MiniLM-L6-v2 模型下载 + 缓存位置

| 项 | 内容 |
|---|---|
| 首次下载 | `pnpm install` 后第一次 `embed()` 触发 `pipeline('feature-extraction', ...)` → 从 HuggingFace 下 ~25MB int8 模型 + tokenizer；可能 30-60s（取决于网络） |
| 中国网络 | `HF_ENDPOINT=https://hf-mirror.com` 走镜像（embedder 已实现 `env.remoteHost` 注入） |
| 缓存位置 | `~/.cache/huggingface/hub/`（transformers.js 默认）或 node_modules 附近；本机已缓存，L2 测试 0.87s |
| 进程重启 | 模型文件不重下，但 pipeline 实例是 in-memory singleton，要重新 build ~1-2s |

风险：首次 `embed()` 慢可能让 live demo 看起来"卡住"。已加 stderr `[cli] retrieve relevant failed: ...` 失败降级；如果 demo 等不及，可能要预热（启动后跑 1 次 dummy embed）。

### 8.3 同 text → 同一 vec 的"全等"假设

L1/L2 测试都断言"同 text 同 vec byte-identical"（用 `toBe` 不是 `toBeCloseTo`）。这对 quantized int8 模型 + 同一进程内的 singleton pipeline 是成立的（无随机性、无批处理差异）。**跨进程**也成立（同一模型 + 同一 ONNX 文件 → bit-deterministic）。**跨模型版本**不成立（升级 quantize 精度或换模型后 vec 会变）。已写注释说明。

### 8.4 占位 summary "(summarization failed)" 不嵌入

如果 summarize 步骤失败（LLM error / JSON parse fail），fallback 写 `(summarization failed)` + `keywords: []`。cli.ts END 块跳过这个 summary 的 embed（不污染向量库）。`listWithEmbeddings` 的 `summary IS NOT NULL` filter 也保证不会返回这条（虽然它有 summary 但语义是垃圾）。

### 8.6 序列化 byte order

`Float32Array.buffer` 在 V8 里是 native byte order，x64 little-endian。SQLite BLOB 也是 byte-faithful（无编码）。`f32ToBuffer` / `bufferToF32` 用 `byteOffset` + `byteLength` 直接切，没有 `.slice().buffer`（会复制），也没有 `new Float32Array([...buf])`（会慢）。当前实现是 0-copy 的视图。**跨平台**：未来 ARM Mac 上 V8 仍然是 little-endian（V8 规范）；所以是安全的。已写注释。
