# Sprint v0.7.3 Scope — `memory_search` Tool (A+B Hybrid Protocol)

<!-- STATUS: PLANNING -->

> **本文件是 Sprint v0.7.3 的计划**。详细设计见 [v0.7.3-design.md](./v0.7.3-design.md)。
>
> v0.7.3 核心：在 v0.7.2 已搭好的 embedder + vector-store + retrieve-relevant 底座上，新增一个 LLM mid-session 可主动调用的 tool —— `memory_search(query, top_k)`。跟 startup 注入（v0.7.2，不需 LLM 决策）互补。Tool 协议选 **A+B 混合**：LLM 仍输出 v0.7 的 `<tool>name({json})</tool>` 文本块（A），但 CLI 把 tool result 拼成 synthetic user message 喂回 LLM 再调 1 次（B）—— 1 round-trip，LLM 看 result 后生成连贯回复给学生。

---

## 1. 背景

v0.7.2 加了 startup 时的"相关历史 session"注入：CLI 启动用 lastReview.keywords embed → 跟 DB 所有已嵌 session 算 cosine → top-2 注入 [System Context]。这是**被动**的：CLI 猜你想用 lastReview 关键词搜，搜完就给 LLM。

但有时候 LLM 中途意识到该查历史：
- 学生说 "I played that game last year too" → LLM 想 "等下，这是不是 Minecraft？我去翻下历史"
- 学生说 "你说过的那个语法点" → LLM 想 "我之前讲过吗？查一下"
- 学生说 "还记得我妹妹吗" → LLM 想 "family 话题查一下"

这些 case startup 注入抓不到（关键词不对、或者已经过好几个 turn）。**v0.7.3 让 LLM 主动调 `memory_search`**。

### 1.1 协议选型（user-confirmed）

v0.7 protocol-compare.md §6 已经预测了 v0.7.3 的形态：

> 如果未来要上 `memory_search` 这种"需要 LLM 看完 result 后继续生成"的 tool——B 更合适……如果 v0.8+ 上 memory_search，再考虑 B 或 A+B 混合（CLI 解析 + 喂 LLM 文本反馈，绕过 tool_use schema）。

**A+B 混合的具体做法**：
- LLM 输出仍走 v0.7 的文本协议：`<tool>memory_search({"query":"...","top_k":2})</tool>`（A 形状，0 fixture 改动）
- CLI 解析 → execute（embed + retrieve，0 外部 LLM call）→ 拿 result
- CLI 把 result 拼成 synthetic user message 推回 history：`[tool_result_v073]\n[formatted result]`
- CLI 调 `client.chat()` 第 2 次（非流式，匹配 summarize() 现有形状）
- LLM 看 result 后生成连贯回复
- CLI 把 2nd-call 的回复 stdout 给学生

**为什么不是 B（Anthropic 原生 function calling）**：
- B 要改 LLMClient（types + anthropic + testing 全动）+ 写多轮 feedback loop + fixture schema 升级
- A+B 复用 v0.7 整套 fixture / LLMClient / parser / registry
- A+B 1 round-trip 对话更连贯（LLM 看完 result 后给连贯回复）
- 1 round-trip 上限明确：2nd-call 不能再调 tool（无递归，安全性可证）

**为什么不是 A 纯文本（v0.7 现行）**：
- A 纯文本适合 sync side-effect tool（mark_mistake 写 DB，LLM 不需要看 result）
- memory_search 是信息检索 tool，LLM 必看 result 才能回复
- A 纯文本下 LLM 搜了等于白搜（CLI execute 后 LLM 不知道搜到啥）

详见 [v0.7.3-design.md §2](./v0.7.3-design.md)。

### 2.2 Out（本次不做）

| # | 项 | 推迟到 |
|---|---|---|
| ❌ | 切到 B（Anthropic 原生 function calling）| v0.8+（如果上 ≥ 3 tool 或 LLM 真实调过不够稳） |
| ❌ | 多 tool 链式调用（memory_search + mark_mistake 同 turn / memory_search → memory_search）| v0.8+（parser 改造）|
| ❌ | Streaming 2nd LLM call | v0.7.3 用 `client.chat` 简化；streaming 加 50 行不值得 |
| ❌ | Tool result 截断 / context budget for tool results | topK=2 × ~100 token = 200 token/search，3 calls/session = 600 token，30-min 上限内 OK |
| ❌ | 异步 / 后台 tool 执行 | 无场景；sync + await 即可 |
| ❌ | mark_mistake 协议改（A → A+B）| mark_mistake 是 sync side-effect，不需要 feedback |
| ❌ | 扩 `Message` type 加 `tool_result` role | synthetic user message 0 type 改动 |
| ❌ | 错例 vector retrieval | v0.8+（用同一 embedder + vector-store 底座）|
| ❌ | `memory_search` 结果跨调用缓存（同一 session 重复查相同 query）| 每次 re-embed，缓存后 <100ms，可接受 |
| ❌ | Tool result 持久化到 messages 表 | 当前不写；学生看不到 tool 过程，只看最终回复；够用 |
| ❌ | 跨 session 累积 tool result 索引（让 memory_search 也能搜到当前 session 已经搜过的内容）| 无场景；当前 session 的 tool result 在 history 里 LLM 自己看 |

```
$ # Pre-req: 至少 1 个已结束 session（v0.7.2 写过 embedding），才能 retrieveRelevant 出非空
$ # 否则 memory_search 返空，2nd-call 仍跑（fixture 通用），但 stdout 文本不同

$   node --import tsx src/cli.ts
$ # Session B: 触发 memory_search
$ printf 'what did i play\nexit\n' | APP_DATA_DIR=./data/live-demo-v073 MINIMAX_API_KEY=sk-test \
$   node --import tsx src/cli.ts
$ # Expect:
$ #   1. stderr 含 "[cli] tool call: memory_search({"query":"...","top_k":2})"
$ #   2. stderr 含 "[cli] tool 2nd-call: memory_search(top_k=2)"
$ #   3. stdout 是 2nd-call fixture text ("I remember you mentioned Minecraft last week! ...")
$ #      NOT 1st-call "Let me check your past sessions..."（1st-call 的 tool 块 strip 后剩这点）
```

## 7. 风险

| 风险 | 触发 | 处置 |
|---|---|---|
| **1 round-trip 加 token** | 每次 memory_search 加 ~200 token result + 2nd-call 完整回复 | topK=2 × ~100 token 可控；3 calls/session = 600 token，30-min 上限内 |
| **2nd-call LLM 又调 tool** | 2nd-call response 含 `<tool>` 块 | prompt 写"do not call any tool in second response" + cli strip safety pass（只 strip，不 execute）|
| **Synthetic user message 污染 history** | `[tool_result_v073]` marker 进入 LLM context | marker 是设计意图，LLM 看到不影响（学 Claude 的 function_calling 训练数据有类似模式）|
| **Fixture match 误中** | 2nd-call fixture 的 `userSaysContains: "[tool_result_v073]"` 跟某 turn 用户输入撞 | marker 唯一，user 不会输入；regex 严格 substring 匹配 |
| **memory_search result 0 时 LLM 行为奇怪** | DB 无 embedding | result text 写 "No relevant past sessions found"；LLM 据此回应（live demo 验证）|
| **embedder 每次 call 都要嵌 query** | 同一 query 重复调 1 session 内 | 缓存后 <100ms；当前不缓存（v0.7.3 范围外）|
