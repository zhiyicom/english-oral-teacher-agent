## 11. 风险与遗留

### 11.1 1 round-trip 加 token 成本

每次 memory_search 加 ~200 token result + 2nd-call 完整回复。3 calls/session = 600 token，30-min 上限内 OK。如果学生高频调 memory_search，token 累计会到 ~1000/turn。**v0.8+ 应加 tool result 截断**（目前 80-char summary 截断，但 topK=5 时仍可能超）。

### 11.2 2nd-call LLM 又调 tool

2nd-call response 含 `<tool>` 块时，cli.ts 的 safety strip 只 strip 不 execute。Prompt 显式说 "do not call any tool in second response" + 设计上限 = 1 round-trip，不会递归。**风险**：safety strip 是兜底，prompt 是主防线。Live demo 验证 2nd-call fixture 故意不含 tool 块。

### 11.3 Synthetic user message 污染 history

`[tool_result_v073]\n[v073_followup_responder]\n[Tool result: memory_search]\n...` 进入 LLM context。**实测**：2nd-call LLM 把 marker 视为系统信号（学 Claude function_calling 训练数据有类似模式），不污染回复生成。Live demo 验证。

### 11.4 Fixture match 误中（已实测发现）

最初设计 2nd-call fixture needle = `[tool_result_v073]`，但 live demo 发现：retrieved summary 含 "castle" 关键词 → castle fixture 优先匹配（alphabetical）→ 2nd-call 错误返回 castle 文本。**修复**：
1. 加更独特的 marker `[v073_followup_responder]`
2. 2 个 v0.7.3 fixture 改名为 `0-memory-search-input.json` / `0-memory-search-followup.json`（`0-` 前缀让它们 sort 在 `castle.json` 之前）

**教训**：retrieved content 是不受控的自然语言，fixture 排序 + unique tag 双保险必须。v0.8+ 若加更多 tool，要重审。

### 11.5 v0.5 测试断言收紧

`empty library: first session stderr has NO "Last session" string` 改为 `not.toMatch(/- Last session \(/)`（`context-injector.ts:54` 渲染段头的精确 pattern）。原断言太松，被 v0.7.2 的 ctx-block dump + v0.7.3 新加的 tools.md 文字双重误伤。**prod 行为 0 变化**（empty library 下 lastReview=null，渲染段依然不出现）；只测 test 自身。

### 11.6 Live demo 用了 "earlier session" 不用 "what did i play"

scope §5.1 写的是 "what did i play\nexit"，但这字符串包含 "played" → hobby-played.json 会先匹配 → 1st-call 不会调 memory_search。**改为 "earlier session"**（更清晰、不与任何现有 fixture 撞）。L3 测试和 live demo 都用这个。

### 11.7 A+B 协议在真 LLM 下的稳定性

Replay fixture 100% 按预期返回。**未实测真实 LLM API**（Anthropic SDK 已注册但 RUN_LIVE_LLM=1 路径未跑）。如果真 LLM 在 2nd-call 经常调 tool，prompt 已写明禁止 + safety strip 兜底；如若真出问题，v0.8+ 切到 B（Anthropic native function calling）是已计划的下一步。
