# Sprint v0.7.4 Design — Live LLM Validation Script

> 本设计是 [v0.7.4-scope.md](./v0.7.4-scope.md) 的细化。范围 / In/Out / DoD 见 scope；本文是实现细节。

---

## 1. 总览

`scripts/live-validate.mjs` 是 1 个 .mjs Node 脚本，做以下事：

1. 检查 `process.env.MINIMAX_API_KEY`（必须由调用方 export，脚本不读 .env contents）
2. 创建 fresh `data/live-validation/<ISO-timestamp>/` 目录
3. 串行跑 5 个 scenario（baseline → mark_mistake → dedup → cross-session retrieval → memory_search A+B）
4. 每个 scenario 启 1（或 2）个独立 CLI 子进程（spawn `tsx src/cli.ts`），喂预设 stdin，写 stdout/stderr 到 `data/live-validation/<ts>/scenario-N/{stdout,stderr}.log`，附加 meta.json
5. 跑完打 1 份 summary（console + `data/live-validation/<ts>/summary.json`）
6. 退出码：0（跑完） / 1（启动 fail / scenario 内部 fail 一次就 bail）

**不是自动化断言**——脚本不判断 pass/fail。Pass/fail 由人读 transcript 后写到 `v0.7.4-validation-report.md`。

## 3. Script 实现

### 3.1 文件结构

```
scripts/
  live-validate.mjs       ← 本脚本（new）
  live-smoke.mjs          ← v0.2 已有（不动）
```

`live-validate.mjs` 用 ESM（"type": "module"），跟 `live-smoke.mjs` 风格一致。**不**加 typecheck（.mjs 不进 tsconfig.test.json）；脚本是开发者工具，不是产品代码。

### 3.2 完整 script（约 200-300 行）

```js
#!/usr/bin/env node
// scripts/live-validate.mjs
// v0.7.4 — live LLM smoke for v0.7 / v0.7.1 / v0.7.2 / v0.7.3.
// Requires: MINIMAX_API_KEY in env. Does NOT read .env (operator provides).
// Outputs: data/live-validation/<ISO-timestamp>/{scenario-N,summary.json}

import { spawn } from 'node:child_process'
import { mkdirSync, writeFileSync, existsSync } from 'node:fs'
import { resolve, join } from 'node:path'
import { fileURLToPath } from 'node:url'
import { execSync } from 'node:child_process'

const __filename = fileURLToPath(import.meta.url)
const ROOT = resolve(__filename, '..', '..')

// ---- Pre-flight checks ----
if (!process.env.MINIMAX_API_KEY) {
  console.error('[live-validate] MINIMAX_API_KEY is not set.')
  console.error('  Run as:  MINIMAX_API_KEY=sk-... node scripts/live-validate.mjs')
  process.exit(1)
}
if (!existsSync(resolve(ROOT, 'node_modules'))) {
  console.error('[live-validate] node_modules not found. Run `pnpm install` first.')
  process.exit(1)
}

// ---- Setup output dir ----
const ts = new Date().toISOString().replace(/[:.]/g, '-')
const outRoot = resolve(ROOT, 'data', 'live-validation', ts)
mkdirSync(outRoot, { recursive: true })
console.log(`[live-validate] output: ${outRoot}`)

// ---- Per-scenario runner ----
async function runProcess({ label, scenarioDir, inputs, timeoutMs = 90_000 }) {
  return new Promise((res) => {
    const proc = spawn(
      process.execPath,
      ['--import', 'tsx', resolve(ROOT, 'src/cli.ts')],
      {
        cwd: ROOT,
        env: {
          ...process.env,
          APP_DATA_DIR: scenarioDir,  // each scenario uses its own data dir
          HF_ENDPOINT: 'https://hf-mirror.com',
        },
        stdio: ['pipe', 'pipe', 'pipe'],
      },
    )
    let stdout = ''
    let stderr = ''
    proc.stdout.on('data', (d) => { stdout += d.toString() })
    proc.stderr.on('data', (d) => { stderr += d.toString() })

    // Write inputs with 4s spacing (LLM calls are 1-5s/turn)
    let i = 0
    const writeNext = () => {
      if (i >= inputs.length) {
        setTimeout(() => proc.stdin.end(), 1000)
        return
      }
      proc.stdin.write(`${inputs[i]}\n`)
      i += 1
      setTimeout(writeNext, 4000)
    }
    setTimeout(writeNext, 1000)  // let CLI print banner first

    const killer = setTimeout(() => {
      console.error(`[live-validate] ${label} timed out after ${timeoutMs}ms, killing`)
      proc.kill('SIGTERM')
    }, timeoutMs)

    proc.on('close', (code) => {
      clearTimeout(killer)
      res({ code, stdout, stderr })
    })
  })
}

async function scenario(label, inputs, opts = {}) {
  const { reuseDataDir = null, subProcesses = 1, perProcessTimeout = 90_000 } = opts
  const dir = reuseDataDir ?? join(outRoot, label)
  mkdirSync(dir, { recursive: true })

  const meta = { label, startedAt: new Date().toISOString(), processes: [] }

  for (let p = 0; p < subProcesses; p++) {
    const procDir = join(dir, `process-${p}`)
    mkdirSync(procDir, { recursive: true })
    const { code, stdout, stderr } = await runProcess({
      label: `${label}/process-${p}`,
      scenarioDir: procDir,  // each process gets its OWN data dir
      inputs: inputs[p] ?? inputs[0],
      timeoutMs: perProcessTimeout,
    })
    writeFileSync(join(procDir, 'stdout.log'), stdout)
    writeFileSync(join(procDir, 'stderr.log'), stderr)
    meta.processes.push({
      process: p,
      exitCode: code,
      stdoutBytes: stdout.length,
      stderrBytes: stderr.length,
    })
    console.log(`  [${label}/p${p}] exit=${code} stdout=${stdout.length}B stderr=${stderr.length}B`)
  }

  meta.finishedAt = new Date().toISOString()
  writeFileSync(join(dir, 'meta.json'), JSON.stringify(meta, null, 2))
  return meta
}

// ---- 5 scenarios (run serially; each spawns 1 or 2 processes) ----
console.log('[live-validate] running 5 scenarios…')

const s1 = await scenario('1-baseline',
  [['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit']])

const s2 = await scenario('2-mark-mistake',
  [['hi', 'i go to school yesterday', 'i am fine thanks', 'exit']])

const s3 = await scenario('3-dedup',
  [['hi', 'i go to school yesterday', 'i go to school yesterday', 'exit']])

// S4: 2 processes share data dir so process-B sees process-A's embedding
const s4Dir = join(outRoot, '4-cross-session-retrieval')
mkdirSync(s4Dir, { recursive: true })
const s4 = await scenario('4-cross-session', [
  ['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit'],  // A: same as S1
  ['hi', 'fine thanks', 'exit'],                                               // B: short, see ctx
], { reuseDataDir: s4Dir, subProcesses: 2 })
// ↑ S4 process-A writes to s4Dir/process-0/; S4 process-B writes to s4Dir/process-1/
// Wait — this won't share data! Need to set APP_DATA_DIR to the SAME dir for both.
// ↑ REDESIGN: instead of `reuseDataDir`, set APP_DATA_DIR=s4Dir for BOTH processes
// (covered in revised version below)

const s5 = await scenario('5-memory-search', [
  ['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit'],  // A: same as S1
  ['earlier session', 'exit'],                                                  // B: trigger memory_search
], { subProcesses: 2 })
// ↑ same data-sharing problem as S4

// ---- Summary ----
const summary = {
  timestamp: ts,
  model: process.env.LLM_MODEL_MAIN ?? 'MiniMax-M3',
  scenarios: [s1, s2, s3, s4, s5],
}
writeFileSync(join(outRoot, 'summary.json'), JSON.stringify(summary, null, 2))
console.log('\n[live-validate] DONE')
console.log(`[live-validate] transcripts: ${outRoot}`)
console.log(`[live-validate] summary:      ${outRoot}/summary.json`)
console.log('\nNext:')
console.log('  1. Read each scenario-N/process-X/{stdout,stderr}.log')
console.log('  2. Write your verdicts in docs/sprint/v0.7.4-validation-report.md')
process.exit(0)
```

### 3.3 Scenario 4/5 共享 data dir 的修正

上面 `scenario()` 函数 `scenarioDir` 是 per-process 的，但 S4/S5 需要 2 个 process 写**同一个** data dir（process-A 写 summary+embedding，process-B 启动时读 lastReview）。

修正：给 `scenario()` 加一个 `sharedDataDir` 选项，让 N 个 process 都用 `APP_DATA_DIR=<sharedDataDir>`。但 transcript 还是写各自目录。

```js
async function scenario(label, inputs, opts = {}) {
  const { sharedDataDir = null, subProcesses = 1, perProcessTimeout = 90_000 } = opts
  const dir = join(outRoot, label)  // transcript root
  mkdirSync(dir, { recursive: true })
  const dataDir = sharedDataDir ?? join(dir, '_data')  // all processes share
  mkdirSync(dataDir, { recursive: true })

  const meta = { label, startedAt: new Date().toISOString(), processes: [] }

  for (let p = 0; p < subProcesses; p++) {
    const procDir = join(dir, `process-${p}`)
    mkdirSync(procDir, { recursive: true })
    const inputsForP = inputs[p] ?? inputs[0]
    const { code, stdout, stderr } = await runProcess({
      label: `${label}/process-${p}`,
      scenarioDir: dataDir,  // ← shared
      inputs: inputsForP,
      timeoutMs: perProcessTimeout,
    })
    writeFileSync(join(procDir, 'stdout.log'), stdout)
    writeFileSync(join(procDir, 'stderr.log'), stderr)
    meta.processes.push({ process: p, exitCode: code, stdoutBytes: stdout.length, stderrBytes: stderr.length })
  }
  // ...
}
```

调用：
```js
const s4 = await scenario('4-cross-session', [
  ['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit'],
  ['hi', 'fine thanks', 'exit'],
], { sharedDataDir: join(outRoot, '4-cross-session', '_data'), subProcesses: 2 })

const s5 = await scenario('5-memory-search', [
  ['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit'],
  ['earlier session', 'exit'],
], { sharedDataDir: join(outRoot, '5-memory-search', '_data'), subProcesses: 2 })
```

### 3.4 Input 节奏

- 每个 turn 间 4s 间隔：M3 一次响应 1-5s；4s 是经验值
- Banner 出来后 1s 再开始写第 1 行（避免 stdin buffer 还没 ready）
- 最后 1 行 `exit` 写完后 1s 再 `stdin.end()`，让 CLI 处理完 exit
- 每个 process 设 90s timeout（覆盖 8-10 轮对话 + summarizer call + embed call）
- Scenario 1/2/3 是 1 process；Scenario 4/5 是 2 process，每个独立 timeout

## 5. 5 个 Scenario 详细输入

### 5.1 Scenario 1: Baseline

| Process | Inputs | Lines | 期望 wall time |
|---|---|---|---|
| 0 | `hi` / `i played minecraft yesterday` / `i usually build castles` / `exit` | 4 | ~30s (3 turns + summarize + embed) |

**Pass 信号**（人眼检 stdout/stderr）：
- ✅ stdout 含 3 段 `[Teacher]: ...`
- ✅ stderr 含 `[cli] ctx: WARM_UP` (第 1 轮)
- ✅ stderr 含 `[cli] summarize ok summary=...c keywords=...`
- ✅ stderr 含 `[cli] markEnded done`
- ✅ stderr 含 `[cli] embedded session.summary (384 dim, 1536 bytes)`
- ✅ stderr 含 `[cli] topic match: ...`（minecraft / 任何种子）
- ✅ 进程 exit 0
- ✅ Teacher 文本连贯（greeting / follow-up / wrap-up）

### 5.2 Scenario 2: mark_mistake

| Process | Inputs | Lines | 期望 wall time |
|---|---|---|---|
| 0 | `hi` / `i go to school yesterday` / `i am fine thanks` / `exit` | 4 | ~30s |

**Pass 信号**：
- ✅ stderr 含 `[cli] tool call: mark_mistake({"original":"...","corrected":"...","category":"grammar"})`
- ✅ DB 中有 1 row mistakes（脚本可选：跑完 SELECT COUNT 写到 meta.json）
- ✅ stdout 不含 `<tool>` 或 `</tool>`
- ✅ Teacher 文本自然（含 "went to school" 这种 rephrase 形态）

**可能 fail 形态**：
- ❌ M3 不调 tool（`[cli] tool call` 不出现）→ Issue: "M3 doesn't emit text tool blocks"
- ❌ M3 调了但 schema 错（`[cli] tool execute error: ...`）→ Issue: "M3 tool call schema deviates"
- ❌ M3 调对了但 stdout 包含 tool 块（`[cli] tool parse error` + `<tool>` leak）→ Issue: "strip pass not working"

### 5.3 Scenario 3: Dedup

| Process | Inputs | Lines | 期望 wall time |
|---|---|---|---|
| 0 | `hi` / `i go to school yesterday` / `i go to school yesterday` / `exit` | 4 | ~30s |

**Pass 信号**：
- ✅ stderr 含 1 个 `[cli] tool call: mark_mistake(...)`（第 2 轮首次）
- ✅ stderr 含 1 个 `[cli] tool dedup: skipped (already marked: "...")`（第 3 轮重复）
- ✅ DB 只有 1 row mistakes

**可能 fail 形态**：
- ❌ 2 个 turn 都调了 tool（dedup 没拦）→ Issue: "dedup 失效"
- ❌ 2 个 turn 都没调 tool → Issue: "M3 不愿标第 1 次"
- ❌ dedup log 出现但 DB 有 2 row → Issue: "dedup 拦 CLI 没拦 DB"

### 5.4 Scenario 4: Cross-session retrieval

| Process | Inputs | Lines | 期望 wall time |
|---|---|---|---|
| 0 (A) | `hi` / `i played minecraft yesterday` / `i usually build castles` / `exit` | 4 | ~30s |
| 1 (B) | `hi` / `fine thanks` / `exit` | 3 | ~25s |

共享 `_data/`：A 写 1 row session + embedding；B 启动时检索。

**Pass 信号**：
- ✅ process-0 stderr 含 `[cli] embedded session.summary`
- ✅ process-1 stderr 含 `[cli] retrieved 1 relevant sessions (query="minecraft ...")`
- ✅ process-1 stderr 含 `[cli] ctx-block:` dump，且 dump 内有 `- Relevant past sessions (N=1):` 段
- ✅ process-1 Teacher 文本能用上 retrieved context（"welcome back!" / 引用 minecraft 主题）

**可能 fail 形态**：
- ❌ retrieved 0（lastReview 没被找到 / embedding 没写好）→ Issue: "v0.7.2 嵌入路径 live 出错"
- ❌ retrieved 1 但 LLM 没说啥相关 → 不算 fail（LLM 可能选择不提），但记 note
- ❌ ctx-block dump 没 Relevant past 段（render bug）→ Issue: "relevantPast 没传进 buildFinalSystem"

### 5.5 Scenario 5: memory_search A+B

| Process | Inputs | Lines | 期望 wall time |
|---|---|---|---|
| 0 (A) | `hi` / `i played minecraft yesterday` / `i usually build castles` / `exit` | 4 | ~30s |
| 1 (B) | `earlier session` / `exit` | 2 | ~30s |

共享 `_data/`：A 写 1 row + embedding；B 用 lastReview 启动 + LLM 调 memory_search。

**Pass 信号**：
- ✅ process-1 stderr 含 `[cli] tool call: memory_search({"query":"...","top_k":...})`
- ✅ process-1 stderr 含 `[cli] tool 2nd-call: memory_search(top_k=...)`
- ✅ process-1 stdout 是连贯英语（含 "Minecraft" 之类的 retrieved 内容）
- ✅ process-1 stdout 不含 `<tool>` / `</tool>`

**可能 fail 形态**：
- ❌ M3 不调 memory_search（"earlier session" 没触发）→ Issue: "M3 不主动调 memory_search"
- ❌ M3 调了但 1st-call 工具块后没 2nd-call（CLI 没识别）→ Issue: "A+B protocol live broken"
- ❌ 2nd-call 后 stdout 仍含 `<tool>`（safety strip 失效）→ Issue: "safety strip 不工作"
- ❌ 2nd-call M3 又调 tool（[cli] tool execute error 出现 2 次）→ Issue: "M3 不遵守 2nd-turn-no-tool prompt"

## 8. 风险

| 风险 | 触发 | 处置 |
|---|---|---|
| **M3 latency 慢** | 单 turn > 30s；90s timeout 不够 | 调大 timeout；记 issue |
| **M3 quota 限速** | 5 scenarios × 8 turns = ~40 LLM calls；可能触发 rate limit | 跑间 1s 间隔；fail 时 retry 0 次；记 issue 给用户 |
| **MINIMAX_API_KEY 无效** | 401 from endpoint | 脚本启动后第 1 个 scenario fail；用户看 stderr 知道 |
