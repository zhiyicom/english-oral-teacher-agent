## V741-001 — Scenario 4 cannot verify v0.7.2 startup retrieval with N=1 prior session

| Field | Value |
|---|---|
| **Sprint link** | v0.7.4 (validation script design); v0.7.2 (behavior under test) |
| **Severity** | Low (no product impact; validation coverage gap) |
| **Discovered by** | Live validation run, 2026-06-10 14:28-14:29 (operator) |
| **Status** | Open — backlog |
| **Repro** | `MINIMAX_API_KEY=<key> node scripts/live-validate.mjs`. In scenario 4, process 0 writes 1 session to a fresh `_data/`, then process 1 starts. Stderr shows: `[cli] retrieved 0 relevant sessions (query="minecraft castle creeper wall build")` even though process 0's session exists with a 1536B embedding (confirmed via `SELECT id, length(embedding) FROM sessions`). |
| **Root cause** | v0.7.2 startup retrieval at `src/cli.ts:228-233` calls `retrieveRelevant({...excludeSessionId: lastReview.sessionId})`. With only 1 prior session, the only candidate is the excluded one → result is always `[]`. This is **intended design** (don't duplicate what the "Last session" block already shows), but the validation scenario doesn't exercise the "≥2 prior sessions" case where retrieval would actually return data. |
| **Workaround (current)** | None needed. v0.7.2 retrieval is exercised end-to-end by scenario 5's `memory_search` tool call (which uses the **same** `listWithEmbeddings` + `retrieveRelevant` primitives, but **without** `excludeSessionId`). Teacher response `"I remember you mentioned Minecraft last week!"` confirms the embedding + cosine path works. |
| **Suggested next sprint** | One of:<br>**A.** Have scenario 4 process 0 do 2 sessions, e.g. `['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit', 'hi', 'i also like roblox', 'exit']`. Then process 1's `lastReview` is the roblox session, and the minecraft session becomes a non-excluded candidate. Cheapest fix.<br>**B.** Seed the scenario's `_data/` from a pre-built fixture (e.g. 3 sessions with embeddings) before spawning process 0/1. Most realistic but requires a fixture file.<br>**C.** Accept the 0 result with N=1 and rely on scenario 5's `memory_search` as the de-facto retrieval verification. Zero code change but loses scenario 4's coverage. |
| **Out of scope this sprint** | Per v0.7.4-scope "0 产品代码改动" and "记录到 backlog" — no fix. |
