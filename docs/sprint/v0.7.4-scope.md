> v0.7.4 核心：在 v0.7.3 已落地的代码底座上，把 v0.7/v0.7.1/v0.7.2/v0.7.3 各 sprint 引入的 LLM-driven 行为在真 API（`RUN_LIVE_LLM=1` 路径）下**走一遍 smoke**。本 sprint **不加新功能、不修产品代码**——只验证 live 路径在每个新增特性上能跑通，发现的问题记录到 backlog（变下个 sprint 候选）。

## 1. 背景

`src/llm/anthropic.ts` 和 `src/cli.ts:46-56 selectClient()` 的 live 路径在 v0.2 sprint 期间跑过 1 次 6-轮 smoke（见 [v0.2-test-report §7.1](./v0.2-test-report.md)），用真实 `MINIMAX_API_KEY` 喂简单对话，persona 一致、6 轮全通。

但 v0.3 之后引入的所有 LLM-driven 行为**从未在真 LLM 下验证过**：

| Sprint | 新增 LLM-driven 行为 | 上次 live 验证 |
|---|---|---|
| v0.3 持久化 | （无新增 LLM 行为，0 验证需求）| — |
| v0.4 状态机 | （无新增 LLM 行为）| — |
| v0.5 记忆 | summarizer LLM call（END 时独立生成 summary + keywords）| ❌ |
| v0.6 话题 | （无新增 LLM 行为，topic 匹配是 deterministic）| — |
| v0.7 工具 | LLM 输出 `<tool>mark_mistake({...})</tool>` 文本块 + 解析 + 写 DB | ❌ |
| v0.7.1 错例注入 + dedup | 注入最近 5 错例到 [System Context]；同 original 二次调标 dedup | ❌ |
| v0.7.2 语义检索 | 启动期 lastReview.keywords 嵌入 + cosine + top-K 注入 | ❌ |
| v0.7.3 memory_search | LLM 输出 `<tool>memory_search({...})</tool>` + 异步执行 + 1 round-trip A+B | ❌ |

到 v0.7.3 末，**所有 LLM-driven 行为都只在 Replay fixture 下验证**。Replay 是确定性的（同一个输入永远返回同一个 fixture），不能反映真 LLM 的实际行为。Live 验证可以暴露：
- Real LLM 是否真的输出 `<tool>...</tool>` 文本块（vs 走 function calling 走 JSON 路径）
- Real LLM 是否遵守 zod schema（category ∈ {grammar, vocabulary, spelling}; query 1-200 chars; top_k 1-5）
- Real LLM 是否遵守 persona（不长篇 lecture / 不承认 AI / 不 quiz）
- Real LLM 在 2nd-call 是否会**违反** "do not call any tool in second response" 约束（safety strip 是否够用）
- Real LLM 是否对 [System Context] 里的 lastReview / relevantPast / activeTopics / recentMistakes 段有合理响应
- Real LLM 跑 summarizer 时是否输出合法 JSON

### 1.1 为什么 v0.7.4 必做

- v0.8 路线图是 Web UI（[DEVELOPMENT_PLAN §3](./../DEVELOPMENT_PLAN.md)）。Web UI 之后所有"用真 LLM"行为都会接触到学生/家长；上线前必须有 1 次真 LLM 烟测
- 3 个 sprint 累积的 LLM-driven 行为（tool calling / dedup / semantic retrieval / A+B protocol）的实际兼容性是 **未知数**
- Replay fixture 是手写样本，可能漏掉真 LLM 实际会做的边缘 case（比如 schema 偏差、prompt 误读）
- v0.7.3 A+B 协议的"1 round-trip"假设依赖 prompt 的"Do not call any tool in second response"——真 LLM 是否遵守，**没测过**

### 1.2 为什么 v0.7.4 是验证 sprint 而非修复 sprint

- 单次 live smoke 暴露的问题可能跨多个 sprint（prompt / schema / fixture）
- 修复可能影响 Replay 测试（fixture 改 / 断言改 / 行为改）→ 牵连太多
- 维护原则（[DEVELOPMENT_PLAN §1.7](./../DEVELOPMENT_PLAN.md)）：sprint 独立可演示 + 测试同步增长。在 v0.7.4 里"演示 1 次真 LLM 跑通"是清晰边界，修复是新 sprint 的事
- Issue → backlog（[[project-sprint-status]] §"v0.7.x deferred" 块）→ 下个 sprint 处理

## 3. 5 个 Scenario 详情

每个 scenario 是 1 个（或 2 个，看跨 session 需求）CLI 进程跑完整次对话。Transcript 存到独立目录。Pass/fail 由真人读 transcript 判定（脚本只负责"跑完"，不负责"判断对错"）。

### 3.1 Scenario 1: Baseline（v0.2-v0.6 路径）

**目的**：验证基础 persona + summarizer + embedding（v0.7.2 写入路径）。

**Setup**：空 `data/live-validation/<timestamp>/` data dir。

**输入序列**（1 进程 1 session）：
```
hi
i played minecraft yesterday
i usually build castles
exit
```

**期望 stderr marker**（人眼检查）：
- `[cli] ctx: WARM_UP` (v0.4)
- `[cli] summarize ok summary=...c keywords=...` (v0.5)
- `[cli] markEnded done` (v0.5)
- `[cli] embedded session.summary (384 dim, 1536 bytes)` (v0.7.2)
- `[cli] topic match: ...` (v0.6)

**Pass 标准**：
- CLI 退出 0
- 4 轮对话 + 1 段 summary + embedding
- Teacher 文本像人话（greeting 友好、follow-up 问题合理、persona 保持）
- 无 `Error` / 无 `tool parse error`

### 3.2 Scenario 2: mark_mistake（v0.7）

**目的**：验证 LLM 真的输出 `<tool>mark_mistake({...})</tool>` 块，且 schema 合规。

**Setup**：空 data dir。

**输入序列**（1 进程）：
```
hi
i go to school yesterday
i am fine
exit
```

**期望 stderr marker**：
- `[cli] tool call: mark_mistake({"original":"i go to school yesterday",...})` (v0.7)
- DB 中有 1 row mistakes（脚本事后 SELECT 验证）

**Pass 标准**：
- mark_mistake tool 块出现在第 2 轮 LLM 响应
- zod schema 接受（不出现 `tool execute error`）
- DB 有 1 row，category ∈ {grammar, vocabulary, spelling}
- Student 看到的 stdout 不含 `<tool>` / `</tool>`

### 3.3 Scenario 3: Mistake dedup（v0.7.1）

**目的**：验证 LLM 第 2 次遇到同样 mistake 时，CLI 正确跳过（in-process dedup set）。

**Setup**：空 data dir。

**输入序列**（1 进程）：
```
hi
i go to school yesterday
i go to school yesterday
exit
```

**期望 stderr marker**：
- 第 2 轮（首次 mistake）：`[cli] tool call: mark_mistake(...)`
- 第 3 轮（重复 mistake）：`[cli] tool dedup: skipped (already marked: "...")`

**Pass 标准**：
- 2 个 turn 都触发了 mark_mistake 路径
- 只有 1 个 tool call log，另 1 个是 dedup log
- DB 只有 1 row mistakes（dedup 真的拦了）

### 3.4 Scenario 4: Cross-session retrieval（v0.7.2）

**目的**：验证真 LLM 在新 session 看到 [System Context] 里的 "Relevant past sessions" 段时，行为合理（提一句 / 引用 / 至少不无视）。

**Setup**：跑 scenario 1 先建立 1 个 session（带 embedding），再开 1 个新 session。

**输入序列**（2 进程，共享 data dir）：
- 进程 A：scenario 1 的 4-轮对话
- 进程 B：`hi\nexit\n`（开 1 轮看 ctx）

**期望 stderr marker**（进程 B）：
- `[cli] retrieved 1 relevant sessions (query="...")` (v0.7.2 startup 注入)
- `[cli] ctx-block:` dump 含 `- Relevant past sessions (N=1):` 段

**Pass 标准**：
- 进程 B 启动 stderr 含 retrieved marker
- ctx-block dump 含 "Relevant past sessions" 段
- 进程 B 的 LLM 响应里能自然用到 retrieved 内容（"welcome back!" / "how was minecraft?" / 类似），或至少 persona 仍正常

**注意**：retrieved 数量取决于 session A 写了啥 summary。本 scenario 借 scenario 1 的 transcript；脚本串行跑 1 → 4 共享同一 data dir。

### 3.5 Scenario 5: memory_search A+B（v0.7.3）

**目的**：验证真 LLM 真的调 `memory_search`、CLI 触发 2nd-call、2nd-call LLM 给连贯回复。

**Setup**：跑 scenario 1 先建立 1 个 session（带 embedding），再开 1 个新 session 触发 memory_search。

**输入序列**（2 进程，共享 data dir）：
- 进程 A：scenario 1 的 4-轮对话
- 进程 B：`what did i do before\nexit\n`（触发 LLM 主动调 memory_search）

**期望 stderr marker**（进程 B）：
- `[cli] tool call: memory_search({"query":"...","top_k":...})`
- `[cli] tool 2nd-call: memory_search(top_k=...)`
- stdout 是 2nd-call 的回复（不是 1st-call 的 "Let me check..." 那种 tool 触发但被 strip 后剩余的衔接语）
- stdout 不含 `<tool>` / `</tool>`

**Pass 标准**：
- 2 个 stderr marker 都在
- stdout 是连贯英语回复（不是裸 tool 块，不是 fallback 到 no-tool 路径）
- 不含 tool 块 leak

**注意**：live 场景下，1st-call LLM **可能不调** memory_search（如果它没觉得"what did i do before"需要查）。这是 live LLM 行为——可能 pass 也可能 fail。脚本不强制要求调；如果没调，scenario fail + 记 issue（"LLM under trigger condition X"）。

## 6. 风险

| 风险 | 触发 | 处置 |
|---|---|---|
| **M3 不输出 `<tool>...</tool>` 文本块** | Real LLM 走自己 internal 路径（emitted JSON function call）；文本里没有 tool 块 | scenario 2/5 fail；记 issue "M3 doesn't emit text tool blocks under current prompt"；下 sprint 决定换 A 协议 / 改 prompt / 切 B 协议 |
| **M3 输出 tool 块但 schema 偏差** | Real LLM 用 `error_type` / `correction` 等非预期字段名 | zod 拒，stderr `tool execute error`；记 issue "zod schema too strict for M3 output"；下 sprint 决定放宽 schema / 加 fallback |
| **M3 2nd-call 调 tool** | 2nd-call LLM 不遵守 "do not call any tool in second response" | safety strip 兜底；stdout 不 leak tool 块；记 issue "M3 doesn't respect 2nd-turn-no-tool rule"；下 sprint 决定改 prompt wording / 强化 strip / 切 B 协议 |
| **M3 summarizer 输出非 JSON** | Real LLM 加 prose 包裹（"Here's the summary: {...}"）| JSON parse fail；记 issue "summarizer prompt not strong enough"；下 sprint 改 prompt |
| **M3 跑得慢** | Latency > 30s/turn；脚本 timeout（设 60s/turn）| 脚本收集 timeout stderr；记 issue "M3 latency on this endpoint is Xs"；下 sprint 决定换 endpoint / 加 retry |
| **M3 输出超过 max_tokens** | Real LLM 长篇 lecture / 一次给 5 个 follow-up | LLM_MAX_TOKENS=2048 拦；记 issue "M3 verbose"；下 sprint 决定加温度 / 改 SOUL 强调 1-3 句 |
| **MINIMAX_API_KEY 不在 .env** | 用户没 setup | 脚本启动就 fail with clear error "MINIMAX_API_KEY missing in .env" |
| **网络不稳** | 调用中途断 | 脚本 retry 0 次（Out §2.2）；记 issue 给用户看 |
| **data dir 残留干扰** | 之前 live smoke / replay 测试在 `./data/` 留数据 | 脚本自动 mkdir `data/live-validation/<timestamp>/`（**全新** data dir，不复用 `./data/`）|
| **Transcript 含学生敏感信息** | USER.md 真实副本可能有真名 | data/ 已 gitignored；transcript 永不进 git；但用户自己备份时要注意 |
