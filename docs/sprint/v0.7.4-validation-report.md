## Issues observed

### Issue 1: v0.7.4 validation script — scenario 4 cannot verify retrieval with N=1 prior session

- **Sprint link**: v0.7.4 (validation script design); v0.7.2 (behavior being tested)
- **Severity**: Low (no product impact; validation coverage gap)
- **Repro**: `node scripts/live-validate.mjs` with valid `MINIMAX_API_KEY`. In scenario 4, process 0 writes 1 session to a fresh `_data/`, then process 1 starts. Stderr shows:
  ```
  [cli] retrieved 0 relevant sessions (query="minecraft castle creeper wall build")
  ```
  despite process 0's session existing with a 1536B embedding (confirmed via `SELECT id, length(embedding) FROM sessions`).
- **Root cause**: v0.7.2 startup retrieval at `src/cli.ts:228-233` calls `retrieveRelevant({...excludeSessionId: lastReview.sessionId})`. With only 1 prior session, the only candidate is the excluded one, so result is always `[]`. This is the **intended design** (don't duplicate what the "Last session" block already shows), but the validation scenario doesn't exercise the "≥2 prior sessions" case where retrieval would actually return data.
- **Suggested next sprint** (v0.7.5 or whichever addresses validation):
  - **Option A** (preferred): Have process 0 do 2 sessions, e.g. `['hi', 'i played minecraft yesterday', 'i usually build castles', 'exit', 'hi', 'i also like roblox', 'exit']`. Then process 1's `lastReview` is the roblox session, and the minecraft session becomes a non-excluded candidate.
  - **Option B**: Seed the scenario's `_data/` from a pre-built fixture (e.g. 3 sessions with embeddings) before spawning process 0/1.
  - **Option C**: Accept the 0 result as expected with N=1, and rely on scenario 5's `memory_search` tool (which has no exclusion) as the de-facto v0.7.2 retrieval verification.
- **Not in this sprint's scope** per "记录到 backlog" policy.

## Cross-cutting observations

These are not issues, but worth noting for the next sprint that touches the v0.7.2 retrieval path:

1. **v0.7.2 startup retrieval is independent from v0.7.3 `memory_search` tool at the candidate-set level**. Both call `sessions.listWithEmbeddings()` and `retrieveRelevant()`. The only difference is `excludeSessionId` — startup retrieval filters out the most recent (to avoid dup with "Last session" block), while `memory_search` does not. A future "show me my most recent sessions" UX could just use `memory_search` with `top_k=5` and an empty query (or a sentinel) — but the embedder has no zero-vector shortcut, so this would embed empty string. Out of scope here.

2. **Embedder model is pre-warmed by scenario 4's process 0**. Scenario 5 process 1 was 5.2s (vs 9.2s for scenario 4 process 1) because the `MiniLM-L6-v2` ONNX pipeline was already loaded in memory from the previous scenario. Real production use would see the same warm-cache behavior since the same `node` process would persist (or the OS would cache the model file). Not a problem, just a timing observation.

3. **LLM latency (M3) is ~2-3s per turn** across all 5 scenarios (process 0 = 13.9s for 3 turns + summarize; process 1 = 9.2s for 2 turns). No scenario hit the 90s timeout. LLM is responsive enough for live classroom-style use. `data/live-validation/<ts>/summary.json` shows all `wallTimeSec` values < 15s.

4. **All exit codes are 0**. No CLI crashes, no uncaught rejections, no `child.kill('SIGTERM')` timeouts. The script's pre-flight check (refuses without `MINIMAX_API_KEY`) and per-process timeout (90s) worked as designed.
