## V751-001 — 80% warn ignored cached tokens (FIXED in v0.7.5.1 hotfix)

| Field | Value |
|---|---|
| **Sprint link** | v0.7.5 |
| **Severity** | Medium (silent fail of warn; would let a user blow past budget undetected) |
| **Discovered by** | Live validation run, 2026-06-10 16:11 (operator) |
| **Status** | **Resolved 2026-06-10** — fix in `src/cli.ts:399-426`, L3 test in `tests/agent/cli-integration.test.ts`, live-verified in scenario 1 (warn fired at 94% on turn 1) |
| **Repro** | First run (`data/live-validation/2026-06-10T16-05-19-253Z/`) showed `[cli] tokens: input=29 cache_read=1901`. Pre-fix code: `29 / 3000 = 0.97%` — would never warn. LLM's actual context: `29 + 1901 = 1930` (64% of 3000). |
| **Root cause** | `src/cli.ts` v0.7.5 (pre-fix) used `usage.inputTokens / env.LLM_CONTEXT_BUDGET_TOKENS` for the 80% check. Anthropic's `input_tokens` is the **fresh** count; the cached portion is reported separately in `cache_read_input_tokens`. With `cache_control: ephemeral` in effect (v0.7.5 added this on the static SOUL+AGENTS+USER prefix), fresh stays small (~30-700) but the cached static portion (~1800-2400) still counts against the LLM's context window. Checking fresh alone would make the warn never fire under caching. |
| **Fix** | `src/cli.ts:411-426` — total = `inputTokens + cacheReadTokens + cacheCreationTokens`; warn compares `total / budget`. |
| **Verification** | L3 test `v0.7.5.1 (V751-001 fix) warn fires when TOTAL context (fresh + cache_read) >= 80% of budget` passes (fixture `budget-warn-cache.json`: input=200, cache_read=1800, budget=2000 → warn at 100%). L3 test `v0.7.5 usage log + 80% warn` still passes (regression: budget-warn.json input=95, cache_read=0, budget=100 → warn at 95%, unchanged). Live scenario 1: warn fired at 94% on turn 1. |
| **Out of scope rationale** | N/A — was in-scope hotfix because the original DoD #6 verification exposed it. |
| **See also** | [[v0.7.5-validation-report]] §4 (full code diff + verification) |

## V751-002 — `chatStream` errors crash the process (OPEN, backlog for v0.7.6)

| Field | Value |
|---|---|
| **Sprint link** | v0.7.5 (and earlier) |
| **Severity** | Low (no product impact at default usage; affects only adversarial conditions like rate-limit bursts or network blips) |
| **Discovered by** | Live validation run, 2026-06-10 16:05-16:11 (operator) — first run's 30-turn scenario 1 hit MiniMax-M3 rate limit at turn ~16-17 |
| **Status** | Open — backlog (v0.7.6 candidate) |
| **Repro** | First run with 30-turn scenario 1: `MINIMAX_API_KEY=<key> node scripts/live-validate-v0.7.5.mjs`. The CLI child spawned for scenario 1 had been running for ~75s (turn 16-17), then stderr shows `Error: 429 {"type":"error","error":{"type":"...","message":"已达到 Token Plan 速率限制..."}}`. The error is unhandled inside the `for await` loop in `src/cli.ts:295-364` — it propagates out of `main()`, the process exits 1 with no graceful user-facing message. |
| **Root cause** | `src/cli.ts:295` — the `for await (const chunk of client.chatStream({...}))` loop has no `try/catch`. If the stream throws mid-iteration (rate limit, network blip, malformed SDK event), the error escapes `main()` and the process crashes. There's also no retry, no user-friendly error message, and no telemetry. The CLI was designed for "happy path" usage; production-style resilience was out of scope for v0.7.5. |
| **Workaround (current)** | None. Operator must restart the session manually. This was the reason I had to reduce scenario 1 from 30 turns to 6 turns in the second run — at 6 turns the run completes well under the rate-limit window. |
| **Suggested next sprint** | v0.7.6 (likely): wrap `chatStream` call in `try/catch`, on caught error: (1) write a user-friendly message to stderr (`[cli] LLM error: <message>`), (2) write a user-friendly message to stdout (`Sorry, I lost my train of thought. Could you say that again?`), (3) `process.exitCode = 1` (don't crash), (4) optionally: 1 automatic retry with `setTimeout(1000)` before bailing (Anthropic SDK's `retry: 2` option already does this for transient errors, but rate-limit 429 is treated as final by default). |
| **Out of scope this sprint** | Per v0.7.5-scope "0 产品代码改动" and "记录到 backlog" — no fix. The hotfix V751-001 was the exception because DoD #6 verification exposed it directly. |
