## V751-002 — `chatStream` errors crash the process (FIXED in v0.7.6 A axis)

| Field | Value |
|---|---|
| **Sprint link** | v0.7.5 (filed) → **v0.7.6 A axis (resolved)** |
| **Severity** | Low → no impact (resolved) |
| **Discovered by** | v0.7.5 live validation run, 2026-06-10 16:05-16:11 — first 30-turn scenario hit MiniMax-M3 rate limit at turn ~16-17, process crashed |
| **Status** | **Resolved 2026-06-11** — fix in `src/cli.ts:444-484` (chatStreamWithRetry try/catch + 1x retry + catch-all fallback + auto-save) |
| **Repro (pre-fix)** | `MINIMAX_API_KEY=<key> node scripts/live-validate-v0.7.5.mjs` with 30-turn scenario 1. After ~75s the SDK throws `Error: 429 ...`. The unhandled error escapes `main()` and the process crashes with exit 1 and no user-friendly message. |
| **Root cause** | `src/cli.ts:295` (pre-v0.7.6) — the `for await (const chunk of client.chatStream({...}))` loop had no `try/catch`. Mid-iteration errors (rate limit / 5xx / network) escaped `main()` and crashed the process. |
| **Fix (v0.7.6 A axis)** | Three layers: |
| | 1. `src/llm/errors.ts` — `classifyLLMError` returns `{ classification, retryable, original }` (5 classes: rate_limit / 5xx / 4xx / network / unknown; 4xx + unknown → retryable=false) |
| | 2. `src/llm/retry.ts` — `chatStreamWithRetry(client, opts, maxAttempts=2)` retries 1x on retryable errors with 1s backoff |
| | 3. `src/cli.ts:444-484` — catch-all around `chatStreamWithRetry`: on persistent failure, write `[Teacher]: Sorry, I lost my train of thought. Could you say that again? 😊` to stdout, auto-save session with `(summarization failed after llm error)` placeholder summary, set `exitCode = 1`, break the loop (not throw) |
| **Verification** | 3 L3 tests cover the full path: 5xx (retryable, both attempts fail) / 4xx (no retry, fail-fast) / 429 (retryable, both attempts fail). All 3 + 322 existing tests pass (325/325). Live demo not re-run (would need to trigger rate limit), but L3 tests use the real `chatStreamWithRetry` wrapper with a `createThrowingProvider(N, ...)` mock. |
| **Out of scope rationale** | N/A — was in-scope A axis work (v0.7.6-scope §1 / §2.1). |
| **See also** | [[v0.7.6-test-report]] §2.4 L3 test list; [[v0.7.6-validation-report]] §3/§4 (live demo stderr confirms catch-all path not triggered under Replay-mode) |

## V752-001 — `[cli] ctx-block:` prints duplicate `[System Context]` prefix (OPEN, cosmetic, deferred)

| Field | Value |
|---|---|
| **Sprint link** | v0.7.6 (discovered) → v0.7.7+ (deferred) |
| **Severity** | **Cosmetic** (no functional impact; only affects operator debug log) |
| **Discovered by** | v0.7.6 live demo dry-run, 2026-06-10 18:02 (operator, reading transcripts) |
| **Status** | Open — backlog (v0.7.7+ candidate) |
| **Repro** | Run any CLI command; stderr shows: |
| | ``` |
| | [cli] ctx-block: |
| | [System Context][System Context]      ← duplicate prefix |
| | - Phase: WARM_UP |
| | - Elapsed: 0.0 min |
| | ... |
| | ``` |
| | The `[System Context]` token appears twice on the same line. |
| **Root cause** | `src/cli.ts:428` writes `\`[cli] ctx-block:\n[System Context]${sysSeg.dynamic}\n\``. But `sysSeg.dynamic` (returned by `buildSystemContext`) **already starts with** the literal `[System Context]` token (it's the first line of the joined text). The CLI adds another prefix in front, producing the double prefix. |
| | Pre-existing bug from v0.7.5 / `buildFinalSystemSplit` — not introduced by v0.7.6 B3. v0.7.6 B3 just exposed it (B3 added the `[cli] ctx-segment:` line and the dump format became more visible). |
| **Why not fixed in v0.7.6** | Cosmetic only. The L3 test that checks the dump (`[cli] ctx-block:\n[System Context]${sysSeg.dynamic}`) currently passes; the v0.7.5 test was written against the same shape. Changing the CLI to drop the redundant prefix would require also updating the L3 test's expected substring. Net effect: clean up 1 string in cli.ts + 1 L3 test assertion. Trivial, but per v0.7.6-scope "no scope creep", deferred. |
| **Workaround (current)** | Operator reads past the duplicate prefix; the actual segment content below the header is correct. |
| **Suggested fix (v0.7.7+)** | `src/cli.ts:428` — change to `\`[cli] ctx-block:\n${sysSeg.dynamic}\n\`` (drop the leading `[System Context]` since `sysSeg.dynamic` already includes it). Update the L3 test (if it asserts the duplicate). 2 lines total. |
