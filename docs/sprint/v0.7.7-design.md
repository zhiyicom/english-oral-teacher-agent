# Sprint v0.7.7 Design — V752-001 fix (1-line template + L3 test)

> 本 sprint 只做 1 件事：消除 stderr `ctx-block` 行的双重 `[System Context]` 前缀。
> 范围：1 个 `src/cli.ts` 字符串改动 + 1 个新增 L3 test。无新 fixture，无协议变更，无架构变更。

---

## 1. 现状（v0.7.6 的 bug）

`src/cli.ts:454` 当前代码：
```ts
process.stderr.write(`[cli] ctx-block:\n[System Context]${sysSeg.dynamic}\n`)
```

`sysSeg.dynamic` 来自 `buildFinalSystemSplit` → `buildSystemContext`（`src/agent/context-injector.ts:64`）→ 返回 string，其第一行就是 `[System Context]`（由 `tests/agent/context-injector.test.ts:50-53` L1 test 锁住）：
```ts
it('starts with [System Context] header', () => {
  const out = ctx(makeState())
  expect(out.startsWith('[System Context]\n')).toBe(true)
})
```

所以模板里再写一次 `[System Context]` 就是写重了 → stderr 出现：
```
[cli] ctx-block:
[System Context][System Context]
- Phase: WARM_UP
...
```

**影响**：纯 cosmetic。LLM 看到的 system block（`sysSeg.dynamic` 直接进 Anthropic SDK）**没**有双前缀——双前缀只出现在 stderr 这条 operator-visible log 上。功能 100% 正确。

**为什么 v0.7.6 没改**：v0.7.6-scope "no scope creep" 政策；当时 L3 test 没用 `ctx-block` 字符串做断言，所以问题被 silent 忽略；live demo 跑出来 operator 才看到。

---

## 2. 修复（1 字符 = 1 行改动）

`src/cli.ts:454` 改成：
```ts
process.stderr.write(`[cli] ctx-block:\n${sysSeg.dynamic}\n`)
```

去掉模板里的 `[System Context]` token；`sysSeg.dynamic` 自带 header。

**Diff**（概念）：
```diff
-process.stderr.write(`[cli] ctx-block:\n[System Context]${sysSeg.dynamic}\n`)
+process.stderr.write(`[cli] ctx-block:\n${sysSeg.dynamic}\n`)
```

`process.stderr.write` 调用总行数不变（1 行 → 1 行），调用次数不变（1 → 1）。无任何运行时行为变化除了 stderr 输出。

## 3. 测试

### 3.1 新增 L3 test（1 个 it block）

在 `tests/agent/cli-integration.test.ts` 末尾（或 "empty library" 测试附近，因为那也是 fresh-session first-turn 触发 ctx-block 的场景）新增：

```ts
it('first-turn ctx-block dump has NO duplicate [System Context] prefix (v0.7.7 / V752-001 fix)', async () => {
  // Fresh session → first turn → triggers the first-turn-only ctx-block dump
  // on stderr. The dump must show [System Context] exactly once at the
  // start of the dynamic block (V752-001 fix: previously the CLI added an
  // extra "[System Context]" token in front of sysSeg.dynamic, which itself
  // already starts with that token, producing a double prefix).
  const result = await runCli('hi\nexit\n', {
    MINIMAX_API_KEY: 'sk-test',
    APP_DATA_DIR: dataDir,
  })
  expect(result.exitCode).toBe(0)

  // Extract the ctx-block dump from stderr.
  const stderrLines = result.stderr.split('\n')
  const ctxBlockIdx = stderrLines.findIndex((l) => l.startsWith('[cli] ctx-block:'))
  expect(ctxBlockIdx).toBeGreaterThanOrEqual(0)  // first-turn dump must be present
  // The dump runs from `ctxBlockIdx` until the next `[cli]` line or EOF.
  // We assert the FIXED shape: line right after `[cli] ctx-block:` is the
  // dynamic block header `[System Context]` (NOT `[System Context][System Context]`).
  const firstContentLine = stderrLines[ctxBlockIdx + 1]
  expect(firstContentLine).toBe('[System Context]')  // exact match, not regex
  // Belt-and-suspenders: ensure the doubled prefix is nowhere in the dump.
  const dumpBody = stderrLines.slice(ctxBlockIdx).join('\n')
  expect(dumpBody).not.toContain('[System Context][System Context]')
}, 20000)
```

**为什么是 `toBe('[System Context]')` 精确匹配，不是 regex**：
- 精确匹配 catch 任何意外重复（包括 `[System Context] [System Context]` 带空格的变种）
- regex `^\[System Context\]$` 也行，但 `toBe` 锁得更死
- 双重 assert（`toBe` + `not.toContain` 双前缀）防御未来 refactor

**为什么用 `'hi\nexit\n'`**：
- 2 turns，first turn 触发 ctx-block dump
- "hi" 匹配 v0.2 fixture（`0-hi-input.json` → "Hi there! How are you doing today?"）
- "exit" 触发 `markEnded`，session 正常结束，exit code 0

**DataDir**：用 `beforeEach` 已经建好的 `dataDir`（每个 it 独立 tmpdir）。无需 setup。
