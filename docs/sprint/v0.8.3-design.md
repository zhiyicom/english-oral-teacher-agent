## 1. 设计决策

### 1.1 不重构核心 agent

`runTurn()` 目前缓冲 LLM 响应并输出单个 `student-text` 事件（而非逐字 `text-chunk` 事件）。v0.8.3 保持此行为。SSE 协议按原样发送生成器产出的 `TurnEvent` 类型——服务端是一个透明管道。

**用户体验**：turn 事件按序列到达（ctx-segment → ctx → student-text → done），每个事件更新 UI。老师消息一次性出现而非逐字显示——对 v0.8.3 来说体验仍不错。

真正的流式输出（text-chunk 增量）需要重构 `src/llm/retry.ts` 中的 `chatStreamWithRetry()` 使其不缓冲，风险较高，延至 v0.8.5。

### 1.2 内存会话状态存储

`SessionState`（phase、elapsedMin、silenceMin 等）不持久化到 SQLite。跨 turn 之间，服务端通过 `createApp()` 内部的 `Map<string, { state, phaseHistory, firstPair, isFirstTurn }>` 跟踪状态变化。

服务端重启时，进行中的会话丢失状态需重新初始化——对单用户 localhost 工具可接受，会话通常持续 30 分钟以内。

### 1.3 通过原生 EventSource 实现 SSE

浏览器使用 `new EventSource(url)` 连接到 SSE 端点。每轮对话是一个独立的 EventSource 连接（带 `?action=turn&input=...`）。避免了自定义 SSE 解析库——零新依赖。

### 1.4 Replay 模式下的 E2E 测试

服务端默认使用 replay 模式（使用 `tests/fixtures/replay/` fixtures）。E2E 测试输入 "hi" → 服务端匹配 `greeting.json` fixture → 返回预录制响应。无真实 LLM 调用。与现有 L3 CLI 测试模式一致。

## 3. 服务端：SSE 端点重写

### 3.1 当前状态（v0.8.2）

`src/server.ts:210-228` 处的处理器为一个桩：
```typescript
app.get('/api/sessions/:id/stream', (c) => {
  const id = c.req.param('id')
  if (!sessions.get(id)) {
    return c.json({ error: 'session not found', id }, 404)
  }
  return streamSSE(c, async (stream) => {
    await stream.writeSSE({
      event: 'done',
      data: JSON.stringify({
        sessionId: id,
        endedReason: 'stub',
        note: 'v0.8.1 SSE stub — full turn loop is v0.8.3',
      }),
    })
  })
})
```

### 3.2 目标状态（v0.8.3）

Query 参数：
- `action=init` — 以 SSE 事件返回当前阶段
- `action=turn&input=...` — 运行一轮对话，流式输出所有 TurnEvent

处理器逻辑：

1. 解析 `?action=`（必填：`init` 或 `turn`）和 `?input=`（`turn` 时必填）。
2. 验证会话存在 → 不存在返回 404。
3. **`action=init`**：以 SSE `phase` + `done` 事件返回当前阶段。
4. **`action=turn`**：
   - 验证输入非空 → 空返回 400。
   - 检查会话未结束 → 已结束返回 SSE `error` + `done`。
   - 从内存存储重建 `SessionState`（首 turn 从 DB 重建）。
   - 从 `messages.getBySession()` 加载消息历史。
   - 使用现有单例（client、embedder、toolRegistry、DAOs、clock）构建 `TurnInput` + `TurnDeps`。
   - 消费 `runTurn()` 生成器：
     ```typescript
     const gen = runTurn(turnInput, deps)
     while (true) {
       const next = await gen.next()
       if (next.done) {
         // 以 next.value.endedReason 写入最终 done 事件
         await stream.writeSSE({
           event: 'done',
           data: JSON.stringify({ endedReason: next.value.endedReason }),
         })
         break
       }
       await stream.writeSSE({
         event: next.value.type,
         data: JSON.stringify(next.value),
       })
     }
     ```
   - 将新的 `SessionState` 存回内存存储。

### 3.3 内存会话存储

```typescript
interface SessionRuntime {
  state: SessionState
  phaseHistory: PhaseTransition[]
  firstPair: Message[]
  isFirstTurn: boolean
}

const sessionStore = new Map<string, SessionRuntime>()
```

在 `createApp()` 顶部创建。每轮读取，完成后将 `TurnOutput` 中的更新值写回。

### 3.4 SessionState 重建

```typescript
function reconstructSessionState(
  session: Session,
  clock: Clock,
  store: Map<string, SessionRuntime>,
): SessionRuntime
```

若存在已存储的 `SessionRuntime` 则返回，否则从 DB 行创建新的：
- `startedAt` 取自 `session.started_at`
- 当前阶段取自 `JSON.parse(session.phase_history)`（最后一个条目的 phase）
- `elapsedMin` 取自 `(clock.now() - startedAt) / 60000`
- 全新的 `phaseHistory`、`firstPair`、`isFirstTurn`

## 4. SessionPage 组件契约

### 4.1 状态

| 状态 | 触发条件 | UI |
|---|---|---|
| **加载中** | 页面挂载，等待 `action=init` SSE | "加载中…" 文字 |
| **活跃** | `init` 完成，会话未结束 | 完整 UI：阶段标签 + 计时器 + 消息 + 输入框 |
| **处理中** | 用户发送输入，等待 `done` 事件 | 输入框锁定，"老师正在回复…" 占位文字 |
| **已结束** | `done` 事件带 `endedReason` | 消息只读 + [返回主界面] |
| **错误** | SSE `error` 事件或 fetch 失败 | 错误横幅 + 重试按钮 |

### 4.2 组件树

```
SessionPage
├── 顶栏
│   ├── ← 返回（Link 到 /）
│   ├── 阶段标签（热身 / 主体练习 / 总结 / 已结束）
│   ├── 计时器（mm:ss 自会话开始）
│   └── [结束本次] 按钮（调用结束会话）
├── 消息列表（flex-col、overflow-y-auto、自动滚到底部）
│   ├── 消息气泡（学生）— 右对齐、bg-blue-100
│   └── 消息气泡（老师）— 左对齐、bg-gray-100
├── 输入区域（底部固定）
│   ├── <textarea> — Enter 发送、Shift+Enter 换行
│   └── [发送] 按钮
└── 错误横幅（条件渲染、可关闭）
```

### 4.3 数据流

```
SessionPage 挂载
  → EventSource(/api/sessions/:id/stream?action=init)
  → SSE phase 事件 → setPhase(currentPhase)
  → SSE done 事件 → 关闭 EventSource，setReady(true)

用户输入 "hi" + Enter
  → 追加学生消息气泡到状态
  → 锁定输入（isTurning = true）
  → EventSource(/api/sessions/:id/stream?action=turn&input=hi)
  → SSE ctx-segment 事件 → （暂时忽略，仅供调试）
  → SSE ctx 事件 → 更新计时器
  → SSE student-text 事件 → 追加老师消息气泡
  → SSE done 事件 → 关闭 EventSource，解锁输入
  → 如有 endedReason：显示已结束状态
```

### 4.4 输入锁定

从用户发送输入到 `done` 事件到达期间，输入框**禁用**。这防止了重复发送的竞态条件（v0.8-design §9.3）。锁定期间占位文字变为"老师正在回复…"。

### 4.5 计时器

使用 `setInterval(1000)` 的 `useEffect` 计算 `elapsed = Math.floor((Date.now() - sessionStartedAt) / 60000)` 并显示为 `mm:ss`。当有 `ctx` 事件的 `elapsed` 字段时以此更新。

## 5. Web lib 补充

### 5.1 `web/src/lib/api.ts`

```typescript
export function getSessionStreamUrl(
  sessionId: string,
  action: 'init' | 'turn',
  input?: string,
): string {
  const params = new URLSearchParams({ action })
  if (input) params.set('input', input)
  return `/api/sessions/${encodeURIComponent(sessionId)}/stream?${params}`
}
```

## 9. 风险与缓解

| 风险 | 缓解措施 |
|---|---|
| `runTurn()` 集成破坏现有服务端测试 | 更新 SSE 桩测试为断言 greeting fixture 的真实事件 |
| 从 DB 重建 SessionState 较脆弱 | 首 turn 使用 `initState()`。后续 turn 使用内存存储。DB 重建仅为兜底 |
| EventSource URL 长度限制（长输入）| 对话输入通常 < 500 字符。超出时客户端验证 |
| E2E 等待 SSE 响应超时 | Replay fixtures 即时响应（无网络）。15s timeout 充裕 |
| 输入框竞态条件（重复发送）| 发送后到 `done` 事件前锁定输入（disabled）。视觉提示："老师正在回复…" |
| 状态泄漏（Map 无限增长）| 会话 30 分钟后自动结束。v0.8.3 暂不清理（顶多几个条目）|
