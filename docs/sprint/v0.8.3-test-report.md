## 5. Design decisions and issues

### 5.1 No core agent refactoring (text-chunk deferred)

`runTurn()` still buffers the LLM response and yields a single `student-text` event. The SSE protocol sends the exact TurnEvent types — the server is a transparent pipe. True character-by-character streaming requires refactoring `chatStreamWithRetry()` in `src/llm/retry.ts` and is deferred to v0.8.5.

### 5.2 In-memory session state store

`SessionState` is tracked in a `Map<string, SessionRuntime>` inside `createApp()`. Not persisted across server restarts — acceptable for single-user localhost with sessions ≤30 min. The `reconstructSessionState()` function rebuilds state from DB on first turn, then uses the in-memory store for subsequent turns.

### 5.3 SSE via native EventSource

No custom SSE parser library. Each turn opens a fresh `EventSource` connection with query params (`?action=turn&input=...`). The browser parses the SSE protocol natively. Input locking prevents race conditions (double-send).

### 5.4 biome-ignore comments

Two `useExhaustiveDependencies` suppressions in SessionPage.tsx:
- `[messages]` — intentional: scroll on new messages
- `[id, closeES]` — `closeES` is stable (useCallback with `[]`)

One `noArrayIndexKey` suppression — messages are append-only, never reordered or deleted.
