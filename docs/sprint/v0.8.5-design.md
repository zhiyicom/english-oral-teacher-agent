## 1. 真正逐字流式对话

### 1.1 当前状态

```
LLM SDK stream → chatStreamWithRetry() → 缓冲拼接 → { response, usage }
→ runTurn() → 取 response → yield student-text（一次性全文）
→ SSE → 浏览器 → 整条消息渲染
```

### 1.2 目标状态

```
LLM SDK stream → chatStreamWithRetryGen() → yield ChatChunk（不缓冲）
→ runTurn() → yield text-chunk（逐 delta）→ ... → yield student-text（全文，保留兼容）
→ SSE → 浏览器 → 逐 delta 追加到气泡
```

### 1.3 实现方案

**新增 `chatStreamWithRetryGen` async generator**（`src/llm/retry.ts`）：

```typescript
export async function* chatStreamWithRetryGen(
  client: LLMClient,
  opts: ChatOpts,
  maxAttempts = 2,
): AsyncGenerator<ChatChunk, void, void> {
  let lastErr: unknown = null
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      for await (const chunk of client.chatStream(opts)) {
        yield chunk
      }
      return
    } catch (err) {
      lastErr = err
      const classified = classifyLLMError(err)
      process.stderr.write(
        `[cli] llm error: ${classified.classification} (${(err as Error).message})\n`,
      )
      if (!classified.retryable) break
      if (attempt < maxAttempts) {
        process.stderr.write(`[cli] retrying in 1s... (attempt ${attempt + 1}/${maxAttempts})\n`)
        await new Promise((r) => setTimeout(r, 1000))
      }
    }
  }
  throw lastErr
}
```

**保持 `chatStreamWithRetry` 兼容**（用作 wrapper）：

```typescript
export async function chatStreamWithRetry(...): Promise<StreamResult> {
  let response = ''
  let usage = null
  for await (const chunk of chatStreamWithRetryGen(client, opts, maxAttempts)) {
    if (chunk.type === 'text') response += chunk.delta
    else if (chunk.type === 'usage') usage = chunk
  }
  return { response, usage }
}
```

**修改 `runTurn`**（`src/agent/turn.ts`）：
- 新增 `text-chunk` 到 `TurnEvent` union
- 将 `chatStreamWithRetry` 调用替换为 `chatStreamWithRetryGen`
- 循环中 yield `{ type: 'text-chunk', delta }` 给每个 text chunk
- 收集 usage chunk
- 循环后 yield `student-text`（全文，保持向后兼容）

**修改 CLI `forwardEvent`**（`src/cli.ts`）：
- 新增 `text-chunk` case：`process.stdout.write(event.delta)`

**修改 SessionPage**（`web/src/components/SessionPage.tsx`）：
- 监听 `text-chunk` SSE event
- 实时追加 delta 到当前 assistant 气泡（不新建，修改末尾）

### 1.4 向后兼容

- `student-text` 事件保持不变——旧版客户端忽略 `text-chunk` 仍能正常工作
- `chatStreamWithRetry` 函数签名不变——被 `runTurn` 保留为可用路径

## 8. Phase 推进增强（v0.8.5 后续）

### 8.1 问题

初始设计将阶段行为指令放在 `prompts/SOUL.md` 中，依赖 LLM 阅读 prompt 前部的指令后记住并在后续遵从。实测发现 DeepSeek chat 模型几乎完全忽略 system prompt 中的阶段指令，始终保持在闲聊模式。

### 8.2 三层注入策略

为解决 chat 模型对 system block 遵从度低的问题，采用了三层阶段信息注入：

**第一层：System Context 动态块中的阶段指令**

`src/agent/context-injector.ts` — 不再只放 `Phase: WARM_UP`，而是注入完整的阶段行为指令：

```
## You are in WRAP_UP phase (25-30 min). CRITICAL — follow these steps NOW:
- DO NOT introduce new topics or ask open-ended questions
- Summarize 1-2 things practiced or improved today
- Point out 1 thing the student did well
- Mention 1 thing to work on next time
- Suggest a mini practice task
- Move the conversation toward a natural close
```

**第二层：用户消息前缀注入**

`src/agent/turn.ts` — 在发给 LLM 的消息副本中，在最后一条用户消息前注入简短提醒：

```
[Phase: END — say goodbye now. NO questions. Final message.] almost everyday
```

chat 模型对用户消息的遵从度远高于 system block。原始 history 数组不被修改，只传浅拷贝给 LLM。

**第三层：SOUL.md 中的详细阶段说明**

`prompts/SOUL.md` Session Phases 部分保留，作为静态参考文档。LLM 在需要更多上下文时可以从这里获取。

### 8.3 沉默跳过 WRAP_UP 保护

`src/agent/turn.ts` — 当检测到长时间沉默导致阶段从 MAIN_ACTIVITY 直接跳到 END（跳过 WRAP_UP），插入一个 catch-up WRAP_UP 轮次：

- 条件：`phase === END` AND `phaseBeforeTick ∈ {WARM_UP, MAIN_ACTIVITY}` AND `silenceMin ≥ 10`
- 动作：将 phase 覆写为 WRAP_UP，LLM 做一轮总结，下一轮自然进入 END 告别

### 8.4 话题库注入

`prompts/topic-library.md` — 按 A1-A2/B1/B2 三级分类的英语口语话题库。

`src/prompts/loader.ts` — `loadSystemPrompt()` 读取 `topic-library.md`，`buildSystemString()` 将其作为 `# TOPIC_LIBRARY` 部分注入 system prompt。LLM 在 MAIN_ACTIVITY 阶段不再需要"读文件"——话题库内容直接在 prompt 中。

## 9. 调试工具

### 会话结束流水线

会话在 Web UI 结束时（用户说 "stop" 或 30min phase_end），服务端执行完整的收尾流水线：

```
endedReason 非空
  → summarize(全部消息, LLM) → summary + keywords
  → logSummarize() → data/llm-debug/*_summarize.txt
  → extractStudentDiscoveries(summary, interests, LLM) → 提取学生新信息
  → updateUserSettings({interests, bodyAppend}) → 原子更新 USER.md
  → matchTopic(keywords, topicLibrary) → 更新 topic_stats
  → embedder.embed(summary) → sessions.setEmbedding() → 跨会话检索
  → sessions.markEnded()
```

### 跨会话记忆

`loadLastReview(db)` 在每个新会话首轮动态查询（非服务端启动时一次性缓存），确保最新摘要可见。首轮 `[System Context]` 注入 `Last session` 摘要和关键词。

### 话题追踪与去重

`topic-matcher.ts` 使用 Jaccard + keyword-hit ratio 双重评分匹配摘要关键词到话题库。匹配上的话题记录到 `topic_stats` 表，注入 `Active topics` 供 LLM 参考。`prompts/topic-library.md` 作为 `# TOPIC_LIBRARY` 注入 system prompt，LLM 在 MAIN_ACTIVITY 中从中选题。

### 用户画像自动更新

`profile-extractor.ts` 在每次摘要生成后分析摘要文本，提取学生的新技能、兴趣、偏好，通过 `updateUserSettings()` 合并到 `prompts/USER.md` 的 frontmatter（interests）和 body 中。

### Phase 指令文件化

阶段行为指令从 TypeScript 硬编码移至可编辑的 `prompts/phases.md`。每个阶段包含 Context（注入 System Context 动态块）和 Reminder（注入用户消息前缀）两部分。阶段首次切换时使用完整 Context 文本以打断 LLM 惯性。

### DEBUG_LOG_LLM

`.env` 中设置 `DEBUG_LOG_LLM=1`，每个 LLM 请求的完整内容写入 `data/llm-debug/<timestamp>_<session>_turn<N>.txt`，包含所有 system blocks 和消息历史。

### selectClient 优先级修复

`RUN_LIVE_LLM=1` 优先级最高——始终使用真实 API。fixtures 目录存在时默认 replay 模式。测试环境通过 `delete process.env.RUN_LIVE_LLM` / `RUN_LIVE_LLM='0'` 强制 replay。
