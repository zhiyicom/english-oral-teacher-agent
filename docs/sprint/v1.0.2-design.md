## 1. HOTFIX 阶段（已落地，见 commit `e35d70e`）

### 1.1 Bug A — 频繁切换 topic

**症状**：session `ddb32b4f` 27 分钟内 topic_select 工具触发 11 次，每次只聊 2-3 轮就切。LLM 缺乏"在同一话题停留足够久"的约束。

**根因**：`topic_select` 工具无频率控制，LLM 每次 MAIN_ACTIVITY 进入都倾向换一个 topic。

**方案**：客户端"年龄门"（client-side age gate）

- 新建 `src/agent/topic-counter.ts`：进程内 `Map<sessionId, number>`，每次成功 user turn 递增；成功 `topic_select` 重置为 0
- `src/agent/turn.ts` 在 `topic_select` 分支前检查：`currentCount < MIN_TOPIC_AGE` 时 yield `{error: "Topic too young (X/Y)..."}` 走 2nd-call，让 LLM 重新措辞而非切换
- 旁路：用户输入匹配 `/\b(switch|change|new|another|fresh|different)\s+topic\b|换个?话题|换个新话题/i` 时直接放行
- 阈值默认 5，可通过 `TOPIC_AGE_MIN` 环境变量覆盖（0 = 禁用，用于测试）

**验证**：`tests/agent/tools/topic-counter.test.ts` 20 个测试全过。

### 1.2 Bug B — SSE 2nd-call 响应被前端丢弃

**症状**：session `ddb32b4f` 11 of 33 turns 出现"无回复 / 只显示 preamble"。

**根因**：`web/src/components/SessionPage.tsx` 的 student-text handler 有一段 v0.8 兼容 shim：

```typescript
// 旧代码（v0.8 兼容，2026-06-27 之前）
if (!streamingRef.current) {
  streamingRef.current = data.text
  setStreamingText(stripInternal(data.text))
}
```

意图：老客户端（v0.8 之前，无 text-chunk 流式）只接收 student-text 事件，新客户端会先收到 text-chunk 再收 student-text（后者是最终完整文本）。

**但实际**：当 1st-call 流式输出了任何 text-chunk（包括纯 `<tool>...</tool>` 调用，strip 后为空字符串），`streamingRef.current` 就是 truthy（即使内容为空），导致 2nd-call 的 student-text 被 **完全丢弃**。下次 `done` 事件把空字符串写入 bubble。

**方案**：移除 shim，student-text 始终替换 `streamingRef`：

```typescript
// 新代码
streamingRef.current = data.text
setStreamingText(stripInternal(data.text))
```

text-chunk 流式期间会先累积 delta，student-text 到达时整体替换（语义上就是"最终确认版本"）。

**验证**：SessionPage typecheck + build pass；ddb32b4f 历史数据中 11 个受影响 turn 在 DB 中文本完整，修复后应能正常显示。

### 1.3 诊断基础设施

## 2. 新功能阶段：per-keyword 命中统计 + 选择策略升级

### 2.1 问题

`topic_stats` 只记录每个 topic 的 `discussion_count`。带来的两个盲区：

1. **关键词 drift 不可见**：例如 `travel` 被命中 3 次，但实际驱动它的可能只有 `trip / vacation / holiday` 三个词，topic 内其他 27 个 keyword 从未触发
2. **选择算法粗粒度**：`selectTopic()` 的 D2（"讨论次数最少优先"）只能区分 topic 级别。一个被命中 5 次的 topic 和一个被命中 1 次的 topic 都比 0 次"老"，但前者内部也许仍有大量未聊的角度

### 2.2 数据模型：新增 `keyword_hits` 表

```sql
-- 006_keyword_hits.sql
CREATE TABLE IF NOT EXISTS keyword_hits (
  topic         TEXT NOT NULL,
  keyword       TEXT NOT NULL,
  hit_count     INTEGER NOT NULL DEFAULT 0,
  first_hit_at  TEXT,
  last_hit_at   TEXT,
  PRIMARY KEY (topic, keyword)
);
CREATE INDEX IF NOT EXISTS idx_keyword_hits_topic ON keyword_hits(topic);
```

**粒度选择**：per-(topic, keyword)，而非全局 per-keyword。原因：

- 全局 keyword 计数会混淆归属（`music` 同时出现在 `entertainment` 和 `art_culture`，合并计数无意义）
- 当前 `matchTopic()` 的 `shared[]` 已经按 topic 划分，直接落表一致

**不用 JSON 列**：JSON 列无法高效做"全局 dormant keyword 排行"，且读-改-写需要事务。分离表支持 `ON CONFLICT(topic, keyword) DO UPDATE`，原子 UPSERT。

### 2.3 写入路径

在 `src/cli.ts:519` 和 `src/server.ts:491` 现有的 `topicStats.incrementAndUpdate` 后追加：

```typescript
const match = matchTopic(summaryKeywords, topics.list())
if (match) {
  topicStats.incrementAndUpdate(match.topic, new Date())
  // NEW:
  keywordHits.upsertMany(match.topic, match.shared, new Date())
}
```

`match.shared` 由 `topic-matcher.ts:42` 计算（`t.keywords.filter((k) => normSession.includes(k.toLowerCase()))`），case 规范化与 storage 一致。

DAO 接口（`src/storage/topics.ts`）：

```typescript
export interface KeywordHit {
  topic: string
  keyword: string
  hitCount: number
  firstHitAt: string | null
  lastHitAt: string | null
}

export interface KeywordHitsDao {
  upsertMany(topic: string, keywords: string[], now: Date): void
  getAll(): KeywordHit[]
  getByTopic(topic: string): KeywordHit[]
}
```

### 2.4 选择算法升级：keyword-freshness 评分

`src/agent/topic-engine.ts` 当前 score 公式：

```
score = -count * W_TOPIC + interest * W_INTEREST + noise
       (-0.1)            (+0.5)            (±0.2)
```

新公式：

```
score = -count * W_TOPIC - avgKeywordHit * W_KEYWORD + interest * W_INTEREST + noise
       (-0.1)            (-0.05)              (+0.5)              (±0.2)
```

其中：

- `avgKeywordHit(t)` = topic `t` 的 keywords 在 `keyword_hits` 表中的 hit_count 平均值；keyword 缺失（未命中过）视为 0
- `W_KEYWORD = 0.05` 初值

**效果**：
- 0 命中 topic：count=0, avg=0，score 由兴趣 + 噪声决定
- 被讨论 3 次但内部 keyword 全新：count=-0.3, avg=0，仍有竞争力
- 被讨论 3 次且内部 keyword 都被反复聊：count=-0.3, avg=-0.15，总分更低，被新 topic 超越

**权重初值取舍**：`W_KEYWORD=0.05` 仅为 `W_TOPIC=0.1` 的一半。理由：

- 太小：选择行为几乎不变，feature 没存在感
- 太大：盖过兴趣匹配（interests 是 F6 的明确信号）
- 当前值：让"keyword 全新"和"被反复聊"产生可观测差异，但不主导决策

后续根据 §5 backlog B1 实际命中分布再调。

### 2.5 工具返回扩展：`suggested_keyword`

`topic_select` 工具当前返回 `{slug, title, est_minutes: 15}`。LLM 拿到 slug 后自行决定聊什么。

新增 `suggested_keyword` 字段：

```typescript
// src/agent/tools/topic-select.ts
export interface TopicSelectResult {
  slug: string
  title: string
  est_minutes: number
  suggested_keyword: string  // NEW
}
```

**计算**：选 topic 内 hit_count 最小的 keyword。命中数为 0 的 keyword 排前面；并列时按字母序。

**LLM 使用建议**：`prompts/SOUL.md` 不变（不强制 LLM 用 suggested_keyword），但 LLM 拿到后自然会倾向以该词为切入点。这是个软引导而非硬约束。

**向后兼容**：新增字段不影响现有 LLM 处理逻辑——LLM 已经会忽略不认识的字段。

### 2.6 API 响应扩展：`GET /api/topics`

当前返回：
```json
[{ "name": "...", "keywords": [...], "description": "..." }]
```

新返回：
```json
[{
  "name": "travel",
  "description": "Travel & Adventure",
  "keywords": ["trip", "vacation", "holiday", ...],
  "hitCount": 3,
  "keywordHits": { "trip": 3, "vacation": 2, "holiday": 1 }
}, {
  "name": "tech_future",
  "description": "Tech & future",
  "keywords": ["ai", "robot", ...],
  "hitCount": 0,
  "keywordHits": {}
}]
```

**0 命中 topic 仍返回**（不隐藏），便于用户在 UI 上看到"哪些话题从没被聊过"。

**`PUT /api/topics` 字段白名单**：客户端发送 PUT 时只携带 `name, keywords, description`；服务端收到 body 后**只取这三个字段**，忽略 `hitCount` / `keywordHits`。即使前端误传也不会污染统计。

```typescript
// src/server.ts
const body = (await c.req.json()) as Array<...>
const clean = body.map(({name, keywords, description}) => ({name, keywords, description}))
```

### 2.7 Web UI 展示

`web/src/components/TopicLibraryPage.tsx` 每个话题卡片改为：

```
+----------------------------------------------------------+
| Travel & Adventure  (travel)            (3)     [编辑]  |
| [trip (3)] [vacation (2)] [holiday (1)] [beach (0)] ... |
+----------------------------------------------------------+
```

具体规则（已与用户确认）：

| 元素 | 位置 | 样式 |
|---|---|---|
| `(hitCount)` | 话题名右侧，编辑按钮左侧 | 与 slug `(travel)` 同样的灰字小字 |
| `keyword (N)` | keyword chip 内 | chip 文本格式：`keyword (hitCount)` |
| 0 命中 | 不淡化 | 与其他命中数完全相同的样式（用户确认） |
| 排序 | 字母序 | 保持当前 |

**为什么 chip 内而非 chip 外**：用户确认（v2 设计确认点 #1）。chip 外（角标）需要新设计 chip 角标位置，复杂度更高。

## 4. 风险与边界

### 4.1 风险

| 风险 | 缓解 |
|---|---|
| Topic 重命名 → orphan keyword_hits 行 | 低频操作；v1.1 清理（backlog B2）|
| Topic 删除 → orphan keyword_hits 行 | 同上 |
| `W_KEYWORD=0.05` 调得不准，导致选择行为异常 | backlog B1：根据实际命中分布动态调 |
| 0 命中 topic 数量大（当前 23/30），UI 噪音 | 用户已确认不淡化、不隐藏；信息可见性优先 |
| `keyword_hits` 累积后 `GET /api/topics` 返回体积膨胀 | 30 topic × ~20 keyword = 600 行，最坏情况几 KB，可接受 |

### 4.2 边界

- 写入只在 session 正常结束时（CLI `finally`、Web 流水线）。V751-002 catch-all 路径走 placeholder summary，keywords 为空，不触发 keyword_hits 写入
- keyword 匹配仍走 `matchTopic` 的 case-insensitive 规范化；`upsertMany` 入参也用小写，与存储一致
- `topic-counter.ts` 内存 Map 在 server restart 时清空。当前 v1.0.2 不做持久化（hotfix 阶段已说明接受此限制）
