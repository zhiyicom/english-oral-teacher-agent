## 背景

2026-06-27 实际对话（session `ddb32b4f`，27 min / 33 turns）中观察到三类问题：
1. **频繁切换 topic** — 27 分钟内切换 11 次，远超合理节奏
2. **assistant bubble 为空 / 只显示 preamble** — 11 of 33 turns 受影响
3. **刷新页面后才看到完整回复** — SSE 2nd-call 响应被前端兼容代码静默丢弃

第 1 类影响"是否过度切题"，属于长期话题选择策略问题——`selectTopic()` 只考虑 `topic_stats.discussion_count`，不知道每个 topic 内部哪些 sub-keyword 已被频繁驱动。

v1.0.2 sprint 包含两部分：

**HOTFIX 阶段**（已 commit `e35d70e`）：
- Bug B：SSE student-text handler 的 `if (!streamingRef.current)` 兼容 shim 删除，2nd-call 响应不再丢失
- 诊断基础设施：per-turn `logTurnDiagnostic` + `POST /api/diagnostic/log` + SessionPage SSE event 计数（opt-in `debug:web_diag=1`）
- Bug A：`topic-counter.ts` 引入 MIN_TOPIC_AGE=5 阈值，强制 topic 在当前至少 5 轮内不切换（显式 `switch topic` / `换个话题` 旁路）

**新功能阶段**（本 sprint 待实施）：
- `keyword_hits` 表记录每个 (topic, keyword) 命中次数
- `selectTopic()` 加 keyword-freshness 评分，倾向选 keyword 命中少的 topic
- `topic_select` 工具返回 `suggested_keyword` 给 LLM 作开场角度
- Web UI `/topics` 页面在话题名右侧展示 `(N)` 命中次数，每个 keyword chip 内显示 `(N)` 子话题命中次数
