## 3. Key achievements

### 3.1 数据层：`keyword_hits` 表 + DAO

- Migration `006_keyword_hits.sql` 热加载，PK `(topic, keyword)` 保证幂等
- `upsertMany(topic, keywords, now)` 整个 topic 内所有 keyword 在一个 SQLite 事务内写入，避免半写状态
- `getAll()` 用于 tool 计算 `avgKeywordHit`，`getByTopic(topic)` 可用于未来 B3 "挂名 keyword 告警"

### 3.2 选择算法：keyword-freshness 评分

- `selectTopic()` score 公式升级：
  ```
  score = -count * 0.1          (W_TOPIC, v0.7.6)
        - avgKeywordHit * 0.05  (W_KEYWORD, v1.0.2 NEW)
        + interest * 0.5        (W_INTEREST, v0.7.6)
        + noise ∈ [-0.2, 0.2]   (v0.7.6)
  ```
- 效果：当两 topic 计数相当时，keyword 命中少者优先，间接压制 "同一个 topic 内反复用同一组 keywords" 倾向
- 向后兼容：未传 `keywordStats` 的旧 caller 行为完全不变（`avgKeywordHit=0`）

### 3.3 Tool 返回扩展：`suggested_keyword`

- `pickFreshestKeyword(topic, stats)`：取 topic 内 hit_count 最小的 keyword（首字母升序打破并列）
- 旧 caller 不传 `keywordStats` 时仍然返回 `suggested_keyword`（从 0 起步的 alphabetical tiebreak）
- 字段为 `?` 可选，对忽略未知字段的 LLM 完全向后兼容

### 3.4 API + UI 展示

- `GET /api/topics` JOIN topic_stats + keyword_hits：每个 topic 都返回统计字段（即使为 0 / {}）
- `PUT /api/topics` 字段白名单：服务端只接受 `name/keywords/description`，即使 client 误传 `hitCount/keywordHits` 也被忽略（防止编辑器误覆盖统计）
- Web UI `/topics`：话题卡片右侧 `(N)` 灰色小字（讨论总次数）；keyword chip 内 `keyword (N)`；保留字母序，不按命中排序
