### 1.2 设置页"取消"按钮

**方案**：在 `web/src/components/SettingsPage.tsx` 的"保存"按钮右侧对称加"取消"按钮（`rounded border border-slate-300 bg-white px-4 py-2 text-slate-700 hover:bg-slate-100`），点击触发 `handleCancel()`：

```ts
async function handleCancel() {
  try {
    const sessions = await listSessions()
    const ended = sessions.find((s) => s.endedAt !== null)
    const target = ended ?? sessions[0]
    if (target) navigate(`/session/${target.id}`)
  } catch { /* best-effort, stay on /settings */ }
}
```

`listSessions()` 在 `src/server.ts` 已按 `started_at DESC` 排序，所以第一个 `endedAt !== null` 的就是"最近已结束"。

## 2. 阶段化话题策略

### 2.1 问题与动机

当前 `topic_select` 工具和 WARM_UP 阶段各自有问题：

| 现象 | 根因 |
|---|---|
| WARM_UP 开场完全靠 LLM 即兴，缺少与上次会话的明确衔接 | `lastReview` 注入存在（v0.5 起）但只给 LLM 自由发挥，无定向 seed |
| `topic_select` 工具的 D3 (interest boost) 是 dead code | `interests: []` 在 `cli.ts:336` / `server.ts:279` 硬编码，user.interests 数据从不流到算法层 |
| `user.interests` 仅在 WARM_UP 的 prompt 文本里被 LLM 读 | 由 `SOUL.md:34` 的"smoothly transition to a new topic from `# STUDENT` interests"承担，无算法强制 |
| 用户期望 WARM_UP 用"上次会话关键词"作开场，MAIN_ACTIVITY 纯按调用次数选 topic | 当前两边都做不到位 |

### 2.2 现状盘点（v1.0.2 已有的相关机制）

**WARM_UP 已有 `lastReview` 注入**（v0.5 + v1.0.1 加强）：

- 数据源：`src/agent/retrieval.ts` `loadLastReview(db)`，返回最近一条 `summary IS NOT NULL AND length(summary) > 30` 的 session
- 注入点 1：`src/agent/turn.ts:429-442` 首轮 stand-alone user message：
  ```
  Last session (X days ago): {summary}
  Keywords: {keywords[0..3]}
  Greet the student first, then make a natural connection to something from last session before moving to a new topic.
  ```
- 注入点 2：`src/agent/context-injector.ts:87-94` `[System Context]` 区块：
  ```
  - Last session (X days ago, Y min): {summary}
  - Last session keywords: {keywords.join(', ')}
  ```

**MAIN_ACTIVITY `topic_select` 工具已有 D3 接口**（v0.7.6 + v1.0.2 加强），但数据没接：

- `src/agent/topic-engine.ts:55-59` `computeInterest()` 实现正确
- `src/agent/tools/topic-select.ts:77-83` `createTopicSelectTool({ interests })` 接受参数
- `src/cli.ts:336` / `src/server.ts:279` 都硬编码 `interests: []`

### 2.3 设计目标

1. **WARM_UP**：在现有 `lastReview` 注入基础上，**额外**给 LLM 一个聚焦的"开场钩子"——由 session-end LLM 一次性从当前 summary 里挑一个适合做社交开场的 keyword。LLM 看到 hook 后第一句就围绕它展开
2. **MAIN_ACTIVITY**：`topic_select` 工具永久不接 user.interests；显式声明这一点（既清掉 dead code 的语义混乱，也给未来重开 D3 留接口）
3. **边界清晰**：WARM_UP 负责兴趣 + 历史衔接；MAIN_ACTIVITY 负责基于调用次数选 topic；两端职责不重叠

### 2.5 详细设计 — WARM_UP 种子钩子

#### 2.5.1 数据来源：profile-extract 输出扩展

`src/agent/profile-extractor.ts` 当前输出：

```ts
export interface StudentDiscoveries {
  newInterests: string[]
  bodyUpdate: string | null
}
```

扩展为：

```ts
export interface StudentDiscoveries {
  newInterests: string[]
  bodyUpdate: string | null
  /** v1.0.3 — single keyword picked by LLM as next session's WARM_UP opener hook. */
  nextWarmUpSeed: string | null
}
```

`EXTRACTION_PROMPT` 末尾追加：

```
- next_warm_up_seed: ONE short keyword (1-3 words) from this session that would
  make a natural, friendly opener for the NEXT session. Prefer something
  light/social (e.g. a hobby, place, food, daily activity), not a heavy topic.
  Pick from the keywords above if any fits; otherwise pick any concrete noun
  from the summary. Return null if nothing suitable.
```

JSON 输出示例：

```json
{
  "new_interests": ["cello", "rock music"],
  "body_update": "Student has been playing cello for 8 years...",
  "next_warm_up_seed": "cello"
}
```

解析处增加对 `next_warm_up_seed` 字段的容错：

```ts
const parsed = JSON.parse(result.content)
return {
  newInterests: Array.isArray(parsed.new_interests) ? parsed.new_interests : [],
  bodyUpdate: typeof parsed.body_update === 'string' ? parsed.body_update : null,
  nextWarmUpSeed: typeof parsed.next_warm_up_seed === 'string' && parsed.next_warm_up_seed.trim()
    ? parsed.next_warm_up_seed.trim()
    : null,
}
```

#### 2.5.2 Server 模块作用域状态

`src/server.ts` 顶层加：

```ts
// v1.0.3 §1.3 — WARM_UP opener hook seeded by last session's profile-extract.
// In-memory only: server restart loses it (acceptable, falls back to no-hook).
let pendingWarmUpSeed: string | null = null
```

session-end 流水线（`server.ts` 大约 460-510 行，profile-extract 调用处附近）写入：

```ts
const discoveries = await extractStudentDiscoveries(
  review.summary,
  systemPrompt.userProfile.interests,
  client,
)
await updateUserSettings({
  interests: discoveries.newInterests,
  bodyAppend: discoveries.bodyUpdate ?? undefined,
})
// v1.0.3 — write next session's WARM_UP opener hook
pendingWarmUpSeed = discoveries.nextWarmUpSeed
```

session-start 流水线读取并清空：

`POST /api/sessions` handler：

```ts
app.post('/api/sessions', (c) => {
  const row = sessions.create()
  // v1.0.3 — return the WARM_UP opener hook so web can pre-warm its state
  // (read-once semantics: cleared after read).
  const warmUpHook = pendingWarmUpSeed
  pendingWarmUpSeed = null
  return c.json({ id: row.id, warmUpHook }, 201)
})
```

为什么在 POST 返回里塞 warmUpHook：前端需要把 hook 跟着 session 走，存到 React state 里。后续 first-turn 调用 `POST /api/sessions/:id/stream` 时，前端把 hook 作为 body 字段回传。详见 §2.5.4。

#### 2.5.4 WARM_UP 注入点

**核心约束**：现有 `lastReview` 注入已包含"Last session" 整段，**不应替换**——它的信息密度高（summary 全文 + 4 keywords），是 LLM 自由发挥的素材。新的 `warm_up_hook` 是**叠加**在它之上的定向 seed。

`src/agent/turn.ts:429-442` 调整为：

```ts
if (wasFirstTurn && input.lastReview?.summary) {
  const summary = input.lastReview.summary
  const kws = (input.lastReview.keywords ?? []).slice(0, 4).join(', ')
  // v1.0.3 — focus the WARM_UP opener with a curated hook (LLM-picked
  // at session-end from the previous summary). Fallback: no directive
  // when no hook is available (e.g. first-ever session, server restart).
  const opener = input.warmUpHook
    ? `Your opener topic for today: "${input.warmUpHook}". Make the first question naturally about this — then chat around the student's interests.`
    : `Make a natural connection to something from last session before moving to a new topic.`
  const hint = [
    `Last session (${input.lastReview.daysAgo} days ago): ${summary}`,
    `Keywords: ${kws}`,
    opener,
  ].join('\n')
  process.stderr.write(`[turn] WARM_UP hint as standalone message\n`)
  callMessages = [
    ...history.slice(0, -1),
    { role: 'user' as const, content: hint },
    lastMsg,
  ]
}
```

**TurnInput 新增字段**：

```ts
export interface TurnInput {
  // ... existing fields ...
  /** v1.0.3 §1.3 — LLM-curated single keyword from session-end, used as
   *  the WARM_UP opener directive. Null when not available (first session,
   *  server restart, LLM failed to pick). */
  warmUpHook: string | null
}
```

**CLI 注入**（`src/cli.ts` 启动时）：

构造 turn input 时加 `warmUpHook` 字段。

#### 2.5.5 注入格式

完整 WARM_UP hint 形态（first-turn-only）：

```
Last session (3 days ago): Student practiced discussing weekend routines and
described their favorite café.
Keywords: café, weekend, routine, food
Your opener topic for today: "café". Make the first question naturally about
this — then chat around the student's interests.
```

vs 无 hook fallback：

```
Last session (3 days ago): Student practiced discussing weekend routines and
described their favorite café.
Keywords: café, weekend, routine, food
Make a natural connection to something from last session before moving to a
new topic.
```

差异只在第三行——前面的 summary + keywords 完全保留。LLM 既能拿到全量素材（自由发挥空间），也有明确的"今天先聊这个"的方向。

### 2.6 详细设计 — MAIN_ACTIVITY 算法调整

#### 2.6.1 `selectTopic()` 接口扩展

`src/agent/topic-engine.ts`：

```ts
export function selectTopic(opts: {
  topics: Topic[]
  stats: TopicStat[]
  interests: readonly string[]
  keywordStats?: KeywordHit[]
  excludeDays?: number
  now?: Date
  rng?: () => number
  /** v1.0.3 §1.3 — default false. When false, D3 (interest boost) is
   *  ignored even if `interests` is non-empty. v1.0.3 call sites all
   *  pass false explicitly; default keeps backwards compat with tests. */
  useInterestBoost?: boolean
}): Topic | null {
  // ... existing code ...
  const keywordStats = opts.keywordStats ?? []
  const useInterestBoost = opts.useInterestBoost ?? false

  // ...

  const scored: TopicCandidate[] = pool.map((t) => {
    const stat = opts.stats.find((s) => s.topic === t.name) ?? null
    const count = stat?.discussionCount ?? 0
    const interest = useInterestBoost ? computeInterest(t, opts.interests) : 0
    const kAvg = avgKeywordHit(t, keywordStats)
    const noise = (rng() * 2 - 1) * NOISE_RANGE
    return {
      topic: t,
      count,
      interest,
      avgKeywordHit: kAvg,
      score: -count * COUNT_WEIGHT - kAvg * KEYWORD_WEIGHT + interest * INTEREST_WEIGHT + noise,
    }
  })

  // ... existing sort + return ...
}
```

`TopicCandidate.interest` 字段在 `useInterestBoost: false` 时为 0；`interest * INTEREST_WEIGHT` 自然归零。

## 4. 风险与边界

### 4.1 风险

| 风险 | 缓解 |
|---|---|
| profile-extract LLM 不稳定，可能不输出 `next_warm_up_seed` | 容错解析：缺字段时写 `null`，session-start fallback 到无 hook 的现有文案 |
| LLM 输出的 hook 太抽象 / 太冷门 / 不适合社交开场 | 文案 prompt 明确要求"1-3 词、轻/社交、具体名词"；失败时 fallback；用户后续可手动调整（v2+ 在 Settings 加 override）|
| `pendingWarmUpSeed` 进程重启丢失 | 设计上接受：fallback 到无 hook，UX 不致命。DB 持久化留 backlog |
| Web 模式下 hook 跟 session 走但 POST /api/sessions/:id/stream 当前 body 可能没解析 | 现有 stream handler 已经做 `c.req.json().catch(() => ({}))`，只需扩 `warmUpHook` 字段 |
| Web 模式下 hook 跨页面刷新丢失 | React state 在 MainPage 启动新会话时拿到 → 跳转 /session/:id 时一并保留（Web 已有 session state 传递机制） |
| topic_select description 改文案后，LLM 仍尝试传 `interests` 之类参数 | tool schema 不接受 `interests`，Zod 会 reject，不影响 |
| 改了 selectTopic 默认值后，老测试期望 D3 起作用 | 加新测试 + 跑 `pnpm test` 全量回归；旧测试用 `useInterestBoost: true` 显式打开 D3 即可（默认 false 不影响显式 true 的路径）|

### 4.2 边界

- WARM_UP 多轮对话切换种子：v1.0.3 一个 session 只用一个种子；多轮随下次会话再重选
- DB 持久化 `pendingWarmUpSeed`：v1.0.3 用模块作用域内存；server 重启丢失可接受
- D1 时间窗口可配置：v1.0.3 硬编码 30 天；配置化属于 backlog
- 按 interests 排序 topic 库：v1.0.3 不做；WARM_UP prompt 文本引导足够
- LLM 实时从 injected keywords 重选种子：v1.0.3 在 session-end 一次性选好存起来；避免 session-start 1-3s 延迟
- 让 `topic_select` 工具接受 `useInterests` 参数：v1.0.3 算法永远关闭 D3；未来要打开，由 `selectTopic.useInterestBoost` 接口控制，不需要外露到工具
