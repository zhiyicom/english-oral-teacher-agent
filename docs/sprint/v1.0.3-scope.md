## 背景

v1.0.2 sprint 落地后立即从用户反馈捕获到一个 Web UI 体验问题：
- **删除会话需要二次确认** — Sidebar 列表点 × 时弹出 `window.confirm()` 弹窗，用户觉得多余，每次删除多一次点击成本。

随后追加：
- **设置页缺退出入口** — 改了一半想退出没有快捷路径（→ §1.2）
- **话题选择未显式区分阶段** — `topic_select` 工具的 interest boost 是 dead code，WARM_UP 开场完全靠 LLM 即兴；用户期望 WARM_UP 用上次摘要的关键词做种子钩子并围绕兴趣闲聊，MAIN_ACTIVITY 则纯基于调用次数选 topic（→ §1.3）

本 sprint 定位为轻量级 UI 打磨 + 小范围话题策略调整。结构上保持可扩展：本文件作为 scope，单独需求用编号记录，后续统一设计 + 测试。

### 1.3 阶段化话题策略：WARM_UP 种子钩子 + MAIN_ACTIVITY 纯计数选 topic

> **来源**：用户反馈。当前 `topic_select` 工具的 D3 (interest boost) 在算法层是 dead code (`interests: []` 硬编码)，同时 WARM_UP 阶段的开场完全靠 LLM 即兴。本节把"基于兴趣的开场"和"基于调用次数的选 topic"显式分工到两个阶段。

| # | 事项 | 文件 |
|---|---|---|
| 6 | ~~`SessionsDao` 新增 `getLatestWithSummary()`~~ → **design 阶段去掉**：现有 `loadLastReview(db)`（`src/agent/retrieval.ts`）已实现此能力，profile-extract 复用其 `LastReview.keywords` 即可，不新增 DAO 方法 | — |
| 7 | profile-extract 在 session-end 输出增加字段 `nextWarmUpSeed`：从当前 summary 里挑一个适合做社交开场的 keyword；prompt 加输出行 `- next_warm_up_seed: <single keyword>` | `src/agent/profile-extractor.ts` |
| 8 | server 模块作用域加 `pendingWarmUpSeed: string \| null`；session-end 写入；session-start 读取后清空 | `src/server.ts` |
| 9 | ~~`prompts/phases.md` WARM_UP Context 增加 `{warm_up_hook}` 占位符~~ → **design 阶段去掉**：WARM_UP 注入已在 `turn.ts:429-442` 走 stand-alone user message 路径，不依赖 phases.md 的 Context 区块机制 | — |
| 10 | `turn.ts` WARM_UP hint 增加定向 seed 行：`Your opener topic for today: "{warmUpHook}". ...`（fallback 到原"natural connection"文案）| `src/agent/turn.ts` |
| 11 | `topic_select` tool description 改文案：删除 "interest boost" 字样，明确说明 D3 永久禁用 | `src/agent/tools/topic-select.ts` |
| 12 | `selectTopic()` 接口新增可选 `useInterestBoost?: boolean`（默认 `false`）；CLI/Server 调用显式 `false` + 注释指向本节 | `src/agent/topic-engine.ts`, `src/server.ts`, `src/cli.ts` |
| 13 | 测试：profile-extractor 返回 `nextWarmUpSeed`、`selectTopic({useInterestBoost:false})` 不受 interests 影响、`POST /api/sessions` 返回 `warmUpHook` 字段并清空模块作用域 | `tests/agent/profile-extractor.test.ts`, `tests/agent/topic-engine.test.ts`, `tests/server/app.test.ts` |
| 14 | 验证：跑完一次有摘要的会话 → 启动新会话 → WARM_UP 第一句应提到上次会话的种子词；多次 `topic_select` 工具调用 description 不含 "interest" 字样 | manual smoke + E2E |

**WARM_UP（0-5 min）行为**：
- 流程：`POST /api/sessions` 时 server 读 `pendingWarmUpSeed`（来自上次 session-end 的 profile-extract），注入 phase context → LLM 在 WARM_UP 第一轮看到形如 `warm_up_hook: 上次 session 聊过的 music` 的钩子
- WARM_UP prompt 文本（追加）："Use `{warm_up_hook}` as your conversation opener; smoothly transition to chatting around the student's interests."
- 无种子 fallback：首次启动 / `pendingWarmUpSeed === null` → 占位符被替换为 `(no prior session)` → 退化为现有 WARM_UP 行为（greet + open question）
- "围绕兴趣闲聊"：现有 `SOUL.md:34` 已写明"smoothly transition to a new topic from `# STUDENT` interests"，无需新 prompt 文本

**MAIN_ACTIVITY（5-25 min）行为**：
- `topic_select` 工具的算法选择：
  - D1 hard exclude（30 天内讨论过的话题）—— **保留**，理由是 UX 防重复安全网，与"基于调用次数"的精神一致
  - D2 `discussion_count` 排序
  - D4 噪声 ±0.2
  - D5 keyword freshness（`avgKeywordHit * KEYWORD_WEIGHT`）
  - D3 interest boost —— **永久禁用**（即使将来 `interests: []` 被填上也不参与）
- `topic_select` tool description 改为：
  > "Pick the next topic based on call-count signals: discussion_count, keyword freshness, and a 30-day hard exclude to avoid repeats. Interest matching happens in WARM_UP phase via prompt — this tool does NOT consider user.interests."
- `selectTopic()` 接口新增可选 `useInterestBoost?: boolean`（默认 `false`）；CLI / Server 调用方全部显式 `false`，并加注释指向 `§1.3`

**边界情况**：

| 情况 | 行为 |
|---|---|
| 首次启动，从未有过 session | `pendingWarmUpSeed === null` → WARM_UP 退化为现状 |
| 上次会话 `summary === '(summarization failed)'` | `getLatestWithSummary()` 跳过该会话，往前找 |
| 上次会话 `keywords === null` 或 `[]` | 同上跳过；最坏情况找不到任何 session → fallback |
| Server 重启 | `pendingWarmUpSeed` 丢失（模块作用域内存）→ 同首次启动 fallback。可接受，UX hint 不是关键路径 |
| USER.md 已声明 interests 但没聊过 session | interests 仅在 WARM_UP prompt 文本作为引导，对 MAIN_ACTIVITY 工具无任何影响 |
| session-end LLM 拒绝输出 `nextWarmUpSeed` | 字段缺失 → server 写入 `null` → 下次 session WARM_UP fallback |

**风险评估**：低-中
- 新增 1 个 DAO 方法 + profile-extract 输出字段 + 模块作用域字段 + WARM_UP 注入：影响面集中在 session 启动/收尾
- topic_select description 改文案 + 显式 `useInterestBoost: false`：算法等价现状（D3 本来就是 dead code），但语义清晰化
- 中风险点：profile-extract 加字段需要 LLM 配合输出新格式 → 通过回归测试 + few-shot 例子兜底；如果 LLM 输出格式不稳定，session-end 流程需 catch 异常并写 null

**范围外**：
- WARM_UP 多轮对话切换种子（v1.0.3 一个 session 只用一个种子）
- DB 持久化 `pendingWarmUpSeed`（v1.0.3 用内存；server 重启丢失可接受，v2+ 再考虑）
- D1 时间窗口可配置（v1.0.3 硬编码 30 天）
- 按 interests 排序 topic 库（v1.0.3 不做；现有 WARM_UP 文本引导足够）
- LLM 实时从 injected keywords 重选（v1.0.3 在 session-end 一次性选好存起来；避免 session-start 1-3s 延迟）
