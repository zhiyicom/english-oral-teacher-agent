## 1. 现状盘点（v1.0.3 baseline `2661f13`）

### 1.1 System Block 0 H1 重复

抽样 turn1 文件 `data/llm-debug/2026-06-27T13-46-54-461Z_ddb32b4f_turn1.txt`，Block 0（cached）实际结构：

```
[Block 0 [CACHED]]:
# SOUL                                              ← 第 10 行（loader 拼）

# SOUL                                              ← 第 11 行（SOUL.md H1）

You are **Alex**, a warm, patient English oral practice partner...

...

# AGENTS                                            ← 第 62 行（loader 拼）

# AGENTS — operating manual                         ← 第 64 行（AGENTS.md H1）

...

# STUDENT                                           ← 第 73 行（loader 拼）

Jeremy is a 13-year-old Chinese middle schooler...  ← USER.md 没 H1

...

# TOOLS                                             ← 第 88 行（loader 拼）

# Tool Calling                                      ← 第 90 行（TOOLS.md H1）
```

**根因**：`src/prompts/loader.ts:109-127` 的 `buildSystemString()` 用硬编码 H1 拼接，但 4 个源文件自身已含 H1（USER.md 例外，它没有 H1）。

**4 个源文件的真实 H1 vs loader 拼的 H1 对照**：

| 源文件 | 文件实际 H1 | loader 拼的 H1 | 一致性 |
|---|---|---|---|
| `SOUL.md` | `# SOUL` | `# SOUL` | ✅ 字面一致 → 完全重复 |
| `AGENTS.md` | `# AGENTS — operating manual` | `# AGENTS` | ❌ 措辞不同 → 重复但带歧义 |
| `USER.md` | （无 H1，YAML frontmatter，被 gray-matter 解析）| `# STUDENT` | ⚠️ loader 必须拼（userProfile 字段无 H1）|
| `TOOLS.md` | `# Tool Calling` | `# TOOLS` | ❌ 措辞不同 → 重复但带歧义 |

**字节影响**：每次 turn 都重复 H1（4 处 × ~12 字节 = **~48 字节/turn**）。纯视觉冗余 + LLM 已忽略，**收益极小**，但语义清洁价值高。

### 1.2 Last session 摘要双处注入 + keywords 数量不对称

WARM_UP turn1 文件的实际产出（`data/llm-debug/..._turn1.txt`）：

**Block 1 `[System Context]` 内（line 287-290）**：
```
[System Context] Phase: WARM_UP | Elapsed: 0.2 min | Silence: 0.0 min | Entered WARM_UP 0.2 min ago
- Last session (11 days ago, 25 min): Jeremy had a low-energy session focused on free conversation rather than structured practice. The teacher attempted to engage him with open-ended questions about his weekend, school, and daily activities, but responses were consistently minimal. The session was brief and ended quickly with a short homework task about naming one of his fish.
- Last session keywords: short responses, low engagement, free conversation, cello, throat, homework task   ← 6 项
- Active topics: travel (3 times, 4 days ago), ...
```

**Messages[0] user 内（line 294-296）**：
```
Last session (11 days ago): Jeremy had a low-energy session focused on free conversation rather than structured practice. The teacher attempted to engage him with open-ended questions about his weekend, school, and daily activities, but responses were consistently minimal. The session was brief and ended quickly with a short homework task about naming one of his fish.
Keywords: short responses, low engagement, free conversation, cello                                        ← 4 项
Greet the student first, then make a natural connection to something from last session before moving to a new topic.
```

**关键发现**：
- 摘要全文重复（Block 1 与 Messages[0] 各发一次）
- Block 1 的 keywords 列表 = `short responses, low engagement, free conversation, cello, throat, homework task`（**6 项**）
- Messages[0] 的 keywords 列表 = `short responses, low engagement, free conversation, cello`（**4 项**，`slice(0, 4)`）
- 这种"Block 1 比 Messages[0] 多 2 个 keywords"的不对称，可能让 LLM 对哪些是"权威 keyword"产生困惑
- Messages[0] 多了 `opener` 指令行（warmUpHook 注入点），Block 1 没有

**两份来源**：
- Block 1：`src/agent/context-injector.ts:87-95` `buildSystemContext()` 在 lastReview 非 null 时输出
- Messages[0]：`src/agent/turn.ts:436-455` 首轮 WARM_UP 注入的 stand-alone user message

**字节影响**：每次 WARM_UP 第一轮，Last session 摘要全文重复一次（~250-400 字节/turn），且 keywords 数量不对称 → LLM 看到 6 项 vs 4 项。

### 1.3 注入路径梳理（v0.5 → v1.0.3 演化）

```
v0.5: 引入 lastReview，仅注入 Messages[0]（作为 WARM_UP 开场引导）
v0.7.5: 拆分 static/dynamic，Block 1 在 dynamic 段
v0.7.6 B3: block 1 添加 per-segment token 计数
v1.0.1: 加重 Block 1 的 Last session 段（加入 keywords 行 + duration 元数据）
v1.0.3 §1.3: 加 warmUpHook 到 Messages[0] 的 opener 行
```

→ Block 1 与 Messages[0] 在 lastReview 处理上**演化到冗余但又各有侧重**（Block 1 有 duration + 6 keyword；Messages[0] 有 opener 指令 + 4 keyword）。

## 2. 设计目标

> **核心约束**（用户审查反馈，2026-06-28）：不能影响任何 LLM 回复效果，不能丢失当前 prompt 的任何语义。

1. **§1.1 拼装层 H1 单一来源**：让源文件自身 H1 是唯一权威；loader 不再硬拼
2. **§1.2 Last session 摘要单一来源 + keywords 数量对齐**：摘要全文只在 Messages[0] 出现一次；Block 1 改为元数据指针；两处 keywords 列表数量一致（统一 6 项）
3. **语义零丢失**：所有关键词、所有时间戳、每段摘要全文都保留在 prompt
4. **行为零回归**：现有所有 turn1 / turn4 测试断言不变（除非显式测试新行为）；A/B replay 对比 LLM 第一句开场无差异
5. **防御性**：H1 缺失时 loader 抛错而非静默输出空白

## 3. §1.1 — `buildSystemString()` 重构 + USER.md 加 H1

### 3.1 目标策略

**H1 责任单一化**：4 个源文件自身都含 H1，loader 不拼任何 H1。这是 v1.0.4 §1.1 的最终策略。

| 源文件 | H1 来源 | v1.0.4 责任 |
|---|---|---|
| `SOUL.md` | 文件已有 `# SOUL` | 不变；loader 不拼 |
| `AGENTS.md` | 文件已有 `# AGENTS — operating manual` | 不变；loader 不拼 |
| `USER.md` | **v1.0.4 新增 `# STUDENT`** | 在 YAML frontmatter 之后、body 第一行之前加 `# STUDENT` H1 |
| `TOOLS.md` | 文件已有 `# Tool Calling` | 不变；loader 不拼 |

**设计权衡（USER.md 是否加 H1）**：

| 候选 | 评价 |
|---|---|
| A. 不加，loader 拼特例 `# STUDENT` | 保留 cue，但 §1.1 仍有 1 个特例，破坏"loader 不拼 H1"原则 |
| B. 不加，loader 不拼，用户用 body 第一行 `Jeremy is a 13-year-old...` 自带 cue | 简洁但 cue 不明显，违反"语义零丢失"严格解读 |
| **C. 加，USER.md 自身含 `# STUDENT` H1** ✅ | 4 个源文件统一，loader 真正不拼任何 H1，原则落地 |

**决定**：采用 C —— 改 `prompts/USER.md` 增加 H1。这是**唯一改动 prompt 文件**的项。

### 3.2 USER.md 改动

**改动前**（line 31-32）：
```
---
name: Jeremy
...
voice_enabled: true
---
Jeremy is a 13-year-old Chinese middle schooler...
```

**改动后**：
```
---
name: Jeremy
...
voice_enabled: true
---
# STUDENT

Jeremy is a 13-year-old Chinese middle schooler...
```

仅在 YAML closing `---` 与 body 之间插入 `# STUDENT` H1。loader 不需要任何代码变更处理此特例。

### 3.3 目标签名

```typescript
export function buildSystemString(sp: SystemPrompt): string {
  const sections = [sp.soul.trim(), '', sp.agents.trim(), '', sp.user]
  if (sp.tools) sections.push('', sp.tools)
  return sections.join('\n')
}
```

**变化点**：
- 删除所有 4 个硬拼 H1（`'# SOUL'`、`'# AGENTS'`、`'# STUDENT'`、`'# TOOLS'`）
- 直接拼接 4 个 trim 后的源文件内容
- 段落间留一个空行（与原版一致）

### 3.4 运行时检查（H1 防御）

新增 helper（loader.ts 内）：

```typescript
function assertHasH1(field: string, body: string, fileName: string): void {
  const firstLine = body.trim().split('\n', 1)[0]?.trim() ?? ''
  if (!/^# \S/.test(firstLine)) {
    throw new Error(
      `[loader] ${fileName} must start with a '# <Title>' heading after YAML frontmatter. ` +
      `Got: ${JSON.stringify(firstLine.slice(0, 40))}. ` +
      `See v1.0.4-design.md §3.4.`,
    )
  }
}
```

在 `loadSystemPrompt()` 里对 4 个段都调用。`USER.md` 因为有 gray-matter 解析后的 body（不含 frontmatter），其首行 `Jeremy is a 13-year-old...` **不匹配** `^# \S` → 会失败。

**所以 `USER.md` 必须先加 `# STUDENT` H1 才能通过 runtime check**。这是 §3.1 设计策略 C 的强制性体现 —— 不加 H1 就启动失败。

### 3.5 边界情况

| 情况 | 行为 |
|---|---|
| 4 个源文件都有 H1 | 通过 runtime check，正常输出 |
| 某源文件忘写 H1 | `loadSystemPrompt()` 抛错 `[loader] <file> must start with...`，启动失败 |
| `tools` 为 null | 保持现状不输出 `# TOOLS` 段；runtime check 只检查 4 个段 |
| `user` (STUDENT) 字段为空 | runtime check 仍然要求 H1 存在；body 为空也通过（firstLine 仍取到 H1）|

### 3.6 测试覆盖

`tests/prompts/loader.test.ts`（可能新增，或合并到现有 loader 测试）：

| Test | 验证 |
|---|---|
| `buildSystemString: 4 段顺序正确且每段 H1 只出现 1 次` | regex 匹配 `# SOUL` count=1, `# AGENTS` count=1, `# STUDENT` count=1, `# Tool Calling` count=1 |
| `buildSystemString: 工具段为空时不输出 # TOOLS 段` | `tools: null` → 输出不含 `# Tool Calling` 也不含空行 |
| `buildSystemString: USER.md body 直接拼接在 # STUDENT 后` | 输出含 `# STUDENT\nJeremy is a...` 完整链 |
| `loadSystemPrompt: 源文件缺 H1 时抛错` | mock SOUL.md 删掉首行 → expect throw 含 "must start with '# <Title>'" |

### 3.7 字节节省

每 turn Block 0 节省 = 4 个 H1 × 平均 ~12 字节 = **~48 字节/turn**。
LLM 计费时实际是按 token，48 字节 ≈ 12 tokens（英文）或 ~16 tokens（中英混）。**节省极小**。

本节价值是**清洁** + **未来源文件改 H1 时只改一处**（如把 `# AGENTS — operating manual` 改成 `# OPERATING MANUAL`，loader 不需要同步）。

## 4. §1.2 — Last session 摘要单一来源 + keywords 对齐

### 4.1 设计原则

**Block 1 仅做"上下文指针"，Messages[0] 做"实际开场"**：
- Block 1（dynamic 段）：输出元数据 + 6 个 keywords + pointer 措辞 → 让 LLM 知道"上次会话聊过这些关键词"+"摘要全文在 Messages[0]"
- Messages[0]（首轮 WARM_UP stand-alone user message）：输出摘要全文 + **6** 个 keywords + opener 指令 → LLM 实际对话时用的内容

**核心约束**：
- **两个位置不能包含同一段摘要文本**（避免冗余）
- **两处的 keywords 数量必须一致**（v1.0.4 统一为 6 项，修复 v1.0.3 的 4 vs 6 不对称）

### 4.2 Block 1 新格式（`context-injector.ts:87-95` 改造）

**改造前**（v1.0.3）：
```typescript
if (lastReview) {
  const dayWord = lastReview.daysAgo === 1 ? 'day' : 'days'
  const durStr = lastReview.durationMin != null ? `${lastReview.durationMin} min` : 'unknown'
  lastSeg = [
    `- Last session (${lastReview.daysAgo} ${dayWord} ago, ${durStr}): ${lastReview.summary}`,
    `- Last session keywords: ${lastReview.keywords.join(', ')}`,
  ].join('\n')
  lines.push(lastSeg)
}
```

**改造后**（v1.0.4）：
```typescript
if (lastReview) {
  const dayWord = lastReview.daysAgo === 1 ? 'day' : 'days'
  const durStr = lastReview.durationMin != null ? `${lastReview.durationMin} min` : 'unknown'
  const kwList = lastReview.keywords.join(', ')
  lastSeg = [
    `- Last session (${lastReview.daysAgo} ${dayWord} ago, ${durStr}) — keywords: ${kwList} (full summary in opening user message)`,
  ].join('\n')
  lines.push(lastSeg)
}
```

**输出对比**（基于 turn1 实际数据）：

| 版本 | 输出 |
|---|---|
| v1.0.3 | `- Last session (11 days ago, 25 min): Jeremy had a low-energy session focused on free conversation...homework task about naming one of his fish.\n- Last session keywords: short responses, low engagement, free conversation, cello, throat, homework task` |
| v1.0.4 | `- Last session (11 days ago, 25 min) — keywords: short responses, low engagement, free conversation, cello, throat, homework task (full summary in opening user message)` |

字节差异：~250-400 字节/turn（仅 WARM_UP 第一轮）。

### 4.3 Messages[0] 改造（`turn.ts:438` keywords slice）

**改造前**（v1.0.3）：
```typescript
const kws = (input.lastReview.keywords ?? []).slice(0, 4).join(', ')
```

**改造后**（v1.0.4）：
```typescript
const kws = (input.lastReview.keywords ?? []).slice(0, 6).join(', ')
```

**仅此一行改动**。其它 hint 逻辑（`Last session (Nd ago): summary` + opener）保持不变。

**输出对比**（基于 turn1 实际数据）：

| 版本 | Keywords 行 |
|---|---|
| v1.0.3 | `Keywords: short responses, low engagement, free conversation, cello`（4 项）|
| v1.0.4 | `Keywords: short responses, low engagement, free conversation, cello, throat, homework task`（6 项）|

→ 与 Block 1 的 keywords 数量对齐。

### 4.4 Messages[0] 其它字段保持现状

`src/agent/turn.ts:436-455` 的 WARM_UP hint 输出**保持现状**：

```typescript
const opener = input.warmUpHook
  ? `Your opener topic for today: "${input.warmUpHook}". Make the first question naturally about this — then chat around the student's interests.`
  : `Greet the student first, then make a natural connection to something from last session before moving to a new topic.`
const hint = [
  `Last session (${input.lastReview.daysAgo} days ago): ${summary}`,
  `Keywords: ${kws}`,   // ← v1.0.4 改 slice 0..6
  opener,
].join('\n')
```

**保留**：
- 摘要全文注入（v1.0.3 已有）
- opener 指令（warmUpHook 有/无分支）

**为什么 Messages[0] 的 `Last session` line 不含 duration？**：
- 历史差异：v0.5 注入 Messages[0] 时没考虑 duration，v1.0.1 加重 Block 1 时加了 duration
- v1.0.4 不强制对齐（收益小，违反"语义零丢失"严格解读不适用 —— duration 信息在 Block 1 已保留）

### 4.5 边界情况

| 情况 | Block 1 输出 | Messages[0] 注入 | 备注 |
|---|---|---|---|
| `lastReview === null`（首次会话）| 不输出 lastSeg（保持现状）| 不注入 hint（保持现状）| 行为零变化 |
| `lastReview.summary` 非空，`keywords: []` | `- Last session (Nd ago, Nm min) — keywords:  (full summary in opening user message)` | `Keywords: `（空字符串）| OK；保持 v1.0.3 行为 |
| `lastReview.durationMin === null` | `Nm min` → `unknown min`（保持 v1.0.3 fallback）| 同 v1.0.3，无 duration | OK |
| 非 WARM_UP 阶段 | Block 1 仍输出 lastSeg（dynamic）| turn.ts 仅 WARM_UP 注入 hint | 保持现状 |
| WARM_UP 但 turn >= 2 | Block 1 输出 lastSeg | turn.ts 不注入 hint（首轮 only）| 保持现状 |

### 4.7 字节节省

**仅 WARM_UP 第一轮**：
- Block 1 减少：~250-400 字节（摘要全文删除，但 keywords 仍 6 项）
- Messages[0] 增加：~30-50 字节（2 个新 keyword + 列表长度扩展）

**净减**：~220-350 字节/turn × 1 turn / session。
LLM token：~55-90 tokens / session。

### 4.8 LLM 行为影响评估

| 风险点 | 评估 |
|---|---|
| LLM 不再从 Block 1 读摘要 → 困惑 | **低风险**：Messages[0] 仍是完整摘要 + Block 1 明确说"full summary in opening user message"；A/B replay 验证必做 |
| LLM 不再看到 duration 在 Messages[0] | 已如此（v0.5 起就没在 Messages[0] 加 duration）|
| Block 1 "pointer" 措辞让 LLM 误以为要"寻找"摘要 | 低风险：措辞明确说"in opening user message"，LLM 训练数据常见 |
| Keywords 数量从 4 扩到 6 是否改变 LLM 行为 | **低风险**：v1.0.3 已经在 Block 1 看到 6 项；Messages[0] 多发 2 项是"补齐信息"，不删除任何已有 keyword |
| Keywords 顺序在两处必须一致吗 | v1.0.3 / v1.0.4 都是按 `keywords.join(', ')` 顺序，保持现状 |

**结论**：v1.0.4 §1.2 风险**低**，所有信息完整保留；唯一变化是 Block 1 摘要全文删除（移至 Messages[0]），属于"位置调整"非"内容删除"。**A/B replay 验证是必做的 DoD 项**。
