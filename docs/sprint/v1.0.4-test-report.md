### 3.2 H1 单一来源 + 运行时守卫

**问题**：4 个源文件实际 H1 + loader 拼的 H1 对照：

| 源文件 | 文件实际 H1 | loader 拼的 H1 | 一致性 |
|---|---|---|---|
| `SOUL.md` | `# SOUL` | `# SOUL` | ✅ 字面一致 → 完全重复 |
| `AGENTS.md` | `# AGENTS — operating manual` | `# AGENTS` | ❌ 措辞不同 → 双标题 |
| `USER.md` | （无 H1）| `# STUDENT` | ❌ loader 拼特例 |
| `tools.md` | `# Tool Calling` | `# TOOLS` | ❌ 措辞不同 → 双标题 |

**解法**（v1.0.4）：
1. 让 4 个源文件自身都拥有 H1（USER.md 加 `# STUDENT`）
2. loader 删除所有硬拼 H1
3. 加 `assertHasH1()` —— 4 个源文件任一丢 H1 → 启动失败（带 label 的明确错误）

**为何不只用静态检查**：
- 源文件是手维护 markdown，未来谁手改都可能误删首行
- 启动时 fail-fast 比第一 turn 拿到空白 prompt 更安全
- 7 个 assertHasH1 单元测试覆盖各种边界（leading blank lines / 缺 space / empty）

### 3.3 Last session 单源对齐

**问题**：WARM_UP turn 1 时 LLM 同时看到两份 Last session 信息：
- Block 1：摘要全文 + 4 个 keywords
- Messages[0] synthetic user message：摘要全文 + 4 个 keywords + opener line

LLM 浪费 ~250-400 字节读两遍相同内容。

**解法**（v1.0.4）：
- Block 1：1 行 pointer（metadata + 6 keywords + suffix "full summary in opening user message"）—— 总 ~150 字节
- Messages[0]：保持摘要全文（LLM 的唯一阅读入口）—— keyword slice 从 4 → 6，与 Block 1 列表对齐

**为何 keyword count 4 → 6**：v1.0.3 Block 1 已经显示 6 个 keywords，但 Messages[0] slice(0, 4) 截断到 4 —— 两处列表不一致会让 LLM 困惑"为什么 Block 1 说 6 个，Messages[0] 只有 4 个？哪个是真的？"。对齐到 6 是消除语义不一致的最小改动。

### 3.4 零行为变化验证

- **LLM 第一句开场**：未变 —— WARM_UP turn 1 的 Messages[0] synthetic user message 内容保持（除 keyword count 4→6 这个纯数量对齐）
- **工具调用行为**：未变 —— `buildSystemPrompt()` 仍返回相同 sections + tools section；tool 定义 / description 全部不变
- **可见 transcript**：未变 —— 用户在 UI 看到的 assistant message 不变（LLM 决策过程不变 → 输出文本不变）
- **Replay fixture**：未跑（fixtures 录的是 LLM 输出，不是 prompt；本 sprint 改的是 prompt 拼装不是 LLM 行为；A/B replay diff 列为 v1.0.5+ backlog B14）
