## 1. 现状盘点（v1.0.4 baseline `5f32f42`）

### 1.1 进程模型

```
pnpm dev-web          concurrently 启两个进程：
  ├─ pnpm serve        → tsx watch src/server.ts     （Hono + API + SSE）
  └─ pnpm --dir web dev → vite (port 5173, HMR)      （React SPA）

pnpm start            → node dist/cli.js             （CLI mode, REPL）
```

**v0.8.5 已埋的伏笔**：`src/server.ts:758-784` 在 `existsSync(web/dist/index.html)` 时 fallback SPA：

```typescript
const distIndex = resolve('web/dist/index.html')
if (existsSync(distIndex)) {
  const distDir = resolve('web/dist')
  app.get('/assets/*', /* serve from distDir */)
  app.get('/*', /* return distIndex HTML */)
}
```

**问题**：`pnpm build` 跑完后产物在两处：
- `web/dist/` （Vite 输出）
- `dist/cli.js` + `dist/server.js` （tsc 输出）

`pnpm start` 跑 `dist/server.js` 时，CWD 是项目根，`resolve('web/dist/index.html')` **能找到**（因为 `web/dist` 确实存在），所以 SPA fallback 实际是工作的。但 `pkg` 打包后，**项目根就没有 `web/dist/` 了**（Vite 输出没进 bundle），fallback 失效 → Web UI 404。

### 1.2 数据路径

```
APP_DATA_DIR env var （默认 './data'）
  ├─ oral-teacher.db          （SQLite, openDb 默认 dbFilename）
  ├─ preferences.json         （server.ts:251 loadPrefs/savePrefs）
  ├─ llm-debug/               （turn.ts + cli.ts: logTurnDiagnostic 写入）
  └─ （replay fixtures 不在 APP_DATA_DIR；硬编码在 tests/fixtures/replay/）
```

**调用点**：
- `src/cli.ts:244` `openDb({ dataDir: env.APP_DATA_DIR })`
- `src/server.ts:792` `const dataDir = resolve(env.APP_DATA_DIR)`
- `src/server.ts:251` `const prefsPath = resolve(opts.dataDir, 'preferences.json')`

**问题**：
- 默认 `./data` 在项目根 → git status 会冒出来（虽然 gitignored 但 IDE 仍会扫）
- 装到 `C:\Program Files\` 后：**没写权限**（Program Files 需要 admin 写）+ **卸载清空**（Windows Installer 默认清空 install dir）
- replay fixtures 硬编码 `tests/fixtures/replay/`，**装到用户机器后没有这个路径**

## 2. 设计目标

1. **§1.1 单进程默认**：build 后 `pnpm start` 单进程 serve 完整 SPA；dev 模式仍可双进程 + Vite HMR
2. **§1.2 AppData 默认**：未设 `APP_DATA_DIR` 时走平台 fallback；legacy `./data` 兼容；首次运行自动创建目录
3. **§1.3 USER.md 自动 seed**：build-time 嵌入 `USER.md.example`；首次运行从嵌入内容 seed 到 AppData
4. **§1.4 /setup 向导**：API key + profile 两步表单；写入 AppData/.env + AppData/USER.md；status endpoint 驱动重定向

**核心约束**：
- 现有用户用 `APP_DATA_DIR=./data` 显式设置时，**行为与 v1.0.4 完全一致**（back-compat）
- 现有 L1 / L3 / E2E 测试全通过
- 跨平台（Windows / macOS / Linux）行为一致

## 3. §1.1 — 单进程合并

### 3.1 目标策略

**build 后**：`pnpm build` 把 `web/dist/` 复制到 `dist/web/`，server 启动时 `resolve(__dirname, 'web')` 解析（`__dirname` 在 `dist/server.js` 是 `dist/`，所以是 `dist/web/`）。SPA fallback 不再需要 `existsSync` gate —— 永远 true。

**dev 模式**：`pnpm dev-web` 保持不变（双进程 + Vite HMR）。

### 3.2 目标目录结构

```
dist/                                  ← pnpm build 产物
  ├─ cli.js
  ├─ server.js                         ← __dirname = dist/ here
  ├─ agent/                            ← tsc 输出的子目录
  ├─ storage/
  ├─ prompts/                          ← loader.ts 的 ?raw import 嵌入到 server.js bundle
  └─ web/                              ← postbuild 从 web/dist/ 复制
      ├─ index.html
      └─ assets/
```

### 3.3 package.json 改动

```jsonc
{
  "scripts": {
    "dev": "tsx watch src/cli.ts",
    "serve": "tsx watch src/server.ts",
    "dev-web": "concurrently -k -n srv,web -c blue,magenta \"pnpm serve\" \"pnpm --dir web dev\"",
    "build": "pnpm --dir web build && tsc && pnpm build:copy-web",
    "build:copy-web": "node -e \"const fs=require('fs');const path=require('path');const src='web/dist';const dst='dist/web';fs.rmSync(dst,{recursive:true,force:true});fs.cpSync(src,dst,{recursive:true});\"",
    "start": "node dist/server.js",
    "typecheck": "tsc --noEmit && tsc --noEmit -p tsconfig.test.json && tsc --noEmit -p web/tsconfig.json",
    // ...
  }
}
```

**为什么用 inline node 而不是新脚本文件**：
- `build:copy-web` 只跑一行，避免新增 `scripts/copy-web.js` 维护负担
- Windows 上 `pnpm` 调用 `node -e` 没问题（git-bash / PowerShell 都能跑）

### 3.4 src/server.ts 改动

**移除**：
```typescript
// src/server.ts:758-784
const distIndex = resolve('web/dist/index.html')
if (existsSync(distIndex)) {
  // ... SPA fallback
}
```

**替换为**（简化版）：
```typescript
// v1.0.5 §1.1 — always-on SPA fallback. The build script copies web/dist
// to dist/web/, so distDir resolution is always valid in production. Dev
// mode uses `pnpm dev-web` (Vite dev server on a separate port) and does
// not hit this code path.
const distDir = resolve(__dirname, 'web')
const distIndex = join(distDir, 'index.html')
app.get('/assets/*', (c) => {
  const filePath = resolve(distDir, c.req.path.slice(1))
  if (!filePath.startsWith(distDir)) return c.notFound()
  if (!existsSync(filePath)) return c.notFound()
  // ... serve
})
app.get('/*', (c) => {
  if (c.req.path.startsWith('/api')) return c.notFound()
  return c.html(readFileSync(distIndex, 'utf-8'))
})
```

**关键变化**：
- `distDir` 用 `__dirname` 解析（**不依赖 CWD**）→ `node dist/server.js` 在任何目录跑都对
- 移除 `existsSync(distIndex)` gate → build 没跑时仍然 try serve（会 fail，但失败信息明确）

### 3.5 边界情况

| 情况 | 行为 |
|---|---|
| `pnpm build && pnpm start` | `dist/web/index.html` 存在 → SPA 正常 |
| 仅 `pnpm start`（没 build）| `dist/web/index.html` 不存在 → 500 错误（明确失败） |
| `pnpm dev-web`（dev 模式）| Hono server 不在 hot-reload；Vite dev server 在 5173 → 浏览器开 5173；不走 dist/web |
| `pnpm --dir web build` 后没跑 `tsc` | 没有 dist/server.js → 用户必须跑 `pnpm build` 全流程；README 标注 |

### 3.6 测试覆盖

`tests/server/app.test.ts` 新增：

| Test | 验证 |
|---|---|
| `GET / 路径返回 SPA index.html` | `pnpm build` 后，dist/web 存在 → `GET /` 返回 HTML |
| `GET /assets/main.js 返回 JS` | `GET /assets/<file>` 返回对应 MIME |
| `GET /api/* 仍走 API 路由` | `GET /api/health` 不返回 HTML |
| `GET /unknown-spa-path 返回 index.html` | `/history/abc` → HTML（前端路由）|

- 不实现 SSR / prerender（纯 SPA）
- 不内嵌 Vite dev server（复杂、收益小）

## 4. §1.2 — AppData 路径解析

### 4.1 设计原则

**优先级**：
1. `process.env.APP_DATA_DIR` 显式设置 → 用显式值（向后兼容 dev + installer 可指定）
2. 未设置 + `./data/oral-teacher.db` 存在 → 用 `./data`（legacy back-compat + warning）
3. 未设置 + 无 legacy → 走平台 fallback（Windows / macOS / Linux 各异）

**关键不变量**：路径始终是**绝对路径**（不依赖 CWD），调用方无需 `resolve()`。

### 4.2 目标签名

`src/config/paths.ts`：

```typescript
import { existsSync, mkdirSync } from 'node:fs'
import { join, resolve } from 'node:path'
import { homedir, platform } from 'node:os'

/**
 * v1.0.5 §1.2 — resolve the per-user data directory with priority:
 *   1. APP_DATA_DIR env var (explicit override)
 *   2. ./data if it already contains a sessions.db (legacy back-compat)
 *   3. Platform default:
 *      - Windows: %APPDATA%\EnglishOralTeacher\
 *      - macOS:   ~/Library/Application Support/EnglishOralTeacher/
 *      - Linux:   ${XDG_CONFIG_HOME:-~/.config}/english-oral-teacher/
 *
 * Creates the directory (recursively) on first call. Returns an absolute path.
 */
export function getAppDataDir(): string {
  const explicit = process.env.APP_DATA_DIR?.trim()
  if (explicit) {
    const resolved = resolve(explicit)
    ensureDir(resolved)
    return resolved
  }

  const legacyPath = resolve('./data')
  const legacyDb = join(legacyPath, 'oral-teacher.db')
  if (existsSync(legacyDb)) {
    process.stderr.write(
      `[startup] WARN: legacy data found at ${legacyPath}. ` +
      `Set APP_DATA_DIR=./data in .env to keep using it, ` +
      `or move ./data/* to the platform default location.\n`,
    )
    ensureDir(legacyPath)
    return legacyPath
  }

  const platformDefault = platformAppDataDir()
  ensureDir(platformDefault)
  return platformDefault
}

export function getSubDir(name: string): string {
  const base = getAppDataDir()
  const sub = join(base, name)
  ensureDir(sub)
  return sub
}

function platformAppDataDir(): string {
  const name = platform() === 'win32'
    ? 'EnglishOralTeacher'
    : 'english-oral-teacher'
  switch (platform()) {
    case 'win32':
      return join(process.env.APPDATA ?? join(homedir(), 'AppData', 'Roaming'), name)
    case 'darwin':
      return join(homedir(), 'Library', 'Application Support', name)
    default: {
      const xdg = process.env.XDG_CONFIG_HOME?.trim()
      const base = xdg && xdg.length > 0 ? xdg : join(homedir(), '.config')
      return join(base, name)
    }
  }
}

function ensureDir(path: string): void {
  if (!existsSync(path)) mkdirSync(path, { recursive: true })
}
```

## 5. §1.3 — USER.md 自动 seed

### 5.1 设计原则

**问题**：`src/prompts/loader.ts:34-53` 用 `__dirname/../../prompts/USER.md` 路径。pkg 打包后源文件可能不在真实文件系统（snapshot 在 .exe 内）。

**解决方案**：build-time 用 esbuild `?raw` import 把 `USER.md.example` 嵌入到 bundle。运行时不需要 fs 访问该文件。

### 5.2 目标签名

`src/prompts/loader.ts`：

```typescript
import { existsSync, mkdirSync, readFileSync, writeFileSync } from 'node:fs'
import { join } from 'node:path'
// v1.0.5 §1.3 — build-time embed of the example USER.md. esbuild's `?raw`
// suffix embeds the file content as a string at build time, so we don't
// need fs access at runtime (works correctly after pkg snapshot).
import exampleText from './USER.md.example?raw'
import matter from 'gray-matter'
import lockfile from 'proper-lockfile'
import { getAppDataDir } from '../config/paths.js'

export interface UserProfile {
  // ... (unchanged)
}

export interface LoadUserResult {
  body: string
  data: Record<string, unknown>
  seededFromExample: boolean
}

export function loadUserFile(): LoadUserResult {
  const userMdPath = join(getAppDataDir(), 'USER.md')

  if (existsSync(userMdPath)) {
    const raw = readFileSync(userMdPath, 'utf-8')
    const parsed = matter(raw)
    return { body: parsed.content, data: parsed.data, seededFromExample: false }
  }

  // First run: seed USER.md from the embedded example so subsequent
  // settings writes have a real file to update. Use atomic write.
  seedUserMdFromExample(userMdPath)
  const parsed = matter(exampleText)
  return { body: parsed.content, data: parsed.data, seededFromExample: true }
}

function seedUserMdFromExample(targetPath: string): void {
  mkdirSync(getAppDataDir(), { recursive: true })
  const tmpPath = `${targetPath}.tmp.${Date.now()}`
  writeFileSync(tmpPath, exampleText, 'utf-8')
  // Atomic rename (NTFS supports rename-over-existing)
  const { renameSync } = require('node:fs')
  renameSync(tmpPath, targetPath)
}

export async function updateUserSettings(
  updates: Partial<{ /* ... */ }>,
): Promise<void> {
  // v1.0.5 §1.3 — USER.md lives in AppData (was __dirname/../../prompts/USER.md).
  const path = join(getAppDataDir(), 'USER.md')

  // proper-lockfile requires the file to exist before locking.
  // If USER.md doesn't exist yet, seed it first.
  if (!existsSync(path)) {
    loadUserFile()  // Triggers seed
  }

  const release = await lockfile.lock(path, { retries: 3 })
  try {
    const raw = await readFile(path, 'utf8')
    const { data, content } = matter(raw)
    // ... merge logic unchanged ...
  } finally {
    await release()
  }
}
```

**关键变化**：
- `?raw` import 在 `tsconfig` 里需要 `compilerOptions.module = "esnext"` + `moduleResolution = "bundler"`（vitest 用 esbuild bundler 模式，已默认）
- `renameSync` 用 `require('node:fs')` 因为已经 import 了 `existsSync/mkdirSync/readFileSync/writeFileSync` 各自一行（也可以 `import { renameSync }` 加一行）
- `loadUserFile()` 新增 `seededFromExample` 字段；调用方 `parseUserProfile` 不变（不影响行为）

### 5.3 tsconfig 改动

如果当前 `moduleResolution` 不支持 `?raw`：

```jsonc
// tsconfig.json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "Bundler",  // 关键：允许 ?raw import
    // ...
  }
}
```

**验证**：vitest 内部用 esbuild bundler 模式，`?raw` 直接支持。`tsc --noEmit` 需要 `moduleResolution: Bundler`。

### 5.4 边界情况

| 情况 | 行为 |
|---|---|
| AppData/USER.md 已存在 | 正常读取；不动 |
| AppData/USER.md 不存在 | 自动 seed；返回 seededFromExample=true |
| AppData 目录不可写 | mkdir/writeFile 抛错；明确失败 |
| `?raw` import 失败（构建错）| build 阶段就报错（比 runtime fail 好）|
| pkg 打包后 `?raw` 内容正确嵌入 | 验证：手动 pkg 一次后 dist/.../server.js 中 grep `Jeremy is a 13-year-old` 能找到 |
| 现有用户 prompts/USER.md 还在 | 第一次跑新版时不被读取；prompt 系统从 AppData/USER.md 读；旧文件无效。**CHANGELOG 标注** |
| 并发：两个 server 进程同时启动 + 同时 seed | rename atomic；两个都尝试 seed → 第二个覆盖第一个；结果一致（example 内容相同）|

### 5.5 prompts/USER.md.example 文件头注释


## 6. §1.4 — /setup 向导

### 6.1 设计原则

**API Key 加载优先级链**：
1. `process.env.API_KEY`（L3 tests + dev env var injection）
2. `{APP_DATA_DIR}/.env`（wizard 写入路径）
3. `<cwd>/.env`（legacy dev 配置）

**当全部缺失时**：server 启动成功（不抛错），但 `/api/setup/status` 返回 `needsApiKey: true`。Web 端 fetch status 后重定向到 `/setup` 页。

**Profile 加载**：复用 §1.3 的 `loadUserFile()`。首次运行无 USER.md → 自动 seed → `hasUserProfile: true`。

### 6.2 src/config/secrets.ts

```typescript
import { existsSync, readFileSync, writeFileSync, renameSync } from 'node:fs'
import { join } from 'node:path'
import { getAppDataDir } from './paths.js'

/**
 * v1.0.5 §1.4 — API key resolution priority chain:
 *   1. process.env.API_KEY (test + dev injection)
 *   2. {APP_DATA_DIR}/.env (wizard writes here)
 *   3. <cwd>/.env (legacy dev fallback)
 *
 * Returns null when no source has a non-empty value.
 */
export function getApiKey(): string | null {
  const fromEnv = process.env.API_KEY?.trim()
  if (fromEnv) return fromEnv

  const fromAppData = readEnvFile(join(getAppDataDir(), '.env')).API_KEY
  if (fromAppData?.trim()) return fromAppData.trim()

  const fromCwd = readEnvFile(join(process.cwd(), '.env')).API_KEY
  if (fromCwd?.trim()) return fromCwd.trim()

  return null
}

export function isSetupNeeded(): boolean {
  return getApiKey() === null
}

export function setApiKey(key: string): { persisted: string[] } {
  if (!key.trim()) throw new Error('API key cannot be empty')
  const envPath = join(getAppDataDir(), '.env')

  let current: Record<string, string> = {}
  if (existsSync(envPath)) current = readEnvFile(envPath)

  const persisted: string[] = []
  if (current.API_KEY !== key) {
    current.API_KEY = key
    persisted.push('API_KEY')
  }

  const content =
    Object.entries(current)
      .map(([k, v]) => {
        const needsQuote = /\s|["'=#]/.test(v)
        return needsQuote ? `${k}="${v.replace(/"/g, '\\"')}"` : `${k}=${v}`
      })
      .join('\n') + '\n'

  const tmp = `${envPath}.tmp.${Date.now()}`
  writeFileSync(tmp, content, 'utf-8')
  renameSync(tmp, envPath)
  process.env.API_KEY = key  // Update process env so subsequent getApiKey() sees it
  return { persisted }
}

function readEnvFile(path: string): Record<string, string> {
  if (!existsSync(path)) return {}
  const content = readFileSync(path, 'utf-8')
  const result: Record<string, string> = {}
  for (const line of content.split('\n')) {
    const trimmed = line.trim()
    if (!trimmed || trimmed.startsWith('#')) continue
    const m = trimmed.match(/^([A-Z_][A-Z0-9_]*)=(.*)$/)
    if (m) {
      const val = m[2].replace(/^["']|["']$/g, '')
      result[m[1]] = val
    }
  }
  return result
}
```

### 6.5 server.ts 新增 endpoints

```typescript
// v1.0.5 §1.4 — /setup endpoints. The web UI fetches /setup/status on
// load and redirects to /setup if needsApiKey is true. The wizard
// posts to /setup/api-key and /setup/profile to write to AppData.

app.get('/api/setup/status', (c) => {
  const needsApiKey = isSetupNeeded()
  let hasUserProfile = false
  try {
    const { data } = loadUserFile()
    hasUserProfile = Boolean(
      typeof data.name === 'string' && data.name.length > 0 &&
      typeof data.age === 'number'
    )
  } catch { /* missing = false */ }
  return c.json({
    needsApiKey,
    hasUserProfile,
    appDataDir: getAppDataDir(),
    version: getVersion(),
  })
})

app.get('/api/setup/profile-default', (c) => {
  // Use the embedded example as default. UI pre-fills the form.
  const { data } = loadUserFile()
  return c.json({
    name: data.name ?? '',
    age: data.age ?? 13,
    level: data.level ?? 'intermediate',
    goals: data.goals ?? [],
    interests: data.interests ?? [],
  })
})

app.post('/api/setup/api-key', async (c) => {
  const body = await c.req.json()
  const key = typeof body.apiKey === 'string' ? body.apiKey.trim() : ''
  if (!key) return c.json({ error: 'apiKey required' }, 400)
  try {
    const { persisted } = setApiKey(key)
    return c.json({ ok: true, persisted })
  } catch (err) {
    return c.json({ error: (err as Error).message }, 500)
  }
})

app.post('/api/setup/profile', async (c) => {
  const body = await c.req.json()
  const name = typeof body.name === 'string' ? body.name.trim() : ''
  const age = Number(body.age)
  const level = body.level
  if (!name) return c.json({ error: 'name required' }, 400)
  if (!Number.isFinite(age) || age < 3 || age > 120) {
    return c.json({ error: 'age must be 3..120' }, 400)
  }
  if (level !== 'beginner' && level !== 'intermediate' && level !== 'advanced') {
    return c.json({ error: 'level must be beginner|intermediate|advanced' }, 400)
  }
  await updateUserSettings({
    // updateUserSettings merges into frontmatter; need to ensure clean values
    bodyAppend: undefined,  // not appending to body here
    interests: Array.isArray(body.interests) ? body.interests.map(String) : [],
  })
  // Note: updateUserSettings doesn't expose name/age/level directly.
  // We need a richer API or do a manual write here.
  await writeProfileToUserMd({ name, age, level, goals: body.goals, interests: body.interests })
  return c.json({ ok: true })
})
```

### 6.7 web/src/App.tsx 改动

```tsx
import { useEffect, useState } from 'react'
import { BrowserRouter, Route, Routes, useNavigate } from 'react-router-dom'
// ... 现有 imports ...
import SetupPage from './components/SetupPage.tsx'

function SetupGate({ children }: { children: React.ReactNode }) {
  const navigate = useNavigate()
  const [checked, setChecked] = useState(false)

  useEffect(() => {
    if (window.location.pathname.startsWith('/setup')) {
      setChecked(true)
      return
    }
    fetch('/api/setup/status')
      .then((r) => r.json())
      .then((s) => {
        if (s.needsApiKey) {
          navigate('/setup', { replace: true })
        }
      })
      .catch(() => { /* network error: let through, may fail later */ })
      .finally(() => setChecked(true))
  }, [navigate])

  if (!checked) return null
  return <>{children}</>
}

export default function App() {
  return (
    <BrowserRouter>
      <SetupGate>
        <div className="flex h-screen bg-white">
          <SessionSidebar />
          <div className="flex-1 overflow-y-auto">
            <Routes>
              <Route path="/" element={<WelcomePage />} />
              <Route path="/session/:id" element={<SessionPage />} />
              <Route path="/history/:id" element={<HistoryPage />} />
              <Route path="/settings" element={<SettingsPage />} />
              <Route path="/topics" element={<TopicLibraryPage />} />
              <Route path="/setup" element={<SetupPage />} />
            </Routes>
          </div>
        </div>
      </SetupGate>
    </BrowserRouter>
  )
}
```

**为什么用 gate 而不是 redirect-only**：
- 避免 race condition（routes 已 render 时还没收到 status）
- 用户直接访问 `/setup?key=xxx` 不被 gate 拦截
- Gate 失败（network error）时不强制 redirect，让用户看到 SPA（LLM 调用时再报错）

### 6.8 边界情况

| 情况 | 行为 |
|---|---|
| 全部 API_KEY 来源缺失 | server 启动 OK；`/api/setup/status` 返回 needsApiKey=true；UI 跳 `/setup` |
| `process.env.API_KEY` 已设 | `getApiKey()` 直接返回；UI 不跳 `/setup` |
| AppData/.env 有 API_KEY 但 LLM 401 | 用户回 `/setup` 重输（`/` 直接访问也行，会触发 gate）；setApiKey 覆盖写入 |
| Profile 部分字段缺 | POST 400，SetupPage 显示 inline error |
| 用户在 /setup 中途关闭浏览器 | 状态不持久化；下次启动 /setup 又走一遍 |
| `API_KEY=` （空字符串）| `getApiKey()` 视为缺失（trim 后空）|
| API key 含特殊字符（`#`、`=`、`"`）| `setApiKey` 自动加引号转义 |
| 多个进程同时写 AppData/.env | 后写者覆盖（无 lock）；INSTALLER + DEV 不应同时跑 |
| `API_KEY` env var 已设但用户想在 UI 改 | UI 不暴露编辑（env var 优先级最高）；要改需 unset env + restart |

## 8. 风险与回退

| 风险 | 概率 | 影响 | 回退方案 |

| 多个 server 进程同时 setup 写同一文件 | 低 | 中 | rename atomic；内容一致（example 同源）；并发问题通过 Inno Setup "single instance" 避免（v1.0.6）|
