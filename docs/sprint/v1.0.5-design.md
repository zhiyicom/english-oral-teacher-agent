## 1. 现状盘点（v1.0.4 baseline `5f32f42`）

### 1.1 进程模型

```
pnpm dev-web          concurrently 启两个进程：
  ├─ pnpm serve        → tsx watch src/server.ts     （Hono + API + SSE）
  └─ pnpm --dir web dev → vite (port 5173, HMR)      （React SPA）

pnpm start            → node dist/cli.js             （CLI mode, REPL）
```

**v0.8.5 已埋的伏笔**：`src/server.ts:758-784` 在 `existsSync(web/dist/index.html)` 时 fallback SPA：

```typescript
const distIndex = resolve('web/dist/index.html')
if (existsSync(distIndex)) {
  const distDir = resolve('web/dist')
  app.get('/assets/*', /* serve from distDir */)
  app.get('/*', /* return distIndex HTML */)
}
```

**问题**：`pnpm build` 跑完后产物在两处：
- `web/dist/` （Vite 输出）
- `dist/cli.js` + `dist/server.js` （tsc 输出）

`pnpm start` 跑 `dist/server.js` 时，CWD 是项目根，`resolve('web/dist/index.html')` **能找到**（因为 `web/dist` 确实存在），所以 SPA fallback 实际是工作的。但 `pkg` 打包后，**项目根就没有 `web/dist/` 了**（Vite 输出没进 bundle），fallback 失效 → Web UI 404。

### 1.2 数据路径

```
APP_DATA_DIR env var （默认 './data'）
  ├─ oral-teacher.db          （SQLite, openDb 默认 dbFilename）
  ├─ preferences.json         （server.ts:251 loadPrefs/savePrefs）
  ├─ llm-debug/               （turn.ts + cli.ts: logTurnDiagnostic 写入）
  └─ （replay fixtures 不在 APP_DATA_DIR；硬编码在 tests/fixtures/replay/）
```

**调用点**：
- `src/cli.ts:244` `openDb({ dataDir: env.APP_DATA_DIR })`
- `src/server.ts:792` `const dataDir = resolve(env.APP_DATA_DIR)`
- `src/server.ts:251` `const prefsPath = resolve(opts.dataDir, 'preferences.json')`

**问题**：
- 默认 `./data` 在项目根 → git status 会冒出来（虽然 gitignored 但 IDE 仍会扫）
- 装到 `C:\Program Files\` 后：**没写权限**（Program Files 需要 admin 写）+ **卸载清空**（Windows Installer 默认清空 install dir）
- replay fixtures 硬编码 `tests/fixtures/replay/`，**装到用户机器后没有这个路径**

## 2. 设计目标

1. **§1.1 单进程默认**：build 后 `pnpm start` 单进程 serve 完整 SPA；dev 模式仍可双进程 + Vite HMR
2. **§1.2 AppData 默认**：未设 `APP_DATA_DIR` 时走平台 fallback；legacy `./data` 兼容；首次运行自动创建目录

**核心约束**：
- 现有用户用 `APP_DATA_DIR=./data` 显式设置时，**行为与 v1.0.4 完全一致**（back-compat）
- 现有 L1 / L3 / E2E 测试全通过
- 跨平台（Windows / macOS / Linux）行为一致

## 3. §1.1 — 单进程合并

### 3.1 目标策略

**build 后**：`pnpm build` 把 `web/dist/` 复制到 `dist/web/`，server 启动时 `resolve(__dirname, 'web')` 解析（`__dirname` 在 `dist/server.js` 是 `dist/`，所以是 `dist/web/`）。SPA fallback 不再需要 `existsSync` gate —— 永远 true。

**dev 模式**：`pnpm dev-web` 保持不变（双进程 + Vite HMR）。

### 3.2 目标目录结构

```
dist/                                  ← pnpm build 产物
  ├─ cli.js
  ├─ server.js                         ← __dirname = dist/ here
  ├─ agent/                            ← tsc 输出的子目录
  ├─ storage/
  ├─ prompts/                          ← loader.ts 的 ?raw import 嵌入到 server.js bundle
  └─ web/                              ← postbuild 从 web/dist/ 复制
      ├─ index.html
      └─ assets/
```

### 3.3 package.json 改动

```jsonc
{
  "scripts": {
    "dev": "tsx watch src/cli.ts",
    "serve": "tsx watch src/server.ts",
    "dev-web": "concurrently -k -n srv,web -c blue,magenta \"pnpm serve\" \"pnpm --dir web dev\"",
    "build": "pnpm --dir web build && tsc && pnpm build:copy-web",
    "build:copy-web": "node -e \"const fs=require('fs');const path=require('path');const src='web/dist';const dst='dist/web';fs.rmSync(dst,{recursive:true,force:true});fs.cpSync(src,dst,{recursive:true});\"",
    "start": "node dist/server.js",
    "typecheck": "tsc --noEmit && tsc --noEmit -p tsconfig.test.json && tsc --noEmit -p web/tsconfig.json",
    // ...
  }
}
```

**为什么用 inline node 而不是新脚本文件**：
- `build:copy-web` 只跑一行，避免新增 `scripts/copy-web.js` 维护负担
- Windows 上 `pnpm` 调用 `node -e` 没问题（git-bash / PowerShell 都能跑）

### 3.4 src/server.ts 改动

**移除**：
```typescript
// src/server.ts:758-784
const distIndex = resolve('web/dist/index.html')
if (existsSync(distIndex)) {
  // ... SPA fallback
}
```

**替换为**（简化版）：
```typescript
// v1.0.5 §1.1 — always-on SPA fallback. The build script copies web/dist
// to dist/web/, so distDir resolution is always valid in production. Dev
// mode uses `pnpm dev-web` (Vite dev server on a separate port) and does
// not hit this code path.
const distDir = resolve(__dirname, 'web')
const distIndex = join(distDir, 'index.html')
app.get('/assets/*', (c) => {
  const filePath = resolve(distDir, c.req.path.slice(1))
  if (!filePath.startsWith(distDir)) return c.notFound()
  if (!existsSync(filePath)) return c.notFound()
  // ... serve
})
app.get('/*', (c) => {
  if (c.req.path.startsWith('/api')) return c.notFound()
  return c.html(readFileSync(distIndex, 'utf-8'))
})
```

**关键变化**：
- `distDir` 用 `__dirname` 解析（**不依赖 CWD**）→ `node dist/server.js` 在任何目录跑都对
- 移除 `existsSync(distIndex)` gate → build 没跑时仍然 try serve（会 fail，但失败信息明确）

### 3.5 边界情况

| 情况 | 行为 |
|---|---|
| `pnpm build && pnpm start` | `dist/web/index.html` 存在 → SPA 正常 |
| 仅 `pnpm start`（没 build）| `dist/web/index.html` 不存在 → 500 错误（明确失败） |
| `pnpm dev-web`（dev 模式）| Hono server 不在 hot-reload；Vite dev server 在 5173 → 浏览器开 5173；不走 dist/web |
| `pnpm --dir web build` 后没跑 `tsc` | 没有 dist/server.js → 用户必须跑 `pnpm build` 全流程；README 标注 |

### 3.6 测试覆盖

`tests/server/app.test.ts` 新增：

| Test | 验证 |
|---|---|
| `GET / 路径返回 SPA index.html` | `pnpm build` 后，dist/web 存在 → `GET /` 返回 HTML |
| `GET /assets/main.js 返回 JS` | `GET /assets/<file>` 返回对应 MIME |
| `GET /api/* 仍走 API 路由` | `GET /api/health` 不返回 HTML |
| `GET /unknown-spa-path 返回 index.html` | `/history/abc` → HTML（前端路由）|

### 3.7 不在范围

- 不实现 SSR / prerender（纯 SPA）
- 不内嵌 Vite dev server（复杂、收益小）
- 不改 Hono 路由实现

## 4. §1.2 — AppData 路径解析

### 4.1 设计原则

**优先级**：
1. `process.env.APP_DATA_DIR` 显式设置 → 用显式值（向后兼容 dev + installer 可指定）
2. 未设置 + `./data/oral-teacher.db` 存在 → 用 `./data`（legacy back-compat + warning）
3. 未设置 + 无 legacy → 走平台 fallback（Windows / macOS / Linux 各异）

**关键不变量**：路径始终是**绝对路径**（不依赖 CWD），调用方无需 `resolve()`。

### 4.2 目标签名

`src/config/paths.ts`：

```typescript
import { existsSync, mkdirSync } from 'node:fs'
import { join, resolve } from 'node:path'
import { homedir, platform } from 'node:os'

/**
 * v1.0.5 §1.2 — resolve the per-user data directory with priority:
 *   1. APP_DATA_DIR env var (explicit override)
 *   2. ./data if it already contains a sessions.db (legacy back-compat)
 *   3. Platform default:
 *      - Windows: %APPDATA%\EnglishOralTeacher\
 *      - macOS:   ~/Library/Application Support/EnglishOralTeacher/
 *      - Linux:   ${XDG_CONFIG_HOME:-~/.config}/english-oral-teacher/
 *
 * Creates the directory (recursively) on first call. Returns an absolute path.
 */
export function getAppDataDir(): string {
  const explicit = process.env.APP_DATA_DIR?.trim()
  if (explicit) {
    const resolved = resolve(explicit)
    ensureDir(resolved)
    return resolved
  }

  const legacyPath = resolve('./data')
  const legacyDb = join(legacyPath, 'oral-teacher.db')
  if (existsSync(legacyDb)) {
    process.stderr.write(
      `[startup] WARN: legacy data found at ${legacyPath}. ` +
      `Set APP_DATA_DIR=./data in .env to keep using it, ` +
      `or move ./data/* to the platform default location.\n`,
    )
    ensureDir(legacyPath)
    return legacyPath
  }

  const platformDefault = platformAppDataDir()
  ensureDir(platformDefault)
  return platformDefault
}

export function getSubDir(name: string): string {
  const base = getAppDataDir()
  const sub = join(base, name)
  ensureDir(sub)
  return sub
}

function platformAppDataDir(): string {
  const name = platform() === 'win32'
    ? 'EnglishOralTeacher'
    : 'english-oral-teacher'
  switch (platform()) {
    case 'win32':
      return join(process.env.APPDATA ?? join(homedir(), 'AppData', 'Roaming'), name)
    case 'darwin':
      return join(homedir(), 'Library', 'Application Support', name)
    default: {
      const xdg = process.env.XDG_CONFIG_HOME?.trim()
      const base = xdg && xdg.length > 0 ? xdg : join(homedir(), '.config')
      return join(base, name)
    }
  }
}

function ensureDir(path: string): void {
  if (!existsSync(path)) mkdirSync(path, { recursive: true })
}
```

| `src/server.ts:251` | `const prefsPath = join(getAppDataDir(), 'preferences.json')` |

## 5. §1.3 — USER.md 自动 seed

### 5.1 设计原则

**问题**：`src/prompts/loader.ts:34-53` 用 `__dirname/../../prompts/USER.md` 路径。pkg 打包后源文件可能不在真实文件系统（snapshot 在 .exe 内）。

**解决方案**：build-time 用 esbuild `?raw` import 把 `USER.md.example` 嵌入到 bundle。运行时不需要 fs 访问该文件。

### 5.2 目标签名

`src/prompts/loader.ts`：

```typescript
import { existsSync, mkdirSync, readFileSync, writeFileSync } from 'node:fs'
import { join } from 'node:path'
// v1.0.5 §1.3 — build-time embed of the example USER.md. esbuild's `?raw`
// suffix embeds the file content as a string at build time, so we don't
// need fs access at runtime (works correctly after pkg snapshot).
import exampleText from './USER.md.example?raw'
import matter from 'gray-matter'
import lockfile from 'proper-lockfile'
import { getAppDataDir } from '../config/paths.js'

export interface UserProfile {
  // ... (unchanged)
}

export interface LoadUserResult {
  body: string
  data: Record<string, unknown>
  seededFromExample: boolean
}

export function loadUserFile(): LoadUserResult {
  const userMdPath = join(getAppDataDir(), 'USER.md')

  if (existsSync(userMdPath)) {
    const raw = readFileSync(userMdPath, 'utf-8')
    const parsed = matter(raw)
    return { body: parsed.content, data: parsed.data, seededFromExample: false }
  }

  // First run: seed USER.md from the embedded example so subsequent
  // settings writes have a real file to update. Use atomic write.
  seedUserMdFromExample(userMdPath)
  const parsed = matter(exampleText)
  return { body: parsed.content, data: parsed.data, seededFromExample: true }
}

function seedUserMdFromExample(targetPath: string): void {
  mkdirSync(getAppDataDir(), { recursive: true })
  const tmpPath = `${targetPath}.tmp.${Date.now()}`
  writeFileSync(tmpPath, exampleText, 'utf-8')
  // Atomic rename (NTFS supports rename-over-existing)
  const { renameSync } = require('node:fs')
  renameSync(tmpPath, targetPath)
}

export async function updateUserSettings(
  updates: Partial<{ /* ... */ }>,
): Promise<void> {
  // v1.0.5 §1.3 — USER.md lives in AppData (was __dirname/../../prompts/USER.md).
  const path = join(getAppDataDir(), 'USER.md')

  // proper-lockfile requires the file to exist before locking.
  // If USER.md doesn't exist yet, seed it first.
  if (!existsSync(path)) {
    loadUserFile()  // Triggers seed
  }

  const release = await lockfile.lock(path, { retries: 3 })
  try {
    const raw = await readFile(path, 'utf8')
    const { data, content } = matter(raw)
    // ... merge logic unchanged ...
  } finally {
    await release()
  }
}
```

**关键变化**：
- `?raw` import 在 `tsconfig` 里需要 `compilerOptions.module = "esnext"` + `moduleResolution = "bundler"`（vitest 用 esbuild bundler 模式，已默认）
- `renameSync` 用 `require('node:fs')` 因为已经 import 了 `existsSync/mkdirSync/readFileSync/writeFileSync` 各自一行（也可以 `import { renameSync }` 加一行）
- `loadUserFile()` 新增 `seededFromExample` 字段；调用方 `parseUserProfile` 不变（不影响行为）

### 5.3 tsconfig 改动

如果当前 `moduleResolution` 不支持 `?raw`：

```jsonc
// tsconfig.json
{
  "compilerOptions": {
    "module": "ESNext",
    "moduleResolution": "Bundler",  // 关键：允许 ?raw import
    // ...
  }
}
```

**验证**：vitest 内部用 esbuild bundler 模式，`?raw` 直接支持。`tsc --noEmit` 需要 `moduleResolution: Bundler`。

### 5.4 边界情况

| 情况 | 行为 |
|---|---|
| AppData/USER.md 已存在 | 正常读取；不动 |

## 10. §C — 默认 30 话题库 seed

### 10.1 目标

把当前 DB 的 30 个 topic 作为项目出厂快照，commit 到 repo + 用 migration 在新装 DB 时自动 seed，让任何新部署开箱就有 30 个话题。

### 10.2 现状

| 维度 | 现状（v1.0.4 baseline `5f32f42`）|
|---|---|
| 初始 topic 数 | 7（migration `003_topic_stats.sql:21-28` seed：minecraft / school / sports / food / family / movies / music）|
| 用户扩展 | 23 个通过 Web UI 编辑器 + `PUT /api/topics` 增量添加；纯运行时数据，**host 本地**|

| `prompts/topic-library.md` | 81 行，**全部 placeholder**（`### t` / 关键词: placeholder）—— 没人填过 |
| 文档承诺 | `docs/USER_MANUAL.md:259` 写"默认 30 个"——与实际 7 不符，是 doc/code drift |

### 10.3 核心设计

#### 10.3.1 数据流

```
data/topics-default.json                   ← checked-in 30 条（人编辑 / 脚本导出）
       ↓
scripts/export-topics-default.ts           ← tsx 跑一次：DB → JSON + SQL（物理同源）
       ↓
src/storage/migrations/007_topics_default.sql  ← migration 跑一次
       ↓
[新机部署] applyMigrations() → DB 有 30 topics
       ↓
tests/storage/topics-default.test.ts        ← drift test 防止 JSON/SQL 漂移
```

#### 10.3.2 行为矩阵

| 部署前 DB 状态 | 003 跑后 | 007 跑后 | 最终 |
|---|---|---|---|
| 空（新装）| CREATE TABLE only（已移除 7 条 seed）| 30 INSERT OR IGNORE → 30 new | 30 |
| 7 topics（v0.6 老用户 minecraft/school/.../music）| skipped（schema_migrations 已记录）| 30 new（007 名字与 v0.6 无重叠）| 37（用户可手动删 7 个 v0.6 旧名）|
| 30 topics（已编辑过，对应 007 基线名）| skipped | 30 OR IGNORE → 0 new | 30（用户编辑保留）|
| 35 topics（30 基线 + 5 用户自加）| skipped | 30 OR IGNORE → 0 new | 35（用户加的 5 保留）|

**关键不变量**：
- migration 007 **只增不删**：用户 Web UI 后期添加 / 修改的 topic 不会因为重新部署而丢失
- 用户的自定义内容（keywords/description 修改）会**保留**（OR IGNORE 不会用新值覆盖旧值）
- 新装的开箱体验：30 个立刻可用，无需任何配置

#### 10.3.3 文件布局

|---|---|---|

| `src/storage/migrations/007_topics_default.sql` | **NEW** | 30 行 `INSERT OR IGNORE INTO topics (name, keywords_json, description, created_at) VALUES (...)` |
| `src/storage/migrations/003_topic_stats.sql` | **MODIFY** | 移除 7 条 v0.6 starter seed,仅保留 CREATE TABLE（schema_migrations 按文件名去重,旧库保留其 7 个 v0.6 话题）|
| `scripts/export-topics-default.ts` | **NEW** | tsx 跑一次 → DB 读 topics → 写 JSON + 生成 SQL（同源）|

| `data/.gitignore` | **MODIFY** | 加 `!topics-default.json`(内层 .gitignore 优先匹配,需双重开例外) |

### 10.4 data/topics-default.json 格式

```json
[
  {
    "name": "art_culture",
    "keywords": ["art", "culture", "creativity", ...],
    "description": "艺术与文化 (B2)",
    "createdAt": "2026-06-14T16:39:34.703Z"
  },
  ...
]
```

**字段约定**：
- `name`：topic 唯一标识（snake_case），对应 DB `topics.name`（PRIMARY KEY）
- `keywords`：英文词数组，对应 DB `topics.keywords_json`（stringified JSON）
- `description`：显示标题（含水平标签），对应 DB `topics.description`
- `createdAt`：ISO 8601，对应 DB `topics.created_at`

**为什么 keywords 用 array 而不是 stringified JSON**：JSON 文件是给人 / 脚本读的，array 更可读。导出脚本负责 stringify 进 SQL；测试反过来 parse。

### 10.5 migration 007 SQL 模板

```sql
-- 007_topics_default.sql
-- v1.0.5 §C — ship 30 default topics as the project baseline.
-- Companion to data/topics-default.json (the human-readable form).
-- INSERT OR IGNORE makes this idempotent:
--   * Fresh install: 003 仅 CREATE TABLE, 007 seeds 30 → 30
--   * Existing user with v0.6 7 (minecraft/school/.../music): 007 名字无重叠,加 30 → 37
--   * Existing user with 30+ (基线名): OR IGNORE 全部跳过 → 用户编辑保留
--
-- Migration 003 在本 sprint 中已移除 7 条 seed,仅保留 CREATE TABLE
-- (schema_migrations 按文件名去重,旧库保留其 7 个 v0.6 话题)。
--
-- The export script (scripts/export-topics-default.ts) writes both this
-- file and data/topics-default.json from the same in-memory array, so
-- the two are guaranteed to be in sync at generation time.
-- drift test: tests/storage/topics-default.test.ts

INSERT OR IGNORE INTO topics (name, keywords_json, description, created_at) VALUES
  ('art_culture', '["art","culture","creativity",...]', '艺术与文化 (B2)', '2026-06-14T16:39:34.703Z'),
  ('culture_tradition', '["culture","tradition",...]', '风俗与文化 (B1)', '2026-06-14T16:39:34.702Z'),
  -- ... 28 more
  ('work_life_balance', '["balance","work",...]', '工作与生活 (B2)', '2026-06-14T16:39:34.703Z');
```

**关键决策**：

1. **`INSERT OR IGNORE` 而非 `ON CONFLICT DO UPDATE`**：用户的 Web UI 编辑（修 keywords / description）会保留下来——OR IGNORE 跳过冲突行；DO UPDATE 会用 SQL 文件的旧值覆盖用户修改。
2. **移除 migration 003 的 7 条 seed**：原 v0.6 starter topics（minecraft/school/sports/food/family/movies/music）已被 JSON / 007 中的 30 个基线话题（如 school_life、food_drink、sports_health_b2 等）取代，名字无重叠。003 仅保留 CREATE TABLE，由 007 提供全部 30 个基线。schema_migrations 按文件名去重，旧库的 7 个 v0.6 话题不会被自动清理（保留用户选择权，可手动删）。
3. **keywords_json 在 SQL 里手动 stringify**：SQL 不支持 array 类型；JSON 字符串是最紧凑的原生表达。

### 10.6 scripts/export-topics-default.ts

```typescript
/**
 * v1.0.5 §C — one-shot regenerator for the default topic library.
 *
 * Reads the current DB's topics table and writes BOTH:
 *   - data/topics-default.json
 *   - src/storage/migrations/007_topics_default.sql
 *
 * from a single in-memory array, so the two are physically guaranteed
 * to be in sync at generation time. Run after adding/editing topics in
 * the Web UI when you want the new defaults to ship to other hosts.
 *
 * Usage: pnpm tsx scripts/export-topics-default.ts
 *
 * Idempotent: re-running with the same DB overwrites both files with
 * identical content. The drift test (tests/storage/topics-default.test.ts)
 * catches divergence if someone hand-edits one file without the other.
 */
import { Database } from 'better-sqlite3'
import { readFileSync, writeFileSync, mkdirSync } from 'node:fs'
import { join } from 'node:path'

interface SeedTopic {
  name: string
  keywords: string[]
  description: string
  createdAt: string
}

function main() {
  const dbPath = process.env.DB_PATH ?? './data/oral-teacher.db'
  const db = new Database(dbPath)
  const rows = db.prepare(
    'SELECT name, keywords_json, description, created_at FROM topics ORDER BY name'
  ).all() as Array<{ name: string; keywords_json: string; description: string; created_at: string }>
  db.close()

  const seeds: SeedTopic[] = rows.map(r => ({
    name: r.name,
    keywords: JSON.parse(r.keywords_json),
    description: r.description,
    createdAt: r.created_at,
  }))

  // 1. Write JSON
  const jsonPath = './data/topics-default.json'
  writeFileSync(jsonPath, JSON.stringify(seeds, null, 2) + '\n', 'utf-8')
  process.stderr.write(`[export] wrote ${seeds.length} topics to ${jsonPath}\n`)

  // 2. Generate SQL (same in-memory array → physical sync)
  const sqlHeader = `-- 007_topics_default.sql
-- v1.0.5 §C — ship ${seeds.length} default topics as the project baseline.
-- Generated by scripts/export-topics-default.ts — DO NOT EDIT BY HAND.
-- Companion to data/topics-default.json.
-- See tests/storage/topics-default.test.ts for drift test.

INSERT OR IGNORE INTO topics (name, keywords_json, description, created_at) VALUES\n`

  const rows2 = seeds.map(s => {
    const kj = JSON.stringify(s.keywords)
    const desc = s.description.replace(/'/g, "''")  // SQL escape
    return `  ('${s.name}', '${kj}', '${desc}', '${s.createdAt}')`
  }).join(',\n')

  const sql = sqlHeader + rows2 + ';\n'
  const sqlPath = './src/storage/migrations/007_topics_default.sql'
  mkdirSync('./src/storage/migrations', { recursive: true })
  writeFileSync(sqlPath, sql, 'utf-8')
  process.stderr.write(`[export] wrote ${seeds.length} topics to ${sqlPath}\n`)
}

main()
```

**为什么 SQL 文件也由脚本生成而不是手写**：
- JSON 和 SQL 是同一份数据的两种表达——手工维护两份容易漂移
- 脚本从同一 in-memory array 写两份，**物理保证同步**
- 手工编辑 SQL 后漂移由 drift test 兜底

```typescript
})
```

**为什么 drift test 要 apply migrations 到 temp DB**：直接读 SQL 文件 parse `INSERT OR IGNORE` 然后跟 JSON 对比是 fragile（regex 解析 SQL 易错）。跑真 migration 拿真 DB 状态，物理保证等价。

### 10.8 .gitignore 改动

需要**两个文件**都改（git 的目录匹配规则有坑）：

**.gitignore**（根）：

```diff
 # Runtime data (NEVER commit student data) — `data/*` not `data/`, so
 # the !data/topics-default.json exception below can re-include the JSON
 # (a `!` rule cannot un-ignore a file inside an ignored directory).
-data/
+data/*
 *.db
 ...
+
+# v1.0.5 §C — checked-in default topic library (project asset, not user data)
+!data/topics-default.json
```

**data/.gitignore**（内层）：

```diff
 *
 !.gitignore
+# v1.0.5 §C — checked-in default topic library (project asset, not user data)
+!topics-default.json
```

**为什么需要双层例外 + `data/*` 而非 `data/`**：
- `data/` 是运行时数据目录（oral-teacher.db / preferences.json / llm-debug/）—— 都是 host 本地数据，**绝对不能** commit
- `data/topics-default.json` 是项目资产，不是用户数据——它和 SQL migration 内容一致
- git 的 `!` 例外**不能**重新包含一个被忽略目录里的文件——必须把 `data/` 改成 `data/*`（忽略内容而非目录）才生效
- 项目里还有 `data/.gitignore`，它的 `*` 规则会**覆盖**根 `.gitignore` 的 `!data/topics-default.json`，必须在内层也加例外
- 经验证 `git check-ignore -v data/topics-default.json` 输出为空 = 文件未被忽略

### 10.9 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| JSON 和 SQL 文件漂移（人编辑 SQL 但忘改 JSON）| 低 | 中 | drift test 在 CI 失败时阻止 merge；export 脚本可一键重生 |
| 用户后期 Web UI 编辑的 topic 在 DB 里比 JSON 多，重新部署后多出来的不会自动 seed 也不会被删 | 低 | 低 | 设计本意——用户的自定义保留；如需 reset，v1.0.5+ 加 reset 端点（backlog D2）|
