> **分支**: `main`

## 背景

v1.0.6 sprint 计划交付 **Windows 一键安装包**（pkg + Inno Setup）。在动手打包前，业务架构层必须先做 4 项适配，否则安装器面临三个根本性阻塞：

| 阻塞 | 现状 | 装到 Windows 后的后果 |
|---|---|---|
| **必须双进程** | `pnpm serve` + `pnpm --dir web dev` 并行；v0.8.5 加了 SPA fallback 但 gate 在 `existsSync(web/dist/index.html)` 后 | pkg 打成单 .exe 后只能跑 `node dist/server.js`，没法同时启 Vite dev server，**Web UI 404** |
| **数据写在项目根** | `APP_DATA_DIR` 默认 `./data` | 安装到 `C:\Program Files\EnglishOralTeacher\` 后，SQLite / preferences.json / llm-debug/ 全在 Program Files 下，**没写权限 + 卸载清空** |
| **API key 必须手改 .env** | `.env` 是 `API_KEY` 唯一来源 | 安装用户拿到 .exe 后不会编辑 .env，**首启即崩** |
| **prompts/USER.md 写死在 `__dirname/../../prompts/`** | 解析路径相对源文件目录 | pkg 打包后源文件可能不在真实文件系统上（snapshot 在 .exe 内），路径解析失败，**首启即崩** |

这 4 项同时也是 **dev 体验改进**（单进程简化开发；AppData 隔离让 git status 干净；/setup 让新克隆仓库的开发者也能跑），所以独立 sprint。

**核心约束**：
- **行为零回归**：现有 L1 / L3 / E2E 测试全通过；现有用户使用 `APP_DATA_DIR=./data` 显式设置时与 v1.0.4 行为完全一致
- **平台无关**：macOS / Linux 上同等行为（AppData fallback 用 XDG / Library 路径）
- **不做打包**：pkg / Inno Setup / 代码签名 / 自动更新 = v1.0.6 范围

**变更详情**：

`getAppDataDir()` 目标签名：

```typescript
export function getAppDataDir(): string {
  const explicit = process.env.APP_DATA_DIR
  if (explicit && explicit.trim().length > 0) {
    const resolved = resolve(explicit)
    mkdirSync(resolved, { recursive: true })
    return resolved
  }
  // Legacy back-compat: if ./data exists with sessions.db, warn but use it
  const legacyData = resolve('./data')
  if (existsSync(join(legacyData, 'oral-teacher.db'))) {
    process.stderr.write(
      `[startup] WARN: legacy data found at ${legacyData}. ` +
      `To keep using it, set APP_DATA_DIR=./data in .env. ` +
      `Otherwise, new data will be written to the platform-default location.\n`,
    )
    mkdirSync(legacyData, { recursive: true })
    return legacyData
  }
  // Platform default
  const platformDefault = platformAppDataDir()  // Windows: %APPDATA%\EnglishOralTeacher, etc.
  mkdirSync(platformDefault, { recursive: true })
  return platformDefault
}
```

**边界情况**：

| 情况 | 行为 |
|---|---|
| `APP_DATA_DIR` 显式设置 | 用显式值；确保目录存在 |
| `APP_DATA_DIR` 未设，`./data/oral-teacher.db` 存在 | warning + 用 `./data`（向后兼容已有用户）|
| `APP_DATA_DIR` 未设，`./data/` 不存在 | 走 platform fallback |
| platform fallback 路径不可写 | `mkdirSync` 抛错（明确失败，不静默）|
| `%APPDATA%` 未设置（罕见 Linux 容器）| 走 XDG default `${XDG_CONFIG_HOME:-~/.config}/english-oral-teacher/` |

**范围外**：
- 自动把 `./data` 内容迁移到 AppData（仅 warning，迁移是 v1.0.6 安装器或用户手动）
- 把 `APP_DATA_DIR` 路径做成可点击的 Web UI 设置项

**变更详情**：

`loadUserFile()` 改造后：

```typescript
import exampleText from './USER.md.example?raw'

export function loadUserFile(): { body: string; data: Record<string, unknown>; seededFromExample: boolean } {
  const userMdPath = join(getAppDataDir(), 'USER.md')

  if (existsSync(userMdPath)) {
    const raw = readFileSync(userMdPath, 'utf-8')
    const parsed = matter(raw)
    return { body: parsed.content, data: parsed.data, seededFromExample: false }
  }

  // First run: seed from build-time embedded example
  const parsed = matter(exampleText)
  // Atomic write to AppData
  writeFileSync(userMdPath, exampleText, 'utf-8')
  return { body: parsed.content, data: parsed.data, seededFromExample: true }
}
```

**为什么用 `?raw` 而不是 fs.readFileSync**：
- pkg 打包后，`__dirname/../../prompts/USER.md.example` 路径在 snapshot 里，不在真实文件系统 → `fs.readFileSync` 失败
- esbuild 的 `?raw` suffix 把文件内容作为字符串嵌入 bundle，运行时不需要 fs 访问

**边界情况**：

| 情况 | 行为 |
|---|---|
| AppData/USER.md 存在 | 正常读取；不动 |
| AppData/USER.md 不存在，example 可读 | 自动 seed；返回 seededFromExample=true |
| AppData 目录不可写 | 抛错 `[loader] cannot seed USER.md to <path>: <err>`；不静默 |
| `?raw` import 失败（构建错）| 启动失败（build-time error，比 runtime fail 好）|
| 已有用户用 prompts/USER.md（gitignored）| 第一次跑新版时被忽略；新的写入路径在 AppData；旧的 prompts/USER.md 保留但无效。**警告**：CHANGELOG 标注"行为变更" |

**范围外**：
- 把 `prompts/USER.md` 也搬到 AppData（已经是 gitignored，新机制下完全不用）
- USER.md 字段动态发现（保持 REQUIRED_USER_FIELDS 写死）
- 多用户配置（多份 USER.md）

`src/config/secrets.ts`：

```typescript
import { existsSync, readFileSync, writeFileSync } from 'node:fs'
import { join } from 'node:path'
import { getAppDataDir } from './paths.js'

const ENV_FILE_PRIORITY = [
  join(getAppDataDir(), '.env'),
  join(process.cwd(), '.env'),
]

function readEnvFile(path: string): Record<string, string> {
  if (!existsSync(path)) return {}
  const content = readFileSync(path, 'utf-8')
  const result: Record<string, string> = {}
  for (const line of content.split('\n')) {
    const m = line.match(/^([A-Z_][A-Z0-9_]*)=(.*)$/)
    if (m) result[m[1]] = m[2].replace(/^["']|["']$/g, '')
  }
  return result
}

export function getApiKey(): string | null {
  if (process.env.API_KEY?.trim()) return process.env.API_KEY.trim()
  for (const path of ENV_FILE_PRIORITY) {
    const env = readEnvFile(path)
    if (env.API_KEY?.trim()) return env.API_KEY.trim()
  }
  return null
}

export function setApiKey(key: string): void {
  const path = ENV_FILE_PRIORITY[0]  // AppData/.env
  let current: Record<string, string> = {}
  if (existsSync(path)) current = readEnvFile(path)
  current.API_KEY = key
  const content =
    Object.entries(current)
      .map(([k, v]) => `${k}=${v.includes(' ') ? `"${v}"` : v}`)
      .join('\n') + '\n'
  // Atomic write
  const tmp = `${path}.tmp.${Date.now()}`
  writeFileSync(tmp, content, 'utf-8')
  renameSync(tmp, path)
  process.env.API_KEY = key  // Update process env so subsequent getApiKey() sees it
}

export function isSetupNeeded(): boolean {
  return getApiKey() === null
}
```

app.get('/api/setup/status', (c) => {
