### 1.1 触发场景

v1.0.6 §1.13 在 `web/src/components/VoiceInput.tsx` 添加了 Web Speech API 错误提示，但实施粗糙：

### 1.2 暴露的两个问题

#### Bug A：误导性文案

| 场景 | 当前提示 | 实际原因 |
|---|---|---|
| 国内版 Edge + 网络抖动 | "Try Microsoft Edge" | 用户已经在用 Edge，问题不是浏览器 |
| 拒绝麦克风权限 | "Try Microsoft Edge" | 实际是权限问题 |
| 浏览器全局禁用语音识别 | "Try Microsoft Edge" | 实际是浏览器设置问题 |
| 启动后静默不说话 | "Try Microsoft Edge" | 实际是没说话（`no-speech`）|

W3C SpeechRecognitionErrorEvent 定义 6 个错误码 + Chromium 扩展 4 个：

| 错误码 | 含义 |
|---|---|
| `no-speech` | 没有检测到语音 |
| `aborted` | 主动 abort() 打断 |
| `audio-capture` | 麦克风获取失败 |
| `not-allowed` | 用户拒绝权限 |
| `bad-grammar` | SRGS 语法错误（开发者问题）|
| `language-not-supported` | lang 不支持 |
| `network` (Chromium 扩展) | 云端不可达 |
| `service-not-allowed` (Chromium 扩展) | 浏览器全局禁用 |
| `phrase-too-long` (Chromium 扩展) | 单段太长 |
| `audio-busy` (Chromium 扩展) | TTS 占用音频设备 |

当前实现一刀切 = "Try Microsoft Edge"，对所有错误码都成立。

#### Bug B：布局挤压

`<VoiceInput>` 用 `<>...</>` 平级渲染 button + span，外层是 flex：

```tsx
// web/src/components/SessionPage.tsx:574-586
{!ended && (
  <div className="flex gap-2 border-t pt-3">
    <VoiceInput ... />
    <textarea ... />
    <button ...>发送</button>
  </div>
)}
```

`<VoiceInput>` 内部:
```tsx
<>
  <button>🎤</button>
  {showEdgeHint && <span>Speech unavailable in Chrome. Try Microsoft Edge.</span>}
</>
```

→ flex 容器把 `<button>`, `<span>`, `<textarea>`, `<button>发送</button>` 排成一行，提示把 textarea 挤到右边。

## 3. §1.1 — 错误码 → 细分提示

### 3.1 映射表

| 错误码 | STRINGS key | 中文文案 |
|---|---|---|
| `audio-capture` | `voiceErrorAudioCapture` | 无法访问麦克风 |
| `not-allowed` | `voiceErrorNotAllowed` | 需要麦克风权限 |
| `service-not-allowed` | `voiceErrorServiceNotAllowed` | 浏览器禁用了语音识别 |
| `network` | `voiceErrorNetwork` | 识别服务不可达，请检查网络 |
| `no-speech` | `voiceErrorNoSpeech` | 没有听到声音 |
| `language-not-supported` | `voiceErrorLangNotSupported` | 暂不支持该语言 |
| `unknown` / 其他 | `voiceErrorUnknown` | 语音输入出错，请重试 |

### 3.2 函数签名

```ts
// web/src/components/VoiceInput.tsx（内部）
function hintForError(err: string): string {
  switch (err) {
    case 'audio-capture':
      return STRINGS.voiceErrorAudioCapture
    case 'not-allowed':
      return STRINGS.voiceErrorNotAllowed
    case 'service-not-allowed':
      return STRINGS.voiceErrorServiceNotAllowed
    case 'network':
      return STRINGS.voiceErrorNetwork
    case 'no-speech':
      return STRINGS.voiceErrorNoSpeech
    case 'language-not-supported':
      return STRINGS.voiceErrorLangNotSupported
    default:
      return STRINGS.voiceErrorUnknown
  }
}
```

`switch` 而不是对象 lookup —— TypeScript 会在新增错误码时给出 exhaustive 检查提示。


## 4. §1.2 — 提示位置修复

### 4.1 VoiceInput 接口扩展

```ts
// 新增 onHint prop
interface VoiceInputProps {
  onResult: (text: string) => void
  onInterim: (text: string) => void
  onHint?: (msg: string | null) => void  // NEW

```

`onHint(msg | null)` 双向 —— 错误时传 `msg`，清空时传 `null`（重试/超时）。

### 4.2 VoiceInput 内部改造

```tsx
// 旧

  const err = rec.error ?? 'unknown'
  setErrorMsg(err)
  setState('error')
  recognitionRef.current = null
  if (err === 'unknown' || err === 'network') {
    setShowEdgeHint(true)
  }
  setTimeout(() => {
    setState('idle')
    setErrorMsg(null)
  }, 2500)
}

return (
  <>
    <button>🎤</button>
    {showEdgeHint && <span>Speech unavailable in Chrome. Try Microsoft Edge.</span>}
  </>
)

// 新
rec.onerror = () => {
  const err = rec.error ?? 'unknown'
  console.error(`[VoiceInput] SpeechRecognition error: ${err}`)
  setErrorMsg(err)
  setState('error')
  recognitionRef.current = null
  onHint?.(hintForError(err))
  setTimeout(() => {
    setState('idle')
    setErrorMsg(null)
    onHint?.(null)
  }, 2500)
}

// start() 开头加：
const start = useCallback(() => {
  if (!SpeechRecognitionCtor || disabled) return
  onHint?.(null)  // 主动重试时清掉旧提示
  // ...
}, [...])

return <button>🎤</button>  // 不再渲染任何 hint
```

## 7. 边界情况

### 7.1 错误码未来扩展

W3C spec 是 stable 的，但 Chromium 可能在未来加新的扩展码。`hintForError` 的 `default` 分支保证未知错误码也能显示"语音输入出错，请重试"，不至于崩或显示 undefined。

### 7.2 提示期间用户重复点麦克风

- 用户看到 "需要麦克风权限" → 立刻点麦克风按钮 → 重试
- 行为：`onHint?.(null)` 在 `start()` 开头调用 → 旧提示立刻消失
- 下次 `onerror` 时再次触发新的 `onHint?.(msg)`

### 7.3 用户从横线下方 textarea 输入文字

- `voiceHint` 只影响顶部分割线上方的小字
- 不影响 textarea 焦点、键盘事件、disabled 状态

### 7.4 多次连续错误

- 每次 `onerror` 都覆盖 `voiceHint`
- 2.5 秒计时器不重置 —— 第一次的 timer 触发后清空，第二次的 timer 也触发空操作（幂等）

### 7.5 abort() 主动打断

- `aborted` 不在 switch case 中 → 走 default → "语音输入出错，请重试"
- 这是预期行为（用户主动 stop 不应该弹错误）
- 进一步优化可在 `start()` 开头加 `onHint?.(null)`，让 stop 时清空（v1.0.7 不做）

## 11. 话题命中统计与去重重构（F1–F4）

### 11.1 现状与目标

根因诊断见 [v1.0.7-scope.md §8](./v1.0.7-scope.md#8-backlog-详情话题命中统计与去重失效f1f4待确认未实施)。一句话总结：

| # | 根因 | 后果 |
|---|---|---|
| F1 | END 摘要关键词是教学元词（"engagement challenges"、"topic refusal"），与话题库内容关键词几乎不交集 | ~50% 会话 END 时颗粒无收 |
| F2 | `topic_select` 工具只读，`sessions.topics_used` 是死列 | 实际选中的话题全部丢失，唯一记账通道只剩摘要撞库 |
| F3 | `topic_stats` 常年近空 → D1/D2/D5 去重信号饿死 → 选题退化为噪声随机 + LLM 惯性回老话题 | 用户观感"总重复那几个固定话题" |
| F4 | `matchTopic` 只返单个最佳，多话题会话欠计数 | 进一步稀释统计信号 |

**本节设计目标**：

1. **解决 F2**：让 `topic_select` 真正选中的话题在调用时就落账，不再依赖 END 摘要撞库
2. **解决 F1**：让落账口径直接来自工具的 `suggestedKeyword` / `keywords`，与摘要元词脱钩
3. **解决 F4**：每节课记多个话题（而非单 best），保留多话题覆盖
4. **缓解 F3**（统计层）：让 `topic_stats` / `keyword_hits` 真正随每次会话增长，让 D1/D2/D5 三档去重有信号可吃
5. **缓解 F3**（行为层）：在 prompt 侧约束 LLM 不要每轮重选 / 尊重 minAge 拦截后的兜底

业务核心不变量：**现有的 `selectTopic()` 评分公式 / `MIN_TOPIC_AGE=5` gate / `topic_stats` / `keyword_hits` schema 一行不动**，只在 END pipeline 与"采用 → 落账"之间补一层 ledger。

### 11.2 核心思路：write-on-selection

把"采用 → 落账"这一步从 END 后置挪到"采用当时"：

- **何时算"采用"**：
  - **触发点 A**：`turn.ts` MAIN_ACTIVITY 自动注入的话题（保证每节课至少 1 个）
  - **触发点 B**：`turn.ts` A+B 协议里 `topic_select` 真正成功（`minAge` 未拦截、`selectedSlug` 非空）
- **落账内容**：`{ slug, suggestedKeyword, source: 'auto' | 'llm' }`
- **累计位置**：本会话的 `adoptedTopics: Map<slug, { suggestedKeyword, source }>`（dedup by slug）
- **END 时**：遍历 ledger，逐 slug 调 `topicStats.incrementAndUpdate` + `keywordHits.upsertMany(slug, [suggestedKeyword])`；同时写 `sessions.topics_used`

**为什么 dedup by slug**：

- 同一会话里 LLM 可能调 `topic_select` 多次（被 minAge 挡回再调），最终只有 1 个 slug 真正被采用
- 单 slug 在该会话里只算一次 `topic_stats` increment（避免一节课重复刷同一个话题）

**为什么保留 `suggestedKeyword` 不直接散喂所有 `keywords`**：

- 工具的 `keywords[]` 是话题的候选关键词集（~5–8 个），`suggestedKeyword` 是选出的代表词
- 一节课对一个 slug 只记 1 个代表关键词最干净，避免一次会话对 `keyword_hits` 写 5–8 行
- `topics_used` 列里也只存 slug，便于 UI 渲染

### 11.3 数据模型变更

#### 11.3.1 `sessions.topics_used` 列启用

- 列已存在（schema 早期就加了），但目前**零写入路径**：
  - `sessions.ts` `markEnded` 的 `updateEnd` SQL 不包含 `topics_used`
  - `MarkEndedInput` 类型没有 `topics_used` 字段
  - `SELECT_COLS` 也没取这列（UI 端看不到）
- 本次启用：**纯代码层补全，不需 schema migration**。

```ts
// src/storage/sessions.ts
export interface MarkEndedInput {
  sessionId: string
  endReason: SessionEndReason
  summary: string
  keywords: string[]
  discoveries?: StudentDiscovery[]
  endedAt?: number
  topicsUsed?: string[]  // NEW
}

// updateEnd SQL 新增 topics_used = ?
const updateEnd = db.prepare(`
  UPDATE sessions
  SET ended_at = @ended_at,
      end_reason = @end_reason,
      summary = @summary,
      keywords_json = @keywords_json,
      topics_used = @topics_used  -- NEW
  WHERE id = @id
`)
```

`SELECT_COLS` 也加 `topics_used`，UI 端 `GET /api/sessions/:id` 即可拿到。

启动时一次性补丁（防历史 NULL 干扰读取）：

```sql
UPDATE sessions SET topics_used = '[]' WHERE topics_used IS NULL;
```

放到 `src/storage/migrations.ts` 或现有初始化脚本里，幂等可重跑。

```ts
  runtime.adoptedTopics.set(autoTopic.slug, {

```

- 这个 slug 是真实进入对话的话题，比 END 摘要撞库准确得多

### 11.8 END pipeline 重写（`src/server.ts:500-572` 与 `src/cli.ts` endSession）

当前 END 流程：

```
summarize → extractDiscoveries → markEnded → matchTopic(review.keywords) →
  topicStats.incrementAndUpdate(oneSlug) → keywordHits.upsertMany(sumKeywordHits) →
  embed → setEmbedding
```

改造后：

```
summarize → extractDiscoveries →
  IF adoptedTopics.size > 0:
    FOR (slug, {suggestedKeyword}) IN adoptedTopics:
      topicStats.incrementAndUpdate(slug, sessionId, endedAt)
      keywordHits.upsertMany(slug, [suggestedKeyword])  // 仅 1 个代表词
    markEnded({ ..., topicsUsed: [...adoptedTopics.keys()] })
  ELSE:  // §11.9 fallback
    best = matchTopic(review.keywords, topics.list())
    if best:
      topicStats.incrementAndUpdate(best.slug, ...)
      keywordHits.upsertMany(best.slug, review.keywords)
    markEnded({ ..., topicsUsed: best ? [best.slug] : [] })
embed → setEmbedding
```

代码层面：

```ts
// src/server.ts endSession
const adopted = runtime.adoptedTopics

if (adopted.size > 0) {
  for (const [slug, info] of adopted) {
    topicStats.incrementAndUpdate(slug, sessionId, endedAt)
    if (info.suggestedKeyword) {
      keywordHits.upsertMany(slug, [info.suggestedKeyword])
    }
  }
  await sessions.markEnded({
    sessionId,
    endReason,
    summary,
    keywords,
    discoveries,
    endedAt,
    topicsUsed: [...adopted.keys()],
  })
} else {
  // Fallback — §11.9
  const best = matchTopic(review.keywords, topics.list())
  if (best) {
    topicStats.incrementAndUpdate(best.slug, sessionId, endedAt)
    keywordHits.upsertMany(best.slug, review.keywords)
  }
  await sessions.markEnded({
    sessionId, endReason, summary, keywords, discoveries, endedAt,
    topicsUsed: best ? [best.slug] : [],
  })
}
```

抽共享函数 `src/agent/topic-recorder.ts`：

```ts
// src/agent/topic-recorder.ts NEW
export async function recordAdoptedTopics(
  adopted: Map<string, { suggestedKeyword: string; source: 'auto' | 'llm' }>,
  fallbackKeywords: string[],
  ctx: { sessionId: string; endedAt: number },
): Promise<string[]> {
  if (adopted.size > 0) {
    for (const [slug, info] of adopted) {
      topicStats.incrementAndUpdate(slug, ctx.sessionId, ctx.endedAt)
      if (info.suggestedKeyword) {
        keywordHits.upsertMany(slug, [info.suggestedKeyword])
      }
    }
    return [...adopted.keys()]
  }
  const best = matchTopic(fallbackKeywords, topics.list())
  if (best) {
    topicStats.incrementAndUpdate(best.slug, ctx.sessionId, ctx.endedAt)
    keywordHits.upsertMany(best.slug, fallbackKeywords)
    return [best.slug]
  }
  return []
}
```

`src/server.ts` 与 `src/cli.ts` 的 endSession 都调这个函数，避免双份实现漂移。

### 11.9 Fallback：旧 matchTopic 路径保留

如果一节课真的**一次都没采用话题**（极少见：WARM_UP 之后直接 END、或者 LLM 死循环被 abort），`adoptedTopics` 为空。

此时回退到 v1.0.6 行为：

- 走 `matchTopic(review.keywords, topics.list())`
- 仍可能命中（取决于摘要质量）→ 至少有一点统计
- 完全没命中 → `topics_used` 写空数组、`topic_stats` 不动

**不静默失败**：END 响应里返回 `topicsUsed: []`，UI 可显示"本节未匹配话题"。

### 11.10 行为层修复（`prompts/phases.md`）

F3 一半是统计问题（F1/F2/F4），一半是行为问题：诊断日志显示 LLM 在学生拒绝话题后反复调 `topic_select`、被 `minAge=5` gate 挡下、又重选。统计层修好后这些 select 行为本身仍浪费 token。

**Prompt 加固**（MAIN_ACTIVITY 段）：

```markdown
## Topic selection discipline (v1.0.7)

- After a topic is selected (either by `topic_select` tool or auto-injected in MAIN_ACTIVITY),
  **stick with it for the rest of the phase** unless the student explicitly asks to change.
- If `topic_select` returns `topic_select_blocked`, **do not retry in the same phase** —
  respect the gate; either continue with the current topic or use the student-facing fallback
  phrasing suggested by the tool.
- Re-selecting only when: student says "换个话题" / "switch topic" / "another subject", or
  the current topic has been exhausted (≥ 8 turns, no new content).
```

**为什么放 `phases.md` 而不是 `system.md`**：

- `phases.md` 是 phase-specific 指令的现有承载点
- 不污染全局 system prompt
- 与 v0.5 "model C wall-clock phase" 的架构一致（phase-specific 指令由 phase prompt 注入）

### 11.11 关键不变量

| 不变量 | 如何保证 |
|---|---|
| `topic_stats` 单调递增 | END 时逐 adopted slug `incrementAndUpdate`；同 slug 在同一会话 dedup（`adopted: Map` 自动去重） |
| `keyword_hits` 不爆炸 | 每个 adopted slug 只写 1 个代表词（`suggestedKeyword`），不写 `topics[slug].keywords[]` 整组 |
| `sessions.topics_used` 不为 NULL | `markEnded` 总写 `topicsUsed: [...]`（空数组也是数组，不是 NULL）；启动时一次性 `UPDATE topics_used = '[]' WHERE topics_used IS NULL` |
| 现有 happy path 不破坏 | MAIN_ACTIVITY auto-inject、topic_select A+B 既有逻辑一行不动；只多了 ledger set |
| `MIN_TOPIC_AGE=5` gate 不动 | 触发点 A 已经在 gate 之外；触发点 B 已经过 gate 拦截；都不影响 |
| `selectTopic()` 评分公式不动 | D1/D2/D5 信号现在能涨，公式不变，效果自然改善 |
| server 重启丢 ledger = 可接受 | `adoptedTopics` 是运行时 in-memory；server 重启期间进行中的会话本来就在 v0.6 起就用 `topic-counter` 持久化 turn-count 但 in-memory state 本就允许丢（同 F3 类） |
| 前端不强制改 UI | `topics_used` 列就绪即可，UI 端 v1.0.7 不强制展示（v1.0.8 再做） |
| `topic_select` 工具只读契约保留 | 工具本身仍不写库，写库是 caller（`turn.ts`）的责任；这样工具仍可在单元测试中独立验证 |

### 11.12 边界情况

| 场景 | 行为 |
|---|---|
| LLM 调 topic_select 后又被 minAge 挡回 | 不记账（`result.blocked === true`），与既有行为一致 |
| LLM 调 topic_select 同一 slug 选 2 次 | 第 2 次 `adopted.has(slug)` 跳过（Map 自动 dedup） |
| LLM 调 topic_select 选 A，被拒，再选 B | A 不记账（blocked）、B 记账；`adoptedTopics = { B }` |
| MAIN_ACTIVITY auto-inject 后 LLM 又调 topic_select 选同 slug | dedup：`adoptedTopics = { slug }` 单条 |
| 学生说"换个话题"触发显式 switch | `EXPLICIT_SWITCH_RE` 命中 → minAge 旁路 → topic_select 成功 → 记账 |
| WARM_UP 后直接 END（学生跑了） | `adoptedTopics.size === 0` → fallback 走 matchTopic |
| server restart 中途 crash | 进行中会话 ledger 丢失 → END 时 fallback 到 matchTopic（best effort） |
| `suggestedKeyword` 为空字符串 | 防御：`if (info.suggestedKeyword)` 跳过 `keyword_hits` 写入；`topic_stats.incrementAndUpdate` 不受影响 |
| 同 slug 在不同会话重复 | 正常：`incrementAndUpdate` 用 `last_discussed_at` 跨会话累加 |
| `topics_used` 列早已在表里但 SELECT_COLS 未取 | 启用 SELECT_COLS 后 UI 即可读，不需要列 migration |
| LLM 在 MAIN_ACTIVITY auto-inject 同一 turn 内又调 topic_select 选别的 slug | A 点先记账 → B 点成功记账不同 slug → `adoptedTopics` 两条（罕见，OK） |
