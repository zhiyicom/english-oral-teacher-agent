## 背景

v1.0.6 §1.13（commit `56e1348`，Web 侧 Chrome voice Edge hint）引入了语音识别失败时的提示：

```ts
// web/src/components/VoiceInput.tsx
if (err === 'unknown' || err === 'network') {
  setShowEdgeHint(true)
}
// ...
{showEdgeHint && (
  <span className="text-xs text-amber-600">
    Speech unavailable in Chrome. Try Microsoft Edge.
  </span>
)}
```

实际使用中暴露两个问题：

### Bug A：提示对中国 Edge 用户具有误导性

- Edge 浏览器在中国使用的是 **Azure 中国节点**（`*.azure.cn`），不需要翻墙即可直连
- 但 v1.0.6 代码把 `network`/`unknown` 错误一律归到"用错浏览器"
- 国内版 Edge 用户明明在用 Edge，也看到 "Try Microsoft Edge"，**完全没解决实际问题**
- 用户已反馈（2026-07-06）：在 Edge 下开/关 Clash Verge 时，语音输入时而工作、时而失败，提示信息毫无帮助

### Bug B：布局挤压

- `<VoiceInput>` 内部用 `<></>` 把 `<button>` 和错误提示 `<span>` 平级渲染
- 外层 `SessionPage.tsx:576` 的 `<div className="flex gap-2 border-t pt-3">` 是 flex 布局
- 错误提示直接出现在按钮和 `<textarea>` 之间，**把输入框挤到右边**
- 视觉上很乱，影响正常输入

### 根因分析

W3C Web Speech API spec 定义了 6 个 `SpeechRecognitionErrorEvent.error` 错误码 + Chromium 扩展 4 个。当前实现只粗粒度判了 `unknown`/`network`，把所有网络/云端故障一刀切 = "换浏览器"。

**用户实际遇到的错误分布（推测）**：
- 多数情况 = `network`（云端不可达）—— 国内 Edge 应走 Azure 中国节点，直连即可；不应触发"换浏览器"
- 麦克风权限问题 = `not-allowed` 或 `audio-capture` —— 应明确提示"给权限"而不是"换浏览器"
- 浏览器全局禁用 = `service-not-allowed` —— 应明确提示去设置里开
- 静音 = `no-speech` —— 应提示"没听到声音"而不是"换浏览器"

**错误码 → 文案映射**：

| 错误码 | STRINGS key | 中文文案 |
|---|---|---|
| `audio-capture` | `voiceErrorAudioCapture` | 无法访问麦克风 |
| `not-allowed` | `voiceErrorNotAllowed` | 需要麦克风权限 |

**变更详情**：
- `hintForError` 是 `VoiceInput` 内部纯函数，便于以后写单测

- 拔掉网络 / 关掉 Edge STT 访问 → 显示"识别服务不可达，请检查网络"

### 1.2 提示位置移到横线上方居中（解决 Bug B）

| # | 事项 | 文件 |
|---|---|---|
| 5 | `<VoiceInput>` 增加 `onHint?: (msg: string \| null) => void` prop，把错误文案通过 callback 传出，组件本身不再渲染任何提示 | `web/src/components/VoiceInput.tsx` MODIFY |
| 6 | `SessionPage` 新增 `voiceHint: string \| null` state，接收 VoiceInput 的 hint 回调 | `web/src/components/SessionPage.tsx` MODIFY |
| 7 | 输入区容器结构调整：外层 `<div className="border-t pt-3">`（横线在这里），内层 `<div className="flex gap-2">`（按钮 + textarea + send），**上方**插入 `<div className="text-center text-xs text-amber-600 pb-2">{voiceHint}</div>` | `web/src/components/SessionPage.tsx` MODIFY |
| 8 | 启动识别 / 主动重试时调 `onHint?.(null)` 清空旧提示 | `web/src/components/VoiceInput.tsx` MODIFY |

**变更详情**：

旧结构（VoiceInput 内部渲染 hint，会挤压 textarea）：
```tsx
<div className="flex gap-2 border-t pt-3">
  <VoiceInput /> {/* 内部: <button/> <span>hint</span> 平级 */}
  <textarea />
  <button>发送</button>
</div>
```

新结构（hint 提升到 SessionPage，渲染在 border-t 上方居中）：
```tsx
<div className="border-t pt-3">
  {voiceHint && (
    <div className="text-center text-xs text-amber-600 pb-2" data-testid="voice-hint">
      {voiceHint}
    </div>
  )}
  <div className="flex gap-2">
    <VoiceInput onHint={setVoiceHint} />
    <textarea />
    <button>发送</button>
  </div>
</div>
```

**自动消失机制**：
- 沿用 v1.0.6 的 2.5 秒 `setTimeout` —— 但要把清空 hint 也加进去
- `rec.onerror` 内：`onHint?.(hintForError(err)); setTimeout(() => onHint?.(null), 2500)`

**验证**：
- 横线之上（不是行内）显示提示
- 水平居中（`text-center`）
- 输入框不被挤压，宽度正常
- 2.5 秒后自动消失

### 1.3 话题命中统计与去重重构（F1–F4）

**根因诊断完整说明见 §8**。本节给出修复方向概要，详细设计见 [v1.0.7-design.md §11](./v1.0.7-design.md#11-话题命中统计与去重重构f1f4)。

#### 1.3.1 根因四象限

| # | 根因 | 位置 |
|---|---|---|
| **F1** | END 摘要关键词是教学元词（"engagement challenges"、"topic refusal"），与话题库内容关键词几乎不交集 | `src/server.ts:534` `matchTopic(review.keywords, topics.list())` |
| **F2** | `topic_select` 工具只读、`sessions.topics_used` 是死列 | `src/agent/tools/topic-select.ts` + `src/storage/sessions.ts` `markEnded` |

| **F4** | `matchTopic` 只返单个最佳，多话题会话欠计数 | `src/agent/topic-matcher.ts:40-54` |

#### 1.3.2 修复策略：write-on-selection + 共享 ledger

- **核心**：把"采用 → 落账"从 END 后置挪到"采用当时"，落账口径直接来自工具的 `suggestedKeyword` / `keywords`，与摘要元词脱钩。
- **两个触发点**：
  - **A 点**（`turn.ts:392-413`）MAIN_ACTIVITY 自动注入的话题 —— 保证每节课至少 1 个
  - **B 点**（`turn.ts:809-813`）`topic_select` 真正成功（`minAge` 未拦截、`selectedSlug` 非空）—— 记录 LLM 显式采用
- **累计位置**：`adoptedTopics: Map<slug, { suggestedKeyword, source: 'auto' | 'llm' }>`，dedup by slug
- **END 流程重写**：遍历 ledger → `topicStats.incrementAndUpdate(slug)` + `keywordHits.upsertMany(slug, [suggestedKeyword])` → 写 `sessions.topics_used`
- **行为层修复**：`prompts/phases.md` MAIN_ACTIVITY 段加 "Topic selection discipline"，约束 LLM 不在 `topic_select_blocked` 后重试

#### 1.3.4 核心约束

- **不动 `selectTopic()` 评分公式**（D1/D2/D5 三档）；只让信号开始涨
- **不动 `MIN_TOPIC_AGE=5` gate**；A 点本来就在 gate 外，B 点已经过 gate
- **不动 `topic_stats` / `keyword_hits` schema**；只调 `upsertMany` 的入参

- **前端 UI 不强制改**：v1.0.7 把 `topics_used` 数据通道接通即可，UI 端展示留 v1.0.8

## 8. F1–F4 根因诊断（已纳入 §1.3 实施）

> **来源**：2026-07-08 用户实测反馈"选题总是重复那几个固定话题" + "会话结束后 Web UI 话题库命中记录几乎不变"。经代码走查 + `data/oral-teacher.db` 数据核查 + 9 次历史会话重放，两个现象均**确认属实**。**2026-07-08 用户确认纳入 v1.0.7 §1.3 实施**；修复设计见 [v1.0.7-design.md §11](./v1.0.7-design.md#11-话题命中统计与去重重构f1f4)。本节保留作为"问题诊断"的可追溯记录。

### 8.0 现象与证据

**数据库现状**（`data/oral-teacher.db`，9 个已结束会话）：
- `topic_stats` 仅 4 个话题，计数极低：`daily_routine=2`、`food_drink=1`、`friends_social=1`、`school_life=1`
- `keyword_hits` 仅 5 行、命中总和 6
- **`sessions.topics_used` 全部为 NULL**

**重放验证**（用线上同一套 `matchTopic`，threshold=0.1，撞 30 话题库）：

| 会话日期 | 摘要 keywords（节选） | 匹配结果 |
|---|---|---|
| 07-08 efa317d2 | hotpot, friends, school, past tense, conversation, **topic refusal, engagement challenges** | daily_routine，仅 shared=[school] |
| 07-07 0a3396c8 | summer vacation, future tense, picnic, friends, **grammar corrections** | friends_social，仅 shared=[friends] |
| 07-06 20a486ca | summer, favorite activities, **conversation skills, low engagement, speaking practice** | food_drink，仅 shared=[summer] |
| 07-06 124aa05a | （空）| **NO MATCH** |
| 07-05 8b41ff29 | **no practice, brief session, disengagement, early termination** | **NO MATCH** |
| 07-05 d5475672 | free conversation, **warm-up**, cello, daily activities, **casual talk, student-led** | **NO MATCH** |
| 07-05 95e3d209 | **warm-up conversation, past experiences, response clarification, speaking fluency** | **NO MATCH** |
| 07-05 37666d25 | school, summer vacation, folk music, best friend, **scaffolding, low engagement** | daily_routine，仅 shared=[school] |
| 07-04 3a626d17 | exam, math, gaming, gta5, delta force, **grammar correction** | school_life，shared=[exam,math] |

**结论**：9 次会话只有 5 次能记上账、累计只加 6 个 keyword-hit，与数据库现状**逐字吻合**——即"命中记录几乎不变"不是 UI 未刷新，而是库里本来就没涨。

### 8.1 F1 — 命中统计喂错口径（摘要元词撞不上话题库）

- 落库入口：`src/server.ts:534` `matchTopic(review.keywords, topics.list())`（CLI 端 `src/cli.ts` 同构）。
- Summarizer（`prompts/summarizer-system.md` + `src/agent/summarizer.ts`）产出的 keywords 大量是**关于"这节课上得怎么样"的教学元描述**：`engagement challenges`、`topic refusal`、`low engagement`、`scaffolding`、`grammar corrections`、`warm-up`、`beginner-level engagement`、`response clarification` 等。话题库里没有这些词。
- `matchTopic`（`src/agent/topic-matcher.ts:33`）靠这些词与话题库关键词求交集，元词永远不中；命中的几次也常是 7 个词只撞中 1 个内容词（`shared=[school]`）。
- 后果：约一半会话颗粒无收，`topic_stats` / `keyword_hits` 几乎不增长。

### 8.2 F2 — 实际选中的话题从不落账

- `topic_select` 工具（`src/agent/tools/topic-select.ts`）的 `execute()` **纯计算、不写库**。
- `sessions.topics_used` 列全 NULL——LLM 在 MAIN_ACTIVITY 真正选中/讨论的话题，没有任何一条进 `topic_stats`。
- 诊断日志 `data/llm-debug/diag-efa317d2.jsonl` 显示：单节课 `topic_select` 被调十余次（还大量被 `minAge=5` gate 挡下 `topic_select_blocked`），但这些"实际选了什么"全部丢失。
- 唯一记账通道是 END 时那次不可靠的摘要撞库（F1）。**统计量测的是一个很吵的代理指标，而非真实发生的选题。**

### 8.3 F3 — 去重信号被饿死，选题退化

- `selectTopic()`（`src/agent/topic-engine.ts:95`）三个去重档位全靠 `topic_stats` / `keyword_hits` 供数：
  - **D1 30 天硬排除**（`filterHardExclude`）：只有 4 个话题有 `last_discussed_at`，其余 26 个永不被排除；
  - **D2 讨论次数惩罚** + **D5 关键词新鲜度惩罚**：对 26 个话题恒为 0。
- 于是 30 个话题里 26 个分数几乎相等，最终由 `±0.2` 噪声（`NOISE_RANGE`）决定 → **去重实际未生效，近似均匀随机**。
- 叠加行为层问题（见诊断日志）：学生反复拒绝时，LLM 每轮重复调 `topic_select`、被 `minAge` gate 挡回、然后绕回 WARM_UP 锚定的 friends/school/hotpot/游戏；即使 tool 选到 environment/space/art，也常无视 slug 拉回老话题。这是用户观感"总是那几个固定话题"的直接来源。

### 8.4 F4 — 单会话只记 1 个 best 话题

- `matchTopic` 只返回**单个**最佳匹配（`src/agent/topic-matcher.ts:40-54`），END 时也只 `incrementAndUpdate` 一个话题。
- 一节课若真实覆盖多个话题，其余话题被欠计数，进一步稀释统计信号。

### 8.5 候选修复方向（未决，仅备选）

> 以下为讨论素材，**尚未选定**，等用户确认后再细化设计。

1. **让 `topic_select` 选中即写账**（针对 F2/F3）：工具或其 caller（`turn.ts` MAIN_ACTIVITY 分支 / server endSession）在实际采用某 topic 时直接 `topicStats.incrementAndUpdate` + 写 `sessions.topics_used`，不再单纯依赖 END 摘要撞库。
2. **拆分摘要关键词口径**（针对 F1）：Summarizer 输出区分"内容词 / 教学元词"，`matchTopic` 只吃内容词；或改由 LLM 直接输出命中的 topic slug。
3. **END 支持多话题匹配**（针对 F4）：`matchTopic` 返回 Top-N 而非单个，逐个累加。
4. **行为层**（针对 F3 观感）：约束 LLM 在 MAIN_ACTIVITY 内不要每轮重选、尊重 tool 返回的 slug；或复核 `minAge` gate 被挡时的兜底话术。

**风险提示**：改动涉及 `topic_stats` / `keyword_hits` 语义与 `selectTopic` 评分，属核心记忆/去重路径，需要配套 L1 测试与一次历史数据回放校验，不宜与语音提示 sprint 混做，建议单开 sprint。
