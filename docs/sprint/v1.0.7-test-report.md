## 4. 代码审查要点（自审）

### 4.1 `src/agent/topic-recorder.ts` — 核心抽象

**导出**：`AdoptedTopic`、`AdoptedTopicsMap`、`RecordContext`、`recordAdoptedTopics`

**关键签名**：
```ts
recordAdoptedTopics(
  adopted: AdoptedTopicsMap,                 // in-memory ledger
  fallbackKeywords: string[],                // summary.keywords（兜底用）
  ctx: { sessionId, now },
  deps: { topics, topicStats, keywordHits, topicsDao }
): string[]                                  // 返回写入的 slug 列表
```

**逻辑分支**：
- ledger 非空（happy path）→ 遍历 Map，对每个 slug 调 `topicStats.incrementAndUpdate(slug)` + `keywordHits.upsertMany(slug, [suggestedKeyword])`，返回 `[...keys]`
- ledger 为空（cold path：MAIN_ACTIVITY 没注入 + LLM 也没成功调 topic_select）→ 调 `matchTopic(fallbackKeywords, topics)`，命中则增量 + 共享 upsertMany，返回 `[best.topic]`
- 都无命中 → 返回 `[]`，END pipeline 仍照常 `markEnded`

**设计取舍**：
- `upsertMany(slug, [suggestedKeyword])` **只写一个 representative keyword**，不写整个 `keywords[]` 数组。原因：`keyword_hits` 喂的是 D5 dedup 信号（slug 级频次），多写变体词会让表膨胀但去重效果无提升。
- slug-level dedup 由 Map key 自然保证（Hook B 不会重复写同 slug）。

### 4.2 `src/agent/turn.ts` — Hook A / Hook B

**Hook A**（MAIN_ACTIVITY 自动注入分支，~L404）：
- 触发：WARM_UP → MAIN_ACTIVITY 相位切换时刻的 auto-`topic_select`
- 行为：`input.adoptedTopics?.set(slug, { suggestedKeyword, source: 'auto' })` + `yield { type: 'topic-adopted', source: 'auto' }`
- 不受 `MIN_TOPIC_AGE=5min` gate 约束（phase-driven，不是 LLM-driven）

**Hook B**（`topic_select` 工具成功路径，~L846）：
- 触发：LLM 主动调工具且未返回 `error`
- 行为：`if (!adopted.has(slug)) set(...)` — 同 slug 不重复写 + `yield { type: 'topic-adopted', source: 'llm' }`
- 受 5min gate 约束（与之前一致，Hook B 复用原有 gate 路径）

**`TurnEvent` 扩展**：新增 `'topic-adopted'` 联合变体供 SSE/CLI 消费。

### 4.3 `src/storage/sessions.ts` — `topics_used` 列激活

**schema 现状**：`topics_used TEXT` 列在 `001_init.sql`（v0.6）就已存在，但一直是"死列"——没人写、没人读。

**改动**：
- `Session` interface 加 `topics_used: string[] | null`
- `SELECT_COLS` 加 `topics_used`
- 启动幂等 patch（`createSessionsDao()` 内）：
  ```sql
  UPDATE sessions SET topics_used = '[]' WHERE topics_used IS NULL
  ```
  兼容 legacy NULL 行 + 多进程先后打开 DB 不报错。
- `updateEnd` SQL：`... topics_used = COALESCE(?, topics_used) WHERE id = ?` —— 多次 markEnded 不覆盖先前写入。
- `serializeTopicsUsed(undefined)` 返回 `null`（让 COALESCE 兜底）；提供时返回 JSON.stringify。
- `parseTopicsUsed(raw)` 失败回退 `[]`。

### 4.4 `src/server.ts` + `src/cli.ts` — END pipeline 改写

**旧路径**：END 时直接拿 `summary.keywords` 调 `matchTopic()` → 命中就增量 1 个 slug（且常匹不中或匹错）。

**新路径**：END 时调 `recordAdoptedTopics(rt.adoptedTopics, review.keywords, ctx, deps)` → ledger 优先 / fallback 兜底 → 返回 slug 列表 → 透传给 `markEnded({ topicsUsed })`。

**`SessionRuntime.adoptedTopics`**：
- 初始化在 `reconstructSessionState` 返回值
- 每次 `sessionStore.set(rt)` 都保留（不重置）
- 注入 `turnInput.adoptedTopics`（Hook A/B 才能写入）

### 4.5 `prompts/phases.md` — Prompt 加固

新增 "Topic selection discipline (v1.0.7 §11)" 子节，告诉 LLM：
- 选定话题后**不要轻易重新选**（除非学生显式要求 / 当前话题 ≥8 轮无新材料）
- `topic_select_blocked` 时**不要重试**（尊重 gate）
- 反复重新选会污染 D1/D2/D5 去重信号

| `upsertMany` 单 keyword 不足以代表 topic | ✅ `suggested_keyword` 是人工策划的开场 phrase，本身就是高代表性词；D5 信号只需 slug 级频次 |
| `matchTopic` fallback 仍可能匹错（教学元词干扰） | ✅ Fallback 仅在 ledger 空时跑，且每次 END 只会跑一次（不是每轮）。正常 session 走 ledger 路径 |

| `recordAdoptedTopics` 依赖注入不完整时静默失败 | ✅ 三个 deps 全部由 server.ts / cli.ts 已有的实例传入，零新增模块依赖 |
