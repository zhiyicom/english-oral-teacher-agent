## 1. 现状盘点

### 1.1 TTS 代码现状（v1.0.7 之前的 `SessionPage.tsx:97-128`）

```ts
function speakAssistant(text: string) {
  const v = voiceRef.current
  if (!v.enabled) return
  const clean = stripInternal(text).trim()
  if (!clean) return
  speechSynthesis.cancel()
  const u = new SpeechSynthesisUtterance(clean)
  u.rate = v.speed
  const allVoices = speechSynthesis.getVoices())
  const localVoices = allVoices.filter(
    (x) => 'local' in x && (x as SpeechSynthesisVoice & { local: boolean }).local
  )
  const isMultilingual = (x: SpeechSynthesisVoice) => /multilingual/i.test(x.name)
  const onlineNonMulti = allVoices.filter((x) => !isMultilingual(x))
  const baseLang = v.accent.split('-')[0]
  const voice =
    localVoices.find((x) => x.lang === v.accent) ??
    localVoices.find((x) => x.lang.startsWith(`${baseLang}-`)) ??
    localVoices.find((x) => x.lang === 'en-US') ??
    localVoices.find((x) => x.lang.startsWith('en')) ??
    onlineNonMulti.find((x) => x.lang === v.accent) ??
    onlineNonMulti.find((x) => x.lang.startsWith(`${baseLang}-`)) ??
    onlineNonMulti.find((x) => x.lang === 'en-US') ??
    onlineNonMulti.find((x) => x.lang.startsWith('en')) ??
    allVoices.find((x) => x.lang === v.accent)
  if (voice) {
    u.voice = voice
    u.lang = voice.lang
  }
  speechSynthesis.speak(u)
}
```

### 1.2 暴露的问题

#### Bug A：过滤器用错字段

| W3C 标准 | 当前代码 |
|---|---|
| `SpeechSynthesisVoice.localService: boolean` | `'local' in x && x.local` |

Chromium 内核（Edge / Chrome）只暴露 `localService`，**没有** `local` 属性 → `localVoices` 永远为空 → 永远走到 `onlineNonMulti` 分支。

#### Bug B：fallback 链跨 source

当前 fallback 链是混在一起的：**先 local 4 档 → 再 online 4 档 → 再 any 1 档**。这意味着：

- 用户无法选择只用本地 / 只用在线
- 在某些网络环境下（如本机）会被强制走云端，即使本地语音可用
- 反之亦然（某些机器本地 OneCore 全无，但被云端兜底）

#### Bug C：UI 缺控制项

- Settings 页只有 voice_enabled / voice_speed / voice_accent
- 没有"语音源"选项
- 用户不知道自己用的是本地还是在线，也无法切换

### 5.3 重写 speakAssistant

```ts
function speakAssistant(text: string) {
  const v = voiceRef.current
  if (!v.enabled) return
  const clean = stripInternal(text).trim()
  if (!clean) return
  speechSynthesis.cancel()

  const allVoices = speechSynthesis.getVoices()
  const isMultilingual = (x: SpeechSynthesisVoice) => /multilingual/i.test(x.name)

  // 严格按 voiceSource 过滤 — 不跨 source 降级
  const pool: SpeechSynthesisVoice[] = v.source === 'local'
    ? allVoices.filter((x) => x.localService === true)
    : allVoices.filter((x) => !x.localService && !isMultilingual(x))

  const baseLang = v.accent.split('-')[0]

  // 单一 source 内 4 档回退
  const voice =
    pool.find((x) => x.lang === v.accent) ??
    pool.find((x) => x.lang.startsWith(`${baseLang}-`)) ??
    pool.find((x) => x.lang === 'en-US') ??
    pool.find((x) => x.lang.startsWith('en'))

  if (!voice) {
    const msg = v.source === 'local'
      ? STRINGS.voiceSourceNoMatchLocal.replace('{accent}', v.accent)
      : STRINGS.voiceSourceNoMatchOnline.replace('{accent}', v.accent)
    setVoiceHint(msg)
    setTimeout(() => setVoiceHint(null), 3000)
    return
  }

  const u = new SpeechSynthesisUtterance(clean)
  u.rate = v.speed
  u.voice = voice
  u.lang = voice.lang
  speechSynthesis.speak(u)
}
```

### 5.4 与 v1.0.7 行为的对照表

| 场景 | v1.0.7（混合 fallback）| v1.0.8（严格 source 过滤）|
|---|---|---|
| source=local + 本机有 en-GB OneCore、accent=en-US | 走 online（因为 localVoices 为空）| 走 local en-GB（4 档内回退）|
| source=local + accent=en-GB | 同上（误走到 online）| 走 local en-GB 直接匹配 |
| source=local + 本机无任何 OneCore | online 兜底 → 走云端 | hint 提示，无声 |
| source=online + accent=en-US | 走云端（受网络影响）| 走云端（同左，行为不变）|
| source=online + 浏览器无在线语音 | any fallback 兜底 | hint 提示，无声 |

注意：**默认 source='online' 时，v1.0.8 行为与 v1.0.7 完全一致**（都是走云端）。区别只在用户主动切到 'local' 之后。

## 8. 边界情况

### 8.1 source='local' 但本机完全无 OneCore 语音

| 现象 | 行为 |
|---|---|
| `allVoices.filter(x => x.localService === true)` 返回 `[]` | pool 为空 |
| 4 档内找不到匹配 voice | `setVoiceHint(voiceSourceNoMatchLocal...)` + 3 秒后清空 |
| 用户看到提示文字后 | 自行切到 'online' 或调整 voice_accent |

### 8.2 source='local' + accent=en-GB，本机有 en-GB OneCore

| 现象 | 行为 |
|---|---|
| pool 含 en-GB OneCore | 第 1 档匹配命中 |
| 立刻 `speechSynthesis.speak()` | 本地音频引擎播放，无网络请求 |

### 8.3 source='local' + accent=en-US，本机只有 en-GB OneCore

| 现象 | 行为 |
|---|---|
| pool 含 en-GB OneCore | 第 1 档（精确 en-US）失败 → 第 2 档（en-*）失败（因为 en-GB 的 lang='en-GB'，不以 'en-' 前缀但满足 startsWith('en')）→ 第 3 档（en-US）失败 → 第 4 档（startsWith('en')）命中 en-GB |
| `u.lang = voice.lang` | lang='en-GB'，浏览器按 en-GB 音素发音，但文本可能拼写偏美式 |
| 用户听到英式口音 | 这是预期行为（v1.0.8 §1.3 4 档内回退的设计选择）|

### 8.4 source='online' + accent=en-US，本机在线语音只有 zh-CN

| 现象 | 行为 |
|---|---|
| pool 含 zh-CN 在线 | 4 档内都失败（无 en-* 在线语音）|
| hint 提示"没有可用的在线 en-US 语音" | 用户切到 'local' 或换 accent |

### 8.5 voice_enabled = false

| 现象 | 行为 |
|---|---|
| speakAssistant 入口直接 return | 无声音、无 hint（与 v1.0.7 一致） |

### 8.6 用户连续切换 source

| 现象 | 行为 |
|---|---|
| 切到 'local' → 助手回复 → hint 出现（无匹配）| 3 秒后 hint 自动清空 |
| 用户立刻切到 'online' | 下次助手回复走新 source；旧 hint 还在 3 秒内显示 |

### 8.7 STT 错误 + TTS 无匹配同时发生

| 现象 | 行为 |
|---|---|
| 用户按麦克风 → STT 报 network 错 → voiceHint 显示"识别服务不可达" | 同时助手回复结束 → TTS 无匹配 → voiceHint 被覆盖成"没有可用的在线 en-US 语音" |
| 2.5 秒后 STT 错误定时器清空 voiceHint → 之后 TTS 的 3 秒定时器仍会清空一次 | 幂等 |

### 8.8 voice_accent 是 voiceRef 之外的值（理论上不会发生）

| 现象 | 行为 |
|---|---|
| baseLang = v.accent.split('-')[0] = 'en' | 4 档回退仍能命中 en-* 语音 |

### 8.9 SpeechSynthesisVoice.localService 不存在的浏览器（极端）

| 现象 | 行为 |
|---|---|
| 所有 voice 的 `localService` 为 undefined | source='local' 时 pool 为空 → hint 提示；source='online' 时 `!x.localService` 全为 true（undefined falsy）→ pool 包含全部 voice，但过滤掉 Multilingual 后仍可能有可用语音 |

### 8.10 getVoices() 在异步加载时为空

| 现象 | 行为 |
|---|---|
| `speechSynthesis.getVoices()` 返回 `[]`（首次调用时机太早）| pool 为空 → hint 提示 |
| 解决（不在本 sprint 范围）| 监听 `voiceschanged` 事件 + 重试；v1.0.9+ 评估 |

### 8.11 STT 错误被 onend 立刻覆盖（Bug #4）

> v1.0.7 §1.1 修复错误码细分提示时，遗漏了这个竞态。本 sprint §13 修复。

| 现象 | v1.0.7 行为 | v1.0.8 行为 |
|---|---|---|
| onerror('no-speech') 触发 | state='error'，schedule 2.5s 清空 | state='error'，schedule 2.5s 清空，`erroredRef=true` |
| onend 紧接着触发 | state='idle'（**立刻覆盖**），ref=null | ref=null，**但** `if (!erroredRef.current)` 守卫跳过 `setState('idle')` |
| 2.5s timer | state='idle'（无操作）| state='idle'，`erroredRef=false` |
| 用户可见 | toast 闪现 ~0ms → 消失 | toast 完整显示 2.5 秒 |

### 8.12 STT 错误码永远显示 'unknown'（Bug #1）

| 现象 | v1.0.7 行为 | v1.0.8 行为 |
|---|---|---|
| onerror 回调签名 | `() => void`（不接 event） | `(event: SpeechRecognitionErrorEvent) => void` |
| 错误码读取 | `rec.error ?? 'unknown'`（instance 上的字段） | `event.error ?? 'unknown'`（W3C 规范位置） |
| Edge 150 / Chromium 150 实际值 | undefined（被吃掉） | `'network'` / `'no-speech'` / `'not-allowed'` 等真实值 |
| toast 显示 | "语音输入出错，请重试" | 具体错误码对应文案 |
| DevTools console | `[VoiceInput] SpeechRecognition error: unknown` | `[VoiceInput] SpeechRecognition error: network` 等真实码 |

## 11. 风险与回退

同 `v1.0.8-scope.md` §7。补充几点：

### 11.1 关于"严格不降级" vs 4 档内回退

- **严格不降级**：跨 source 不降级（local → online 不行；反之亦然）
- **4 档内回退**：单一 source 内仍有 4 档匹配（accent → base lang → en-US → 任意 en）
- 这两个不冲突。用户说"出声失败，显示提示"指的是 pool 内 4 档全失败的情况，4 档内回退到 en-GB 不算"出声失败"

### 11.2 en-US → en-GB 口音变化的预期

- 本机只有 en-GB OneCore 时，source='local' + en-US 会落到 en-GB
- 用户会听到英式口音（Susan / George / Hazel 之一）
- 这是**预期行为**，不是 bug。Settings 提示文案可后续加一句"切换 source 或调整口音"

### 11.3 voiceHint 3 秒 vs 2.5 秒不一致

- VoiceInput 的 STT 错误 2.5 秒后清空
- speakAssistant 的 TTS 无匹配 3 秒后清空
- 不一致是**有意为之**：STT 错误更紧急（用户主动触发的反馈），TTS 无匹配是辅助信息
- 用户同时遇到两者时后者覆盖前者，符合预期

### 11.4 ls voice_source 缓存的同步问题

- localStorage 写 voice_source 后立即生效（重新进入 Settings 时读取）
- 但 server 的 voice_source 可能未及时同步（如 save 失败）
- 当前 .catch 块已处理 server 不可达的情况，localStorage 是 fallback
- 跨标签页同步不在本 sprint 范围

### 11.5 STT 修复相关风险

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `erroredRef` 守卫漏掉 onerror → onend 中间有其他事件（罕见）| 极低 | 低 | 加 mock 测试；如漏可改用 ref 清空时机 |
| aborted 跳过错误后用户主动 stop 不再有 toast | 低 | 低 | 预期行为；如果用户希望 stop 也反馈，v1.0.9+ 评估 |
| 修复后用户还是看到错误码（因为云端 STT 端点访问不到）| 高 | 低 | 本次只修"看清错误"，不修"消除错误"；真根因（DNS / 云端服务）属于环境问题 |
| `SpeechRecognitionErrorEvent` 类型定义与浏览器实现略有差异 | 低 | 中 | 字段只读 `error: string` 和 `message: string`，W3C spec 稳定 |

## 13. STT 错误提示修复（Bug #1 + Bug #4）

### 13.1 现状盘点

`web/src/components/VoiceInput.tsx` 当前实现：

```ts
// L6-16: ISpeechRecognition 接口
interface ISpeechRecognition extends EventTarget {
  continuous: boolean
  interimResults: boolean
  lang: string
  start(): void
  stop(): void
  onresult: ((event: SpeechRecognitionEvent) => void) | null
  onerror: (() => void) | null                          // ← 没有 event 参数
  onend: (() => void) | null
  readonly error?: string                                // ← instance 上的非标字段
}

// L111-127: handlers
rec.onerror = () => {
  const err = rec.error ?? 'unknown'                     // ← 永远显示 'unknown'（Bug #1）
  console.error(`[VoiceInput] SpeechRecognition error: ${err}`)
  setErrorMsg(err)
  setState('error')
  recognitionRef.current = null
  onHint?.(hintForError(err))
  setTimeout(() => {
    setState('idle')                                     // ← 2.5s 后清空
    setErrorMsg(null)
    onHint?.(null)
  }, 2500)
}
rec.onend = () => {
  setState('idle')                                       // ← 立刻覆盖错误态（Bug #4）
  recognitionRef.current = null
}
```

#### Bug #1 — onerror 读错字段

W3C Web Speech API spec：

```
[Constructor(optional SpeechRecognitionEventInit eventInitDict)]
interface SpeechRecognitionErrorEvent : Event {
  readonly attribute SpeechRecognitionErrorCode error;
  readonly attribute DOMString message;
};

enum SpeechRecognitionErrorCode {
  "no-speech",
  "aborted",
  "audio-capture",
  "not-allowed",
  "bad-grammar",
  "language-not-supported",
  // ... 后续可能扩展
};
```

**关键**：`SpeechRecognitionErrorCode error` 在 **event** 上，不在 instance 上。`ISpeechRecognition.error` 是 Chromium 私有扩展字段，且在 Edge 150 / Chromium 150 上**通常为 undefined**。

后果：用户看到的错误永远是 `unknown`，DevTools console 日志永远是 `[VoiceInput] SpeechRecognition error: unknown`。

#### Bug #4 — onend 覆盖 onerror 设的错误态

浏览器触发顺序：`onstart → onerror → onend`（间隔几毫秒）。当前代码：

1. onerror 触发 → state='error'，schedules 2.5s 清空
2. onend 触发 → state='idle'（**立即覆盖**）
3. 2.5s timer → state='idle'（无操作）

结果：用户看到 toast 闪现 ~0ms，根本来不及读错误信息。

#### 顺手修复 — aborted 错误码

`aborted` 在用户主动点麦克风停止时触发（`rec.stop()`）。当前会走 `hintForError` 的 `default` 分支，显示"语音输入出错，请重试"，这是**误导**——用户明明是自己停的。

修复：`onerror` 内 `if (err === 'aborted') return` 跳过。

### 13.2 设计目标

1. **真实错误码透传**：从 `event.error` 读取，UI 显示对应文案
2. **错误态不被覆盖**：`onend` 在出错后不再 `setState('idle')`
3. **aborted 不报错**：用户主动 stop 路径无 toast
4. **不动其他逻辑**：`start()` / `stop()` / hotkey / 既有 i18n 字段全部保留

### 13.3 接口扩展

```ts
// 新增 SpeechRecognitionErrorEvent 接口（W3C spec 子集）
interface SpeechRecognitionErrorEvent extends Event {
  readonly error: string           // W3C SpeechRecognitionErrorCode
  readonly message: string         // 浏览器给的额外描述
}

// 修改 onerror 签名
interface ISpeechRecognition extends EventTarget {
  // ... 既有
  onerror: ((event: SpeechRecognitionErrorEvent) => void) | null  // CHANGED
  // ... 既有
  // readonly error?: string  ← 删除，不再使用
}
```

### 13.4 handler 改造

```ts
// 新增 erroredRef 追踪本轮是否出过错
const erroredRef = useRef(false)

rec.onerror = (event: SpeechRecognitionErrorEvent) => {
  const err = event.error ?? 'unknown'
  console.error(`[VoiceInput] SpeechRecognition error: ${err}`, event.message ?? '')

  // aborted = 用户主动 stop，不弹错误
  if (err === 'aborted') {
    setState('idle')
    setErrorMsg(null)
    recognitionRef.current = null
    return
  }

  setErrorMsg(err)
  setState('error')
  erroredRef.current = true                              // NEW: 标记出错
  onHint?.(hintForError(err))
  setTimeout(() => {
    setState('idle')
    setErrorMsg(null)
    onHint?.(null)
    erroredRef.current = false                            // NEW: 重置标记
  }, 2500)
}

rec.onend = () => {
  recognitionRef.current = null
  if (!erroredRef.current) {                             // NEW: 仅无错时才清态
    setState('idle')
  }
}
```

**关键点**：
- `onerror` **不再** null ref —— 让 `onend` 统一负责清理
- `onend` **不再** 无条件清态 —— 用 `erroredRef` 守卫
- `onerror` 的 2.5s timer 同时清态 + 重置 `erroredRef`，保证下一轮识别干净
