# Sprint v1.0.8 范围 — TTS 语音源可配置化（本地 / 在线）

> **Sprint**: v1.0.8
> **上一 sprint**: v1.0.7（语音错误码细分提示 + 话题命中统计与去重重构 — 已合 main）
> **分支**: `main`
> **日期**: 2026-07-11
> **状态**: 1 项 TTS 体验类功能（Settings 页加语音源选择，让用户自行决定 TTS 走本机 OneCore 还是浏览器云端）+ 2 项 STT 错误提示修复（让用户看清真实的 STT 失败原因）

---

## 背景

v1.0.7 修完了语音**输入**的错误码细分提示，但本机出现 TTS 静默无声的根因还没解决。

### 现象

2026-07-10 用户实测：在 Edge 150 / Chrome 150 上，按下设置页"预听语音"按钮 → `speechSynthesis.speak()` 返回 `speaking: true, paused: false`，但**耳机里完全没有声音**。助手回复结束后 `speakAssistant()` 同样静默。同一个安装包在另一台机器上 TTS 工作正常，所以是**环境**问题，不是浏览器/代码整体 bug。

### 根因（代码侧）

代码路径在 `web/src/components/SessionPage.tsx:106`：

```ts
const localVoices = allVoices.filter(
  (x) => 'local' in x && (x as SpeechSynthesisVoice & { local: boolean }).local
)
```

W3C 标准的本地语音字段是 **`localService: boolean`**，不是 `local`。Chromium 内核（Edge / Chrome）只暴露 `localService`，**没有** `local` 属性。所以这条过滤器在 Chromium 上**永远返回空数组**。

后果：`speakAssistant()` 走完整条 fallback 链一路回退到 `onlineNonMulti.find(...)`，最后挑中 Microsoft 在线语音，`speech.platform.bing.com` 走云端 TTS：

- 在国内：本机访问 `speech.platform.bing.com` 被 DNS 污染 / 网络层拦截 → 请求失败 → `speak()` 返回 true 但实际无声
- 在境外：能出声，但依赖云端，断网即废

### 为什么用户要改设计

直接把过滤器修成 `localService === true` 虽然能立刻让 TTS 走本机 OneCore（George/Hazel/Susan / Huihui/Kangkang/Yaoyao），但这是**一刀切**：所有用户都被迫用本机语音，看不到在线语音选项（可能在某些机器上本机语音更难听，或者用户就是想要云端的某款声音）。

正确的做法是把"本地 vs 在线"做成**用户可配置的选项**，让用户根据自己的机器环境自行选择。

### 1.3 SessionPage 改造：speakAssistant 严格按 voice_source 过滤

| # | 事项 | 文件 |
|---|---|---|
| 11 | SessionPage 加 `voiceSource` state + ref（与 voiceEnabled/voiceSpeed/voiceAccent 同组） | `web/src/components/SessionPage.tsx` MODIFY |
| 12 | `getSettings().then` 读取 `voice_source` | `web/src/components/SessionPage.tsx` MODIFY |
| 13 | 重写 `speakAssistant`：根据 voiceSource 严格过滤 voice pool，去掉当前混在一起的 fallback 链 | `web/src/components/SessionPage.tsx` MODIFY |
| 14 | 找不到匹配 voice 时 `setVoiceHint(...)` 显示提示（沿用 VoiceInput 的 voiceHint state 机制），3 秒后清空 | `web/src/components/SessionPage.tsx` MODIFY |

**过滤逻辑**（伪代码）：

```ts
function speakAssistant(text: string) {
  const v = voiceRef.current
  if (!v.enabled) return
  const clean = stripInternal(text).trim()
  if (!clean) return
  speechSynthesis.cancel()

  const allVoices = speechSynthesis.getVoices()
  const isMultilingual = (x) => /multilingual/i.test(x.name)
  const baseLang = v.accent.split('-')[0]

  // 严格按 voiceSource 过滤
  const pool = v.source === 'local'
    ? allVoices.filter((x) => x.localService === true)
    : allVoices.filter((x) => !x.localService && !isMultilingual(x))

  const voice =
    pool.find((x) => x.lang === v.accent) ??
    pool.find((x) => x.lang.startsWith(`${baseLang}-`)) ??
    pool.find((x) => x.lang === 'en-US') ??
    pool.find((x) => x.lang.startsWith('en'))

  if (!voice) {
    setVoiceHint(
      v.source === 'local'
        ? STRINGS.voiceSourceNoMatchLocal.replace('{accent}', v.accent)
        : STRINGS.voiceSourceNoMatchOnline.replace('{accent}', v.accent)
    )
    setTimeout(() => setVoiceHint(null), 3000)
    return
  }

  const u = new SpeechSynthesisUtterance(clean)
  u.rate = v.speed
  u.voice = voice
  u.lang = voice.lang
  speechSynthesis.speak(u)
}
```

**关键变化**：
- 不再用混合 fallback 链（先 local 再 online 再随便选）
- 每个 source 内仍保留 4 档回退（精确 accent → base lang → en-US → 任意 en），确保能出声
- 找不到 voice 时**不降级到另一种 source**，按用户决策严格走：出声失败 + 显示提示

### 1.5 STT 错误提示修复（Bug #1 + Bug #4）

> 根因诊断见 [v1.0.8-design.md §13](./v1.0.8-design.md#13-stt-错误提示修复bug-1--bug-4)。本次只修复代码侧的两个 latent bug；STT 真正失败的原因是浏览器云端 STT 端点访问不到，任何前端修复都无法绕过。

#### 1.5.1 根因

**Bug #1**：`web/src/components/VoiceInput.tsx:111-127` 的 `rec.onerror` 回调读错了错误字段：

```ts
rec.onerror = () => {                       // ← 没有接 event 参数
  const err = rec.error ?? 'unknown'        // ← 读 instance，W3C 规范里这个字段是空的
  // ...
}
```

W3C 规范规定 `SpeechRecognitionErrorEvent.error` 在 **event 对象**上，instance 上的 `error` 属性是 Chromium 私有扩展、且在 Edge 150 / Chromium 150 上**通常为空**。后果：

- 用户控制台永远显示 `[VoiceInput] SpeechRecognition error: unknown`
- toast 永远显示"语音输入出错，请重试"（`voiceErrorUnknown` 文案）
- 即使是 `network`（云端不可达）、`not-allowed`（权限拒绝）、`service-not-allowed`（浏览器禁用）等具体错误码，全部被吃掉
- 调试和定位问题失去信号

**Bug #4**：`rec.onend` 立刻覆盖 `rec.onerror` 设的错误态：

```ts
rec.onerror = () => {
  setState('error')
  setTimeout(() => setState('idle'), 2500)  // 2.5 秒后清掉错误态
}
rec.onend = () => {
  setState('idle')                          // ← 立刻覆盖，2.5 秒定时器作废
}
```

浏览器在 STT 失败时按 `onstart → onerror → onend` 顺序触发（中间间隔几毫秒）。所以：

1. onerror 触发：state='error'，schedules 2.5 秒后清空
2. onend 触发：state='idle'（覆盖错误态），ref=null
3. 2.5 秒后：state='idle'（无操作）

用户**完全看不到错误提示**——state 在 0ms 就被切回 idle。

#### 1.5.2 修复策略

**Bug #1 修复**：回调接 `event` 参数，从 `event.error` 读取。同时补一个 `SpeechRecognitionErrorEvent` 类型定义（参考 W3C spec）。

**Bug #4 修复**：引入 `erroredRef: useRef<boolean>` 追踪本轮识别是否出过错：

- `rec.onerror` 标记 `erroredRef.current = true`
- `rec.onend` 仅在 `!erroredRef.current` 时才 `setState('idle')`
- 错误态由 `onerror` 的 2.5 秒 timer 负责清空（timer 也重置 `erroredRef.current = false`）

**顺手修复**：`aborted` 错误码不应弹错误提示（用户主动点麦克风停止时会被触发）。在 `onerror` 里加 `if (err === 'aborted') return` 跳过。

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `voice_source` 字段持久化失败（USER.md 损坏）| 极低 | 低 | `loadUserFile` 校验时 fallback 到 `'online'` |
| 旧用户升级：USER.md 缺 voice_source | 中 | 低 | `data.voice_source ?? 'online'`，行为不变 |
| 用户选 'local' 但本机没装任何本地语音 | 中 | 低 | hint 提示后无声音，用户可切换 |
| 用户选 'local' + en-US，但本机只有 en-GB OneCore | 高 | 中 | 当前 4 档回退（精确 → base lang → en-US → 任意 en）会落到 en-GB，**这违反了"严格不降级"决策** |
| `SpeechSynthesisVoice.localService` 在某些浏览器不存在 | 低 | 中 | `'localService' in x` 防御；不存在时 `pool` 为空，必然走 hint |
| voiceHint 复用 VoiceInput 的 state，3 秒清空可能与 STT 错误 2.5 秒冲突 | 低 | 低 | 用户同时遇到 STT 错误 + TTS 无匹配时，后者覆盖前者；后到的覆盖前者是符合预期的 |
| Settings 页加载时 server 不可达 | 低 | 低 | 已有 `.catch()` 块，从 localStorage fallback |
| STT 修复后用户还是看到错误码（因为云端 STT 端点访问不到）| 高 | 低 | 本次只修"看清错误"，不修"消除错误"；真根因（DNS / 云端服务）属于环境问题 |
| `erroredRef` 守卫漏掉 onerror 触发的边界情况 | 低 | 中 | 单元测试覆盖；manual smoke §5.5 验证 |
| aborted 跳过错误后用户主动 stop 不再有 toast | 低 | 低 | 预期行为；如果用户希望 stop 也反馈，v1.0.9+ 评估 |

### 关于"严格不降级 vs 4 档内回退"的口径澄清

用户的"严格不降级"是指**跨 source 不降级**（local 找不到不降级到 online）。但**单一 source 内**仍保留 4 档回退：

1. 精确 accent 匹配（如 en-GB → en-GB）
2. base lang 前缀匹配（如 en-GB → en-*）
3. en-US 兜底（任何 source 内有 en-US 都用）
4. 任意 en 前缀兜底

这与"严格不降级"不冲突——所有匹配都在**用户选定的 source pool 内**进行。如果连第 4 档都找不到（如本机装的是非英语 OneCore），才走 hint 提示。
