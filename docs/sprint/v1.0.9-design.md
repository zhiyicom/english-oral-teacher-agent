# Sprint v1.0.9 设计 — 话题选择去重机制修复

> **Sprint**: v1.0.9
> **依赖**: v1.0.7（话题统计重构 write-on-selection）+ v1.0.8（已合 main）
> **日期**: 2026-07-12
> **范围**: [v1.0.9-scope.md](./v1.0.9-scope.md)
> **一句话**：话题"重复被抽中 + 命中分布不均"不是单点 bug，而是**四个缺陷叠加**让 D1/D2/D5 三档去重信号集体失效，选择退化为随机、且计数还是错的。本 sprint 一次修四处。

---

## 0. 根因链复盘

分析阶段（未改码）从"用户反馈重复 + 命中分布不均"倒推，串出一条链：

```
缺陷1 stale快照 ──► D1/D2/D5 读到启动时旧数据 ──► 三档去重信号失效
                                                     │
缺陷2 噪声±0.2 ──► 唯一"活着"的 D4 噪声盖过 count(0.1)/kAvg(0.05) ──┤
                                                     ▼
                                          选择 ≈ 带轻微偏置的随机 ──► 小库短期重复
                                                     │
缺陷3 Hook A 纯随机 ──► 注入话题与热身脱节 ──► LLM 忽略它、聊别的
                                                     │
缺陷4 write-on-selection ──► 被忽略的抽签也计数+1 ──► 统计失真
```

**关键判断**：四者必须一起修。

- 只修缺陷 1（快照）不修缺陷 2（噪声）：信号能读到了，但仍被 `±0.2` 噪声主导，赢家还是随机。
- 只修缺陷 4（计数口径）不修缺陷 1（快照）：计数即使准了，运行中的工具也读不到（还是启动快照）。
- 缺陷 3、4 是一对：Hook A 抽了不相关话题（3）→ LLM 忽略 → 但仍被计数（4）。修 3 降低"被忽略"概率；修 4 的判定（`isTurnOnTopic` 关键词子串命中）保证"即使 LLM 真的没在聊注入话题，也因命中数为 0 而不虚计"。

**为什么撤回原分析方向 3（30 天→7 天）**：用户在分析阶段直接指出——`filterHardExclude` 只要真正生效，"30 天内讨论过就排除"本身就保证了 30 天不重复；把窗口缩到 7 天，只会让第 8～30 天的话题**提前**被放回池子，**削弱**去重。重复的真因是缺陷 1 让 D1 根本没生效，不是窗口太大。故 30 天保持不变。

### 0.1 关键澄清：两套独立计数，LLM 看不到 hit_count

写文档时容易把"计数"说成单一概念，实际上**存在两套互相独立的计数器**，在分析阶段必须分清：

| 计数器 | 存储 | 粒度 | 谁写 | 谁读 |
|---|---|---|---|---|
| `onTopicTurns`（v1.0.9 §1.4 新增）| 内存 ledger（`adoptedTopics` Map 的 entry 字段） | per (slug, session) | turn.ts 每轮结束后调 `isTurnOnTopic` 决定是否 +1 | `recordAdoptedTopics` 判定 adoption 阈值 |
| `keyword_hits.hit_count`（v1.0.2 已有）| SQLite `keyword_hits(topic, keyword)` 行 | per (topic, keyword) | session END 时 `upsertMany`（原子 +1）| `selectTopic` D5 打分 |

**两套计数互不耦合**：
- `isTurnOnTopic` 是**纯字符串匹配**（子串命中 `topic.keywords ∪ topic.description`），不读 `keyword_hits`；
- `upsertMany` 不读 `onTopicTurns`，它只看 ledger 里的 `suggestedKeyword` 字段。

**LLM 的 prompt 里**没有任何 hit_count 字段**——**`topic_select` 工具的返回值（`src/agent/tools/topic-select.ts:122-130`）只有 `slug` / `title` / `est_minutes` / `suggested_keyword` / `keywords` 五项。`hit_count` 只在 `selectTopic` **本地打分**时使用，**不打入 prompt、不调 LLM**。

链路：

```
[场内每轮]
  isTurnOnTopic(activeTopic, userMsg, assistantMsg)  ← 纯字符串匹配
    → true:  onTopicTurns += 1
    → false: 冻结（不归零）

[session END]
  recordAdoptedTopics(adopted)
    → for [slug, info] of adopted:           ← Map 天然 dedup
        if (onTopicTurns >= ADOPTION_MIN_TURNS) {
          topicStats.incrementAndUpdate(slug)
          keywordHits.upsertMany(slug, [info.suggestedKeyword])
        }

[下次 session, 工具调用]
  topic_select.execute()
    → selectTopic() → D5 avgKeywordHit(topic) = mean of hit_count
      ↑ 读到 session 1..N-1 累计（§1.1 修复后）
```

### 0.2 double-count 防护（两层）

1. **场内 dedup**：`adoptedTopics` 是 `Map<slug, AdoptedTopic>`，同一 slug 在一场会话里最多一个 entry。即使 Hook A 注入后 LLM 又 Hook B 选了同话题，Map.set() 覆盖不会重复 +1。
2. **场间 UPSERT 累加**：(topic, keyword) 主键冲突时 `hit_count = hit_count + 1`——**累加是 D5 设计的语义**，不是 bug。"用得越多分越低"必须靠跨会话单调递增才能体现。

### 0.3 v1.0.9 之前的"假 double-count"现象

旧代码下用户可能误以为"每次都重置计数"：session 1 结束时 `upsertMany` 写了 `hit_count = 5`，但因为 `keywordStats` 是启动快照（缺陷 1），session 2 的工具里 `hit_count` **仍是 0**——看上去像 reset，实际是**永远看不到新写入**。§1.1 getter 修了之后，session N 的工具稳定读到 session 1..N-1 的累计写入，计数单调递增。

## 1. §1.1 stale 快照修复

### 1.1 现状

`src/agent/tools/topic-select.ts:92-133`：

```ts
export function createTopicSelectTool(opts: {
  topics: Topic[]
  stats: TopicStat[]          // ← 数组，创建时捕获
  interests: string[]
  keywordStats?: KeywordHit[] // ← 数组，创建时捕获
  ...
}): Tool {
  const keywordStats = opts.keywordStats ?? []   // 闭包变量，永不更新
  return {
    execute(args) {
      const winner = selectTopic({
        topics: opts.topics,
        stats: opts.stats,        // 每次都是启动那一刻
        keywordStats,             // 同上
        ...
      })
    }
  }
}
```

`src/server.ts:302-315` 启动时一次性喂快照。会话结束时 `topicStats.incrementAndUpdate()` 写的是 **DB**；`opts.stats` 是**内存数组**，两者不是同一份 → 工具永远看不到更新。

### 1.2 方案：数组字段升级为 "数组或 getter"

签名改为接受二者之一，工具内归一：

```ts
type OrGetter<T> = T | (() => T)
function resolve<T>(x: OrGetter<T>): T {
  return typeof x === 'function' ? (x as () => T)() : x
}

export function createTopicSelectTool(opts: {
  topics: Topic[]
  stats: OrGetter<TopicStat[]>
  interests: string[]
  keywordStats?: OrGetter<KeywordHit[]>
  ...
}): Tool {
  return {
    execute(args) {
      const stats = resolve(opts.stats)                 // ← 每次 execute 重新求值
      const keywordStats = resolve(opts.keywordStats ?? [])
      const winner = selectTopic({ topics: opts.topics, stats, keywordStats, ... })
      // pickFreshestKeyword 也用同一份 keywordStats
    }
  }
}
```

**注册端（`src/server.ts`）**：

```ts
toolRegistry.register(
  createTopicSelectTool({
    topics: topics.list(),                    // 话题库运行期不变，快照 OK
    stats: () => topicStats.all(),            // ← getter：每次 execute 读最新
    interests: [],
    keywordStats: () => keywordHits.getAll(), // ← getter
    useInterestBoost: false,
  }),
)
```

`src/cli.ts` 的注册同样改为 getter（保持双端一致）。

### 1.3 为什么 topics 仍可快照

`topics.list()` 是话题库（`topics-default.json` 导入的静态定义），运行期不增删。只有 `stats` / `keywordStats` 会随会话结束变化。保持 topics 快照可减少不必要的读，也让改动最小。

### 1.4 向后兼容

既有单测直接传数组（`stats: [...]`）。`resolve()` 对数组直接返回，对函数才调用——数组调用零改动，只需把类型放宽为 `OrGetter<T>`。新增 U2（getter 变更可见）+ U3（数组重载仍可用）。

### 1.5 性能

`topicStats.all()` / `keywordHits.getAll()` 是小表全量 SELECT。单用户系统，`topic_select` 每场会话至多几次调用。额外开销可忽略；若未来成瓶颈，可加 session 级缓存（本 sprint 不做，避免又引入"缓存过期 = stale"的同类问题）。

## 2. §1.2 噪声修复

### 2.1 现状数字

`topic-engine.ts:75-78, 130`：

```
COUNT_WEIGHT   = 0.1
KEYWORD_WEIGHT = 0.05
NOISE_RANGE    = 0.2
score = -count*0.1 - kAvg*0.05 + interest*0.5 + noise,  noise ∈ [-0.2, +0.2]
```

| 信号 | 单位差带来的 score 变化 |
|---|---|
| count 差 1 次 | 0.1 |
| kAvg 差 1 次命中 | 0.05 |
| **噪声峰峰值** | **0.4** |

噪声峰峰值是最强信号（count）的 **4 倍**。一个讨论过 3 次的话题（-0.3）完全可能因为一次 +0.2 噪声、对手 -0.2 噪声而反超一个从未讨论的话题（0）：`-0.3+0.2 = -0.1 > 0-0.2 = -0.2`。

### 2.2 改动

```ts
const NOISE_RANGE = 0.05   // was 0.2
```

峰峰值降到 **0.1**，恰好 ≈ 一次 count 差。效果：

- count 明确不同的话题：count 主导，噪声只在 count 极接近时起扰动作用；
- count 相同的话题：kAvg（0.05）与噪声（±0.05）同量级，主要看关键词新鲜度 + 小随机——正是 D4"打破平局"的本意；
- 完全平局（count、kAvg 都相同）：噪声决定，避免每次同一个赢家。

### 2.3 为什么不动权重

改 `COUNT_WEIGHT` / `KEYWORD_WEIGHT` 会改变既有测试里"谁排第一"的期望，牵连面大。只调噪声是让"信号浮出水面"的最小充分改动。同步更新 `topic-engine.ts:12-14` D4 注释里的数值描述。

### 2.4 与固定 rng 测试的关系

现有单测用 `rng: () => 0.5`（noise = 0）等固定值验证确定性排序，噪声幅度变化不影响这些用例。新增 U1：用能产生最大噪声的 rng（0 和 1）验证 count=0 的话题仍稳定压过 count=3——证明噪声不再翻盘。

## 3. §1.3 Hook A 上下文化

### 3.1 现状

`turn.ts:404-440`，进入 MAIN_ACTIVITY：

```ts
const topicResult = await topicTool.execute({
  phase: 'MAIN_ACTIVITY',
  exclude_recent_days: 30,
})   // ← 没有当前会话上下文
```

抽到的话题与学生刚在 WARM_UP 聊的东西无关 → LLM 倾向忽略。

### 3.2 方案：软加权（contextBoost），不硬过滤、不越权去重

三步：

**(a) turn.ts 提取上下文关键词**：从 `history` 中 WARM_UP 阶段的 user 消息文本，提取关键词。v1.0.9 用轻量提取（复用项目已有的分词/关键词工具；若无则空白+标点切词 → 去停用词 → 取名词性/高频词）。产出 `context_keywords: string[]`。

**(b) topic_select schema 新增可选入参**：

```ts
export const TopicSelectArgsSchema = z.object({
  phase: z.enum([...]).default('WARM_UP'),
  exclude_recent_days: z.number().int().min(0).max(365).default(30),
  context_keywords: z.array(z.string()).optional(),   // ← 新增
})
```

**(c) topic-engine 新增独立 contextBoost 通道**：

```ts
const CONTEXT_WEIGHT = 0.15   // 与 count(0.1) 同量级，能倾斜但压不过多次 count 差

export function computeContextOverlap(topic: Topic, contextKeywords: readonly string[]): number {
  if (contextKeywords.length === 0) return 0
  const norm = new Set(contextKeywords.map((s) => s.toLowerCase()))
  return topic.keywords.filter((k) => norm.has(k.toLowerCase())).length
}

// score 里追加： + contextOverlap * CONTEXT_WEIGHT
```

**独立于 interest 通道**：不复用 `computeInterest` / `INTEREST_WEIGHT`——那条通道被 v1.0.3 §1.3 明确禁用（`useInterestBoost=false`），语义是"学生长期兴趣画像"。contextBoost 语义不同：是"本场会话刚聊到的即时上下文"。混用会让 D3 死灰复燃、语义混乱。

### 3.3 关键约束：去重优先级 > 相关性

contextBoost 加在 `score` 上，而 `filterHardExclude`（D1）在**打分之前**就把"最近 N 天讨论过"的话题移出池子。所以：

- 一个话题即使跟热身**高度相关**，只要它在最近 30 天讨论过，就**进不了候选池**，contextBoost 根本没机会作用于它；
- contextBoost 只在"D1 幸存者"之间做倾斜。

这保证"相关性不能翻越去重"——这是 §1.3 与 §1.1 协同的核心。

### 3.4 软加权而非硬过滤的理由

若热身关键词与库中任何话题都无交集（`contextOverlap` 全 0），score 退化为纯 count/kAvg/noise，仍能选出话题。硬过滤（"只在相关话题里选"）会在无相关话题时**无题可注入**，破坏 Hook A 的"必然注入一个话题"契约。

### 3.5 权重 0.15 的取舍

- 大于 count 权重（0.1）：一次相关命中能压过"对手少讨论一次"的优势——相关性适度优先；
- 但只要对手多讨论 2 次以上（-0.2 以上差距）就压不过，去重仍是主导；
- 与噪声（现 ±0.05）拉开量级，相关信号不被随机淹没。

CLI 端不注入 context_keywords（CLI 无 WARM_UP 关键词提取链路时，`context_keywords` 省略 → contextOverlap=0 → 行为不变）。

## 4. §1.4 write-on-adoption

### 4.1 现状

`topic-recorder.ts:42-61`，v1.0.7 §11 的 write-on-selection：只要话题进了 `adoptedTopics` ledger（被 Hook A 自动注入 或 Hook B LLM 手动选），END 就 `incrementAndUpdate` +1，不管实际是否讨论。

`turn.ts:427-434` Hook A 注入后立即 `input.adoptedTopics?.set(slug, {...})`——**注入即记账**。

### 4.2 方案：以"对话关键词命中"作为 adoption 判定

**核心判定（`topic-engine.ts` 新增纯函数）**：

```ts
/**
 * v1.0.9 §1.4 — A turn is "on the topic" when the user message OR the
 * assistant message contains (case-insensitive substring) any of
 * `topic.keywords` ∪ `topic.description`.
 *
 * description (人话长串) 作为 keywords 子串命中的兜底，避免 keywords
 * 列表短时误判率过高。null/空 description 不参与。
 *
 * 任一方命中即返回 true。用于 write-on-adoption 的轮次计数。
 */
export function isTurnOnTopic(
  topic: Topic,
  userContent: string,
  assistantContent: string,
): boolean {
  const needles = [
    ...topic.keywords,
    ...(topic.description ? [topic.description] : []),
  ]
    .map((n) => n.toLowerCase().trim())
    .filter((n) => n.length > 0)
  if (needles.length === 0) return false
  const haystack = `${userContent} ${assistantContent}`.toLowerCase()
  return needles.some((needle) => haystack.includes(needle))
}
```

**示例**（`topic = boeing_737`, `keywords = ["boeing", "737", "cockpit"]`, `description = "波音 737 驾驶舱"`）：

| User | Assistant | `isTurnOnTopic` | 原因 |
|---|---|---|---|
| `"I watched a Boeing cockpit video"` | `"Great! What caught your attention?"` | ✓ | user 命中 `boeing` / `cockpit` |
| `"嗯"` | `"Tell me more."` | ✗ | 双方都无关键词 |
| `"我昨天看了波音 737 的视频"` | `"哦？是哪个型号？"` | ✓ | user 命中 `波音 737 驾驶舱` |
| `"ok"` | `"Anyway, the cockpit has three displays..."` | ✓ | assistant 命中 `cockpit` |
| `"yeah"` | `"Got it. So what do you want to do today?"` | ✗ | 双方都无关键词 |

**数据结构**（`topic-recorder.ts`）：

```ts
export interface AdoptedTopic {
  suggestedKeyword: string
  source: 'auto' | 'llm'
  onTopicTurns: number   // ← 累计 isTurnOnTopic 返回 true 的轮数
}
```

**追踪逻辑**（turn.ts）：

- 话题被 Hook A / Hook B 选中时，`onTopicTurns = 0`，标记为"当前活跃话题"；
- 每完成一个**非阶段切换**的正常对话轮，调用 `isTurnOnTopic(activeTopic, userMsg, assistantMsg)`：
  - true → `onTopicTurns += 1`；
  - false → 不增加（旧轮数**冻结**——不归零，因为本轮不命中不代表话题已放弃；后续若再命中会继续累加）。
- 阶段切换（如 MAIN_ACTIVITY→WRAP_UP）或新话题被选中时，旧话题停止累加。

**计数判定**（`recordAdoptedTopics`）：

```ts
const ADOPTION_MIN_TURNS = 2

for (const [slug, info] of adopted) {
  if (info.onTopicTurns < ADOPTION_MIN_TURNS) continue   // ← 未达标不计数
  deps.topicStats.incrementAndUpdate(slug, ctx.now)
  if (info.suggestedKeyword) deps.keywordHits.upsertMany(slug, [info.suggestedKeyword], ctx.now)
  adoptedSlugs.push(slug)
}

if (adoptedSlugs.length === 0) {
  // 全场无达标话题 → 保留既有 matchTopic fallback，best-effort 记一条
  const best = matchTopic(fallbackKeywords, deps.topics)
  if (best) { ...; return [best.topic] }
  return []
}
return adoptedSlugs
```

### 4.3 阈值 N=2 的语义

不再是 "停留轮数 ≥ 2"，而是 **"user/assistant 真正触及话题关键词的轮数 ≥ 2"**：

- N=1 太松：注入后学生一句 `"嗯"` 就达标——几乎等于 write-on-selection 的老问题。
- **N=2（选定）**：注入后学生至少在两轮对话里**说了或听到了与话题相关的词**。足以排除"立刻跑题 + 敷衍应答"的虚计场景。
- N≥3 太严：用户做短会话（5 轮以内结束 MAIN_ACTIVITY）容易全场无达标，频繁走 fallback，反而失真。

阈值集中为常量 `ADOPTION_MIN_TURNS`，便于后续按实测调整。

### 4.4 边界

- **描述含停用词无法避免**：如 description 为 "关于日常生活的对话" 包含"关于"等。子串匹配仍要求该子串出现在 user/assistant 文本中，"关于"出现在用户输入里极少，几乎不误命中；若实际命中多了，描述作为"补充池"的兜底价值就在那里。
- **多关键词 / 多语言**：keywords 与 description 拼接后做 OR 匹配；中英文混合（如 `"波音 737 驾驶舱"`）也能命中对应文本（子串不区分语言）。
- **空关键词库 / 缺 description**：needles 为空直接 `return false`——防御退化，但**实际不应出现**（话题库导入校验）。
- **阶段切换轮不计入 onTopicTurns**：MAIN_ACTIVITY 入口的"注入轮"不计；WRAP_UP 切换轮不计（避免总结阶段误算进 MAIN_ACTIVITY 话题的讨论）。
- **多话题**：一场里若 Hook B 让 LLM 换了话题，各话题各自累计 `onTopicTurns`，分别判定；ledger 是 Map，互不干扰。
- **fallback 不变**：全场无达标话题时仍走 `matchTopic(summaryKeywords)`，保证不会"整场零记录"。

### 4.5 为什么不做 LLM 语义判定

更准的做法是让 LLM/分类器判断"这一轮是否真在聊注入话题"。但每轮多一次分类调用，成本与延迟都高，且失败模式复杂（LLM 自己评估自己，循环依赖）。v1.0.9 用 `isTurnOnTopic` 关键词子串命中作为**代理指标**——

- 实现简单：纯字符串匹配，零 LLM 调用、零 embedder 调用；
- 复用现有数据：`topic.keywords` + `topic.description` 已在 Engine / Recorder 里使用；
- 可测：`isTurnOnTopic` 是纯函数，单测喂固定文本断言即可；
- 可解释：用户事后查 `topic_stats` 时，可对照话题 keywords 复盘"为什么这次算聊过"。

语义判定留 §2 占位。

## 5. 四项协同关系

| 修复 | 单独修的局限 | 协同后 |
|---|---|---|
| §1.1 快照 | 信号读到了，但仍被噪声主导 | + §1.2 让信号真正决定排序 |
| §1.2 噪声 | 信号浮出，但读的还是旧数据 | + §1.1 让信号是最新的 |
| §1.3 Hook A | 注入相关了，但被忽略仍虚计 | + §1.4 忽略的不计数 |
| §1.4 计数 | 计数准了，但工具读不到 | + §1.1 计数进得了运行中的工具 |

**闭环**：§1.1+§1.2 让"选得对"，§1.3+§1.4 让"记得对"，且"记得对"的数据经 §1.1 回流到"选得对"，形成正确的反馈回路。
