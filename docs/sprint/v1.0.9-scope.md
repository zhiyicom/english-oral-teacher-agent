> **Sprint**: v1.0.9

> **分支**: `main`
> **日期**: 2026-07-12

## 背景

2026-07-12 用户（Jeremy 的教练视角）反馈：最近几次练习里，LLM 每次选的 topic **仍然会重复**。同时查看话题库命中记录发现：

- 少数话题命中 2 次、3 次；
- **大量话题命中数为 0**（从未被抽到过）；
- 库里明明还有没命中过的话题，为什么有的话题会被反复命中？

这说明"少讨论优先 + 近期不重复"的去重机制没有按设计工作。经代码走查（**本 sprint 前的分析阶段，未改码**），定位到一条**根因链**：四个独立缺陷叠加，使得 D1（硬排除）/ D2（少讨论优先）/ D5（关键词新鲜度）三档信号**几乎全部失效**，选择退化成"带轻微偏置的随机"，于是同一批话题被反复抽中。

### 根因链（四个缺陷）

#### 缺陷 1 — `topic_select` 工具持有 stale 快照（最致命）

`src/server.ts:302-315` 在**服务器启动时**注册 `topic_select` 工具，把 `stats` 和 `keywordStats` 作为**闭包快照**一次性传入：

```ts
toolRegistry.register(
  createTopicSelectTool({
    topics: topics.list(),
    stats: topicStats.all(),          // ← 启动时快照，之后永不刷新
    interests: [],
    keywordStats: keywordHits.getAll(), // ← 同样是启动快照
    useInterestBoost: false,
  }),
)
```

`createTopicSelectTool`（`src/agent/tools/topic-select.ts:92-133`）把这两个数组**捕获进闭包**，每次 `execute()` 都用**启动那一刻**的数据。

后果：会话结束时 END 管线确实调用 `topicStats.incrementAndUpdate()` / `keywordHits.upsertMany()` 写了 DB，但**运行中的工具永远看不到这些新写入**。于是：

- **D1 硬排除失效** — `filterHardExclude` 用 `stat.lastDiscussedAt` 判断"最近 N 天是否讨论过"。快照里 `lastDiscussedAt` 停留在启动时刻，新讨论的话题在快照里仍是"很久没讨论"甚至"从未讨论"，**排除不掉** → 同一话题下一次照样进池子。
- **D2 计数偏置失效** — `sortByCountAsc` 和 score 里的 `-count*0.1` 用的也是快照 count，新增的讨论计数看不到 → 已经讨论 3 次的话题在工具眼里还是 0 次。
- **D5 关键词新鲜度失效** — 同理，`avgKeywordHit` 用快照的 `keywordStats`，命中过的关键词在工具眼里 hit_count 仍是旧值。

> **注**：`src/server.ts:307-309` 的注释写着 *"Read once at startup; only mutated in the session-end finally block (same module-scoped DB), so reading once is correct"* —— 这个假设是**错的**：session-end 写的是 **DB**，而工具读的是**内存里的启动快照数组**，两者不是同一份数据。除非重启 server，工具永远看不到更新。这正是"重启前一直重复、重启后短暂正常又开始重复"现象的来源。

#### 缺陷 2 — 噪声幅度盖过所有信号

`src/agent/topic-engine.ts:75-78`：

```ts
const COUNT_WEIGHT = 0.1
const KEYWORD_WEIGHT = 0.05
const INTEREST_WEIGHT = 0.5   // 默认关闭
const NOISE_RANGE = 0.2       // ← 噪声幅度
```

score 公式（`topic-engine.ts:130`）：

```
score = -count*0.1 - kAvg*0.05 + interest*0.5 + noise
noise ∈ [-0.2, +0.2]   （峰峰值 0.4）
```

问题：噪声峰峰值 **0.4**，而"讨论 1 次 vs 0 次"带来的 count 信号只有 **0.1**，"关键词命中差 1 次"只有 **0.05**。也就是说噪声比最强的有效信号还大 4 倍。即使 D1/D2/D5 都正常工作，**噪声也能轻松翻盘**，让讨论过 3 次的话题（count 信号 -0.3）盖过一个从未讨论的话题（0），排到前面。设计里 D4 的本意是"打破平局、避免每次同一个赢家"，但 `±0.2` 把它从"平局扰动"放大成了"主导项"。

#### 缺陷 3（分析方向 4）— Hook A 自动注入纯随机，与当前会话脱节

`src/agent/turn.ts:404-440`：进入 MAIN_ACTIVITY 时，Hook A 调用 `topic_select` 自动注入一个话题。但它传的是：

```ts
await topicTool.execute({ phase: 'MAIN_ACTIVITY', exclude_recent_days: 30 })
```

**没有传入任何当前会话上下文**（WARM_UP 阶段学生刚聊过的内容 / 关键词）。工具内部走的是纯 count/noise 抽签，抽到的话题往往跟学生刚才热身聊的东西**毫不相关**。LLM 拿到一个突兀的话题后，倾向于**忽略**它、继续顺着热身的话头聊——这又引出缺陷 4。

#### 缺陷 4（分析方向 5）— write-on-selection 让计数与"真实讨论"脱钩

`src/agent/topic-recorder.ts:42-61` 是 v1.0.7 §11 引入的 **write-on-selection**：只要话题被 `topic_select` 选中（进了 `adoptedTopics` ledger），END 管线就 `incrementAndUpdate` 计数 +1，不管 LLM **是否真的讨论了它**。

结合缺陷 3：Hook A 自动注入一个不相关话题 → 记进 ledger → LLM 实际忽略它、聊了别的 → 但 END 仍给这个"被忽略的话题"计数 +1。于是：

- `topic_stats.discussion_count` 反映的是"**抽签抽中了几次**"，不是"**真的聊了几次**"；
- 被"抽中但忽略"的话题计数虚高，真正聊了很多的内容（顺着热身聊的）可能根本没进任何话题的计数；
- 命中分布因此既不均、也不真实。

### 四个缺陷如何叠加成"重复"

1. 缺陷 1 让 D1/D2/D5 三档去重信号全部读旧快照 → 失效；
2. 剩下唯一"活着"的是 D4 噪声，而缺陷 2 又让噪声幅度过大 → 选择≈纯随机；
3. 纯随机 + 小话题库，自然会短期内重复抽到同几个；
4. 缺陷 3 让 Hook A 抽的话题与会话无关 → LLM 忽略；
5. 缺陷 4 又把"被忽略的抽签"计成真实讨论 → 统计失真，且即使缺陷 1 修好，D1/D2/D5 依据的也是错误计数。

所以四个必须**一起修**：只修缺陷 1 而不修缺陷 2，选择仍被噪声主导；只修计数（缺陷 4）而不修 stale 快照（缺陷 1），新计数照样进不了运行中的工具。

> 分析阶段结论详见 [v1.0.9-design.md §0](./v1.0.9-design.md#0-根因链复盘)。用户在分析阶段确认："方向 3（把硬排除窗口 30 天改 7 天）撤回"——因为只要 D1 真正生效，30 天窗口严格优于 7 天，缩窗只会**削弱**去重。故本 sprint **不含**"改窗口"这一项。

## 1. 范围内

### 1.1 修复 stale 快照 — 让 D1/D5（及 D2）读到最新 DB（分析方向 1）

**目标**：`topic_select` 每次 `execute()` 都基于**当前 DB** 的 stats / keywordStats 评分，而非启动快照。

**方案**：把 `createTopicSelectTool` 的 `stats` / `keywordStats` 从"数组快照"改为"**getter 函数**"，在 `execute()` 内即时拉取。`topics`（话题库本身）仍可保持快照——话题库运行期不变。

| # | 事项 | 文件 |
|---|---|---|
| 1 | `createTopicSelectTool` opts 的 `stats` / `keywordStats` 由 `TopicStat[]` / `KeywordHit[]` 改为 `() => TopicStat[]` / `() => KeywordHit[]`（或新增 getter 字段，保留数组重载给测试） | `src/agent/tools/topic-select.ts` MODIFY |
| 2 | `execute()` 内调用 getter 取最新数据后再传给 `selectTopic()` / `pickFreshestKeyword()` | `src/agent/tools/topic-select.ts` MODIFY |
| 3 | `src/server.ts` 注册时传 `stats: () => topicStats.all()`、`keywordStats: () => keywordHits.getAll()` | `src/server.ts` MODIFY |
| 4 | `src/cli.ts` 同步改为 getter 形式（保持 CLI / Server 一致） | `src/cli.ts` MODIFY |
| 5 | 更新 `src/server.ts:307-309` 的错误注释（"read once at startup is correct" → 说明现在每次 execute 重新读 DB） | `src/server.ts` MODIFY |

**约束**：
- `topic-engine.ts` 的纯函数（`selectTopic` / `filterHardExclude` / `avgKeywordHit` 等）**不动**——它们本来就是纯函数，问题只在调用方喂了旧数据。
- 兼容测试：现有单测直接传数组。为不大改测试，getter 字段接受 `TopicStat[] | (() => TopicStat[])`，工具内 `typeof x === 'function' ? x() : x` 归一。

### 1.2 噪声幅度 `±0.2 → ±0.05`（分析方向 2）

**目标**：让 D4 回归"平局扰动"本职，不再盖过 count（0.1）/ kAvg（0.05）信号。

| # | 事项 | 文件 |
|---|---|---|
| 6 | `NOISE_RANGE` 常量 `0.2 → 0.05` | `src/agent/topic-engine.ts` MODIFY |
| 7 | 更新 `topic-engine.ts:12-14` D4 注释里的 `±NOISE_RANGE` 说明数字 | `src/agent/topic-engine.ts` MODIFY |

**为什么选 0.05**：
- count 权重 0.1 = "多讨论一次"的惩罚；噪声峰峰值应小于它，才不会让"多讨论一次"被一次随机翻盘。`±0.05`（峰峰 0.1）≈ 一次 count 差，是"能打破完全平局、但压不过明确 count 差"的临界值。
- kAvg 权重 0.05：噪声 `±0.05` 与它同量级，两个 count 完全相同的话题之间，仍主要由关键词新鲜度 + 小噪声决定，符合预期。
- 不改 `COUNT_WEIGHT` / `KEYWORD_WEIGHT`：动权重会牵连既有测试的期望排序；只调噪声是最小改动。

### 1.3 Hook A 自动注入参照 WARM_UP 关键词（分析方向 4）

**目标**：MAIN_ACTIVITY 入口自动注入的话题，与学生刚在 WARM_UP 聊过的内容**相关**，降低 LLM 忽略它的概率。

| # | 事项 | 文件 |

| 8 | 从当前会话历史（WARM_UP 轮次的 user 消息）提取关键词/名词，作为"上下文关键词" | `src/agent/turn.ts` MODIFY |
| 9 | `topic_select` schema 新增可选入参 `context_keywords?: string[]` | `src/agent/tools/topic-select.ts` MODIFY |
| 10 | 传入后，在**已通过 D1 硬排除的候选池内**，对与 `context_keywords` 有 keyword 交集的话题给一个软加权（复用 D3 `computeInterest` 的重叠计数思路，但走独立的 `contextBoost`，不复用被禁用的 interest 通道） | `src/agent/topic-engine.ts` MODIFY + `src/agent/tools/topic-select.ts` MODIFY |
| 11 | Hook A 调用处（`turn.ts:408`）把提取到的 `context_keywords` 传进 `execute()` | `src/agent/turn.ts` MODIFY |

**约束**：
- **不破坏 D1**：context 加权只在硬排除**之后**的池子里生效——绝不会把"最近讨论过"的话题因为跟热身相关就拉回来。去重优先级 > 相关性。
- **软加权而非硬过滤**：若没有任何话题与热身关键词相关（交集为空），退化为现有的 count/noise 抽签，不会因"找不到相关话题"而无话题可注入。
- 关键词提取**从简**：v1.0.9 用现有轻量方式（history 里 WARM_UP user 文本 → 复用项目已有的 keyword 提取工具/分词，若无则简单按空白/标点切词并去停用词）。不引入 NLP 依赖。

### 1.4 write-on-adoption 替代 write-on-selection（分析方向 5）

**目标**：`topic_stats.discussion_count` 只统计 LLM **真的聊过**的话题（"聊过"的语义见下），而非"抽签抽中过"。

**"聊过"的判定（v1.0.9 采用）**：

> 一轮对话**算作在话题上**，当且仅当该轮的 user 消息**或** assistant 消息里，**任一**与 `topic.keywords` ∪ `topic.description` 构成**大小写不敏感的子串匹配**。匹配池包含 keywords 和 description，是为了让 `description`（人话长串，如 `"波音 737 驾驶舱"`）作为关键词子串命中的兜底。

```ts
// src/agent/topic-engine.ts — 纯函数（v1.0.9 §1.4）
export function isTurnOnTopic(
  topic: Topic,
  userContent: string,
  assistantContent: string,
): boolean {
  const needles = [
    ...topic.keywords,
    ...(topic.description ? [topic.description] : []),
  ].map((n) => n.toLowerCase().trim()).filter((n) => n.length > 0)
  if (needles.length === 0) return false
  const haystack = `${userContent} ${assistantContent}`.toLowerCase()
  return needles.some((needle) => haystack.includes(needle))
}
```

**示例**（topic = `boeing_737`, keywords = `["boeing", "737", "cockpit"]`, description = `"波音 737 驾驶舱"`）：

| User 消息 | Assistant 消息 | onTopic? |
|---|---|---|
| `"I watched a Boeing cockpit video yesterday"` | `"Great! What caught your attention?"` | ✓（命中 `boeing` / `cockpit`）|
| `"嗯"` | `"Tell me more about it."` | ✗（双方都无关键词）|
| `"我昨天看了波音 737 的视频"` | `"哦？是哪个型号？"` | ✓（命中 `波音 737 驾驶舱` 子串 / `波音`）|
| `"ok"` | `"Anyway, the cockpit has three displays..."` | ✓（assistant 命中 `cockpit`）|
| `"yeah"` | `"Got it. So what do you want to do today?"` | ✗（双方都无关键词 → "被忽略/敷衍应答"被识别）|

| # | 事项 | 文件 |
|---|---|---|
| 12 | `topic-engine.ts` 新增 `isTurnOnTopic(topic, userContent, assistantContent)` 纯函数（keywords ∪ description 子串匹配） | `src/agent/topic-engine.ts` MODIFY |
| 13 | `AdoptedTopic` 增加 `onTopicTurns: number`（累计 isTurnOnTopic 返回 true 的轮数） | `src/agent/topic-recorder.ts` MODIFY |

**阈值 N=2 的语义**（用方案 C 后的清晰版）：

> "注入后，user/assistant 的对话文本**真正触及**话题关键词至少 **2 轮**" —— 既不是单纯轮数（排除 "ok"/"嗯" 敷衍），也不是 LLM 语义判定（零额外 LLM 调用）。

## 2. 后续需求（占位）

- LLM 语义级"是否跑题 / 是否真的在聊注入话题"的判定（比关键词子串命中更准，但需额外一次分类调用，v1.0.9 范围外）。

- `exclude_recent_days` 做成用户可配（当前硬编码 30）。

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| §1.1 getter 每次 execute 都查 DB，带来额外读开销 | 低 | 低 | `topicStats.all()` / `keywordHits.getAll()` 是小表全量读，单用户系统频率极低；如成瓶颈可加 session 级缓存（本 sprint 不做）|
| §1.1 改 getter 破坏既有单测（传数组）| 中 | 低 | 字段接受 `T[] \| (() => T[])`，工具内归一；既有数组调用零改动 |
| §1.2 噪声调小后选择过于确定，"总是同一个赢家" | 低 | 低 | D1 硬排除会持续把上次赢家排除 N 天，配合 count 递增，赢家会自然轮换；`0.05` 仍保留平局扰动 |
| §1.3 关键词提取质量差 → context_keywords 噪声大 | 中 | 低 | 软加权而非硬过滤：提取不准最多退化回现状抽签，不会更糟 |
| §1.3 contextBoost 误把"相关但最近讨论过"的话题拉回 | 低 | 中 | 明确约束：contextBoost 只在 D1 硬排除**之后**的池内生效，绝不翻越去重 |
| §1.4 `ADOPTION_MIN_TURNS=2` 误伤"短会话真讨论" | 中 | 低 | 全场无达标话题时保留 `matchTopic` fallback，仍记一条；阈值集中常量，可调 |
| §1.4 `isTurnOnTopic` 在 topic.description 很长时子串误命中（罕见但可能）| 低 | 低 | 描述为空时直接不纳入；描述含停用词不可避免，但子串需出现在 user/assistant 文本中才算 |
| §1.4 `onTopicTurns` 追踪与阶段切换耦合出错，漏计/多计 | 中 | 中 | 只在"非阶段切换的正常对话轮"累加；U5 覆盖关键词命中 / 不命中 / fallback 三类边界 |
| 四项联动后行为与旧版差异大，用户观感变化 | 中 | 低 | 这正是修复目标；M1/M2 用户侧观察确认方向正确 |

> **DoD（v1.0.9 整体）**：
>
> **§1.1 stale 快照修复**：
> - `createTopicSelectTool` 的 stats/keywordStats 支持 getter，`execute()` 每次读最新 DB；
> - server.ts / cli.ts 注册改为传 getter；过时注释更新；
> - U2/U3/I2 通过——证明不再持有启动快照，且数组重载向后兼容。
>
> **§1.2 噪声修复**：
> - `NOISE_RANGE = 0.05`，注释同步；
> - U1 通过——count 信号不再被噪声翻盘。
>
> **§1.3 Hook A 上下文化**：
> - `topic_select` 接受 `context_keywords?`，`contextBoost` 在硬排除后池内软加权（独立于被禁用的 interest 通道）；
> - turn.ts 提取 WARM_UP 关键词并传入；
> - U4/I3 通过——相关话题被优先、但去重优先级高于相关性。
>
> **§1.4 write-on-adoption**：
> - ledger 记录 `onTopicTurns`（每次正常轮后由 `isTurnOnTopic` 决定是否 +1），`recordAdoptedTopics` 只对 `>= ADOPTION_MIN_TURNS` 的话题计数；
> - 无达标话题时保留 `matchTopic` fallback；
> - U5/I4 通过——计数反映"真的讨论"而非"抽中"。
>
> **不动**：`topic-engine.ts` 既有纯函数签名与 D1/D2/D5 算法结构（除 §1.2 常量、§1.3 新增 contextBoost）；v1.0.8 的语音 / LLM provider / Settings 领域全部代码；`topics-default.json` 话题库内容。
>
> **明确不做**：改硬排除窗口（撤回的方向 3）、改 count/keyword 权重、LLM 语义级跑题判定、话题库扩容。
