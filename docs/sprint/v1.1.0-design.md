# Sprint v1.1.0 设计 — 话题库自动扩展

> **Sprint**: v1.1.0
> **依赖**: v1.0.7（write-on-selection / topic_stats）+ v1.0.9（写选择 + 写 adoption）+ v1.0.5 §C（topics baseline 34）
> **日期**: 2026-07-13
> **范围**: [v1.1.0-scope.md](./v1.1.0-scope.md)
> **一句话**：让话题库"长出来"。开关打开后，session 结束时 summary 关键词**先尝试挂到现有 topic**（matchTopic threshold=0.05）；与所有现有 topic 都无共享 keyword 时，**才用 LLM 整理成新 topic**。开关默认 OFF — fail-safe / 向后兼容。

## 0. 复用与不动

本 sprint **不动**：

- `topic_select` 工具及其算法（D1 hard-exclude / D2 count / D5 keyword freshness）

- baseline 34 topics 的 keywords / description
- `profile-extractor.ts` — 用户 profile 更新走自己链路，与 topic 库独立

## 1. §1.1 开关 `auto_expand_topics`

**为什么不是 `USER.md`**：这是 UI/功能开关，不是 profile 字段（与 `show_debug` 同级）。`preferences.json` 现有结构：


加 `auto_expand_topics?: boolean`，默认 `false`。读取通过 `loadPrefs()` 现有的 `Record<string, unknown>` 返回类型自然兼容；写一个 `isAutoExpandTopicsEnabled(): boolean` 帮助函数避免类型膨胀。

### 1.2 Server 端 PUT /api/settings

仿 `show_debug` 行 794-797：

```ts
if (typeof body.auto_expand_topics === 'boolean') {
  prefsUpdates.auto_expand_topics = body.auto_expand_topics
  persisted.push('auto_expand_topics')
}
```

调用 `savePrefs(prefsUpdates)` 落到磁盘。

### 1.3 CLI 端启动读取

CLI 进程级别 — 启动时 `const autoExpandTopicsEnabled = loadPrefs().auto_expand_topics === true`，传给 session-end 块。CLI 不响应运行期切换（与 `voice_enabled` 等其他 USER.md 字段同模式：进程内常量）。

### 1.5 实时刷新

Server 端 `autoExpandTopicLibrary()` 每次调用前 `loadPrefs()` — 用户在 Web UI 切换开关后，**下一次 session 结束**立即生效。CLI 不支持运行期切换（一致性约束：CLI 是 CLI profile，启动时定）。

## 2. §1.2 `TopicsDao.create()`

### 2.1 DAO 接口扩展

```ts
// src/storage/topics.ts:17
export interface TopicsDao {
  list(): Topic[]
  get(name: string): Topic | null
  create(input: {
    name: string
    keywords: string[]
    description: string | null
    createdAt: string
  }): boolean  // true = 真插入；false = 重名跳过
}
```

### 2.2 实现

```ts
const insertOne = raw.prepare(`
  INSERT OR IGNORE INTO topics (name, keywords_json, description, created_at)
  VALUES (?, ?, ?, ?)
`)
return {
  // ...list / get ...
  create(input) {
    const r = insertOne.run(
      input.name,
      JSON.stringify(input.keywords),
      input.description,
      input.createdAt,
    )
    return r.changes > 0
  },
}
```

**幂等**：与现有 `migration 007` 的 `INSERT OR IGNORE` 模式一致；重名时 `r.changes = 0`，返回 false。

### 2.3 为什么不需要新 SQL 迁移

`topics` 表（`migrations/003_topic_stats.sql`）的 schema 完整支持新行：

```
CREATE TABLE topics (
  name TEXT PRIMARY KEY,
  keywords_json TEXT NOT NULL,
  description TEXT,
  created_at TEXT NOT NULL
)
```

无需新列、无需新表。

### 2.4 与 export 脚本的关系

`scripts/export-topics-default.ts` 按 DB 全量导出。auto-expand 写入运行时 DB 后，**dev 主动跑 export + commit** 才能把新 topic 同步到 `data/topics-default.json` 与 `src/storage/migrations/007_topics_default.sql`。这是有意的：auto-expand 不污染 baseline 34 — baseline 改动需要 dev review 后才能进入 git 历史。

## 3. §1.3 `autoExpandTopicLibrary()` — session-end 管线

### 3.1 函数签名

```ts
// src/agent/auto-expand.ts
export function autoExpandTopicLibrary(
  summaryKeywords: string[],
  adoptedSlugs: string[],
  deps: {
    topics: Topic[]
    topicStats: TopicStatsDao
    keywordHits: KeywordHitsDao
    topicsDao: TopicsDao
    client: LLMClient
  },
  ctx: { now: Date },
  prefs: { enabled: boolean },
): Promise<void>
```

**async**：因为要 await LLM 调用。`enabled = false` 时早返回。

### 3.2 算法

```
async function autoExpandTopicLibrary(...) {
  // 0. 开关检查
  if (!prefs.enabled) return
  if (summaryKeywords.length === 0) return

  // 1. 过滤掉已经 adopted 的 topic 关联 keywords（防止合并路径与
  //    recordAdoptedTopics 在同一个 session 内重复写 stats）
  const adoptedTopics = deps.topics.filter((t) => adoptedSlugs.includes(t.name))
  const adoptedKeywords = new Set(
    adoptedTopics.flatMap((t) => t.keywords.map((k) => k.toLowerCase()))
  )
  const newKeywords = summaryKeywords.filter(
    (k) => !adoptedKeywords.has(k.toLowerCase())
  )
  if (newKeywords.length === 0) return

  // 2. 合并路径：用更宽松的 threshold 找现有 topic
  try {
    const best = matchTopic(newKeywords, deps.topics, 0.05)
    if (best && best.shared.length > 0) {
      // 仅追加 keyword hits，不 increment topic_stats（避免虚高）
      deps.keywordHits.upsertMany(best.topic, best.shared, ctx.now)
      process.stderr.write(
        `[auto-expand] merged [${best.shared.join(',')}] → ${best.topic}\n`,
      )
      return
    }

    // 3. 新建路径：LLM 整理
    const proposed = await extractNewTopicFromKeywords(
      newKeywords,
      deps.topics.map((t) => t.name),
      deps.client,
    )
    if (!proposed) {
      process.stderr.write('[auto-expand] LLM declined / invalid, skip\n')
      return
    }
    const inserted = deps.topicsDao.create({
      name: proposed.name,
      keywords: proposed.keywords,
      description: proposed.description,
      createdAt: ctx.now.toISOString(),
    })
    if (!inserted) {
      // OR IGNORE 跳过（理论上前置校验已挡住，但保留兜底）
      process.stderr.write(`[auto-expand] duplicate name ${proposed.name}, skip\n`)
      return
    }
    deps.topicStats.incrementAndUpdate(proposed.name, ctx.now)
    deps.keywordHits.upsertMany(proposed.name, proposed.keywords, ctx.now)
    process.stderr.write(
      `[auto-expand] created: ${proposed.name} (${proposed.keywords.length} kw)\n`,
    )
  } catch (err) {
    process.stderr.write(`[auto-expand] failed: ${(err as Error).message}\n`)
    // 静默放弃，session-end 主流程不受影响
  }
}
```

### 3.3 关键设计点

**（a）不与 `recordAdoptedTopics` 双计**：
- `adoptedSlugs` 内的 topic，stats 已经在 `recordAdoptedTopics` 写过（`incrementAndUpdate` + `upsertMany`）；
- 合并路径**只追加** `keyword_hits`、**不 increment** `topic_stats`；
- 新建路径（LLM 决定的新 topic）天然不在 adoptedSlugs 里，可以完整写三件。

**（b）threshold 0.05 比 baseline 0.1 宽松**：
- `matchTopic` 的 `score = max(jaccard, shared.length / sessionKeywords.length)`
- session 5 个关键词、与现有 topic 共享 1 个：`shared_count / session_count = 1/5 = 0.20` → 0.20 > 0.05 → 命中
- session 10 个关键词、共享 1 个：`1/10 = 0.10` → 0.10 > 0.05 → 命中
- session 20 个关键词、共享 1 个：`1/20 = 0.05` → 刚好 ≥ 0.05 → 命中
- session 30 个关键词、共享 1 个：`1/30 = 0.033` → < 0.05 → 不命中

阈值选 0.05 是"每个 session 平均共享 1 个关键词就挂"的程度 — 与用户"看是不是有 topic 可加入"的直觉一致。如果共享关键词 ≥ 1 就挂（threshold=0），会过度合并；0.05 是经验值。

**（c）LLM 失败兜底**：
- LLM `client.chat` 抛错 → 整段 try/catch 捕获 → stderr 一行 → return；
- session-end 主流程不受影响（与 `recordAdoptedTopics` 同样的 best-effort 模式）。

**（d）服务端 vs CLI 端调用位置**：

| 入口 | 调用位置 | 现有模式 |
|---|---|---|
| Server | `src/server.ts:555` 之后、`markEnded` 之前 | 包在同一个 try/catch 里 |
| Server（summarize 失败兜底）| `src/server.ts:601` 之后 | 同上 |
| CLI | `src/cli.ts:533` 之后、`markEnded` 之前 | 包在 `finally` 的 try/catch 里 |

server.ts:601 兜底路径**没有** summary keywords，所以 auto-expand 在那里是 no-op（`summaryKeywords.length === 0` → return）— 不需要额外分支。

## 4. §1.4 `extractNewTopicFromKeywords()` — LLM 整理

### 4.1 函数签名

```ts
// src/agent/topic-builder.ts
export interface ProposedTopic {
  name: string
  keywords: string[]
  description: string
}

export async function extractNewTopicFromKeywords(
  keywords: string[],
  existingTopicNames: readonly string[],
  client: LLMClient,
): Promise<ProposedTopic | null>  // null = decline / invalid
```

### 4.2 Prompt（仿 profile-extractor.ts）

```ts
const TOPIC_BUILDER_PROMPT = [
  'You are a topic taxonomy builder for an English oral-teacher agent.',
  'Given a list of keywords that did NOT match any existing topic, decide',
  'whether they describe a coherent new topic worth adding to the library.',
  '',
  'Return ONLY a JSON object (no markdown, no explanation):',
  '{',
  '  "should_create": true | false,',
  '  "name": "topic_slug_snake_case",',
  '  "keywords": ["keyword1", "keyword2", ...],',
  '  "description": "Short English label (B1/B2 hint)"',
  '}',
  '',
  'Rules:',
  '- When in doubt, return should_create: false (we can re-evaluate next session).',
  '- Only create when keywords describe a coherent topic (≥2 keywords, each ≥2 chars).',
  '- Reject scattered / 1-word trivia / transient events (e.g. "ok", "yes", "really").',
  '- Slug: 3-30 chars, lowercase, snake_case, English-only, must be unique vs existing.',
  '- keywords: 3-15 English lowercase tokens. Include meaningful original inputs;',
  '  drop noise (1-2 char words, pure numbers, interjections).',
  '- description: 1 short English sentence with B1/B2 level hint (max 200 chars).',
].join('\n')
```

完整 prompt 在调用时附加：
```
Existing topic names (DO NOT reuse): ${existingTopicNames.join(', ')}

Keywords to evaluate: ${keywords.join(', ')}

Return ONLY the JSON object.
```

### 4.3 响应解析 + 校验链

```ts
const result = await client.chat({
  system: 'You build structured topic taxonomy. Always output valid JSON.',
  messages: [{ role: 'user', content: prompt }],
  temperature: 0.3,
  maxTokens: 250,
})

let parsed: any
try {
  parsed = JSON.parse(result.content)
} catch {
  return null  // decline on parse failure
}

// 校验链（任一失败 → return null）
if (parsed.should_create !== true) return null
if (typeof parsed.name !== 'string') return null
if (!/^[a-z][a-z0-9_]{2,29}$/.test(parsed.name)) return null  // slug 格式
if (existingTopicNames.includes(parsed.name)) return null       // 重名
if (!Array.isArray(parsed.keywords)) return null
const kwClean = parsed.keywords
  .filter((k: unknown): k is string => typeof k === 'string')
  .map((k: string) => k.toLowerCase().trim())
  .filter((k: string) => k.length >= 2 && !/^\d+$/.test(k))
  .slice(0, 15)
if (kwClean.length < 2) return null  // keywords 不足
if (typeof parsed.description !== 'string') return null
const descClean = parsed.description.trim()
if (descClean.length === 0 || descClean.length > 200) return null

return { name: parsed.name, keywords: kwClean, description: descClean }
```

### 4.4 为什么 temperature=0.3

参考 profile-extractor.ts:55 — 既不是 0（完全确定性，会重复相同错误）也不是 1（随机性过高）。0.3 让 LLM 在拿不准时倾向给 false / 给不同答案，避免每次返回同样错误的命名。

### 4.5 maxTokens=250

JSON 对象只有 4 个字段，最多 ~300 字符；250 是余量上限，防止 LLM 啰嗦。

## 7. 例子（端到端）

### 7.1 场景 A：合并路径

session 内容：聊波音 737 驾驶舱、aerospace engineering。

```
summarize.keywords = ['boeing', '737', 'cockpit', 'aerospace', 'engineering']
recordAdoptedTopics(adopted={})  → [] (空)
autoExpandTopicLibrary(['boeing', '737', 'cockpit', 'aerospace', 'engineering'],
                       [],
                       { topics: baseline34, ... },
                       { now: '2026-07-13T...' },
                       { enabled: true })

→ step 1: adoptedTopics=空 → newKeywords 不变
→ step 2: matchTopic(['boeing', ...], baseline34, threshold=0.05)
   ├─ 'aviation' topic 已存在，shared = ['boeing', 'cockpit']  (2/5 = 0.4 > 0.05)
   └─ best = { topic: 'aviation', shared: ['boeing', 'cockpit'] }
→ keywordHits.upsertMany('aviation', ['boeing', 'cockpit'], now)
→ stderr: "[auto-expand] merged [boeing,cockpit] → aviation"
```

DB 状态：`aviation` 的 keyword_hits 表多 2 行（boeing, cockpit），`topic_stats` 不变。**与 `recordAdoptedTopics` 路径完全正交**。

### 7.2 场景 B：新建路径

session 内容：聊 pottery / ceramics / kiln / glaze。

```
summarize.keywords = ['pottery', 'ceramics', 'kiln', 'glaze']
→ step 2: matchTopic(...threshold=0.05)
   ├─ 现有 34 topics 中没有 shared >= 1 的
   └─ best = null
→ step 3: extractNewTopicFromKeywords(['pottery', 'ceramics', 'kiln', 'glaze'],
                                       existingNames=[34 个 slug],
                                       client)
   └─ LLM 返回: {"should_create": true,
                  "name": "ceramics",
                  "keywords": ["pottery", "ceramics", "kiln", "glaze", "clay"],
                  "description": "Ceramics & pottery — making, firing, glazing (A2-B1)"}
   └─ 校验链全通过
→ topicsDao.create({ name: 'ceramics', keywords: [...], description: '...' })
   → true (新插入)
→ topicStats.incrementAndUpdate('ceramics', now)
→ keywordHits.upsertMany('ceramics', [...], now)
→ stderr: "[auto-expand] created: ceramics (5 kw)"
```

DB 状态：`topics` 表多 1 行，`topic_stats` 多 1 行 (count=1)，`keyword_hits` 多 5 行。

### 7.3 场景 C：LLM 拒绝

session 内容：聊 yes / ok / 嗯。

```
summarize.keywords = ['yes', 'ok']
→ step 2: matchTopic 全无 shared ≥ 1
→ step 3: extractNewTopicFromKeywords(['yes', 'ok'], ...)
   └─ LLM 返回: {"should_create": false, ...}
   └─ 校验链: parsed.should_create !== true → return null
→ stderr: "[auto-expand] LLM declined / invalid, skip"
```

DB 状态：无变化。**符合用户"倾向拒绝"决策**。

## 8. 与 v1.0.9 的协同

| 维度 | v1.0.9 | v1.1.0 |
|---|---|---|
| 关注点 | 选择 / 计数 / 匹配 | 库覆盖 |
| 解决的问题 | "重复抽同一批" | "库里没有该主题" |
| 数据流 | 读 `topics` | 写 `topics`（仅 auto-expand）|
| LLM 调用 | 0（`isTurnOnTopic` 是字符串匹配）| 1（仅当 ON + 无 merge 命中时）|
| 开关 | 无（修选择是默认行为）| `auto_expand_topics`（默认 OFF）|
| 风险面 | 运行时（每次 select）| session 结束（频率低）|

两者**正交**：v1.0.9 让"选得对、记得对"，v1.1.0 让"库够大"。v1.1.0 写出的新 topic 在下一场 session 立即进入 `topic_select` 的候选池（D1 hard-exclude 30 天会跳过它，所以下一场 session 反而**不被选中** — 这恰好符合"刚聊过不要重复"的设计意图）。

## 10. 话题匹配命中率修复（§A–§C）

> **触发**：2026-07-14 真实会话测试。会话覆盖 gaming / travel / aviation / reading / music 五个领域，但 session-end 后 `topic_stats` 几乎无增量。
> **根因诊断**：见 [v1.1.0-scope.md §2](./v1.1.0-scope.md#2-话题匹配命中率修复21a23c)。

### 10.1 §A — 统一关键词格式

**现状**：

| 来源 | 格式 | 示例 |
|------|------|------|
| Summarizer 输出 | 自然英语（空格） | `"delta force"`, `"harry potter"` |
| 话题库 keywords | 下划线连接 | `"delta_force"`, `"harry_potter"` |

`matchTopic` 做精确字符串 `includes` 匹配，`"delta force" !== "delta_force"` → 不命中。

**设计**：在 `matchTopic()` 开头增加归一化步骤，把空格替换为下划线：

```ts
// topic-matcher.ts — normalize before matching
const norm = (s: string) => s.toLowerCase().replace(/\s+/g, '_')
const normSession = sessionKeywords.map(norm)
// topic keywords are already in normalized form (lowercase, underscore)
```

- 不需要改动话题库数据格式（keywords 已是下划线）
- 不需要改动 `keywordHits` 表（已经记的是下划线格式）
- summarizer 输出自然语言短语 → 归一化后自动匹配

**归一化位置**：`matchTopic` 入参处，不改外部调用方的代码。`recordAdoptedTopics` 传给 `matchTopic` 的 `fallbackKeywords` 无需预处理。

### 10.2 §B — 优化 summarizer prompt

**现状**：`prompts/summarizer-system.md` 未区分"内容词"和"教学元词"。实际输出中，`correction`, `series`, `character`, `episode` 等教学元词频繁出现，稀释了内容命中率。

**设计**：在 `summarizer-system.md` 的 JSON schema 说明中增加两条约束：

```markdown
## keywords rules
- Keywords MUST describe **what the student talked about** (conversation content),
  NOT what was taught or corrected.
- Use **underscores for multi-word phrases**: `summer_vacation`, `delta_force`,
  `harry_potter`, `chongqing_food`.
```

并在 JSON 输出示例中展示正确格式：

```json
{
  "summary": "The student discussed...",
  "keywords": ["summer_vacation", "delta_force", "harry_potter", "airplane_travel", "pop_music"]
}
```

**注意**：这是一个**软约束**——LLM 不一定 100% 遵循。§A 的归一化是硬兜底。两处改动互相配合：prompt 优化减少元词 + 归一化消除残余格式差异。

### 10.3 §C — matchTopic 返回 Top-N

**现状**：

```ts
// topic-matcher.ts:33 — single-best return
export function matchTopic(
  sessionKeywords: string[],
  topics: Topic[],
  threshold = 0.1,
): TopicMatch | null {  // ← 只返回一个
  // ...
  return best
}
```

`recordAdoptedTopics` 的 fallback 路径（line 87-92）只处理 `best: TopicMatch | null`。

**设计**：

**10.3.1 修改 `matchTopic` 签名**：

```ts
export function matchTopic(
  sessionKeywords: string[],
  topics: Topic[],
  threshold = 0.1,
): TopicMatch[] {  // ← 返回数组，按 jaccard 分值降序
  // ...
  return results.sort((a, b) => b.jaccard - a.jaccard)
}
```

**10.3.2 修改 `recordAdoptedTopics` fallback**：

```ts
// Before (v1.0.9):
const best = matchTopic(fallbackKeywords, deps.topics)
if (best) {
  deps.topicStats.incrementAndUpdate(best.topic, ctx.now)
  deps.keywordHits.upsertMany(best.topic, best.shared, ctx.now)
  return [best.topic]
}

// After (v1.1.0 §C):
const matches = matchTopic(fallbackKeywords, deps.topics)
for (const m of matches) {
  deps.topicStats.incrementAndUpdate(m.topic, ctx.now)
  deps.keywordHits.upsertMany(m.topic, [m.shared[0]], ctx.now)
}
return matches.map(m => m.topic)
```

**10.3.3 `keywordHits` 写入策略**：

每个匹配只写 `shared[0]`（第一个共有关键词），不写全部 shared。理由：
- `keyword_hits` 的 D5 信号只需 slug 级频次
- 多写变体词让表膨胀但去重效果不提升
- 与主路径 `recordAdoptedTopics` 的 `upsertMany(slug, [suggestedKeyword])` 保持一致

**10.3.4 有其他调用方吗？**

检查 `matchTopic` 的所有引用：
- `topic-recorder.ts:87` — 已改
- `auto-expand.ts`（如存在）— 使用自己的 threshold=0.05 调用，返回类型改成数组后需同步更新
- 其他文件 — grep 确认

## 11. ledger+fallback 合并 + auto-expand MERGE 遍历 Top-N（§D + §E）

> **触发**：2026-07-14 第二次真实会话（session2）。会话覆盖 school_life / shopping / gaming / friends_social 四个话题，session-end 后只命中 1 个。auto-expand 开关已打开但关键词也未补充到话题库。
> **根因诊断**：见 [v1.1.0-scope.md §5](./v1.1.0-scope.md#5-ledgerfallback-合并--auto-expand-merge-遍历-top-n51a52b)。

### 11.1 §D — `recordAdoptedTopics` 取消 early return

**现状**：

```ts
// topic-recorder.ts:74-84 (v1.0.9)
if (adopted.size > 0) {
  for (const [slug, info] of adopted) {
    if (info.onTopicTurns < ADOPTION_MIN_TURNS) continue
    deps.topicStats.incrementAndUpdate(slug, ctx.now)
    deps.keywordHits.upsertMany(slug, [info.suggestedKeyword], ctx.now)
    adoptedSlugs.push(slug)
  }
  if (adoptedSlugs.length > 0) return adoptedSlugs  // ← 问题行
}
// fallback — 只有 adoptedSlugs 为空才跑
```

后果：ledger 有 1 个话题（如 school_life）达标时，fallback 完全不执行。其余讨论过的话题（shopping、gaming）不进入 `topic_stats`，也不传给 `autoExpandTopicLibrary` 的 `adoptedSlugs`。

**修复**：

```ts
// topic-recorder.ts — after fix
const writtenSlugs = new Set<string>()

// 1. Write ledger topics that meet threshold
if (adopted.size > 0) {
  for (const [slug, info] of adopted) {
    if (info.onTopicTurns < ADOPTION_MIN_TURNS) continue
    deps.topicStats.incrementAndUpdate(slug, ctx.now)
    if (info.suggestedKeyword) {
      deps.keywordHits.upsertMany(slug, [info.suggestedKeyword], ctx.now)
    }
    writtenSlugs.add(slug)
  }
}

// 2. Run fallback for any topics NOT already in the ledger
const matches = matchTopic(fallbackKeywords, deps.topics)
for (const m of matches) {
  if (writtenSlugs.has(m.topic)) continue  // dedup
  deps.topicStats.incrementAndUpdate(m.topic, ctx.now)
  deps.keywordHits.upsertMany(m.topic, [m.shared[0]!], ctx.now)
  writtenSlugs.add(m.topic)
}

return [...writtenSlugs]
```

**关键变化**：不再区分"ledger 路径"和"fallback 路径"，两者一起跑，用 `writtenSlugs` Set 去重。

**对 `autoExpandTopicLibrary` 的影响**：`adoptedSlugs` 现在包含 ledger + fallback 的全部话题，Step 1 的过滤更准确（更多 keywords 被标记为"已采纳"），减少了误 merge。

### 11.2 §E — auto-expand MERGE 遍历全部 Top-N 匹配

**现状**：

```ts
// auto-expand.ts:77-87 (v1.1.0 §1.3)
const matches = matchTopic(newKeywords, deps.topics, MERGE_THRESHOLD)
const best = matches[0]  // ← 只取最优
if (best && best.shared.length > 0) {
  deps.keywordHits.upsertMany(best.topic, best.shared, ctx.now)
  return  // ← 其余关键词全丢
}
```

session2 场景：7 个剩余关键词中有 `delta_force`（匹配 gaming）和 `school_subjects`（匹配 school_life），但只取 matches[0] → gaming 得到 1 个 keyword，school_life 被跳过，其余 `grandparent`、`cola`、`supermarket`、`sports_drinks`、`shooting_game` 全丢。

**修复**：

```ts
// auto-expand.ts — MERGE step rewritten
const matches = matchTopic(newKeywords, deps.topics, MERGE_THRESHOLD)
const mergedKwSet = new Set<string>()

for (const m of matches) {
  // Each match contributes its shared keywords to the existing topic.
  deps.keywordHits.upsertMany(m.topic, m.shared, ctx.now)
  for (const k of m.shared) mergedKwSet.add(k.toLowerCase())
}

if (matches.length > 0) {
  process.stderr.write(
    `[auto-expand] merged into ${matches.length} topic(s): ` +
    matches.map(m => `${m.topic}[${m.shared.join(',')}]`).join('; ') + '\n'
  )
}

// Only keywords that appeared in ZERO matches are candidates for CREATE.
const unmerged = newKeywords.filter(
  k => !mergedKwSet.has(k.toLowerCase())
)
if (unmerged.length === 0) return

// Step 3: CREATE — only for truly unmatched keywords
const proposed = await extractNewTopicFromKeywords(
  unmerged, deps.topics.map(t => t.name), deps.client
)
// ... rest unchanged
```

**关键变化**：MERGE 不再取 `matches[0]`，而是遍历全部匹配，将每个匹配的 shared 关键词写入对应 topic。只有**所有匹配中都没出现的关键词**才走 CREATE。

### 11.3 session2 模拟验证

| 步骤 | 关键词 | 结果 |
|------|--------|------|
| 剩余关键词 | supermarket, grandparent, cola, sports_drinks, shooting_game, delta_force, school_subjects | — |
| §E MERGE → gaming | `delta_force` | keyword_hits gaming +1 |
| §E MERGE → school_life | `school_subjects` | keyword_hits school_life +1 |
| mergedKwSet | {delta_force, school_subjects} | — |
| §E unmerged | supermarket, grandparent, cola, sports_drinks, shooting_game | → CREATE candidate |

合并后 2 个关键词挂到了已有 topic，5 个真正新关键词传送给 LLM 做 CREATE 判断。
