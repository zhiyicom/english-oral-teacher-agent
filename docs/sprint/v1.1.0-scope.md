# Sprint v1.1.0 范围 — 话题库自动扩展（按需新建 / 合并关键词）

> **Sprint**: v1.1.0
> **上一 sprint**: v1.0.9（话题选择去重机制修复 — 已合 main）
> **分支**: `main`
> **日期**: 2026-07-13
> **状态**: 1 个开关 + 1 条 session-end 管线 + 1 次 LLM 调用（开关 OFF 时完全无副作用）

---

## 背景

v1.0.9 把"选择重复 + 命中分布不均"的四个根因修完了，**已有 34 个 baseline topics 的选择机制**不再退化。但库的**覆盖面**是另一个独立问题：

- 2026-07-13 当天练习覆盖了"飞机 / 波音 737 / 三角洲行动 / minecraft / 赵雷 / 程艾影"等内容 — 库里完全没有匹配；
- 库里"music" / "tech" 等覆盖太宽，但没有任何"boeing 737"、"delta force"、"zhaolei" 这种细分命中；
- 这导致 **selectTopic 的 D1 hard-exclude 形同虚设**（话题从未被采纳 = 从未进入 30 天窗口 = 永远会被选中）— Jeremy 的体感是"练来练去就在那几个话题里转"。

修选择机制（v1.0.9）能解决"重复抽同一批"，但要让库**长出新的话题 / 新关键词**来覆盖学生的真实兴趣，必须有"自动扩展"通道。

目前库的演进路径是**手工**：开发者看 llm-debug 日志，发现新内容 → 写 SQL `INSERT` → 跑 `scripts/export-topics-default.ts` 重生成 baseline → 写 commit。**慢 / 不连续 / 不覆盖**。

## 目标

加一个**开关式自动扩展**：开关打开后，每次 session 结束会分析 summary 关键词，**先尝试合并到现有 topic；不能合并再用 LLM 整理成新 topic**。开关默认 OFF — 向后兼容零风险。

### 关键决策（已与用户对齐）

| 决策点 | 选择 | 理由 |
|---|---|---|
| LLM 拿不准时 | **倾向拒绝**（`should_create: false`） | 不污染库；用户可在 Web UI 手动管理 |
| 开关默认 | **OFF** | 向后兼容；冷启动不建无意义 topic |
| 关键词挂靠策略 | **先用更宽松的 `matchTopic` 挂到现有 topic；都没命中再 LLM 新建独立 topic** | 用户的原话："看是不是有 topic 可加入" |

## 1. 范围内

### 1.1 设置开关 `auto_expand_topics`

**目标**：在 Settings 页加一个开关，控制"session 结束后是否自动扩展话题库"。开关默认 OFF，OFF 时**整条管线不触发，行为完全等同 v1.0.9**。

| # | 事项 | 文件 |
|---|---|---|
| 1 | `preferences.json` schema 新增 `auto_expand_topics: boolean`（默认 `false`） | `src/server.ts`（在 `loadPrefs` 默认值 + `PUT /api/settings` 分支）|
| 2 | CLI 端启动时读取 `preferences.json` 的 `auto_expand_topics` 值（CLI 同样的开关语义） | `src/cli.ts` |
| 3 | Web Settings 页加 ON/OFF 按钮（仿 `show_debug` 行 277-288 模板），调用 `updateField('auto_expand_topics', ...)` | `web/src/components/SettingsPage.tsx` |
| 4 | `STRINGS` 新增 2 个 key（标签 + 描述，仿 `settingsShowDebug`） | `web/src/components/SettingsPage.tsx` 或 i18n 文件（若有）|

**约束**：
- 持久化放 `preferences.json`，**不动 `USER.md`**：这是 UI/功能开关，不是 profile 字段。
- 开关状态变更后**不需要重启 server**：CLI 一次性读取（CLI 进程级别生效即可）；server 每次 `recordAdoptedTopics` 调用前查 `loadPrefs()` 即可。
- 不引入新的 SQL 表 / 列 — 现有 `topics` / `topic_stats` / `keyword_hits` 表已能容纳。

### 1.2 `TopicsDao.create()` — 新建独立 topic

**目标**：让 auto-expand 能 INSERT 一条新 topic 到 `topics` 表。**`OR IGNORE` 幂等**，已有同名 topic 时静默跳过。

| # | 事项 | 文件 |
|---|---|---|
| 5 | `TopicsDao` 接口加 `create(input)` 方法；`createTopicsDao` 实现：单条 `INSERT OR IGNORE INTO topics (name, keywords_json, description, created_at) VALUES (?, ?, ?, ?)`；返回 `changes > 0`（即真插入了） | `src/storage/topics.ts` |

**约束**：
- 不动 schema：复用现有 `topics` 表；不需要新 SQL migration。
- 不动 baseline 34：`scripts/export-topics-default.ts` 仍按 DB 全量导出，但新 topic 进运行时 DB 之后，下次跑 export 会**包含**到 `topics-default.json` 与 `007_topics_default.sql`（dev 主动 commit 即"接受"为 baseline；不 commit 也不会丢，因为运行时 DB 是单一事实源）。
- 不做批量 INSERT：一次只插一条（每次 session 最多 1 个新 topic）。

### 1.3 `autoExpandTopicLibrary()` — session-end 管线

**目标**：在 `recordAdoptedTopics()` 之后（同一个 finally / SSE-done 块里），调用新的 `autoExpandTopicLibrary(...)`。**只在开关 ON 时调用**，OFF 时函数体直接 `return`。

| # | 事项 | 文件 |
|---|---|---|
| 6 | 新建 `src/agent/auto-expand.ts`：`autoExpandTopicLibrary(summaryKeywords, adoptedSlugs, deps, ctx)` 函数 | NEW |
| 7 | server.ts: 现有 `recordAdoptedTopics` 调用之后，新增 `autoExpandTopicLibrary(...)` 调用（用同一个 try/catch 包住）；CLI:533 同步 | `src/server.ts`, `src/cli.ts` |

**函数签名**：
```ts
export function autoExpandTopicLibrary(
  summaryKeywords: string[],
  adoptedSlugs: string[],
  deps: {
    topics: Topic[]
    topicStats: TopicStatsDao
    keywordHits: KeywordHitsDao
    topicsDao: TopicsDao
    client: LLMClient
  },
  ctx: { now: Date },
): void
```

**算法**（用户已对齐的"先合并 / 后新建"策略）：

```
1. 过滤 summaryKeywords，去掉已经在 adoptedSlugs 关联话题的 keywords
   （避免双重计数；合并路径只看"还没被采纳的额外关键词"）
2. 用 matchTopic(summaryKeywords, topics, threshold=0.05) 跑一遍
   ├─ best != null（shared.length >= 1）:
   │   └─ keywordHits.upsertMany(best.topic, best.shared, ctx.now)
   │     ↑ 只追加关键词到 hit_count，不 increment topic_stats（避免虚高）
   │     stderr: "[auto-expand] merged <keywords> → <topic>"
   └─ best == null（与所有现有 topic 都无 keyword 共享）:
       └─ 调 LLM extractNewTopicFromKeywords(summaryKeywords, deps.client)
           └─ { should_create, name, keywords[], description }
               ├─ should_create=true + slug 合法 + name 不与现有 topic 重名:
               │     ├─ topicsDao.create({name, keywords, description, createdAt: now})
               │     ├─ topicStats.incrementAndUpdate(name, ctx.now)
               │     ├─ keywordHits.upsertMany(name, keywords, ctx.now)
               │     └─ stderr: "[auto-expand] created: <name>"
               └─ 其他:
                   └─ stderr: "[auto-expand] LLM declined / invalid, skip"
```

**关键不变量**：
- **不开关不触发**：函数体首行 `if (!prefs.autoExpandTopics) return`。
- **不写已 adopted 的 topic 的 stats**：`adoptedSlugs` 内的 topic 不进 step 2 的合并路径（已经在 `recordAdoptedTopics` 写过 stats）。
- **不重复插入**：LLM 输出的 name 与 `topics.list()` 重名时，`create()` 内部 `OR IGNORE` 自动跳过；上层显式校验更清楚，重复时拒绝。
- **失败兜底**：整条管线用 try/catch 包住，任何错误 → stderr 一行 + 静默放弃，**不影响 session 结束主流程**。

### 1.4 LLM 整理新 topic：`extractNewTopicFromKeywords()`

**目标**：把"未匹配的关键词"扔给 LLM，让它决定"应不应该建新 topic"。用户决策：**拿不准时倾向不建**（fail-safe）。

| # | 事项 | 文件 |
|---|---|---|
| 8 | 新建 `src/agent/topic-builder.ts`：`extractNewTopicFromKeywords(keywords, existingTopicNames, client)` | NEW |
| 9 | Prompt：参考 `profile-extractor.ts:16-38` 的 JSON-only 模板（"Always output valid JSON" + 强约束 `when in doubt return should_create: false`）| 同上 |
| 10 | 响应解析：`try JSON.parse / catch → null`（解析失败视为 decline） | 同上 |

**Prompt 关键约束**：
- `should_create: false` 多于 `true` — 用户决策（fail-safe）
- slug：`/^[a-z][a-z0-9_]{2,29}$/`（3-30 字符，snake_case，英文小写）
- keywords：3-15 个英文 token，去掉 1-2 字符单词、纯数字、纯语气词
- description：1 句英文 + B1/B2 等级提示
- 若不确定 → `should_create: false`（"we can re-evaluate next session"）

**校验链**（在 prompt 之后、create 之前）：
1. slug 格式正则
2. 与 `existingTopicNames` 数组对比，重名 reject
3. keywords 数组长度 ≥ 2 且每个 token ≥ 2 字符
4. description 非空且 ≤ 200 字符

## 2. 后续需求（占位）

- Web UI 的 topic 管理页（**列出 / 编辑 / 删除**当前库；用户对 auto-expand 建出来的 topic 有最终话语权）。v1.1.0 仅做 DB 写入，UI 让用户能看到但**不能编辑**。
- LLM 整理新 topic 时，把 `description` 也写进 `topic.description` 字段（参与 `isTurnOnTopic` substring 匹配）— v1.1.0 description **写但不参与匹配**（描述会出现在 keywords 字符串里太细，误命中风险高），留 §2 占位。
- 阈值 0.05 / maxTokens 250 等常量改成可配（写在 `preferences.json` 高级设置里）。
- 批量 / 离线整理：把过去 N 场 session 的 summary 一次性喂给 LLM，发现并补全历史遗漏的 topic。
- auto-expand 的覆盖率可视化：Settings 页显示"最近 N 次 session 自动扩展了多少新 topic / 合并了多少关键词"。

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| LLM 整理出的 topic 命名混乱（`ceramic_art` vs `pottery`）| 中 | 低 | slug 重复时 reject；用户在 Web UI 手动 delete |
| 开关 ON 后冷启动期数据库膨胀 | 中 | 中 | 每次 session 最多建 1 个 topic；用户一键关闭；`loadPrefs` 默认 OFF |
| `matchTopic` 阈值 0.05 过松 → 误把无关关键词挂到错误 topic | 低 | 低 | 阈值仍 ≥ `shared.length ≥ 1`；keywords 数组 OR 关系不会消除其他词 |
| `matchTopic` 阈值 0.05 过严 → 实际聊过的细分话题也不挂靠，频繁进 LLM 路径 | 中 | 低 | LLM 路径是 fail-safe，浪费一次调用但不影响功能；阈值常量可调 |
| LLM 调用增加 session-end 延迟 | 中 | 低 | 单次 LLM 调用 maxTokens=250、温度 0.3，与 profile-extractor 同量级 |
| LLM 调用失败（网络 / 配额）→ 阻塞 session-end | 低 | 中 | 整段 try/catch 包住；失败时 stderr 一行 + 静默放弃，session-end 主流程不受影响（与 `recordAdoptedTopics` 同样的 best-effort 模式）|
| `topicsDao.create()` 与 `OR IGNORE` 行为有出入 | 极低 | 低 | 单测 T1/T2 覆盖插入与重名 |
| 开关状态未实时刷新（CLI 启动后改 settings.json 不生效）| 中 | 低 | CLI 是单 session 进程，启动时读一次可接受；server 每次 `autoExpandTopicLibrary` 内重新 `loadPrefs` |

> **DoD（v1.1.0 整体）**：
>
> **§1.1 开关**：
> - `preferences.json` 默认 `auto_expand_topics: false`；
> - `PUT /api/settings` 接受并持久化；
> - Web Settings 页 ON/OFF 按钮渲染 + 写入；
> - CLI 启动时读取。
>
> **§1.2 `TopicsDao.create()`**：
> - 单测 T1（插入）/ T2（重名不重复）通过；
> - `topicsDao.create({...})` 返回 boolean 表示"真插入 / 重复跳过"。
>
> **§1.3 `autoExpandTopicLibrary()`**：
> - 开关 OFF 时早返回，零副作用（A1 通过）；
> - 合并路径：仅 `keywordHits.upsertMany`，`topicStats` 不变（A8 通过）；
> - 整段 try/catch，LLM 抛错不向上传播（A7 通过）；
> - server / cli 两端 session-end 都接入。
>
> **§1.4 `extractNewTopicFromKeywords()`**：
> - `should_create=true` + slug 合法 + 不重名 → 三个 DAO 各 1 次（A2 通过）；
> - `should_create=false` / 非法 slug / 重名 → 三个 DAO 全不调（A3/A4/A5 通过）；
> - 校验链：slug 正则 + keywords 长度 + description 非空。

## 2. 话题匹配命中率修复（§2.1–§2.3）

> **背景**：2026-07-14 真实会话测试暴露了三个叠加问题，导致多话题会话的 `matchTopic` fallback 几乎颗粒无收。详见 [v1.1.0-design.md §10](./v1.1.0-design.md#10-话题匹配命中率修复ab)。

### 2.1 §A — 统一关键词格式（消除空格 vs 下划线不匹配）

| # | 事项 | 文件 |
|---|---|---|
| 1 | `matchTopic()` 匹配前将 `sessionKeywords` 和 `topic.keywords` 都归一化为小写 + 空格→下划线替换 | `src/agent/topic-matcher.ts` MODIFY |

- 会话关键词 `"summer vacation"` + 话题 `travel`（含 `vacation`）→ 匹配命中

### 2.2 §B — 优化 summarizer prompt（减少教学元词）

| # | 事项 | 文件 |
|---|---|---|
