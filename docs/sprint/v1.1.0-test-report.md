## 6. 代码审查要点（自审）

### 6.1 `auto-expand.ts` — 关键不变量

1. **不重复 increment 已 adopted topic 的 stats**：
   - filter 阶段用 `adoptedSlugs` 排除关联词（L62-67）
   - merge 阶段只调 `keywordHits.upsertMany`，**不调** `topicStats.incrementAndUpdate`（L88-99）
2. **fail-safe 兜底**：`extractNewTopicFromKeywords` 返回 `null` 时（任何错误路径），整个 auto-expand 块不抛、不写库（L113-118）
3. **slug 重名校验**：传给 LLM 前先 `existingTopicNames = new Set(topics.list().map(t => t.name))`，避免 LLM 提议重名后被静默 `INSERT OR IGNORE` 吞掉（L70-73）

### 6.2 `topic-builder.ts` — 校验链顺序

按失败率从高到低排序（前置 fail-fast 减少 LLM 调用浪费）：

1. JSON parse 失败 → null（L84）
2. `should_create === true` 严格判断（非 truthy coerce）→ null（L88）
3. slug regex `/^[a-z][a-z0-9_]{2,29}$/` → null（L93）
4. `existingNames` 重名 → null（L97）
5. keywords 清理 + 长度 ≥ 2 → null（L100-108）
6. description 非空 + 长度 ≤ 200 → null（L111-115）

### 6.3 `preferences.ts` — strict 比较

```ts
export function isAutoExpandTopicsEnabled(dataDir: string): boolean {
  return loadPreferences(dataDir).auto_expand_topics === true
}
```

**为什么不直接 `!!prefs.auto_expand_topics`**：与用户"开关 OFF 默认"决策对齐 —— 即使用户手编辑 `preferences.json` 写成 `"yes"`，系统行为仍是 OFF，避免 truthy coerce 导致 surprise。`tests/config/preferences.test.ts:50-53` 显式覆盖此场景。

### 6.4 与现有资产的复用率

- **`matchTopic`**：直接复用，未改一行。auto-expand 的 merge 路径 = `matchTopic(keywords, topics, threshold=0.05)` + `keywordHits.upsertMany`（已有方法）。
- **`topic-recorder.recordAdoptedTopics`**：未改一行。auto-expand 完全是 recordAdoptedTopics 的下游补充，不替代。
- **`profile-extractor`**：复用其"system prompt + temperature 0.3 + maxTokens + try JSON.parse"模式，prompt 模板从 `extractStudentDiscoveries` 派生。
- **`TopicsDao` 接口**：在原有 `list` / `get` 上加 `create`，零破坏性变更。
- **DAO 注入方式**：与 v1.0.9 §1.1 一致，DAO 通过依赖注入传入 `autoExpandTopicLibrary`，CLI 和 server 共享同一函数。
