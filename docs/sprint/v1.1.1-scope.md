## 背景

2026-07-15 两次真实练习（`data/llm-debug/diag-0f7bf740.jsonl`、`diag-bfc000c5.jsonl`）暴露出 3 类 4 个 **LLM 输出异常触发的 UI 故障**：

| # | 学生侧现象 | 根因 |
|---|---|---|
| **P0-#2** | UI 出现 `[Tool call: topic_select]\n{"phase": ...}` 裸文本，无对话回复 | `tool-parser.ts:29` 只匹配 `<tool>...</tool>`；`turn.ts:677-682` 把整个 response 原样推给学生 |
| **P0-#4** | wrap-up / end 阶段对话开头有 `[Phase: WRAP_UP — Time remaining: ~1.6 min — ...]` 元数据 | LLM 偶尔把 `[System Context]` 块或内联前缀回显；`turn.ts:642` 只诊断日志不剥除 |
| **P1-#5** | 同一段 followup 写进 history 两次（`Nice, playing... Nice, playing...`）| `turn.ts:959-991` blocked 路径把 `response` 和 `followup` 都 push 进 history |
| **P1-#6** | topic_select blocked 后 fallback "复读机"（连续 2-3 轮基本一样的 followup）| blocked 时 2nd LLM call 没收到"不要重复"指令，倾向于复制最近对话风格 |

**用户原话（7/15 反馈）**：「使用过程中多次出现以下问题：1）话题调用切换失败 2）回复卡住，web界面只出现出现[]工具调用内容，没有对话回复 3）最后一次对话没有达到结束时间LLM开始结束对话」。

注：用户提到的「3）提前结束」属于 P2 范围（premature wrap-up 涉及 LLM 自发结论与状态机阈值的偏差），本次 sprint 暂不修，纳入 backlog。

### 关键决策（已与用户对齐）

| 决策点 | 选择 | 理由 |
|---|---|---|
| P0-#2 修复策略 | **识别 + 防御剥离**（parser 优先 + displayText 兜底）| LLM 偶尔失误时话题仍能切换；UI 永远不出现裸 `[Tool call: ...]` |
| P1-#5/#6 修复策略 | **短路 2nd call + 确定性 fallback** | 彻底消灭复读机 + 省一次 LLM 调用；displayText 必非空 |
| 发布方式 | **只推 main，不打 tag 不发 Release** | 沿用 v1.1.0 hotfix `bfb6680` 流程；最小干扰 |

## 1. 范围内

### 1.1 P0-#2 — Bracket 形式工具调用识别 + 防御剥离

**目标**：LLM 偶尔输出 `[Tool call: topic_select]\n{"phase": ...}` 而忘记 `<tool>...</tool>` 包装时，**优先尝试当成有效工具调用执行**；若 JSON 解析失败或 LLM 只输出了 `[Tool call: name]` 没有 json，**至少从 displayText 里剥掉这段前缀**，学生永远不会看到裸 `[Tool call: ...]`。

| # | 事项 | 文件 |
|---|---|---|
| 1 | 新增 `parseBracketToolCall(response)` —— 正则 `/^\[Tool call:\s*(\w+)\s*\]\s*(\{[\s\S]*?\})\s*$/`；JSON.parse 失败时**不抛错**，返回 `null` | `src/agent/tool-parser.ts` MODIFY |
| 2 | 新增 `stripBracketToolCall(response)` —— 即使 JSON 坏，也能剥掉 `[Tool call: name]\n{...}` 前缀；返回剥除后的字符串 | 同上 |
| 3 | `turn.ts:677` 的 `if (!parsed)` fall-through 之前加 bracket 探测链：先试 `parseBracketToolCall`，成功 → `parsed = bracketParsed`（走正常分支执行工具）；失败 → `response = stripBracketToolCall(response)` + `displayText = response` | `src/agent/turn.ts` MODIFY |

**约束**：
- **不动**现有 `parseToolCall` / `stripToolCall` API（80+ 既有测试不退化）
- bracket 形式 parser **不抛错** —— bracket form 是 LLM 失误的兜底路径，抛错会让 `turn.ts` 的 try/catch 之外再炸
- bracket 正则 `^...$` 锚定 —— LLM 闲聊里出现 `[Tool call: ...]` 字样不会误匹配
- `name` 必须匹配 `\w+`（字母/数字/下划线），非法字符如 `[Tool call: my-tool]` 不匹配
- 防御剥离的 fallback：剥完前缀后如果 `displayText` 为空字符串，仍 yield 空 student-text（不抛错；UI 显示空白即可，下次 LLM 响应正常）

### 1.2 P0-#4 — Phase prefix 回显剥除

**目标**：当 LLM 把 `[Phase: WRAP_UP — Time remaining: ~1.6 min — ...]` 这样的元数据前缀回显到回复开头时，**自动剥除**，学生看到的是干净的对话文本。

| # | 事项 | 文件 |
|---|---|---|
| 4 | 新增 `stripEchoedPhasePrefix(response)` —— 返回 `{cleaned, stripped}`；正则 `/^\[Phase:[^\]]*\]\s*/` 覆盖 diag 中所有 5 种 observed 格式 | `src/agent/tool-parser.ts` MODIFY |
| 5 | `turn.ts:642` 在 `let parsed = parseToolCall(response)` **之前**剥除：`response = stripEchoedPhasePrefix(response).cleaned`；日志新增 `phasePrefixStripped: boolean` 字段 | `src/agent/turn.ts` MODIFY |

**约束**：
- 正则 `^` 锚定行首 —— 不剥学生正常对话里讲"phase"这个词
- 不剥用户输入里的 `[Phase: ...]` —— 学生 UI 输入框几乎不可能出现这个前缀；与现有 `rawEchoesPhasePrefix: /^\[Phase:/.test(...)` 同一模式，行为一致
- 必须在 `parseToolCall` 之前剥 —— 否则 stripper 已经删掉的 `[Phase: ...]` 还会被存进 history / 推给学生
- diag 字段 `phasePrefixStripped` 与既有 `rawEchoesPhasePrefix` 共存，便于事后核对 v1.1.1 修复效果（前者描述"原始是否回显"，后者描述"是否被剥除"）

### 1.3 P1-#5/#6 — Blocked topic_select 短路 + 确定性 fallback

**目标**：当 `topic_select` 被 `MIN_TOPIC_AGE` 闸门拒绝（blocked）时，**不再做 2nd LLM call**——直接用 1st 响应的 `strippedPrefix`（LLM 已经在 tool call 之前给出了一段自然对话），或用一段确定性兜底句，避免 followup 重复 + 复读机。

| # | 事项 | 文件 |
|---|---|---|
| 6 | `turn.ts:902-991` blocked 分支改为短路：`displayText = strippedPrefix || "Let's keep going with this — tell me more."`；yield student-text + push history + yield `tool-2nd-call` 事件（带 `argSummary: 'blocked (short-circuit)'`）| `src/agent/turn.ts` MODIFY |
| 7 | blocked 时**不再 push `response` 进 history**（line 959 原行为）—— 否则 history 仍然含 `<tool>...</tool>` 污染，下次 LLM 看到又糊涂 | 同上 |

**fallback 句**：`"Let's keep going with this — tell me more."`
- 英文：与 system prompt 一致（`speaks English with the student`，SOUL.md:16）
- 1 行：未来 i18n 化只需替换这 1 行
- 不用 TODO / placeholder 词（如 `[topic continues]`）—— LLM 学会后会复制，污染 prompt 词表

**约束**：
- displayText 必非空（strippedPrefix 或 fallback 二选一）
- blocked 时**不调** 2nd LLM call → 节省 token + 消除复读机
- blocked 时**不再 yield** `'tool-call'` / `'tool-2nd-call-done'` 事件 —— 避免误导 diag 解读（`'tool-2nd-call'` 事件仍 yield，带特殊 argSummary）

### 1.4 诊断字段扩展

`turn.ts:642` 的 `logTurnDiagnostic(...)` 新增 `phasePrefixStripped: boolean`，向后兼容（已有 diag 解析脚本只读已有字段，新增字段被忽略）：

```ts
logTurnDiagnostic(sessionId, phaseHistory.length, nextState.phase, {
  ev: 'llm_done',
  rawLen,
  rawHead: response.slice(0, 200),
  rawTail: '',
  rawEchoesPhasePrefix,
  phasePrefixStripped,  // NEW
})
```

不动既有 diag 字段 schema，不动 `data/llm-debug/` 下的旧文件。

## 2. 后续需求（占位）

- **P2-#1 话题 spam**：MIN_TOPIC_AGE 闸门之后，LLM 仍反复调 topic_select；可考虑 (a) 在 blocked 时给 LLM 的 prompt 注入明确的"topic_select BLOCKED"信号；(b) `exclude_recent_days` 默认排除本 session 已聊话题

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `[Phase:` 正则把某些正常回复开头误剥 | 低 | 低 | 现有 `rawEchoesPhasePrefix: /^\[Phase:/.test(...)` 已用同一模式检测 6 个月，无误报；保持同一正则 |
| bracket form parser 把 LLM 闲聊里的 `[Tool call: ...]` 当成真工具调用 | 极低 | 中 | 仅当 LLM 输出**完全**由 `[Tool call: name]\n{json}` 构成时才匹配（正则 `^...$` 锚定）；混在自然语言里不会触发 |
| blocked 短路在某些 corner case 漏掉 2nd call 该做的事（如 push 错误信息到 history 供 LLM 看）| 低 | 低 | blocked 时 result 错误信息只 stderr 一行；不进入 history；既然不调 2nd LLM，history 含不含 result 都无所谓 |
| deterministic fallback 句不够自然 | 低 | 低 | 1 行英文、可控；如未来要 i18n 化再扩（T10）|
| 中文学生看不懂 `"Let's keep going with this — tell me more."` | 极低 | 低 | SOUL.md 第 16 行规定 "explanations of grammar may use brief Chinese only when the student is clearly stuck"；这句话既不是 grammar 解释也不是 stuck 场景，保持英文 |
| `phasePrefixStripped` diag 字段被旧 diag 解析脚本忽略 | 低 | 无 | 旧脚本只读已有字段；新增字段被 JSON.parse 自然忽略 |
