## 0. 本 sprint 修了什么

7/15 两次真实练习日志（`data/llm-debug/diag-0f7bf740.jsonl`、`diag-bfc000c5.jsonl`）暴露 3 类 4 个 LLM 输出异常触发的 UI 故障，本 sprint 全部修复：

| # | 现象 | 修复 |
|---|---|---|
| **P0-#2** | UI 出现裸 `[Tool call: topic_select]\n{...}`，无对话回复 | `parseBracketToolCall` 识别 bracket 形式并当真工具调用执行；`stripBracketToolCall` 兜底剥离，UI 永不泄漏 |
| **P0-#4** | wrap-up / end 回复开头带 `[Phase: WRAP_UP — ...]` 元数据 | `stripEchoedPhasePrefix` 在 `parseToolCall` 之前剥离回显的 phase 前缀 |

| **P1-#6** | topic_select 被 blocked 后 fallback "复读机" | blocked 时直接用 1st 响应的 `strippedPrefix`（或确定性兜底句），不再让 2nd-call LLM 复读 |

## 6. 代码审查要点（自审）

### 6.1 bracket 形式解析 — 防误报

- `BRACKET_TOOL_REGEX = /^\[Tool call:\s*(\w+)\s*\]\s*(\{[\s\S]*?\})\s*$/`：`^...$` 锚定整段，只有当响应**完全**由 `[Tool call: name]\n{json}` 构成才匹配 —— LLM 闲聊里夹一句 `[Tool call: ...]` 不会误触发（B7 覆盖）。
- `\w+` 拒绝含 `-` / `.` 的 name（B8 覆盖），与 `<tool>` 形式的 name 规则一致。
- `parseBracketToolCall` JSON 坏时返回 null（**不抛**）—— bracket 是 LLM 忘记 `<tool>` 的兜底路径，不该再抛错阻断（B3 覆盖）；调用方拿到 null 后走 `stripBracketToolCall` 防御剥离。

### 6.2 phase prefix 剥离 — 时序不变量

- 剥离必须在 `let parsed = parseToolCall(response)` **之前**（`turn.ts:644-647`），否则被 stripper 删掉的 `[Phase: ...]` 还会被存进 history / 推给学生。
- `rawEchoesPhasePrefix` 在剥离**之前**计算（记录 pre-strip 事实），`phasePrefixStripped` 记录是否发生剥离 —— 两个字段配合可事后核对修复效果。
- `^` 锚定：句中出现 "phase" 英文词不误剥（P4 覆盖）。

### 6.3 blocked 短路 — 关键不变量

- blocked 时**不再** push `response`（含 `<tool>...</tool>` 污染）进 history，只 push 干净的 `displayText` —— 否则下次 LLM 看到脏 history 又糊涂。
- `displayText` 必为非空：`strippedPrefix || "Let's keep going with this — tell me more."`（T1 走前者，T2 走后者）。
- 兜底句用英文，与 system prompt 的英文环境一致（SOUL.md：grammar 解释才可用简短中文，此处不属该场景）。
- not-blocked 路径（T3）完全保留 v1.1.0 的 A+B 2nd-call + `strippedPrefix` 前置逻辑，回归保护到位。

### 6.4 复用率

- `parseToolCall` / `stripToolCall`：一行未改，bracket / phase 函数平级新增。
- `turn.ts` 既有 `strippedPrefix`（v1.1.0，line 919）：blocked 分支直接复用，无需重算。
- diag schema：只加字段（`phasePrefixStripped`），向后兼容既有 diag 解析脚本。
