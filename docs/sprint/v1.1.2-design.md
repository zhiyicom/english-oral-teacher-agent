## 1. §1.1 `[System note: ...]` 元数据回显剥除

### 1.1 新增函数（同文件 `tool-parser.ts`）

```ts
const ECHOED_SYSTEM_NOTE_PREFIX = /^\[System note:[^\]]*\]\s*/

export function stripEchoedSystemNote(response: string): {
  cleaned: string
  stripped: boolean
} {
  const m = response.match(ECHOED_SYSTEM_NOTE_PREFIX)
  if (!m) return { cleaned: response, stripped: false }
  return { cleaned: response.slice(m[0].length).trim(), stripped: true }
}
```

**正则拆解**：
- `^\[System note:` —— 行首锚定（防止匹配 LLM 闲聊里讲"system note"这个词）
- `[^\]]*\]` —— 匹配到第一个 `]` 为止（覆盖 diag 中观察到的 4 种形态：长版本带 hint、短版本无 hint、单行、多行）
- `\s*` —— 吃掉右括号后的换行 / 空格

**与 v1.1.1 `stripEchoedPhasePrefix` 的对称性**：

| 维度 | `stripEchoedPhasePrefix`（v1.1.1）| `stripEchoedSystemNote`（v1.1.2 NEW）|
|---|---|---|
| 触发前缀 | `[Phase: ...]` | `[System note: ...]` |
| 正则锚定 | `^...$` | `^...$` |
| 返回 shape | `{cleaned, stripped}` | `{cleaned, stripped}` |
| 行中误匹配 | 不剥（`^` 锚定）| 不剥（`^` 锚定）|

**覆盖的 4 种 observed 格式**（来自 `data/llm-debug/diag-7132fcc9.jsonl`）：

| # | 原始 rawHead | cleaned |
|---|---|---|
| 1 | `[System note: you must call \`topic_select\` now. Do NOT repeat the same line again — pick a new topic and start fresh, NOT using Boeing 737 Max 8.]` | `''`（剥完是空字符串）|
| 2 | `[System note: you must call \`topic_select\` now. Pick a new topic unrelated to anything the student already rejected (music instruments, weather, seasons, superpowers, Delta Force). Use \`topic_select\` tool with \`exclude_recent_days=60\`]\n\n<tool>topic_select(...)` | `<tool>topic_select(...)`（system note 后面跟 tool call）|
| 3 | `[System note: you must call \`topic_select\` now. Pick a new topic unrelated to anything the student already rejected.]` | `''` |
| 4 | `[System note: you must call \`topic_select\` now.]`（最短观测） | `''` |

注：当 LLM 输出**完全是** `[System note: ...]`，剥完是空字符串 —— 这是正常的（没有真实对话内容），UI 显示空白即可，下次 LLM 响应正常。**不需要**额外兜底句（学生看到空白不会比看到系统元数据更困惑）。

### 1.2 `turn.ts` 接入

`turn.ts:644-647` 现状（v1.1.1 已加 phase prefix 剥离）：

```ts
const rawEchoesPhasePrefix = /^\[Phase:/.test(response.trim())
const phasePrefixStrip = stripEchoedPhasePrefix(response)
const phasePrefixStripped = phasePrefixStrip.stripped
if (phasePrefixStripped) response = phasePrefixStrip.cleaned
```

调整为（**System note 先，Phase 后**）：

```ts
// v1.1.2 P0-α — strip "[System note: ...]" before "[Phase: ...]". The LLM
// occasionally self-narrates a "[System Context]" reminder as
// "[System note: ...]" text; we strip it defensively so the student
// never sees the meta-instruction in their UI.
const systemNoteStrip = stripEchoedSystemNote(response)
const systemNoteStripped = systemNoteStrip.stripped
if (systemNoteStripped) response = systemNoteStrip.cleaned

// v1.1.1 P0-#4 — strip any echoed "[Phase: ...]" prefix before parsing or
// pushing to history.
const rawEchoesPhasePrefix = /^\[Phase:/.test(response.trim())
const phasePrefixStrip = stripEchoedPhasePrefix(response)
const phasePrefixStripped = phasePrefixStrip.stripped
if (phasePrefixStripped) response = phasePrefixStrip.cleaned
```

**为什么不放 phase prefix 之后**：
- 顺序约束：未观测到 `[System note:` 文本内嵌 `[Phase:` 子串，但**安全顺序是 System note 先剥**——如果未来 LLM 生成 `[System note: now is [Phase: MAIN_ACTIVITY]]` 这种嵌套，phase stripper 只剥外层会留 `MAIN_ACTIVITY]]` 残留；先剥 system note 则外层先没了
- **可逆性**：两个 stripper 都是独立幂等的，调换顺序不影响正常情况

**关键不变量**：
- 在 `parseToolCall(response)` 之前剥（与 v1.1.1 同款）
- 在 `logTurnDiagnostic(... rawHead ...)` 之前剥——`rawHead` 反映剥后的 view（与 v1.1.1 同款）
- `systemNoteStripped` 字段进 diag，与 `rawEchoesPhasePrefix` / `phasePrefixStripped` 共存便于核对

### 2.2 v1.1.2 调整：fallback 选档 + session-level 计数

**新增运行时状态字段**：在 `TurnDeps` interface 加 `blockedCount: number`（默认 0），让 runTurn 自己累加（不污染 input；session reload 时归零即可，本 sprint 接受）。

**文案选档函数**（新文件 `src/agent/blocked-fallback.ts`）：

```ts
export function pickBlockedFallback(blockedCount: number): string {
  if (blockedCount <= 0) return "Let's keep going with this — tell me more."
  if (blockedCount === 1) return "Hmm, that one didn't take. Let's try something different — what haven't we talked about recently?"
  // blockedCount >= 2
  return "We seem to keep circling. Let's pick something completely fresh — what catches your eye right now?"
}
```

### 2.3 `turn.ts:998-1008` 调整

```ts
if (blocked) {
  // v1.1.2 P1-A — session-level block count for fallback dedup.
  deps.blockedCount += 1
  const fallback = pickBlockedFallback(deps.blockedCount - 1)  // 0 → tier 1, etc.
  displayText = strippedPrefix || fallback

  // v1.1.2 P2-A — at tier 2/3, attach a topic+keyword fresh hint so the LLM
  // can see what's available in the library instead of improvising from
  // memory. Tier 1 keeps the conversation natural — no hint.
  let hintSuffix = ''
  let freshHintLayer: 'none' | 'keyword' | 'topic+keyword' = 'none'
  if (deps.blockedCount - 1 >= 1 && (deps.topics?.length ?? 0) > 0) {
    const fresh = pickFreshHints({
      topics: deps.topics,
      stats: deps.topicStats,
      keywordStats: deps.keywordStats,
      excludeDays: 30,
      topicLimit: 3,
      keywordLimit: 5,
    })
    if (fresh.keywords.length > 0) {
      const kwList = fresh.keywords.join(', ')
      if (deps.blockedCount - 1 >= 2 && fresh.topics.length > 0) {
        const tList = fresh.topics
          .map((t) => t.description?.trim() || t.name)
          .join(', ')
        hintSuffix = `\n\nFresh angles to try: ${kwList}.\n\nTry switching to: ${tList}.`
        freshHintLayer = 'topic+keyword'
      } else {
        hintSuffix = `\n\nFresh angles to try: ${kwList}.`
        freshHintLayer = 'keyword'
      }
    }
  }

  yield { type: 'student-text', text: `${displayText}${hintSuffix}\n\n` }
  history.push({ role: 'assistant', content: displayText })  // hint 不入 history
  deps.messages.append({
    sessionId,
    role: 'assistant',
    content: `${displayText}${hintSuffix}`,
  })
  yield {
    type: 'tool-2nd-call',
    name: parsed.name,
    argSummary: `blocked tier=${deps.blockedCount}/layer=${freshHintLayer}`,
  }
}
```

**`deps` 新增字段**：
- `deps.blockedCount: number`（默认 0）
- `deps.topics: Topic[]`（库，14 个话题）
- `deps.topicStats: TopicStat[]`（已有 `activeTopics`，复用即可）
- `deps.keywordStats: KeywordHit[]`（NEW —— 来自 `keyword_hits` 表）

**为什么 hint 不入 `input.history`**：
- hint 是 metadata（"库里有什么"），不是 user-facing 上下文；下次 LLM 看到反而困惑
- 但**入** `deps.messages.append`（持久层）—— 学生 UI 显示完整 hint

**为什么 tier 1 不注入 hint**：
- tier 1 = 首次 blocked，对话应该是自然的（"let's keep going"）；加 hint 立刻变得"机械"
- tier 2 = 二次 blocked，LLM 已经发现 retry，hint 给 LLM 一个方向；只 keyword 层，因为学生可能换角度不换话题
- tier 3 = 多次 blocked，**必须**借调结构化 hint；双层（keyword + topic）让 LLM 既能换角度也能换话题

### 2.4 三个 tier 文案的设计逻辑

| Tier | blockedCount | 文案 | hint |
|---|---|---|---|
| 1 | 0 | `Let's keep going with this — tell me more.` | 无 |
| 2 | 1 | `Hmm, that one didn't take. Let's try something different — what haven't we talked about recently?` | `Fresh angles to try: ...`（keyword 单层）|
| 3 | ≥2 | `We seem to keep circling. Let's pick something completely fresh — what catches your eye right now?` | `Fresh angles to try: ...\n\nTry switching to: ...`（keyword + topic 双层）|

**为什么不用 `tier N: <强制 list>` 编号**：
- LLM 学会后会复制；v1.1.0/1.1.1 设计 prompt 时已踩过这个坑
- 用对话语气（`Hmm, that one didn't take` / `We seem to keep circling`）保持 teacher 人设

## 3. §1.3 Blocked branch 注入 fresh 候选（topic + keyword 双层）

### 3.1 新增公共函数（`src/agent/topic-engine.ts`）

```ts
import type { KeywordHit, Topic, TopicStat } from '../storage/topics.js'

/**
 * v1.1.2 §1.3 — return fresh hints from BOTH layers:
 * - topic: top `topicLimit` fresh topics by `filterHardExclude` + `sortByCountAsc`
 * - keyword: top `keywordLimit` keywords with `hit_count === 0` from `keyword_hits`,
 *   prioritizing those whose parent topic is freshest (low discussion_count first)
 *
 * User correction: hints must surface fresh ANGLES (keywords), not just fresh
 * topics. 7/16 session showed `keyword_hits` only has 33 entries with
 * hit_count=1:30 / =2:3 / =0:0; combined with 8 fresh topics whose keywords
 * are also all unhit, this gives ~673 fresh keywords to anchor the LLM.
 *
 * Used by `turn.ts`'s blocked-branch hint injection. Pure: no I/O, no DB.
 */
export function pickFreshHints(opts: {
  topics: Topic[]
  stats: TopicStat[]
  keywordStats: KeywordHit[]
  excludeDays?: number
  topicLimit?: number
  keywordLimit?: number
  now?: Date
}): { topics: Topic[]; keywords: string[] } {
  const excludeDays = opts.excludeDays ?? 30
  const now = opts.now ?? new Date()
  const topicLimit = opts.topicLimit ?? 3
  const keywordLimit = opts.keywordLimit ?? 5

  // ---- topic layer: filter + sort (same logic as selectTopic pool) ----
  let pool = filterHardExclude(opts.topics, opts.stats, excludeDays, now)
  pool = sortByCountAsc(pool, opts.stats)
  const freshTopics = pool.slice(0, topicLimit)

  // ---- keyword layer: hit_count === 0 across all topics ----
  const zeroHitKw = opts.keywordStats
    .filter((h) => h.hitCount === 0)
    .map((h) => ({
      topic: h.topic,
      keyword: h.keyword,
      // Lower parent-discussion_count ranks first; missing stats → 0
      parentCount:
        opts.stats.find((s) => s.topic === h.topic)?.discussionCount ?? 0,
    }))

  // Sort: parentCount ASC, then topic ASC, then keyword ASC (deterministic)
  zeroHitKw.sort((a, b) => {
    if (a.parentCount !== b.parentCount) return a.parentCount - b.parentCount
    if (a.topic !== b.topic) return a.topic.localeCompare(b.topic)
    return a.keyword.localeCompare(b.keyword)
  })

  const freshKeywords = zeroHitKw.slice(0, keywordLimit).map((h) => h.keyword)

  return { topics: freshTopics, keywords: freshKeywords }
}
```

**为什么 topic 与 keyword 两层各自确定 top-N**：
- topic 层用现成 `filterHardExclude` + `sortByCountAsc`（确定性），与 `selectTopic` pool 一致
- keyword 层从 `keyword_hits` 直接过滤 `hit_count=0`（DB 当前 0 行，但库扩大后不为零），**不**走 `isTurnOnTopic`（那是计算 onTopicTurnCount 的，本 hint 不用）
- 排序：`parentCount ASC`（parent topic 越 fresh，keyword 越 fresh）→ `topic ASC` → `keyword ASC` —— 三级字典序保证确定性测试

**为什么 topic + keyword 两层各自独立**：
- tier 2 只显示 keyword 层（换角度不换话题）
- tier 3 显示双层（换话题 + 换角度）
- 调用方 `turn.ts` 根据 tier 选择哪层进 hintSuffix，算法不变

**为什么不复用 `selectTopic` 的 weighted random**：
- `pickFreshHints` 是**确定 top-N**——LLM hint 不需要随机性，越可预测越好（学生看到的是稳定的"今天有这些题"）
- 测试断言简单：`slice(0, 5)` 是确定性的
- `selectTopic` 的 noise 反而会破坏 hint 的一致性

### 3.2 新增辅助函数（`src/agent/topic-stats-helpers.ts` NEW，本 sprint **不**接入 turn.ts）

```ts
import type { TopicStat } from '../storage/topics.js'

/**
 * v1.1.2 §1.3 — return up to `limit` most-recently-discussed topics with
 * lastDiscussedAt within the last `days` days. Pure: no I/O, no DB.
 * Backlog T13: future sprint may inject this into the blocked-branch hint
 * to tell the LLM what NOT to improvise on. Sorted by lastDiscussedAt DESC.
 */
export function pickRecentDiscussed(
  stats: readonly TopicStat[],
  days: number,
  now: Date = new Date(),
  limit: number = 3,
): TopicStat[] {
  const cutoff = now.getTime() - days * 86_400_000
  return [...stats]
    .filter((s) => s.lastDiscussedAt && Date.parse(s.lastDiscussedAt) >= cutoff)
    .sort((a, b) => Date.parse(b.lastDiscussedAt) - Date.parse(a.lastDiscussedAt))
    .slice(0, limit)
}
```

**为什么不放 `topic-engine.ts`**：
- `topic-engine.ts` 操作 `Topic[]`（库 + stats + keywords），输出 `Topic`
- `pickRecentDiscussed` 只看 `stats`（不需要库），输出 `TopicStat`
- 拆开避免 topic-engine.ts 单文件膨胀；topic-stats-helpers.ts 后续可承接 stats-only 工具

## 10. 风险与回退详解

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `[System note:` 正则把某些正常回复开头误剥 | 低 | 低 | v1.1.1 `stripEchoedPhasePrefix` 已用同款 `^...$` 锚定运行 6 个月无报告误伤；保持同款语义 |
| tier 3 文案 `We seem to keep circling` 被 LLM 学到并复读 | 中 | 中 | 用日常对话语气而非系统词；后续可 i18n |
| session-level `blockedCount` in-memory 不持久化，session reload 清零 | 低 | 低 | 本 sprint 接受"短会话足够"；后续 T11 入 DB |
| `pickFreshHints` 把 NSFW keyword 拉进 hint | 极低 | 中 | 复用既有 `topic_select` tool 用的同一份 `topics: Topic[]` 入参；keyword 是 `topic.keywords[]` 的子集，NSFW 过滤由 `Topic` schema 在库生成时保证 |
| blocked tier ≥ 2 hint 让 student 看到 `_系统_` 提示（如 `topic_select` 关键词漏出）| 低 | 中 | 渲染时**只**用 `topic.description ?? topic.name` + 原始 keyword；hardcode 白名单过滤 `topic_select` / `tool` / `exclude_recent_days` / `phase` 等系统词 |
| tier ≥ 2 注入 `fresh angles` 让 LLM 在下次 input 复读给学生（"Fresh angles are X, Y"）| 低 | 低 | hint 入 `deps.messages.append`（学生 UI 可见）；**不**入 `input.history`（LLM 上下文）|
| diag 字段 `fallbackTier` / `freshHintLayer` 在 non-blocked turn 上为 undefined（旧解析脚本忽略 undefined）| 低 | 无 | 严格 `?: number` / `?: string` 可选字段；旧脚本 JSON.parse 后忽略 undefined |
| `pickFreshHints` keyword 层 `parentCount=0` 排序把所有 hit_count=0 的 keyword 都打散（库有大 fresh topic 时全 keyword 都来自 fresh topic，sort 退化）| 低 | 低 | 三级 sort（parentCount + topic + keyword）保证 deterministic；测试用小型 fixture 验证 |
| 当 `keyword_hits` 表完全为空（新装环境、未有 practice session）时 `pickFreshHints` 把"所有 keyword"当 hit_count=0 | 极低 | 低 | 限定 `hit_count === 0` 显式表达式（不是 `< 1`）；空表 + 库不为空 = 库内 keyword 全当作 fresh kw —— 实际上这是正确行为（新装环境就该把库里全部 keyword 当可选 anchor）|
| tier 3 双层 hint 字数暴增（500+ 字）污染对话节奏 | 低 | 中 | 上限 `topicLimit=3` + `keywordLimit=5` 硬编码；显示字数硬上限 ~80 字（5+3 项 × 平均 8 字 + 标点 ~55 字以内）|
| `hintSuffix` 入 `messages.append` 但不入 `input.history`，未来某 dev 写新 diag 时会困惑 | 低 | 低 | 代码注释 + `docs/sprint/v1.1.2-test-report.md` §6 显式记录这个不变量 |
| `blockedCount` 自增需要在 blocked=true 时执行；如果 dev 改了 blocked 判断逻辑又忘了 +1，会出现 tier 不递增 | 低 | 低 | `deps.blockedCount += 1` 紧贴 `if (blocked)` 第一行，code review 时易见 |
