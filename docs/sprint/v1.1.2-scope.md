## 背景

2026-07-16 一场真实练习（`data/llm-debug/diag-7132fcc9.jsonl`，session `7132fcc9`）与完整对话文本（`data/llm-debug/2026-07-16T09-09-48-304Z_7132fcc9_turn2.txt`）暴露 3 个 **LLM 输出/行为异常 → 学生体验灾难** 的问题。与 v1.1.1 同样属于"LLM 不完美时的鲁棒性"范畴，但触发模式不同。

| # | 学生侧现象 | 根因 |
|---|---|---|
| **P0-α** | UI 出现 `[System note: you must call topic_select now. Do NOT repeat the same line again — pick a new topic and start fresh, NOT using Boeing 737 Max 8.]` 整段元数据 | LLM 自行根据 `[System Context]` 块的"must call tool"要求**自创** `[System note: ...]` 指令文本（确认不在源码里，仅在 PRD.md 出现"System note"语义但不是同字符串），原样写进 history / 推给学生。v1.1.1 的 `stripEchoedPhasePrefix` 正则 `/^\[Phase:[^\]]*\]\s*/` **不覆盖** `[System note: ...]` |
| **P1-A** | topic_select 被 blocked 之后连续 4-5 轮都是同一句话 `"Let's keep going with this — tell me more."`（student 怒回 "Then why don't you give me a new topic then, then, then."）| v1.1.1 短路逻辑用确定性 fallback 句，**没有去重**：同一个 session 内 blocked 计数 ≥ 2 时仍发同样的句子；LLM 也观察到自己的复读，连续用 "Sorry about that" 自我道歉 |
| **P2-A** | LLM 即兴问"favorite weather / season / dreams / superpower / instrument"，学生多次回 `You asked it last time`、`You talked about it last time`，但 LLM 完全不知道——硬要即兴 | 当 `selectTopic: null`（理论上不应发生：本 session 实测 5 次 blocked 全是 MIN_TOPIC_AGE 闸门拦截，**不是** selectTopic 返 null）或 LLM 被闸门拦了不愿调工具时，LLM 只能凭印象编，编的全是真聊过的；它看不见 `topic_stats` / `keyword_hits` 表 |

**用户原话（7/16 反馈）**：「检查7/16的对话，存在严重的问题：1、切换话题不断失败 2、话题选取的都是以前聊过的，命中数不为1的。除了这两个问题，你再对比对话内容和话题库命中情况，看看还有哪些问题。」

**用户后续纠错**：「你说还剩8个fresh的topic，但除了fresh的topic，还有大量从未命中过的关键词，这些也是应该在选话题的时候考虑的。选话题的时候难道不是topic+keyword吗？」

→ 本 sprint 经此纠错，P2-A 由 topic-level 升级为 **topic + keyword 双层 fresh hint**（见 §1.3）。

> 注：本 sprint 不碰 `MIN_TOPIC_AGE` 闸门本身、不动 `recordAdoptedTopics` / `topic-recorder`、不扩大 34 库（DB 中实际还有 8 个 `topic_stats` 不存在的 fresh topic + **673 个 fresh keyword**——后文 §1.3 详述，问题不在数据）。

### 关键决策（已与用户对齐）

| 决策点 | 选择 | 理由 |
|---|---|---|
| P0-α 修复策略 | **`stripEchoedSystemNote` 与 v1.1.1 `stripEchoedPhasePrefix` 对称**（`tool-parser.ts` 加新函数；`turn.ts` 在同位调） | 与 v1.1.1 P0-#4 同类问题，沿用同款剥离模式 |
| P1-A 修复策略 | **session-level block 计数 ≥ 2 时换 fallback 文案** | 行为层防御，与 LLM 是否守规矩解耦 |
| **P2-A 修复策略（升级）** | **topic + keyword 双层 fresh hint**（blocked branch 注入 `Fresh angles: <kw>` + tier ≥ 3 `Try switching to: <topic>`） | 用户指出 LLM 即兴编题时缺的是 keyword 级锚点；topic-level hint（8 fresh）粒度太粗，**不能告诉 LLM"哪些具体角度没用过"** |
| 发布方式 | **只推 main，不打 tag 不发 Release** | 沿用 v1.1.0/v1.1.1 hotfix 流程；最小干扰 |

## 1. 范围内

### 1.1 P0-α — `[System note: ...]` 元数据回显剥除

**目标**：当 LLM 在 main flow 之外（self-instructional turn）输出 `[System note: ...]` 元数据文本时，**自动剥除前缀**（与 v1.1.1 P0-#4 `[Phase: ...]` 同款），学生看到的是干净的回复或完全空——但不让他看见这条"内部告示"。

**确认是 LLM 自发而非 prompt 注入**：
- `Grep "\[System note:"` 在 src/ 下：0 命中（PRD.md 设计文档里有"System note"语义但不是这个完整前缀）
- 每次都跟 tool_use 决策耦合：出现时 LLM 同时（或下一轮）调用 `topic_select`
- 文本由 LLM 即兴构造，**没有**标准化格式（既观察到完整版 `[System note: ...]` 也观察到最短版 `[System note: you must call topic_select now.]`）

| # | 事项 | 文件 |
|---|---|---|
| 1 | 新增 `stripEchoedSystemNote(response)` —— 返回 `{cleaned, stripped}`；正则 `/^\[System note:[^\]]*\]\s*/` 覆盖 diag 中观察到的 4 种形态（带 hint 的长版 / 短版 / 多行 / 行尾变体） | `src/agent/tool-parser.ts` MODIFY |
| 2 | `turn.ts` 在 v1.1.1 已有的 `stripEchoedPhasePrefix` 旁边串联调用（顺序：**System note 先，Phase 后**——避免 `[System note: ...]` 内部恰好含 `[Phase:` 子串时双剥） | `src/agent/turn.ts` MODIFY |
| 3 | `logTurnDiagnostic` 新增 `systemNoteStripped: boolean` 字段 | 同上 |

**约束**：
- **不动** v1.1.1 既有 `stripEchoedPhasePrefix`（已合 `b30f920`）
- 正则 `^` 锚定行首 —— 不剥学生正常对话里讲"system note"这个词
- 不剥用户输入里的 `[System note: ...]` —— 学生 UI 输入框几乎不可能出现
- 必须在 `parseToolCall` **之前**剥 —— 否则 stripper 已删的元数据还会被存进 history / 推给学生
- 必须在 `stripEchoedPhasePrefix` **之前**调 —— `[System note: ...]` 文本可能内部嵌套 `[Phase: ...]` 子串（未观测到，但顺序防御）
- 与 v1.1.1 同款 `^...$` 锚定 —— LLM 在长回复中间穿插一句 `[System note: ...]` 不会被误剥
- diag 字段 `systemNoteStripped` 与既有 `rawEchoesPhasePrefix` / `phasePrefixStripped` 共存，便于核对修复效果

### 1.2 P1-A — Blocked fallback 句去重 + 行为升级

**目标**：当 `topic_select` 被 `MIN_TOPIC_AGE` 闸门拒绝（blocked），且 v1.1.1 短路逻辑用确定性 fallback 句时，**根据 session-level `blockedCount` 切换文案**，避免连续 4-5 轮复读同一句。

| session-level 计数 | 文案 | 行为 |
|---|---|---|
| **0**（首次 blocked） | `"Let's keep going with this — tell me more."` | 与 v1.1.1 完全一致（保留原行为） |
| **1**（再 blocked 一次） | `"Hmm, that one didn't take. Let's try something different — what haven't we talked about recently?"` | 暗示换题；**注入 Fresh angles hint**（§1.3）|
| **≥ 2**（再 blocked）| `"We seem to keep circling. Let's pick something completely fresh — what catches your eye right now?"` | 显式 invite 学生带题；**双层 hint**（keyword + topic）|

| # | 事项 | 文件 |
|---|---|---|
| 4 | `turn.ts` 新增 session-level `blockedCount: number`（`input` 字段或 `deps` 字段，**不**写入 DB）—— 仅运行时内存值 | `src/agent/turn.ts` MODIFY |
| 5 | blocked 分支替换为 `pickBlockedFallback(blockedCount)` —— 三档映射（见上表） | 同上 |
| 6 | `logTurnDiagnostic` 新增 `blockedCount: number` + `fallbackTier: 1\|2\|3` 字段 | 同上 |

**约束**：
- 同一 sprint 的 v1.1.1 测试 T1/T2（blocked + 非空 strippedPrefix / blocked + 空 strippedPrefix）**不退化** —— 改的是文案选择与计数，不改"短路 2nd call"逻辑本身
- `blockedCount` **不入 history**（仅 telemetry / diag），免得污染下次 LLM input
- 文案用英文，与 system prompt 的英文环境一致（SOUL.md:16 `speaks English with the student`）
- `blockedCount` 自增必须**幂等**——同一 turn 内多次走 blocked 路径只 +1（防御 doctype 重复 case）

### 1.3 P2-A — Blocked branch 注入 fresh 候选（topic + keyword 双层）

**目标**：当 blocked 发生或 LLM 不愿调工具时，让 LLM "看见" 库里有哪些**话题级 + 关键词级**两层的 fresh 候选可锚定——避免它凭印象即兴编题而被学生回怼"you asked it last time"。

**为什么必须双层（topic + keyword）**（用户纠错后）：
- 7/16 session 真实数据：库 34 话题 / 706 关键词，`topic_stats` 覆盖 26 话题 / `keyword_hits` 仅 33 关键词（hit_count=1:30, =2:3, **=0:0**）
- **topic-level "fresh"** = 8 个未进 `topic_stats` 的完整话题
- **keyword-level "fresh"** = **673 个**（fresh topic 内 164 + discussed topic 内 509 个从未命中）
- LLM 即兴问时按 `topic.keywords[]` 锚定（`isTurnOnTopic` v1.0.9 §1.4 已经支持子串匹配）—— 但 `topic_select` 只回 `keywords[]`，**不告诉哪些用过了**；hint 必须注入"还没用过哪些 keyword" 才管用
- 单看 topic-level 候选（8 个）会让 LLM 觉得"in 库"，而单看 keyword-level（673 个）会让 LLM 不知哪个话题最相关——**双层才是 actionable hint**

**注入格式**（tier 决定单/双层）：

| Tier | displayText 末尾追加 |
|---|---|
| 1 | （无）|
| **2** | `Fresh angles to try: {kw1}, {kw2}, {kw3}, ...`（最多 5 个 keyword，来自所有 hit_count=0）|
| **3** | `Fresh angles to try: {kw1}, {kw2}, ...` （top 5 keyword）<br>+ `Try switching to: {topic1}, {topic2}, {topic3}`（top 3 fresh topic）|

**hint 内容来源**：

| 层 | 来源 | 排序 | 上限 |
|---|---|---|---|
| **Fresh angles**（keyword 层）| `keyword_hits` 表中 `hit_count = 0` 的全部行（包含 fresh topic 内的 + discussed topic 内未命中的）| 所在 topic 的 `discussion_count` 升序 → keyword 字典序 | **5** |
| **Try switching to**（topic 层）| ① `topic_stats` 不存在（fresh topic）OR ② `last_discussed_at` < `now - excludeDays*86_400_000` | `filterHardExclude` + `sortByCountAsc` | **3** |

**约束**：
- **不**改 `topic_select` tool 成功路径的行为（已观察成功路径 LLM 能正常读 `keywords`）
- 不改 history —— hint 是 metadata，**不**进 `input.history`（免污染下次 LLM input）；**进** `deps.messages.append`（持久层 + 学生 UI 可见）
- `pickFreshHints` 用 `topic_select` 同款 `filterHardExclude` + `sortByCountAsc` 处理 topic 层；keyword 层从 `keyword_hits` 全表过滤 `hit_count=0`
- 渲染时**只**用 `topic.description ?? topic.name` + keyword 原始字符串 —— **不**出现 `topic_select` / `tool` / `exclude_recent_days` / `phase` 等系统词（hardcode 白名单过滤）
- 当 fresh keyword 候选数 < 5 时显示实际数（如 4 个就 4 个），不补默认占位
- 当 fresh topic 候选数 < 3 时显示实际数，不补默认占位
- tier 3 双层注入时 hint 字数控制在 ~80 字内（SOUL.md:18 "1-3 sentences plus one follow-up question"）—— 超出时截断到 5 keyword + 3 topic
- **`pickRecentDiscussed`** 函数可后续接入（本 sprint **不**接入 turn.ts；T12 backlog 留给后续）

| # | 事项 | 文件 |
|---|---|---|
| 7 | 抽公共函数 `pickFreshHints(opts): { topics: Topic[], keywords: string[] }`（双层 return） | `src/agent/topic-engine.ts` MODIFY |
| 7b | 新依赖：`pickFreshHints` 入参加 `keywordStats: KeywordHit[]`（来自 `keyword_hits`），过滤 `hit_count=0` | `src/agent/topic-engine.ts` MODIFY |
| 8 | 抽 `pickRecentDiscussed(stats, days): TopicStat[]` —— 取最近 `days` 天的 N 条（本 sprint **不**被 turn.ts 消费，备后续扩） | `src/agent/topic-stats-helpers.ts` NEW（也可放进 `topic-engine.ts`）|

### 1.4 诊断字段扩展

`turn.ts` 的 `logTurnDiagnostic(...)` 新增 5 个字段（向后兼容，旧解析脚本忽略新字段）：

```ts
logTurnDiagnostic(sessionId, phaseHistory.length, nextState.phase, {
  ev: 'llm_done',
  rawLen,
  rawHead: response.slice(0, 200),
  rawTail: '',
  rawEchoesPhasePrefix,
  phasePrefixStripped,
  // v1.1.2 new:
  systemNoteStripped,                                  // P0-α
  blockedCount,                                        // P1-A
  fallbackTier: 1 | 2 | 3 | undefined,                 // P1-A（仅 blocked=true）

})
```

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `[System note:` 正则把某些正常回复开头误剥 | 低 | 低 | v1.1.1 `stripEchoedPhasePrefix` 已用同款 `^...$` 锚定运行 6 个月无报告误伤；保持同款语义 |
| tier 3 文案变"系统感"——"We seem to keep circling" 让 LLM 学到并复读 | 中 | 中 | tier 文案也走英文对话风格（"Let's pick something fresh"）避开系统化措辞；后续可 i18n |
| session-level `blockedCount` in-memory 不持久化，session reload 清零 | 低 | 低 | 本 sprint 接受"短会话足够"；后续 T11 入 DB |
| `pickFreshHints` 拉出 `keyword_hits` 不存在但实际不该选的 keyword（如 NSFW）| 极低 | 中 | 复用既有 `topic_select` tool 用的同一份 `topics: Topic[]` 入参；keyword 是 `topic.keywords[]` 的子集，NSFW 过滤由 `Topic` schema 在库生成时保证 |
| blocked branch 注入 hint 让 student 看到 `_系统性_` 提示（如果脱敏漏了）| 低 | 中 | 渲染时**只**用 `topic.description ?? topic.name` + 原始 keyword；hardcode 过滤 `topic_select` / `tool` / `exclude_recent_days` / `phase` 等系统词 |
| tier ≥ 2 注入 keyword hint 让 LLM 在下次直接读 hint 复述给学生（"Fresh angles are X, Y"）| 低 | 低 | hint 入 `deps.messages.append`（学生 UI 可见）；不入 `input.history`（LLM 上下文）—— 学生能看到但 LLM 下次 input 里看不到 |
| diag 字段 `fallbackTier` 在 non-blocked turn 上为 undefined（旧解析脚本忽略 undefined 字段）| 低 | 无 | 严格 `?: number` 可选字段；旧脚本 JSON.parse 后忽略 undefined |
| `pickFreshHints` 与 `selectTopic` 算法不一致（如 NOISE_RANGE）| 低 | 低 | `pickFreshHints` 内部显式 `sortByCountAsc` + keyword `hit_count=0` 过滤；**不复用** `selectTopic` 的 noise—— top 3 + top 5 是确定性的，便于测试 |
| `pickFreshHints` 在 keyword 全命中（无 fresh kw）场景下返空数组时 tier 2 不注入（仅 tier 2 hint 是 keyword 单层） | 中 | 低 | tier 2 注入逻辑显式判断 `freshHintLayer === 'keyword'` 才输出那一行；空时跳过整 hint |
| 当 `keyword_hits` 表完全为空（新装环境）时 `pickFreshHints` 会把"所有 keyword"当 hit_count=0 | 极低 | 低 | 限定 `hit_count = 0` 显式表达式（不是 `< 1`），空表场景等于无 fresh kw，tier 2 hint 自动跳过 |
