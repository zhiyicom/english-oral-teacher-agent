## 0. 本 sprint 修了什么

7/16 真实练习（`data/llm-debug/diag-7132fcc9.jsonl`，session `7132fcc9`）暴露 3 类问题，本 sprint 全部修复：

| # | 现象 | 修复 |
|---|---|---|
| **P0-α** | UI 出现 `[System note: you must call topic_select now. ...]` 元数据前缀 | `stripEchoedSystemNote` 在 `parseToolCall` 之前剥离（顺序：System note 先，Phase 后） |
| **P1-A** | blocked 后 fallback "Let's keep going with this — tell me more." 复读 4-5 轮 | session-level `blockedCount` 计数 → `pickBlockedFallback` 三档文案（tier 1 / tier 2 / tier 3）|

**关键决策（用户对齐）**：
- P2-A 由 topic-level hint 升级为 **topic + keyword 双层**——只给 topic 层（8 fresh）粒度太粗，必须给 keyword 层（673 fresh）让 LLM 有 actionable 锚点
- blocked 计数用 **ref 对象**（`blockedCountRef: { value: number }`）传 deps—— primitive 数字无法跨 turn 保留增量

## 6. 代码审查要点（自审）

### 6.1 `[System note: ...]` 剥离 — 与 v1.1.1 phase prefix 对称

- `ECHOED_SYSTEM_NOTE_PREFIX = /^\[System note:[^\]]*\]\s*/` 与 `ECHOED_PHASE_PREFIX` 同款（`^` 锚定 + `[^\]]*` 第一个 `]` 即终止）
- 调用顺序：System note 先，Phase 后——防止 LLM 写出 `[System note: now is [Phase: MAIN_ACTIVITY]]` 这种嵌套（未观测但 paranoid defense）
- 在 `parseToolCall` **之前**调用——否则被 stripper 删的元数据还会被存进 history / 推给学生
- `^` 锚定：句中讲"system note"短语不误剥（S4 覆盖）

### 6.2 blocked 三档 tier ladder — session-level 计数关键不变量

- **必须用 ref 对象**（`{ value: number }`）而非 primitive number 传 deps：primitive 在 turn.ts:1032 拿到的 copy 上 `+= 1`，caller 完全看不到，下次 turn 传进来仍是 0，**tier 永远卡在 1**——这是 ref 对象的核心必要性
- 既有的 `markedOriginals: Set<string>` 与 `warnedRef: { value: boolean }` 已经使用 ref 模式，本 sprint `blockedCountRef` 沿用同款
- `server.ts` 的 `SessionRuntime` 持久化 ref；`sessionStore.set` 写入与读取时都用 `rt.blockedCountRef`（同一个引用，不会重新构造一个新 ref 把计数清零）
- `cli.ts` 用 `const sessionBlockedCountRef = { value: 0 }`（whole-session 作用域，与 `markedOriginals` 同级）
- 既有的 v1.1.1 `T3: not blocked (explicit switch)` 回归测试通过——证明 non-blocked 路径不受 tier ladder 改动影响

### 6.3 topic + keyword 双层 fresh hint — 用户纠错后升级

- **为什么必须双层**（用户原话：「选话题的时候难道不是topic+keyword吗？」）：
  - topic-level "fresh" = 8 个未进 `topic_stats` 的完整话题——粒度粗，LLM 即兴问时不知道哪些 keyword 没碰过
  - keyword-level "fresh" = 673 个（fresh topic 内 164 + discussed topic 内 509 个从未命中）—— LLM 即兴问时按 `topic.keywords[]` 锚定，**必须告诉它哪些 keyword 用过、哪些没用**
  - 双层才是 actionable hint：topic 层给大方向（换这个话题），keyword 层给具体角度（聊这些点）
- 渲染格式：tier 2 keyword-only；tier 3 双层（keyword 行 + topic 行用 `\n\n` 分隔）
- 防御过滤：渲染只取 `topic.description ?? topic.name`，不暴露内部 slug / `topic_select` / `phase` 等系统词

### 6.4 hintSuffix 不入 `input.history` / 入 `deps.messages.append`

- LLM 上下文（`input.history`）**不**含 hint——避免污染下次 LLM input（让 LLM 看见自己的"提示"会迷惑）
- 持久层（`deps.messages.append`）**含** hint——学生 UI 可见（学生能看到 "Fresh angles to try: ..." 的真实可选方向）
- 测试通过 `input.history.filter((m) => m.role === 'assistant').length === 1` 验证历史只有 1 条
