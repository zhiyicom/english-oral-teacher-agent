## 背景

2026-07-23 一场真实练习（`data/llm-debug/diag-86e6f0ff.jsonl`，session `86e6f0ff`）暴露 3 类问题，均与话题选择引擎的"会话内记忆"和"关键词信号"相关。

| # | 学生侧现象 | 根因 |
|---|---|---|
| **P1** | topic_select 在同一次会话中反复选中同一批话题（sports → food → tech → sports again…），学生明确说"这些我们都聊过了" | `selectTopic()` 的硬排除只查 `topic_stats.last_discussed_at`，该表仅在会话结束时由 summarizer 写入。会话中途选过的 topic 无法被排除。`adoptedTopics` Map 只做被动记录（dedup adopt），不反馈给选择算法 |
| **P2** | 学生看到 "Fresh angles to try: habit, internet, playground, project, subject. Try switching to: Daily routine, Education & learning, Friends & social" 这样的系统内部标签，多达 10 次 | `turn.ts` 的 `hintSuffix`（含系统标签 "Fresh angles to try" / "Try switching to"）被直接拼入 `student-text` yield。且 `blockedCountRef` 从不重置——topic_select 成功时不归零，导致一旦进入 tier 3 就永远不退 |
| **P3** | 关键词命中信号在选题评分中被噪声淹没 | `KEYWORD_WEIGHT = 0.05` 与 `NOISE_RANGE = ±0.05` 同级。`avgKeywordHit` 取均值稀释了热点关键词的影响——一个 20 关键词的 topic 即使有 hit_count=5 的关键词，avg 也仅 0.25，惩罚 -0.0125，远小于噪声 |
| **附加** | `prompts/topic-library.md` 全部是占位符（30 个 `### t` / `关键词: placeholder` / `讨论角度: placeholder`） | 文件曾被 DB 异常状态污染后无人重新生成。虽然不注入 LLM prompt（LLM 从 SOUL+AGENTS+USER+tools 获取指令），但作为人类可读参考应保持同步 |

**用户原话（7/23 反馈）**：「7/23日的对话练习出现了挑选话题时反复选取同样话题的现象……而且，为什么要生硬地列出好几个话题让用户挑选呢？」「关键词和命中次数也很重要。请设计改进方案。」

### 关键决策

| 决策点 | 选择 | 理由 |
|---|---|---|
| P1 修复策略 | **`selectTopic()` 加 `excludeSlugs` 参数**——调用方传入本次会话已选的 topic slug 列表，在 hard-exclude 阶段一并排除 | 最小侵入，不改 Tool 接口签名 |
| P2 修复策略 | **`blockedCountRef` 在成功时重置 + fresh hints 从 student-text 移到 [System Context]**——复用并扩展 v1.1.2 T5 模式 | 学生不看到系统标签；hint 仍能到达 LLM |
| P3 修复策略 | **新增 `computeFreshRatio` + 权重调整**（KEYWORD_WEIGHT 0.05→0.10，新增 FRESH_RATIO_WEIGHT 0.15） | 关键词新鲜度从可忽略信号提升到可与 1-2 次 count 差异匹敌 |
| 发布方式 | **只推 main，不打 tag 不发 Release** | 沿用 v1.1.0/v1.1.1/v1.1.2 hotfix 流程 |

## 1. 范围内

### 1.1 P1 — 会话内已选话题硬排除

**目标**：同一会话中 `topic_select` 不会选中已经选过的话题。

| # | 事项 | 文件 |
|---|---|---|
| 1 | `selectTopic()` / `filterHardExclude()` 新增 `excludeSlugs?: string[]` 参数——slug 在列表中的 topic 被硬排除 | `src/agent/topic-engine.ts` MODIFY |
| 2 | `TopicSelectArgsSchema` 新增 `exclude_slugs: z.array(z.string()).optional()`——不在 tool description 中暴露给 LLM | `src/agent/tools/topic-select.ts` MODIFY |
| 3 | `turn.ts` 在 execute topic_select 前从 `input.adoptedTopics` 提取已选 slugs 注入 args | `src/agent/turn.ts` MODIFY |
| 4 | `pickFreshHints` 同样转发 `excludeSlugs` 给 `filterHardExclude`（topic 层保持一致） | `src/agent/topic-engine.ts` MODIFY |

**约束**：
- `excludeSlugs` 参数名用 camelCase（`excludeSlugs`），schema 字段名用 snake_case（`exclude_slugs`），遵循既有 `context_keywords` 约定
- 从 LLM 视角看 schema 无变化（字段默认 `undefined`，zod schema 加 `.optional()` 不破坏旧调用）
- `adoptedTopics.keys()` 在 `execute()` 之前 snapshot，避免执行过程中 Map 突变

### 1.2 P2-1 — `blockedCountRef` 在 topic_select 成功时重置

**目标**：topic_select 成功后 block streak 中断，下一轮 blocked 从 tier 1 重新开始。

| # | 事项 | 文件 |
|---|---|---|
| 5 | topic_select 成功路径（`resetTopicTurnCount` 附近）加 `deps.blockedCountRef.value = 0` | `src/agent/turn.ts` MODIFY |

**约束**：
- 只在 `!('error' in result)` 且 `!blocked` 时重置（即 topic_select 成功执行且未被闸门拦截）
- `blockedCountRef` 可能为 undefined（旧调用方不传），加 guard

### 1.3 P2-2 — Fresh hints 从学生可见文本移到 [System Context]

**目标**：学生永远看不到 "Fresh angles to try" / "Try switching to" 系统标签。LLM 在下一轮通过 [System Context] 获取新鲜话题/关键词提示。

**当前行为**（要移除）：
- blocked branch 拼 `hintSuffix` 到 `student-text` yield → 学生看到
- `hintSuffix` 存入 `deps.messages` → 混入对话历史

**新行为**：
- `student-text` 只输出 `displayText`（fallback 句子）
- `deps.messages` 只存 `displayText`
- Fresh hints 通过扩展的 T5 通道注入下一轮 [System Context]

| # | 事项 | 文件 |
|---|---|---|
| 6 | `TurnInput` 新增 `blockedFreshHints?: { topics: string[]; keywords: string[] } \| null`——blocked 分支计算并写入，下一轮消费 | `src/agent/turn.ts` MODIFY |
| 7 | `SessionRuntime` 新增同名字段，server.ts/cli.ts 每轮 consume→clear→detect→store（镜像 T5 模式） | `src/server.ts` MODIFY, `src/cli.ts` MODIFY |
| 8 | `buildFinalSystemSegments` / `buildSystemContext` 新增 `blockedFreshHints` 参数，有值时在 [System Context] 追加提示行 | `src/agent/prompt-builder.ts` MODIFY, `src/agent/context-injector.ts` MODIFY |
| 9 | blocked branch 移除 `hintSuffix` 相关逻辑，改为写 `input.blockedFreshHints` | `src/agent/turn.ts` MODIFY |

**注入格式**（[System Context] 内追加）：
```
- Fresh keywords (not yet discussed): habit, internet, playground, project, subject
- Fresh topics (less-discussed, not yet covered this session): Daily routine (A1-A2); Education & learning (B1); Friends & social (B1)
- Use these fresh angles to start your questions naturally — do NOT list them to the student.
```

**约束**：
- hints 仅在有值的一轮注入（one-shot，与 T5 信号同款消费模式）
- 不修改 `pickFreshHints` 本身（v1.1.2 §1.6 已正确），只改消费端
- tier 1 不写入 hints（保持纯 continuation 风格）

### 1.4 P3 — 关键词评分信号强化

**目标**：关键词新鲜度在选题评分中有实际影响力，不再被噪声淹没。

| # | 事项 | 文件 |
|---|---|---|
| 10 | 新增 `computeFreshRatio(topic, keywordStats): number`——topic.keywords 中 hit_count=0 的比例，范围 [0,1] | `src/agent/topic-engine.ts` MODIFY |
| 11 | 权重调整：`KEYWORD_WEIGHT` 0.05→0.10；新增 `FRESH_RATIO_WEIGHT = 0.15` | 同上 |
| 12 | 评分公式更新为 `-count*0.10 - avgHit*0.10 + freshRatio*0.15 + interest*0.5 + context*0.15 ± 0.05` | 同上 |

**数值验证**（基于 7/23 生产 DB 实际数据）：

| Topic | count | freshRatio | avgHit | 旧分 | 新分 |
|---|---|---|---|---|---|
| school_life | 1 | 0.90 | 0.10 | -0.105±0.05 | **+0.030±0.05** |
| gaming | 2 | 0.95 | 0.05 | -0.205±0.05 | **-0.060±0.05** |
| music | 7 | 0.83 | 0.39 | -0.739±0.05 | **-0.614±0.05** |

school_life（count=1, 高新鲜度）新分 > 0，明显优于 gaming（count=2）。music（count=7）仍垫底。排序合理。

**约束**：
- `computeFreshRatio` 的 `hitSet` 用 `hitCount > 0` 过滤（与 v1.1.2 §1.6 的 v2 数据模型一致——缺失 = 0 命中 = fresh）
- 空 `topic.keywords` → 返回 0
- 大小写不敏感匹配
- 不改变 `INTEREST_WEIGHT`、`CONTEXT_WEIGHT`、`NOISE_RANGE`

### 1.6 topic_select 成功路径输出 topic + keyword

**目标**：topic_select 成功后，LLM 的 2nd call 能收到 topic 的完整 keywords 列表 + 命中次数最低的 suggested_keyword，而不是只能从 title 硬猜开头角度。

**现状 gap**：`formatToolResult`（`turn.ts:290-291`）当前输出：
```
[v076_topic_select_result]
Selected topic: Sports & health (B2) (slug=sports_health, ~15 min). Start discussing it.
```
但 `TopicSelectResult` 里 `keywords: string[]` 和 `suggested_keyword` 已经计算好了（`pickFreshestKeyword` 取命中次数最低的关键词），只是没被编进输出文本。Tool description 承诺 LLM 会拿到 keywords，实际没拿到。

| # | 事项 | 文件 |
|---|---|---|
| 14 | `formatToolResult` topic_select 分支输出改为包含 `keywords` + `suggested_keyword`；格式如 `Selected topic: ...\nSuggested angle: hiking (least discussed — start here).\nAll keywords: football, exercise, fitness, hiking, team, compete, training, ...` | `src/agent/turn.ts` MODIFY |

**约束**：
- `suggested_keyword` 取自 `pickFreshestKeyword`（已在 topic-select.ts execute() 计算，hitCount 最低 → 最"新鲜"的角度）
- keywords 列表最多截断（前 12 个），避免超过 LLM 注意力窗口
- 不改变 tool 成功路径的 2nd call 逻辑——仍调用 LLM 生成学生面部回复

## 7. 风险与回退

| 风险 | 概率 | 影响 | 回退 |
|---|---|---|---|
| `excludeSlugs` 排除所有 topic 导致 `selectTopic` 返回 null | 低（当前 34 topics，一次会话通常只选 3-5 个） | 中 | 返回 null 后 fallback 到 LLM 即兴（与现有 null 路径一致） |
| `computeFreshRatio` 使部分 topic 排序大变 | 低 | 低 | 新权重经过数值验证，排序更合理而非退化 |
| `blockedFreshHints` 消费模式有遗漏（未 reset / 未传递） | 中 | 中 | 严格镜像 T5 模式（已运行稳定）；测试覆盖 |
| 移除 `hintSuffix` 后学生不知道 LLM 为什么卡住 | 低 | 低 | fallback 句子（tier 1-3）本身已表达"换题"意图，不需要系统标签辅助 |
